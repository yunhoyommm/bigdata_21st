08장: GitHub Actions와 CI/CD

중급

GitHub Actions와 CI/CD¶

학습 목표 - CI/CD의 개념과 GitHub Actions의 역할을 설명할 수 있다 - .github/workflows/ 디렉토리에 워크플로우 YAML을 작성할 수 있다 - Jobs, Steps, Actions의 관계를 이해하고 파이프라인을 구성할 수 있다 - Marketplace의 공식 Actions를 활용하여 CI 파이프라인을 구축할 수 있다 - Secrets와 Environments를 활용한 안전한 CD 파이프라인을 설정할 수 있다

CI/CD의 개념과 GitHub Actions 소개 (ch_08_s01)¶

도입¶

코드를 작성하고 push했을 때 자동으로 테스트가 실행되고, 문제가 없으면 바로 배포되는 것을 상상해보세요. 수동으로 빌드하고 테스트하고 배포하는 반복 작업에서 해방되어, 개발자는 창의적인 작업에 집중할 수 있습니다. 이 절에서는 소프트웨어 개발의 효율을 혁신적으로 높이는 CI/CD의 개념과 GitHub Actions를 배웁니다.

핵심 개념 설명¶

CI/CD가 필요한 이유¶

문제 상황: 수동 통합과 배포의 고통

전통적인 개발 방식에서는 여러 개발자가 각자 개발한 코드를 일주일에 한 번 통합합니다. 이때 무수한 충돌이 발생하고, 누군가의 코드가 다른 사람의 기능을 망가뜨립니다. 밤샘 디버깅 끝에 통합을 마치고 나면, 수동으로 서버에 배포하는 과정에서 또 다른 문제가 발생합니다.

해결책: 자동화된 통합과 배포

CI/CD는 이러한 문제를 자동화로 해결합니다. 작은 변경을 자주 통합하고, 자동 테스트로 품질을 보장하며, 검증된 코드를 자동으로 배포합니다.

CI: 지속적 통합의 개념¶

CI(Continuous Integration, 지속적 통합)은 개발자가 코드를 공유 저장소에 자주 병합하고, 매 병합마다 자동으로 빌드와 테스트를 실행하는 실천 방법입니다.

CI의 핵심 원칙

  1. 자주 통합: 하루에 여러 번 main 브랜치에 병합
  2. 자동 빌드: 코드 변경 시 자동으로 빌드 실행
  3. 자동 테스트: 단위 테스트, 통합 테스트 자동 실행
  4. 빠른 피드백: 문제 발생 시 즉시 개발자에게 알림

비유

조립 라인의 품질 검사와 같습니다. 부품(코드)이 라인에 올라올 때마다 자동으로 검사(테스트)를 거쳐 불량품을 즉시 발견합니다. 최종 제품(릴리스)이 완성될 때까지 기다렸다가 검사하면 모든 부품을 다시 분해해야 하지만, 중간 중간 검사하면 문제를 빠르게 수정할 수 있습니다.

CI의 장점

  • 버그 조기 발견: 코드 작성 직후 문제를 발견하여 수정 비용 최소화
  • 통합 지옥 방지: 작은 단위로 자주 통합하여 대규모 충돌 방지
  • 코드 품질 향상: 자동 테스트로 회귀 버그 차단
  • 팀 생산성 향상: 항상 동작하는 main 브랜치 유지

CD: 지속적 배포/전달의 개념¶

CD는 두 가지 의미를 가집니다.

1. Continuous Delivery (지속적 전달)

검증된 코드를 프로덕션 배포 가능한 상태로 자동 준비하는 것입니다. 최종 배포는 수동 승인 버튼을 누르는 방식입니다.

2. Continuous Deployment (지속적 배포)

검증된 코드를 사람의 개입 없이 자동으로 프로덕션에 배포하는 것입니다. 모든 테스트를 통과한 코드는 즉시 사용자에게 전달됩니다.

CD의 단계

코드 작성 → CI (빌드 + 테스트) → 아티팩트 생성 → 스테이징 배포 → [수동 승인] → 프로덕션 배포

Continuous Delivery는 [수동 승인] 단계가 있고, Continuous Deployment는 자동으로 건너뜁니다.

CD의 장점

  • 빠른 릴리스: 기능을 개발 즉시 사용자에게 전달
  • 위험 감소: 작은 변경을 자주 배포하여 롤백 쉬움
  • 피드백 루프 단축: 사용자 반응을 빠르게 수집하고 개선
  • 배포 스트레스 감소: 자동화로 배포가 일상적인 작업이 됨

팁: 초기에는 Continuous Delivery로 시작하여 수동 승인 단계를 두고, 자신감이 생기면 Continuous Deployment로 전환하는 것을 권장합니다.

GitHub Actions의 위치와 장점¶

GitHub Actions는 GitHub에 내장된 CI/CD 플랫폼으로, 저장소 이벤트(push, PR 등)에 반응하여 워크플로우를 자동 실행합니다.

다른 CI/CD 도구와의 비교

도구 장점 단점
GitHub Actions GitHub 통합, 간편한 설정, Marketplace 복잡한 파이프라인 제한적
Jenkins 강력한 커스터마이징, 플러그인 풍부 설정 복잡, 서버 관리 필요
GitLab CI GitLab 통합, YAML 설정 GitHub과 통합 어려움
CircleCI 빠른 빌드, 병렬 처리 강력 과금, 외부 서비스

GitHub Actions의 장점

  1. 완벽한 GitHub 통합: PR, Issue, Release 등 모든 이벤트에 반응
  2. 간편한 설정: YAML 파일 하나로 워크플로우 정의
  3. Marketplace: 수천 개의 공식/커뮤니티 Actions 재사용
  4. 무료 사용량: Public 저장소 무제한, Private 월 2,000분 무료
  5. 다양한 Runner: Ubuntu, Windows, macOS 환경 지원
  6. 매트릭스 빌드: 여러 버전/환경에서 동시 테스트

GitHub Actions의 구성 요소

Workflow (워크플로우)
  ├── Trigger (트리거): 언제 실행할지
  ├── Jobs (작업): 무엇을 실행할지
  │   ├── Runner (실행 환경): 어디서 실행할지
  │   └── Steps (단계)
  │       ├── Action (재사용 가능한 작업 단위)
  │       └── Run (쉘 명령어)
  └── Secrets (비밀 정보)

예시

절 요약¶

  • CI는 코드 통합과 자동 테스트의 자동화이고, CD는 배포의 자동화입니다.
  • CI/CD는 버그 조기 발견, 빠른 릴리스, 배포 스트레스 감소의 장점을 제공합니다.
  • GitHub Actions는 GitHub에 내장된 CI/CD 플랫폼으로 설정이 간편합니다.
  • Workflow → Jobs → Steps → Actions/Run의 계층 구조로 파이프라인을 구성합니다.
  • Public 저장소는 무제한 무료, Private 저장소는 월 2,000분 무료입니다.

워크플로우 파일 작성하기 (ch_08_s02)¶

도입¶

CI/CD의 개념을 이해했다면 이제 실제로 워크플로우를 작성할 차례입니다. 워크플로우는 YAML 파일로 정의하며, 언제 실행할지, 무엇을 할지, 어떤 환경에서 실행할지를 선언적으로 기술합니다. 이 절에서는 첫 번째 워크플로우를 작성하고 GitHub Actions의 기본 문법을 익힙니다.

핵심 개념 설명¶

워크플로우 파일 구조: name, on, jobs, steps¶

워크플로우 파일 위치

워크플로우 파일은 반드시 .github/workflows/ 디렉토리에 위치해야 합니다.

my-project/
├── .github/
│   └── workflows/
│       ├── ci.yml
│       ├── deploy.yml
│       └── release.yml
├── src/
└── README.md

기본 구조

name: CI                    # 워크플로우 이름 (Actions 탭에 표시)

on: [push, pull_request]   # 트리거: 언제 실행할지

jobs:                       # 작업 목록
  test:                     # Job ID
    runs-on: ubuntu-latest  # 실행 환경
    steps:                  # 실행할 단계들
      - name: Checkout      # 단계 이름
        uses: actions/checkout@v4  # 재사용 Action
      - name: Run tests
        run: npm test       # 쉘 명령어

각 섹션의 역할

섹션 필수 여부 설명 예시
name 선택 워크플로우 표시 이름 name: CI Pipeline
on 필수 트리거 이벤트 on: push
jobs 필수 실행할 작업들 jobs: { test: {...} }
runs-on 필수 Runner 환경 runs-on: ubuntu-latest
steps 필수 순차 실행 단계 steps: [...]

트리거(on) 설정: push, pull_request, schedule, workflow_dispatch¶

1. push 트리거

특정 브랜치에 push할 때 실행합니다.

on:
  push:
    branches:
      - main
      - develop
    paths:
      - 'src/**'      # src 디렉토리 변경 시만
      - '!src/docs/**' # src/docs는 제외

2. pull_request 트리거

PR이 열리거나 업데이트될 때 실행합니다.

on:
  pull_request:
    branches:
      - main
    types:
      - opened       # PR 생성
      - synchronize  # PR 업데이트 (새 커밋 추가)
      - reopened     # PR 재오픈

3. schedule 트리거 (Cron 작업)

정해진 시간에 자동 실행합니다.

on:
  schedule:
    - cron: '0 2 * * *'  # 매일 오전 2시 (UTC)
    - cron: '0 */6 * * *' # 6시간마다

Cron 문법: 분 시 일 월 요일 - 0 2 * * *: 매일 오전 2시 - 0 0 * * 0: 매주 일요일 자정 - 0 0 1 * *: 매월 1일 자정

4. workflow_dispatch 트리거 (수동 실행)

Actions 탭에서 수동으로 실행할 수 있게 합니다.

on:
  workflow_dispatch:
    inputs:
      environment:
        description: 'Deploy environment'
        required: true
        default: 'staging'
        type: choice
        options:
          - staging
          - production

5. 여러 트리거 조합

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]
  schedule:
    - cron: '0 2 * * *'
  workflow_dispatch:

팁: PR에서는 테스트만, main push에서는 테스트와 배포를 모두 실행하도록 조건을 설정할 수 있습니다.

첫 번째 워크플로우 만들기: Hello World¶

1단계: 워크플로우 파일 생성

.github/workflows/hello.yml 파일을 생성합니다.

name: Hello World

on: [push]

jobs:
  greet:
    runs-on: ubuntu-latest
    steps:
      - name: Say Hello
        run: echo "Hello, GitHub Actions!"

      - name: Show Date
        run: date

      - name: Show Environment
        run: |
          echo "Runner OS: $RUNNER_OS"
          echo "GitHub Event: $GITHUB_EVENT_NAME"
          echo "Repository: $GITHUB_REPOSITORY"

2단계: 커밋 및 push

$ git add .github/workflows/hello.yml
$ git commit -m "feat: Hello World 워크플로우 추가"
$ git push

3단계: Actions 탭에서 확인

  1. GitHub 저장소 페이지로 이동
  2. "Actions" 탭 클릭
  3. "Hello World" 워크플로우 클릭
  4. 실행 로그 확인

출력 예시

Say Hello
  Hello, GitHub Actions!

Show Date
  Mon Feb 17 03:45:12 UTC 2025

Show Environment
  Runner OS: Linux
  GitHub Event: push
  Repository: username/my-repo

베스트 프랙티스: 워크플로우 이름(name)은 명확하고 구체적으로 작성하세요. "CI", "Deploy"보다 "Node.js CI", "Production Deploy"가 좋습니다.

워크플로우 문법 검증¶

로컬에서 문법 검증하기

GitHub CLI를 사용하여 로컬에서 문법을 검증할 수 있습니다.

$ gh workflow view
# 워크플로우 목록 표시

$ gh workflow list
# 활성화된 워크플로우 나열

VS Code 확장 (추천)

"GitHub Actions" 확장을 설치하면 YAML 문법 하이라이팅과 자동 완성을 지원합니다.

온라인 검증 도구

  • YAML Lint: YAML 문법 검증
  • GitHub Actions Schema: 스키마 기반 검증

일반적인 문법 오류

# 잘못된 예: 들여쓰기 오류
jobs:
test:  # 잘못된 들여쓰기
  runs-on: ubuntu-latest

# 올바른 예
jobs:
  test:  # 2칸 들여쓰기
    runs-on: ubuntu-latest

# 잘못된 예: 따옴표 없이 특수문자 사용
on:
  push:
    branches:
      - feature/*  # YAML에서 *는 특수문자

# 올바른 예
on:
  push:
    branches:
      - 'feature/*'  # 따옴표로 감싸기

주의: YAML은 들여쓰기에 민감합니다. 탭이 아닌 스페이스 2칸을 사용하세요. 대부분의 에디터에서 자동 변환 설정을 할 수 있습니다.

워크플로우 비활성화/활성화

# 워크플로우 비활성화
$ gh workflow disable <workflow-id>

# 워크플로우 활성화
$ gh workflow enable <workflow-id>

또는 웹 UI에서 워크플로우 상세 페이지의 "..." 메뉴에서 "Disable workflow"를 선택할 수 있습니다.

절 요약¶

  • 워크플로우는 .github/workflows/ 디렉토리의 YAML 파일로 정의합니다.
  • name, on, jobs, runs-on, steps가 기본 구조입니다.
  • 트리거는 push, pull_request, schedule, workflow_dispatch 등이 있습니다.
  • runs-on으로 실행 환경(ubuntu-latest, windows-latest, macos-latest)을 선택합니다.
  • YAML 문법 오류를 방지하려면 들여쓰기와 특수문자 처리에 주의해야 합니다.

Jobs와 Steps 구성 (ch_08_s03)¶

도입¶

워크플로우의 뼈대를 만들었다면 이제 실제 작업을 정의할 차례입니다. Jobs는 워크플로우의 독립적인 작업 단위이고, Steps는 Job 내부의 순차적인 실행 단계입니다. 이 절에서는 Jobs 간 의존성, 병렬 실행, 매트릭스 전략 등 고급 구성 방법을 배웁니다.

핵심 개념 설명¶

Job 정의와 실행 환경(runs-on)¶

Job이란

Job은 워크플로우의 독립적인 작업 단위입니다. 각 Job은 별도의 Runner(가상 머신)에서 실행되며, 기본적으로 병렬로 실행됩니다.

기본 Job 정의

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4
      - name: Build
        run: npm run build

runs-on 옵션

Runner 이미지 용도
ubuntu-latest Ubuntu 22.04 리눅스 빌드 (가장 저렴, 빠름)
ubuntu-20.04 Ubuntu 20.04 특정 Ubuntu 버전 필요 시
windows-latest Windows Server 2022 Windows 전용 빌드
macos-latest macOS 13 (Ventura) iOS/macOS 앱 빌드
macos-13 macOS 13 특정 macOS 버전
self-hosted 자체 호스팅 사용자 정의 환경

여러 Runner에서 실행 (매트릭스 미리보기)

jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest]
    steps:
      - run: echo "Running on $RUNNER_OS"

팁: 비용 절감을 위해 Linux runner를 우선 사용하세요. macOS는 Linux의 10배 비용입니다.

Step 작성: run과 uses¶

Step이란

Step은 Job 내부의 순차적으로 실행되는 개별 작업입니다. 각 Step은 run 명령어 또는 uses Action을 사용합니다.

1. run: 쉘 명령어 실행

steps:
  - name: Install dependencies
    run: npm install

  - name: Run tests
    run: npm test

  - name: Multi-line script
    run: |
      echo "Starting build..."
      npm run build
      echo "Build completed!"

쉘 선택

steps:
  - name: Bash script (Linux/macOS 기본)
    run: echo "Hello from Bash"
    shell: bash

  - name: PowerShell (Windows)
    run: Write-Host "Hello from PowerShell"
    shell: pwsh

  - name: Python script
    run: print("Hello from Python")
    shell: python

2. uses: 재사용 가능한 Action 사용

steps:
  - name: Checkout repository
    uses: actions/checkout@v4

  - name: Setup Node.js
    uses: actions/setup-node@v4
    with:
      node-version: '20'

  - name: Cache dependencies
    uses: actions/cache@v3
    with:
      path: ~/.npm
      key: ${{ runner.os }}-node-${{ hashFiles('**/package-lock.json') }}

uses 문법

uses: {owner}/{repo}@{ref}
  • actions/checkout@v4: GitHub 공식 Action, v4 버전
  • peaceiris/actions-gh-pages@v3: 커뮤니티 Action
  • ./.github/actions/my-action: 로컬 Action
  • docker://alpine:3.8: Docker 컨테이너

베스트 프랙티스: Action 버전은 @v4처럼 메이저 버전을 사용하면 자동으로 최신 패치를 받습니다. @v4.1.2처럼 고정하면 재현 가능한 빌드를 보장합니다.

Job 간 의존성: needs¶

기본 병렬 실행

Job들은 기본적으로 병렬로 실행됩니다.

jobs:
  job1:
    runs-on: ubuntu-latest
    steps:
      - run: echo "Job 1"

  job2:
    runs-on: ubuntu-latest
    steps:
      - run: echo "Job 2"

위 예시에서 job1과 job2는 동시에 실행됩니다.

needs로 순서 지정

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - run: npm run build

  test:
    needs: build  # build가 성공해야 실행
    runs-on: ubuntu-latest
    steps:
      - run: npm test

  deploy:
    needs: [build, test]  # build와 test 모두 성공해야 실행
    runs-on: ubuntu-latest
    steps:
      - run: npm run deploy

실행 흐름

build
  ↓ (성공 시)
test
  ↓ (성공 시)
deploy

조건부 실행: if

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - run: npm test

  deploy:
    needs: test
    if: github.ref == 'refs/heads/main'  # main 브랜치에서만 배포
    runs-on: ubuntu-latest
    steps:
      - run: npm run deploy

주의: needs로 연결된 Job이 실패하면 후속 Job은 건너뜁니다. if: always()를 사용하면 이전 Job 실패 여부와 상관없이 실행할 수 있습니다.

병렬 실행과 순차 실행¶

병렬 실행 (기본)

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - run: npm run lint

  unit-test:
    runs-on: ubuntu-latest
    steps:
      - run: npm run test:unit

  integration-test:
    runs-on: ubuntu-latest
    steps:
      - run: npm run test:integration

위 세 Job은 동시에 실행되어 시간을 절약합니다.

순차 실행

jobs:
  install:
    runs-on: ubuntu-latest
    steps:
      - run: npm install

  lint:
    needs: install
    runs-on: ubuntu-latest
    steps:
      - run: npm run lint

  test:
    needs: lint
    runs-on: ubuntu-latest
    steps:
      - run: npm test

혼합 전략

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - run: npm run build

  test-unit:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - run: npm run test:unit

  test-integration:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - run: npm run test:integration

  deploy:
    needs: [test-unit, test-integration]
    runs-on: ubuntu-latest
    steps:
      - run: npm run deploy

실행 그래프

         build
         /    \
        /      \
  test-unit  test-integration
        \      /
         \    /
         deploy

베스트 프랙티스: 독립적인 작업(lint, test, build)은 병렬로, 의존성이 있는 작업(build → test → deploy)은 순차로 구성하세요.

매트릭스 전략(matrix)으로 다중 환경 테스트¶

매트릭스란

하나의 Job을 여러 환경(Node.js 버전, OS 등)에서 동시에 실행하는 전략입니다.

기본 매트릭스

jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        node-version: [18, 20, 22]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
      - run: npm install
      - run: npm test

위 설정은 Node.js 18, 20, 22 세 가지 버전에서 동시에 테스트를 실행합니다.

다차원 매트릭스

jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest]
        node-version: [18, 20, 22]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
      - run: npm test

위 설정은 3개 OS × 3개 Node.js 버전 = 9개 Job을 동시에 실행합니다.

특정 조합 제외

strategy:
  matrix:
    os: [ubuntu-latest, windows-latest, macos-latest]
    node-version: [18, 20, 22]
    exclude:
      - os: macos-latest
        node-version: 18  # macOS + Node 18 조합 제외

특정 조합 추가

strategy:
  matrix:
    os: [ubuntu-latest]
    node-version: [20]
    include:
      - os: windows-latest
        node-version: 20
        experimental: true

실패 처리

strategy:
  fail-fast: false  # 하나 실패해도 나머지 계속 실행
  matrix:
    node-version: [18, 20, 22]

기본값(fail-fast: true)은 하나라도 실패하면 나머지를 중단합니다.

팁: 매트릭스로 많은 조합을 테스트하면 무료 사용량을 빠르게 소진합니다. 핵심 조합(최신 버전, LTS 버전)만 선택하는 것을 권장합니다.

절 요약¶

  • Job은 독립적인 작업 단위로 별도 Runner에서 실행됩니다.
  • Step은 Job 내부의 순차 실행 단계로 run 또는 uses를 사용합니다.
  • needs로 Job 간 의존성을 정의하여 순차 실행을 구성합니다.
  • 독립적인 Job은 병렬로, 의존성이 있는 Job은 순차로 실행합니다.
  • 매트릭스 전략으로 여러 환경(OS, 언어 버전)에서 동시에 테스트할 수 있습니다.

주요 Actions 활용하기 (ch_08_s04)¶

도입¶

GitHub Actions의 진정한 힘은 재사용 가능한 Actions에 있습니다. 직접 모든 스크립트를 작성하는 대신, 검증된 공식 Actions와 커뮤니티 Actions를 활용하면 빠르고 안전하게 파이프라인을 구축할 수 있습니다. 이 절에서는 가장 많이 사용하는 Actions와 Secrets, 환경 변수를 다루는 방법을 배웁니다.

핵심 개념 설명¶

actions/checkout: 코드 체크아웃¶

왜 필요한가

GitHub Actions Runner는 빈 가상 머신으로 시작합니다. 저장소의 코드에 접근하려면 명시적으로 체크아웃해야 합니다.

기본 사용법

steps:
  - name: Checkout code
    uses: actions/checkout@v4

고급 옵션

steps:
  - name: Checkout with options
    uses: actions/checkout@v4
    with:
      ref: develop              # 특정 브랜치 체크아웃
      fetch-depth: 0            # 전체 히스토리 (기본 1)
      submodules: recursive     # 서브모듈 포함
      token: ${{ secrets.PAT }} # 다른 저장소 접근 시

fetch-depth 설정

  • fetch-depth: 1 (기본): 최신 커밋만 (빠름, 용량 작음)
  • fetch-depth: 0: 전체 히스토리 (git log, git blame 필요 시)
  • fetch-depth: 50: 최근 50개 커밋 (적당한 균형)

여러 저장소 체크아웃

steps:
  - name: Checkout main repo
    uses: actions/checkout@v4
    with:
      path: main-repo

  - name: Checkout shared library
    uses: actions/checkout@v4
    with:
      repository: username/shared-lib
      path: shared-lib
      token: ${{ secrets.PAT }}

베스트 프랙티스: 대부분의 경우 기본 설정(fetch-depth: 1)으로 충분합니다. 전체 히스토리가 필요한 경우에만 fetch-depth: 0을 사용하세요.

actions/setup-node, setup-python 등: 언어 환경 설정¶

Node.js 환경 설정

steps:
  - uses: actions/checkout@v4
  - name: Setup Node.js
    uses: actions/setup-node@v4
    with:
      node-version: '20'
      cache: 'npm'  # npm 캐싱 자동 설정
  - run: npm install
  - run: npm test

여러 버전 설정 (매트릭스와 조합)

strategy:
  matrix:
    node-version: [18, 20, 22]
steps:
  - uses: actions/setup-node@v4
    with:
      node-version: ${{ matrix.node-version }}

Python 환경 설정

steps:
  - uses: actions/checkout@v4
  - name: Setup Python
    uses: actions/setup-python@v5
    with:
      python-version: '3.11'
      cache: 'pip'
  - run: pip install -r requirements.txt
  - run: pytest

Java 환경 설정

steps:
  - uses: actions/checkout@v4
  - name: Setup Java
    uses: actions/setup-java@v4
    with:
      distribution: 'temurin'
      java-version: '17'
      cache: 'maven'
  - run: mvn clean test

Go 환경 설정

steps:
  - uses: actions/checkout@v4
  - name: Setup Go
    uses: actions/setup-go@v5
    with:
      go-version: '1.21'
      cache: true
  - run: go test ./...

팁: cache 옵션을 활성화하면 의존성 설치 시간을 크게 단축할 수 있습니다. npm, pip, maven, gradle 등을 자동으로 캐싱합니다.

actions/cache: 의존성 캐싱¶

왜 캐싱이 필요한가

매번 npm install, pip install 등으로 의존성을 다운로드하면 시간이 오래 걸립니다. 캐싱을 사용하면 이전 빌드의 의존성을 재사용하여 빌드 속도를 2~10배 향상시킬 수 있습니다.

기본 사용법

steps:
  - uses: actions/checkout@v4
  - name: Cache node modules
    uses: actions/cache@v3
    with:
      path: ~/.npm
      key: ${{ runner.os }}-node-${{ hashFiles('**/package-lock.json') }}
      restore-keys: |
        ${{ runner.os }}-node-
  - run: npm install
  - run: npm test

캐시 키 설명

  • key: 캐시의 고유 키 (정확히 일치하는 캐시 찾기)
  • ${{ runner.os }}: OS 구분 (Linux, Windows, macOS)
  • ${{ hashFiles('**/package-lock.json') }}: 의존성 파일의 해시
  • restore-keys: 정확한 키가 없을 때 부분 일치하는 캐시 사용

작동 방식

  1. package-lock.json의 해시 계산
  2. 해시가 같은 캐시가 있으면 복원
  3. 없으면 restore-keys에서 부분 일치 찾기
  4. Job 완료 후 새 캐시 저장

여러 경로 캐싱

- uses: actions/cache@v3
  with:
    path: |
      ~/.npm
      ~/.cache
      node_modules
    key: ${{ runner.os }}-npm-${{ hashFiles('**/package-lock.json') }}

Python 의존성 캐싱

- uses: actions/cache@v3
  with:
    path: ~/.cache/pip
    key: ${{ runner.os }}-pip-${{ hashFiles('**/requirements.txt') }}
    restore-keys: |
      ${{ runner.os }}-pip-

캐시 적중률 확인

Job 로그에서 다음 메시지를 확인할 수 있습니다.

Cache restored successfully
Cache saved successfully

베스트 프랙티스: 의존성 파일(package-lock.json, requirements.txt)의 해시를 키에 포함하여 의존성이 변경될 때만 캐시를 무효화하세요.

Actions Marketplace 탐색하기¶

Marketplace란

GitHub Marketplace는 수천 개의 공식 및 커뮤니티 Actions를 제공하는 플랫폼입니다.

인기 있는 Actions

Action 용도 별점
actions/checkout 코드 체크아웃 ⭐⭐⭐⭐⭐
actions/setup-node Node.js 환경 ⭐⭐⭐⭐⭐
actions/cache 의존성 캐싱 ⭐⭐⭐⭐⭐
docker/build-push-action Docker 이미지 빌드/푸시 ⭐⭐⭐⭐⭐
peaceiris/actions-gh-pages GitHub Pages 배포 ⭐⭐⭐⭐
codecov/codecov-action 코드 커버리지 업로드 ⭐⭐⭐⭐
release-drafter/release-drafter 릴리스 노트 자동 생성 ⭐⭐⭐⭐

Action 선택 기준

  1. 공식 여부: actions/*는 GitHub 공식 Action
  2. 별점과 사용자 수: 많을수록 신뢰성 높음
  3. 최근 업데이트: 6개월 이내 업데이트 확인
  4. 문서 품질: README가 상세한지 확인
  5. 오픈소스: 코드를 검토할 수 있는지 확인

Action 사용 예시

# Slack 알림
- name: Slack notification
  uses: 8398a7/action-slack@v3
  with:
    status: ${{ job.status }}
    text: 'Deployment completed!'
    webhook_url: ${{ secrets.SLACK_WEBHOOK }}
  if: always()

# 코드 커버리지 업로드
- name: Upload coverage to Codecov
  uses: codecov/codecov-action@v3
  with:
    files: ./coverage.xml
    flags: unittests

# Docker 이미지 빌드 및 푸시
- name: Build and push Docker image
  uses: docker/build-push-action@v5
  with:
    context: .
    push: true
    tags: username/app:latest

주의: 커뮤니티 Action을 사용할 때는 소스 코드를 검토하고 신뢰할 수 있는 작성자인지 확인하세요. 악의적인 Action은 비밀 정보를 탈취할 수 있습니다.

환경 변수와 Secrets 사용¶

환경 변수 설정

jobs:
  build:
    runs-on: ubuntu-latest
    env:
      NODE_ENV: production
      API_URL: https://api.example.com
    steps:
      - run: echo "Environment: $NODE_ENV"
      - run: echo "API URL: $API_URL"

Step별 환경 변수

steps:
  - name: Build for staging
    env:
      ENV: staging
    run: npm run build

  - name: Build for production
    env:
      ENV: production
    run: npm run build

GitHub 제공 환경 변수

steps:
  - run: |
      echo "Repository: $GITHUB_REPOSITORY"
      echo "Branch: $GITHUB_REF_NAME"
      echo "Commit SHA: $GITHUB_SHA"
      echo "Actor: $GITHUB_ACTOR"
      echo "Event: $GITHUB_EVENT_NAME"

전체 목록: GitHub 공식 문서

Secrets 설정

민감한 정보(API 키, 비밀번호, 토큰)는 Secrets에 저장해야 합니다.

1. Secrets 추가 (웹 UI)

Settings → Secrets and variables → Actions → New repository secret

2. 워크플로우에서 사용

steps:
  - name: Deploy to server
    env:
      API_KEY: ${{ secrets.API_KEY }}
      DB_PASSWORD: ${{ secrets.DB_PASSWORD }}
    run: ./deploy.sh

Secrets 특징

  • 로그에 자동으로 마스킹됨: ***로 표시
  • 암호화되어 저장
  • 워크플로우에서만 접근 가능 (PR에서는 제한)
  • Organization, Repository, Environment 레벨 지원

Organization Secrets

# Organization에 설정된 Secret 사용
- run: echo "${{ secrets.ORG_LEVEL_SECRET }}"

Environment Secrets

jobs:
  deploy:
    runs-on: ubuntu-latest
    environment: production  # production 환경의 Secrets 사용
    steps:
      - run: echo "${{ secrets.PROD_API_KEY }}"

보안 팁: Secrets를 직접 출력하지 마세요 (echo ${{ secrets.TOKEN }}). 로그에 마스킹되지만, 우회하여 유출할 수 있습니다.

절 요약¶

  • actions/checkout은 저장소 코드를 체크아웃하는 필수 Action입니다.
  • actions/setup-*로 Node.js, Python, Java 등 언어 환경을 간편히 설정합니다.
  • actions/cache로 의존성을 캐싱하여 빌드 속도를 크게 향상시킵니다.
  • GitHub Marketplace에서 검증된 공식/커뮤니티 Actions를 찾아 재사용할 수 있습니다.
  • 환경 변수는 env로, 민감 정보는 secrets로 안전하게 관리합니다.

실전 CI/CD 파이프라인 구축 (ch_08_s05)¶

도입¶

이론과 개별 기능을 익혔다면 이제 실전 파이프라인을 구축할 차례입니다. PR 검증 CI, 빌드 아티팩트 생성, 자동 배포, 릴리스 자동화 등 실무에서 바로 사용할 수 있는 파이프라인을 단계별로 구축합니다. 이 절을 마치면 여러분의 프로젝트에 GitHub Actions CI/CD를 완벽하게 적용할 수 있습니다.

핵심 개념 설명¶

PR 검증 CI 파이프라인: 린트 + 테스트¶

목표

PR이 생성되거나 업데이트되면 자동으로 코드 품질 검사(린트)와 테스트를 실행하여 main 브랜치의 품질을 보장합니다.

.github/workflows/pr-validation.yml

name: PR Validation

on:
  pull_request:
    branches: [main, develop]

jobs:
  lint:
    name: Lint
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Run ESLint
        run: npm run lint

      - name: Run Prettier
        run: npm run format:check

  test:
    name: Test
    runs-on: ubuntu-latest
    strategy:
      matrix:
        node-version: [18, 20, 22]
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js ${{ matrix.node-version }}
        uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Run unit tests
        run: npm run test:unit

      - name: Run integration tests
        run: npm run test:integration

      - name: Upload coverage
        uses: codecov/codecov-action@v3
        if: matrix.node-version == 20  # 하나의 버전에서만 업로드
        with:
          files: ./coverage/coverage-final.json
          flags: unittests

  build:
    name: Build
    runs-on: ubuntu-latest
    needs: [lint, test]
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - run: npm ci

      - name: Build
        run: npm run build

      - name: Check build output
        run: |
          if [ ! -d "dist" ]; then
            echo "Build failed: dist directory not found"
            exit 1
          fi

효과

  • PR 생성 시 자동으로 검증
  • 린트 오류가 있으면 머지 차단
  • 여러 Node.js 버전에서 테스트
  • 빌드 성공 여부 확인

베스트 프랙티스: 브랜치 보호 규칙에서 이 워크플로우를 필수 체크로 설정하면 검증을 통과하지 못한 PR은 머지할 수 없습니다.

빌드 아티팩트 생성과 업로드¶

빌드 아티팩트란

빌드 결과물(컴파일된 바이너리, 번들 파일, 패키지 등)을 저장하여 나중에 다운로드하거나 다른 Job에서 사용할 수 있게 하는 기능입니다.

아티팩트 업로드

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'

      - run: npm ci
      - run: npm run build

      - name: Upload build artifacts
        uses: actions/upload-artifact@v4
        with:
          name: dist-files
          path: dist/
          retention-days: 7  # 7일 후 자동 삭제

아티팩트 다운로드

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - run: npm run build
      - uses: actions/upload-artifact@v4
        with:
          name: dist-files
          path: dist/

  deploy:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - name: Download build artifacts
        uses: actions/download-artifact@v4
        with:
          name: dist-files
          path: dist/

      - name: Deploy
        run: ./deploy.sh dist/

여러 아티팩트 업로드

- uses: actions/upload-artifact@v4
  with:
    name: test-results
    path: |
      test-results/
      coverage/
      logs/*.log

아티팩트 제한

  • 저장소당 최대 용량: 10GB (Free), 100GB (Pro)
  • 개별 아티팩트 최대 크기: 2GB
  • 보관 기간: 기본 90일, 최대 400일

팁: 아티팩트는 빌드 결과물을 배포 Job에 전달하거나, 실패 시 디버깅용 로그를 다운로드하는 데 유용합니다.

GitHub Pages 자동 배포¶

GitHub Pages란

정적 웹사이트를 GitHub에서 무료로 호스팅하는 서비스입니다. 프로젝트 문서, 포트폴리오, 블로그 등에 활용됩니다.

배포 워크플로우

name: Deploy to GitHub Pages

on:
  push:
    branches: [main]
  workflow_dispatch:

permissions:
  contents: read
  pages: write
  id-token: write

concurrency:
  group: "pages"
  cancel-in-progress: true

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Build
        run: npm run build

      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: dist/

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4

GitHub Pages 설정

  1. Settings → Pages
  2. Source: GitHub Actions 선택
  3. 워크플로우 push 후 자동 배포

peaceiris/actions-gh-pages 사용 (대안)

- name: Deploy
  uses: peaceiris/actions-gh-pages@v3
  with:
    github_token: ${{ secrets.GITHUB_TOKEN }}
    publish_dir: ./dist
    cname: docs.example.com  # 커스텀 도메인

베스트 프랙티스: 빌드와 배포를 분리하여 빌드 실패 시 배포를 건너뛰도록 구성하세요.

릴리스 자동화: 태그 기반 배포¶

시나리오

v1.0.0 태그를 push하면 자동으로 빌드하고, GitHub Release를 생성하고, npm에 배포합니다.

워크플로우

name: Release

on:
  push:
    tags:
      - 'v*'  # v로 시작하는 태그 (v1.0.0, v2.1.3 등)

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          registry-url: 'https://registry.npmjs.org'

      - run: npm ci
      - run: npm run build
      - run: npm test

      - name: Upload build artifacts
        uses: actions/upload-artifact@v4
        with:
          name: release-build
          path: dist/

  create-release:
    needs: build
    runs-on: ubuntu-latest
    permissions:
      contents: write
    steps:
      - uses: actions/checkout@v4

      - name: Download build artifacts
        uses: actions/download-artifact@v4
        with:
          name: release-build
          path: dist/

      - name: Create Release
        uses: softprops/action-gh-release@v1
        with:
          draft: false
          prerelease: false
          generate_release_notes: true  # 자동 릴리스 노트
          files: |
            dist/**/*.js
            dist/**/*.css

  publish-npm:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          registry-url: 'https://registry.npmjs.org'

      - run: npm ci
      - run: npm run build

      - name: Publish to npm
        run: npm publish
        env:
          NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}

태그 생성 및 push

$ git tag -a v1.0.0 -m "Release version 1.0.0"
$ git push origin v1.0.0

자동 버전 관리 (semantic-release)

- name: Semantic Release
  uses: cycjimmy/semantic-release-action@v4
  env:
    GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
    NPM_TOKEN: ${{ secrets.NPM_TOKEN }}

semantic-release는 Conventional Commits 기반으로 자동으로 버전을 올리고 릴리스를 생성합니다.

팁: 릴리스 워크플로우는 신중하게 테스트하세요. npm, Docker Hub 등에 잘못된 버전을 배포하면 되돌리기 어렵습니다.

환경(Environment)별 배포 설정¶

Environment란

배포 대상 환경(staging, production)을 정의하고, 환경별 Secrets, 보호 규칙, 승인 프로세스를 설정하는 기능입니다.

Environment 생성

  1. Settings → Environments → New environment
  2. 이름: production
  3. 보호 규칙 설정:
  4. Required reviewers: 승인자 지정
  5. Wait timer: 배포 전 대기 시간
  6. Deployment branches: main 브랜치만 허용

워크플로우에서 Environment 사용

jobs:
  deploy-staging:
    runs-on: ubuntu-latest
    environment: staging
    steps:
      - uses: actions/checkout@v4
      - name: Deploy to staging
        run: ./deploy.sh staging
        env:
          API_KEY: ${{ secrets.STAGING_API_KEY }}

  deploy-production:
    needs: deploy-staging
    runs-on: ubuntu-latest
    environment:
      name: production
      url: https://myapp.com
    steps:
      - uses: actions/checkout@v4
      - name: Deploy to production
        run: ./deploy.sh production
        env:
          API_KEY: ${{ secrets.PROD_API_KEY }}

승인 프로세스

production 환경에 Required reviewers를 설정하면: 1. 워크플로우가 deploy-production Job 전에 일시 정지 2. 지정된 승인자에게 알림 전송 3. 승인자가 "Review deployments" 클릭하여 승인 4. 승인 후 배포 진행

Environment Secrets

Environment별로 다른 Secrets를 설정할 수 있습니다.

Settings → Environments → production → Add secret
  • STAGING_API_KEY: staging 환경 전용
  • PROD_API_KEY: production 환경 전용

배포 이력 추적

Environments 페이지에서 각 환경의 배포 이력을 확인할 수 있습니다. - 누가 배포했는지 - 언제 배포했는지 - 어떤 커밋이 배포되었는지

베스트 프랙티스: production 환경에는 항상 승인 프로세스를 설정하세요. 실수로 버그가 프로덕션에 배포되는 것을 방지할 수 있습니다.

복합 예시: Staging → Production 파이프라인

name: Deploy Pipeline

on:
  push:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm test

  deploy-staging:
    needs: test
    runs-on: ubuntu-latest
    environment: staging
    steps:
      - uses: actions/checkout@v4
      - name: Deploy to Staging
        run: ./deploy.sh staging
        env:
          API_KEY: ${{ secrets.STAGING_API_KEY }}

      - name: Smoke test
        run: curl -f https://staging.myapp.com/health

  deploy-production:
    needs: deploy-staging
    runs-on: ubuntu-latest
    environment:
      name: production
      url: https://myapp.com
    steps:
      - uses: actions/checkout@v4
      - name: Deploy to Production
        run: ./deploy.sh production
        env:
          API_KEY: ${{ secrets.PROD_API_KEY }}

      - name: Smoke test
        run: curl -f https://myapp.com/health

      - name: Notify Slack
        uses: 8398a7/action-slack@v3
        with:
          status: ${{ job.status }}
          text: 'Production deployment completed!'
          webhook_url: ${{ secrets.SLACK_WEBHOOK }}
        if: always()

실행 흐름

test → deploy-staging → [승인 대기] → deploy-production

절 요약¶

  • PR 검증 CI는 lint, test, build를 자동화하여 코드 품질을 보장합니다.
  • 아티팩트로 빌드 결과물을 저장하고 다른 Job에서 재사용할 수 있습니다.
  • GitHub Pages 자동 배포로 문서 사이트를 간편하게 호스팅합니다.
  • 태그 기반 릴리스 자동화로 버전 관리와 배포를 간소화합니다.
  • Environment별 배포 설정과 승인 프로세스로 안전한 프로덕션 배포를 구현합니다.

FAQ¶

Q1: 워크플로우가 실행되지 않습니다.

A1: 다음을 확인하세요. 1. 파일 위치: .github/workflows/ 디렉토리에 있는지 확인 2. YAML 문법: 온라인 YAML 검증 도구로 문법 오류 확인 3. 트리거 조건: on 섹션의 브랜치, 경로 조건이 맞는지 확인 4. Repository Actions 활성화: Settings → Actions → General → "Allow all actions" 선택 5. 파일 확장자: .yml 또는 .yaml인지 확인 (.txt가 아님)

Q2: Private 저장소에서 무료 사용량을 초과했습니다.

A2: 다음 방법으로 사용량을 줄이세요. - Linux runner 우선 사용 (Windows/macOS는 배수 적용) - 캐싱 적극 활용 (actions/cache) - 불필요한 매트릭스 조합 제거 - paths 필터로 특정 파일 변경 시에만 실행 - 개발 중에는 Draft PR 사용하여 워크플로우 건너뛰기 - Self-hosted runner 사용 (자체 서버)

Q3: Secrets가 로그에 노출됩니다.

A3: GitHub은 자동으로 Secrets를 ***로 마스킹합니다. 하지만 다음 경우 우회될 수 있습니다. - Base64 인코딩: echo $SECRET | base64 - JSON 출력: jq 등으로 JSON에 포함 - 환경 변수 전체 출력: env

해결책: Secrets를 절대 직접 출력하지 말고, 필요한 경우 임시 파일에 저장한 후 파일 경로만 전달하세요.

Q4: 다른 저장소의 워크플로우를 트리거하고 싶습니다.

A4: Repository Dispatch 이벤트를 사용하세요.

저장소 A (트리거 발신)

- name: Trigger another workflow
  run: |
    curl -X POST \
      -H "Authorization: token ${{ secrets.PAT }}" \
      -H "Accept: application/vnd.github.v3+json" \
      https://api.github.com/repos/username/repo-b/dispatches \
      -d '{"event_type":"trigger-event"}'

저장소 B (트리거 수신)

on:
  repository_dispatch:
    types: [trigger-event]

Q5: 워크플로우가 너무 느립니다.

A5: 다음 최적화 방법을 시도하세요. 1. 캐싱 활성화: actions/cache, actions/setup-node 등의 cache 옵션 2. Job 병렬화: 의존성 없는 Job은 병렬 실행 3. 불필요한 단계 제거: 빌드 로그 확인하여 시간 소요 파악 4. 조건부 실행: if 조건으로 필요한 경우에만 실행 5. Self-hosted runner: 더 강력한 하드웨어 사용

예시:

- name: Run expensive tests
  if: github.event_name == 'push' && github.ref == 'refs/heads/main'
  run: npm run test:e2e

장 요약¶

이번 장에서는 GitHub Actions로 CI/CD 파이프라인을 마스터했습니다. CI/CD의 개념과 GitHub Actions의 장점을 이해하고, YAML 워크플로우 파일을 작성하는 방법을 배웠습니다. 트리거 설정으로 push, PR, schedule, 수동 실행 등 다양한 시점에 워크플로우를 실행하도록 구성했습니다. Jobs와 Steps의 구조를 이해하고, needs로 의존성을 설정하며, 매트릭스 전략으로 여러 환경을 동시에 테스트하는 기법을 익혔습니다. actions/checkout, actions/setup-node, actions/cache 등 핵심 Actions를 활용하여 효율적인 파이프라인을 구축했습니다. 마지막으로 PR 검증 CI, 아티팩트 관리, GitHub Pages 배포, 릴리스 자동화, Environment별 배포 설정까지 실전 파이프라인을 완성했습니다.
