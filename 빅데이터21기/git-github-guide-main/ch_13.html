13장: 오픈소스 기여하기

실무

오픈소스 기여하기¶

학습 목표 - Fork 기반 워크플로우로 오픈소스 프로젝트에 기여할 수 있다 - CONTRIBUTING.md를 읽고 프로젝트의 기여 규칙을 이해할 수 있다 - 오픈소스 라이선스의 종류와 의무사항을 구분할 수 있다 - PR을 통해 실제 오픈소스 프로젝트에 첫 기여를 할 수 있다 - 자신의 오픈소스 프로젝트를 시작하고 운영할 수 있다

Fork 기반 워크플로우 (ch_13_s01)¶

도입¶

오픈소스 프로젝트에 기여하고 싶은데 저장소에 직접 접근 권한이 없다면 어떻게 해야 할까요? 이 절에서는 누구나 오픈소스에 기여할 수 있게 해주는 Fork 기반 워크플로우를 배웁니다. Fork는 GitHub의 핵심 협업 기능으로, 원본 저장소에 영향을 주지 않고 자유롭게 실험하고 개선할 수 있습니다.

핵심 개념 설명¶

Fork란 무엇인가: 개인 복사본¶

Fork는 다른 사용자의 저장소를 자신의 GitHub 계정으로 완전히 복사하는 기능입니다. 요리 레시피를 복사해서 나만의 레시피북에 적어두는 것과 같습니다. 원본 레시피(원본 저장소)를 바꾸지 않고 자유롭게 변형(수정)할 수 있고, 좋은 변형이 있으면 원본 작성자에게 제안(PR)할 수 있습니다.

Fork의 주요 특징

  1. 완전한 복사본: 원본 저장소의 모든 커밋 히스토리, 브랜치, 태그가 복사됩니다.
  2. 독립적 작업: 원본 저장소에 영향을 주지 않고 자유롭게 수정할 수 있습니다.
  3. 원본과의 연결 유지: 원본 저장소의 업데이트를 쉽게 가져올 수 있습니다.
  4. 기여 경로 제공: Pull Request를 통해 원본 저장소에 변경 사항을 제안할 수 있습니다.

주의: Fork는 GitHub 서버에서의 복사이고 Clone은 로컬로의 복사입니다. Fork 후에는 반드시 로컬로 Clone하여 작업해야 합니다.

Fork -> Clone -> Branch -> PR 워크플로우¶

오픈소스 기여의 표준 워크플로우는 다음과 같습니다.

1단계: 원본 저장소 Fork하기

GitHub 웹 UI에서 기여하고 싶은 저장소로 이동하여 우측 상단의 Fork 버튼을 클릭합니다. 이제 original-owner/repo가 my-username/repo로 복사됩니다.

2단계: Fork한 저장소를 로컬로 Clone하기

$ git clone git@github.com:my-username/awesome-project.git
$ cd awesome-project

팁: SSH URL을 사용하면 매번 비밀번호를 입력하지 않아도 됩니다. SSH 키 설정은 ch_04를 참고하세요.

3단계: Upstream 원격 저장소 추가하기

원본 저장소의 업데이트를 받기 위해 upstream 원격 저장소를 추가합니다.

$ git remote add upstream https://github.com/original-owner/awesome-project.git
$ git remote -v

# 출력:
# origin    git@github.com:my-username/awesome-project.git (fetch)
# origin    git@github.com:my-username/awesome-project.git (push)
# upstream  https://github.com/original-owner/awesome-project.git (fetch)
# upstream  https://github.com/original-owner/awesome-project.git (push)

이제 두 개의 원격 저장소가 설정되었습니다.

  • origin: 자신의 Fork (읽기/쓰기 가능)
  • upstream: 원본 저장소 (읽기만 가능)

4단계: 기능 브랜치 생성하기

main 브랜치에서 직접 작업하지 말고, 새 브랜치를 만들어 작업합니다.

$ git switch -c fix-typo-in-readme

브랜치 이름은 작업 내용을 명확히 표현해야 합니다.

  • fix-typo-in-readme
  • add-korean-translation
  • improve-performance-algorithm

5단계: 변경 사항 커밋하기

$ git add README.md
$ git commit -m "docs: Fix typo in installation section"

베스트 프랙티스: 프로젝트의 커밋 규약(Conventional Commits 등)을 따르세요. CONTRIBUTING.md에 명시되어 있습니다.

6단계: Fork에 Push하기

$ git push origin fix-typo-in-readme

7단계: Pull Request 생성하기

GitHub에서 자신의 Fork 저장소로 이동하면 "Compare & pull request" 버튼이 표시됩니다. 버튼을 클릭하여 PR을 작성합니다.

  • 베이스 저장소: original-owner/awesome-project (main)
  • 헤드 저장소: my-username/awesome-project (fix-typo-in-readme)

PR 설명에는 변경 이유, 관련 Issue 번호, 테스트 방법 등을 포함합니다.

Upstream 설정과 Fork 동기화¶

원본 저장소는 계속 업데이트됩니다. 자신의 Fork를 최신 상태로 유지하려면 정기적으로 upstream의 변경 사항을 가져와야 합니다.

방법 1: 로컬에서 동기화 후 Push

$ git switch main
$ git fetch upstream
$ git merge upstream/main
$ git push origin main

방법 2: GitHub UI에서 Sync Fork 사용

GitHub 웹 UI에서 Fork 저장소로 이동하여 "Sync fork" 버튼을 클릭합니다. "Update branch"를 누르면 원본 저장소의 최신 변경 사항이 Fork에 반영됩니다.

주의: 로컬에서 작업한 내용이 있다면 충돌이 발생할 수 있으므로, 먼저 로컬 작업을 커밋하거나 stash 하세요.

동기화 후 로컬 업데이트

$ git pull origin main

Fork vs Clone: 언제 무엇을 쓸 것인가¶

구분 Fork Clone
위치 GitHub 서버에서 복사 로컬 컴퓨터로 복사
목적 기여를 위한 개인 복사본 생성 로컬 작업 환경 구축
원본과의 관계 연결 유지 (upstream 설정 가능) 연결 없음 (직접 추가해야 함)
쓰기 권한 Fork한 저장소에는 있음 로컬 파일 시스템 권한만
PR 생성 가능 원본 저장소 쓰기 권한 필요

사용 시나리오

  • Fork: 오픈소스 프로젝트에 기여할 때
  • Clone: 직접 쓰기 권한이 있는 저장소(팀 프로젝트 등)를 로컬에서 작업할 때

절 요약¶

  • Fork는 원본 저장소를 자신의 계정으로 완전히 복사하는 기능입니다.
  • Fork -> Clone -> Branch -> PR 워크플로우가 오픈소스 기여의 표준입니다.
  • upstream 원격 저장소를 추가하여 원본 저장소의 업데이트를 가져올 수 있습니다.
  • Fork는 GitHub 서버의 복사, Clone은 로컬의 복사입니다.

기여 준비: 프로젝트 이해하기 (ch_13_s02)¶

도입¶

오픈소스 프로젝트에 기여하기 전에 프로젝트의 문화, 규칙, 구조를 이해하는 것이 중요합니다. 이 절에서는 기여할 프로젝트를 찾고, 기여 가이드라인을 읽고, 프로젝트 환경을 파악하는 방법을 배웁니다. 준비를 철저히 하면 첫 기여가 성공할 확률이 높아집니다.

핵심 개념 설명¶

기여할 프로젝트 찾기¶

좋은 첫 프로젝트의 조건

  1. 활발한 유지보수: 최근 커밋, Issue, PR이 활발한 프로젝트
  2. 초보자 친화적: "Good First Issue", "Help Wanted" 라벨이 있는 프로젝트
  3. 명확한 문서: README, CONTRIBUTING.md가 잘 작성된 프로젝트
  4. 관심 분야: 자신이 사용하거나 관심 있는 기술 스택

프로젝트 찾는 방법

  • GitHub Explore: https://github.com/explore에서 트렌딩 프로젝트 탐색
  • GitHub Topics: good-first-issue, hacktoberfest 등의 토픽으로 검색
  • First Timers Only: https://www.firsttimersonly.com/
  • Up For Grabs: https://up-for-grabs.net/
  • 자신이 사용 중인 도구: 평소 사용하는 라이브러리나 프레임워크

팁: 처음에는 문서 개선, 번역, 오타 수정 등 코드가 아닌 기여로 시작하는 것이 좋습니다.

CONTRIBUTING.md 읽기: 기여 가이드라인¶

CONTRIBUTING.md는 프로젝트의 기여 가이드라인을 담은 문서로, 프로젝트마다 다른 규칙과 절차를 정의합니다.

일반적인 CONTRIBUTING.md 내용

  1. 기여 절차: Fork, 브랜치 생성, PR 작성 방법
  2. 개발 환경 설정: 의존성 설치, 빌드 방법, 테스트 실행
  3. 커밋 규약: 커밋 메시지 형식 (Conventional Commits 등)
  4. 코드 스타일: 린트 규칙, 포맷팅 도구 (Prettier, ESLint 등)
  5. 테스트 요구사항: 새 코드에 대한 테스트 작성 규칙
  6. PR 템플릿: PR 작성 시 포함해야 할 내용
  7. 리뷰 프로세스: 리뷰 대기 시간, 리뷰어 선정 방법

CONTRIBUTING.md 예시

# Contributing to Awesome Project

## Development Setup

1. Fork the repository
2. Clone your fork: `git clone git@github.com:your-username/awesome-project.git`
3. Install dependencies: `npm install`
4. Run tests: `npm test`

## Commit Message Format

We follow Conventional Commits:

- `feat:` New feature
- `fix:` Bug fix
- `docs:` Documentation changes
- `test:` Adding or updating tests

Example: `feat: Add Korean translation for README`

## Pull Request Process

1. Create a feature branch: `git switch -c feature/my-feature`
2. Make your changes and commit
3. Push to your fork: `git push origin feature/my-feature`
4. Open a PR against the `main` branch
5. Wait for CI checks to pass
6. Address review feedback

## Code Style

We use Prettier for formatting. Run `npm run format` before committing.

베스트 프랙티스: CONTRIBUTING.md를 읽지 않고 PR을 만들면 거절당할 가능성이 높습니다. 반드시 먼저 읽으세요.

Code of Conduct: 행동 강령¶

Code of Conduct는 커뮤니티 구성원이 지켜야 할 행동 규칙입니다. 존중, 배려, 포용의 원칙을 명시하여 건강한 커뮤니티를 유지합니다.

일반적인 Code of Conduct 내용

  1. 존중과 배려: 다른 기여자를 존중하고 건설적인 피드백 제공
  2. 차별 금지: 인종, 성별, 종교, 국적, 성적 지향 등에 따른 차별 금지
  3. 괴롭힘 금지: 공격적 언어, 인신공격, 스토킹 등 금지
  4. 신고 절차: 위반 사례 발견 시 신고 방법
  5. 처벌 조치: 규정 위반 시 경고, 차단 등의 조치

대표적인 Code of Conduct

  • Contributor Covenant: https://www.contributor-covenant.org/
  • Citizen Code of Conduct: https://github.com/stumpsyn/policies/blob/master/citizen_code_of_conduct.md

주의: Code of Conduct를 위반하면 프로젝트에서 영구 차단될 수 있습니다. 항상 예의를 지키세요.

Good First Issue로 시작하기¶

Good First Issue는 초보자가 쉽게 시작할 수 있도록 프로젝트 메인테이너가 선별한 이슈입니다.

Good First Issue의 특징

  1. 명확한 범위: 작업 범위가 제한적이고 명확함
  2. 낮은 복잡도: 프로젝트 전체 구조를 이해하지 않아도 해결 가능
  3. 자세한 설명: 문제 상황, 기대 결과, 힌트가 포함됨
  4. 멘토링 가능: 메인테이너가 도움을 줄 준비가 되어 있음

Good First Issue 찾기

# GitHub CLI로 검색
$ gh issue list --repo owner/repo --label "good first issue"

# GitHub UI에서 검색
# Issues 탭 -> Labels -> "good first issue" 클릭

다른 유용한 라벨

  • help wanted: 외부 기여자의 도움이 필요한 이슈
  • documentation: 문서 관련 작업
  • bug: 버그 수정
  • enhancement: 기능 개선

팁: 이슈를 선택했으면 댓글로 "I'd like to work on this"라고 남겨 중복 작업을 방지하세요.

프로젝트 구조와 빌드 환경 파악¶

기여하기 전에 프로젝트의 구조와 빌드 환경을 이해해야 합니다.

프로젝트 구조 파악하기

  1. README.md 읽기: 프로젝트 목적, 주요 기능, 사용법 이해
  2. 디렉토리 구조 확인: 소스 코드, 테스트, 문서 위치 파악
  3. package.json / requirements.txt 등: 의존성 및 스크립트 확인

빌드 환경 설정하기

# Node.js 프로젝트 예시
$ npm install
$ npm test
$ npm run build

# Python 프로젝트 예시
$ python -m venv venv
$ source venv/bin/activate  # Windows: venv\Scripts\activate
$ pip install -r requirements.txt
$ pytest

# Go 프로젝트 예시
$ go mod download
$ go test ./...
$ go build

로컬에서 실행 확인

# 웹 프로젝트
$ npm start
# 브라우저에서 http://localhost:3000 확인

# CLI 도구
$ go run main.go --help

베스트 프랙티스: 코드를 수정하기 전에 테스트가 통과하는지 먼저 확인하세요. 환경 문제를 미리 해결할 수 있습니다.

절 요약¶

  • 기여할 프로젝트는 활발하고, 초보자 친화적이며, 문서가 잘 갖춰진 곳을 선택합니다.
  • CONTRIBUTING.md는 프로젝트의 기여 규칙을 정의하며, 반드시 먼저 읽어야 합니다.
  • Code of Conduct는 커뮤니티 행동 규칙으로, 위반 시 차단될 수 있습니다.
  • Good First Issue로 시작하여 프로젝트에 익숙해지는 것이 좋습니다.
  • 로컬 빌드 환경을 먼저 설정하고 테스트가 통과하는지 확인합니다.

첫 번째 기여 실습 (ch_13_s03)¶

도입¶

이제 실제로 오픈소스 프로젝트에 첫 기여를 할 시간입니다. 이 절에서는 좋은 첫 기여 주제를 선택하고, 기여용 브랜치를 만들고, 프로젝트 규약에 맞춰 커밋하고, PR을 작성하고, 리뷰에 대응하는 전체 과정을 단계별로 안내합니다. 긴장하지 마세요. 모든 오픈소스 기여자는 이 과정을 거쳤습니다.

핵심 개념 설명¶

좋은 첫 기여: 문서 오타 수정, 번역¶

코드가 아닌 기여도 가치 있습니다

첫 기여로 코드를 작성할 필요는 없습니다. 다음과 같은 비코드 기여도 프로젝트에 큰 도움이 됩니다.

1. 문서 오타 수정

# 예시: README.md의 오타 수정
- insatllation  # 틀림
+ installation  # 맞음

2. 번역 추가

많은 프로젝트가 다국어 지원을 환영합니다.

# 한국어 번역 추가
README.md
README.ko.md  # 새로 생성

3. 예제 코드 추가

# 사용 예시 추가
## Examples

### Basic Usage

\`\`\`python
import awesome_lib

result = awesome_lib.process("Hello")
print(result)  # Output: "HELLO"
\`\`\`

4. 링크 수정

404 에러 링크, 깨진 이미지 링크 등을 수정합니다.

5. 코드 주석 개선

# 수정 전
def process(data):
    return data.upper()

# 수정 후
def process(data):
    """
    Convert input string to uppercase.

    Args:
        data (str): Input string to convert

    Returns:
        str: Uppercase version of input
    """
    return data.upper()

팁: 문서 개선은 코드보다 리뷰가 빠르고 승인될 확률이 높아 첫 기여로 적합합니다.

프로젝트 커밋 규약에 맞춰 커밋하기¶

Conventional Commits 규약

대부분의 프로젝트가 Conventional Commits 규약을 따릅니다.

<type>(<scope>): <subject>

<body>

<footer>

주요 타입

  • feat: 새로운 기능
  • fix: 버그 수정
  • docs: 문서 변경
  • style: 코드 포맷팅 (동작 변경 없음)
  • refactor: 리팩터링
  • test: 테스트 추가/수정
  • chore: 빌드 설정, 의존성 업데이트 등

예시

# 문서 오타 수정
$ git commit -m "docs: Fix typo in installation section"

# 한국어 번역 추가
$ git commit -m "docs: Add Korean translation for README"

# 예제 추가
$ git commit -m "docs: Add usage example for basic configuration"

# 버그 수정
$ git commit -m "fix: Resolve crash when input is empty

This commit adds a null check before processing input.

Fixes #123"

주의: 프로젝트마다 규약이 다를 수 있으므로 CONTRIBUTING.md를 확인하세요.

커밋 메시지 작성 팁

  1. 제목은 명령형으로: "Fix bug" (O), "Fixed bug" (X)
  2. 제목은 50자 이내: 간결하게 요약
  3. 본문은 72자마다 줄바꿈: 읽기 쉽게 정렬
  4. Why보다 What과 How: 무엇을, 어떻게 변경했는지 설명
  5. 관련 Issue 번호 포함: Fixes #123, Closes #456

라이선스의 이해 (ch_13_s04)¶

도입¶

오픈소스 프로젝트를 사용하거나 기여할 때 라이선스는 매우 중요한 법적 문서입니다. 이 절에서는 오픈소스 라이선스의 필요성, Permissive와 Copyleft 라이선스의 차이, 주요 라이선스의 특징, 라이선스 선택 가이드, 라이선스 호환성 문제를 배웁니다. 라이선스를 이해하지 못하면 법적 분쟁에 휘말릴 수 있습니다.

핵심 개념 설명¶

왜 라이선스가 중요한가¶

라이선스의 역할

오픈소스 라이선스는 소프트웨어의 사용, 수정, 배포 권한과 의무를 명시하는 법적 문서입니다. Creative Commons와 같은 저작물 이용 약관과 같습니다. '마음대로 써도 됨(MIT)', '출처만 밝히면 됨(Apache)', '같은 조건으로 공유해야 함(GPL)' 등 사용 조건이 다릅니다.

라이선스 없는 코드의 위험

# 라이선스가 없는 GitHub 저장소

MIT LICENSE 파일 없음
Apache LICENSE 파일 없음
GPL LICENSE 파일 없음
→ 기본적으로 저작권법 적용
→ 저작권자의 명시적 허락 없이 사용, 수정, 배포 금지

주의: 라이선스가 없는 코드는 자유롭게 사용할 수 있는 것이 아니라, 오히려 사용이 제한됩니다.

라이선스 선택의 영향

  1. 사용자: 프로젝트를 어떻게 사용할 수 있는가
  2. 기여자: 기여한 코드의 권리는 누구에게 있는가
  3. 상업적 이용: 상업 프로젝트에서 사용 가능한가
  4. 파생 작업: 수정한 코드의 라이선스는 무엇인가

Permissive 라이선스: MIT, Apache 2.0, BSD¶

Permissive 라이선스의 특징

Permissive(허용적) 라이선스는 최소한의 제약으로 자유로운 사용을 허용합니다.

1. MIT License

가장 인기 있는 오픈소스 라이선스로, 매우 간단하고 허용적입니다.

허용사항:
- 상업적 이용 ✓
- 수정 ✓
- 배포 ✓
- 개인 사용 ✓
- 재배포 시 라이선스 변경 ✓

의무사항:
- 라이선스 및 저작권 고지 포함

금지사항:
- 보증 없음 (책임 면책)

사용 예: React, Angular, Rails, jQuery

2. Apache License 2.0

MIT보다 조금 더 상세하며, 특허권 보호 조항이 있습니다.

허용사항:
- 상업적 이용 ✓
- 수정 ✓
- 배포 ✓
- 개인 사용 ✓
- 특허 사용 ✓

의무사항:
- 라이선스 및 저작권 고지 포함
- 변경 사항 명시 (NOTICE 파일)

금지사항:
- 상표권 사용 금지
- 보증 없음 (책임 면책)

사용 예: Kubernetes, TensorFlow, Android

3. BSD License (2-Clause, 3-Clause)

MIT와 유사하나, BSD 3-Clause는 상표권 보호 조항이 추가되었습니다.

허용사항:
- MIT와 거의 동일

의무사항:
- 라이선스 및 저작권 고지 포함

금지사항 (3-Clause):
- 프로젝트 이름이나 기여자 이름을 홍보에 사용 금지

사용 예: Go, FreeBSD, Django

팁: MIT와 Apache 2.0 중 선택이 어렵다면, 특허 관련 프로젝트는 Apache 2.0, 그 외는 MIT를 권장합니다.

Copyleft 라이선스: GPL, LGPL, AGPL¶

Copyleft 라이선스의 특징

Copyleft(상호주의) 라이선스는 파생 작업물도 같은 라이선스로 공개할 것을 요구합니다.

1. GPL v3 (GNU General Public License v3)

가장 강력한 Copyleft 라이선스입니다.

허용사항:
- 상업적 이용 ✓
- 수정 ✓
- 배포 ✓
- 개인 사용 ✓

의무사항:
- 소스 코드 공개 (배포 시)
- 라이선스 및 저작권 고지 포함
- 변경 사항 명시
- 파생 작업물도 GPL로 공개

금지사항:
- 보증 없음 (책임 면책)

사용 예: Linux Kernel, Git, GIMP

2. LGPL (GNU Lesser General Public License)

GPL보다 완화된 버전으로, 라이브러리로 링크만 하는 경우 GPL 의무가 전파되지 않습니다.

허용사항:
- 동적 링크는 LGPL 전파 안 됨
- 정적 링크는 LGPL 전파됨

의무사항:
- 라이브러리 자체 수정 시 소스 공개
- 라이선스 및 저작권 고지 포함

사용 예: Qt (일부), GTK+, GNU C Library

3. AGPL (Affero GPL)

GPL보다 더 강력하며, 네트워크 서비스로 제공 시에도 소스 공개 의무가 있습니다.

허용사항:
- GPL과 동일

의무사항:
- GPL 의무 + 네트워크 서비스 제공 시에도 소스 공개

사용 시나리오:
- 웹 애플리케이션으로 서비스할 때도 소스 공개 필요

사용 예: MongoDB (과거), Nextcloud, Mastodon

주의: GPL 라이브러리를 사용한 상업 소프트웨어는 전체 소스를 공개해야 할 수 있습니다.

라이선스 선택 가이드¶

프로젝트 목적에 따른 라이선스 선택

목적 추천 라이선스 이유
최대한의 자유 MIT 가장 간단하고 제약 없음
특허 보호 필요 Apache 2.0 특허권 명시적 부여
파생 작업물도 오픈소스 유지 GPL v3 강력한 Copyleft
라이브러리 (링크 허용) LGPL 동적 링크는 전파 안 됨
SaaS 서비스 AGPL 네트워크 서비스도 공개
공공 도메인 Unlicense / CC0 모든 권리 포기

주요 고려사항

  1. 상업적 이용 허용 여부: MIT/Apache는 허용, GPL은 제한적
  2. 기여자 유입: MIT/Apache가 GPL보다 기여자가 많은 경향
  3. 기업 친화성: MIT/Apache가 기업에서 선호됨
  4. 철학적 선택: 자유 소프트웨어 운동 지지 시 GPL

GitHub에서 인기 있는 라이선스

  1. MIT License (45%)
  2. Apache License 2.0 (15%)
  3. GPL v3 (10%)
  4. BSD 3-Clause (5%)
  5. LGPL (3%)

베스트 프랙티스: 확신이 서지 않으면 MIT License를 선택하세요. 가장 널리 쓰이고 이해하기 쉽습니다.

라이선스 호환성과 주의사항¶

라이선스 호환성 문제

여러 라이선스의 코드를 섞을 때 충돌이 발생할 수 있습니다.

호환성 매트릭스

프로젝트 라이선스 MIT 코드 사용 Apache 코드 사용 GPL 코드 사용
MIT ✓ ✓ X (GPL로 전환 필요)
Apache 2.0 ✓ ✓ X (GPL로 전환 필요)
GPL v3 ✓ ✓ ✓

주의사항

  1. GPL은 전염성이 있음: GPL 코드를 포함하면 전체 프로젝트가 GPL이 되어야 함
  2. Apache 2.0 + GPL v2 비호환: Apache 2.0과 GPL v2는 호환되지 않음 (GPL v3는 호환)
  3. 의존성 라이선스 확인: npm, pip 등 의존성의 라이선스도 검토 필요
  4. 상표권과 특허권: 라이선스가 코드만 다루고 상표는 별도

라이선스 확인 도구

# Node.js 프로젝트 의존성 라이선스 확인
$ npx license-checker --summary

# Python 프로젝트
$ pip-licenses

# GitHub에서 라이선스 확인
$ gh repo view owner/repo --json licenseInfo

듀얼 라이선싱

일부 프로젝트는 두 가지 라이선스를 제공합니다.

Qt: LGPL (오픈소스) / Commercial License (상업용)
MySQL: GPL / Commercial License

기업은 GPL 의무를 피하기 위해 상업 라이선스를 구매합니다.

주의: 오픈소스 프로젝트를 상업 제품에 사용하기 전에 반드시 법무팀과 상의하세요.

절 요약¶

  • 라이선스는 소프트웨어 사용, 수정, 배포 권한과 의무를 명시합니다.
  • Permissive 라이선스(MIT, Apache)는 자유롭고, Copyleft(GPL)는 파생 작업물 공개를 요구합니다.
  • 프로젝트 목적에 따라 적절한 라이선스를 선택해야 합니다.
  • 라이선스 호환성을 고려하여 의존성을 선택해야 합니다.
  • 라이선스 없는 코드는 사용이 제한됩니다.

오픈소스 프로젝트 운영하기 (ch_13_s05)¶

도입¶

오픈소스에 기여하는 것을 넘어, 자신의 프로젝트를 만들어 운영할 수도 있습니다. 이 절에서는 나만의 오픈소스 프로젝트를 시작하는 방법, 필수 문서 작성, 이슈 및 PR 템플릿 설정, 커뮤니티 빌딩, 릴리스 관리 방법을 배웁니다. 작은 프로젝트라도 체계적으로 운영하면 많은 기여자를 끌어들일 수 있습니다.

핵심 개념 설명¶

나만의 오픈소스 프로젝트 시작하기¶

프로젝트 아이디어 찾기

  1. 자신이 겪은 문제 해결: 직접 필요해서 만든 도구가 최고의 프로젝트
  2. 기존 도구 개선: 불편한 점을 개선한 대안 도구
  3. 학습 목적: 새로운 기술을 배우며 만든 프로젝트
  4. 재미와 실험: 창의적인 아이디어 구현

프로젝트 초기 설정

# GitHub에서 새 저장소 생성
$ gh repo create my-awesome-tool --public --clone

# 기본 파일 생성
$ cd my-awesome-tool
$ touch README.md LICENSE CONTRIBUTING.md CODE_OF_CONDUCT.md .gitignore
$ mkdir .github/ISSUE_TEMPLATE

좋은 프로젝트 이름 짓기

  • 짧고 기억하기 쉬움: axios, jest, vite
  • 의미 전달: prettier, webpack, eslint
  • 검색 가능: 너무 일반적인 단어는 피함
  • 사용 가능: npm, PyPI 등에서 중복 확인

팁: 이름을 짓기 어렵다면 namae.dev에서 사용 가능 여부를 확인하세요.

커뮤니티 빌딩과 기여자 관리¶

초기 기여자 확보하기

  1. README 작성: 명확한 설명과 예제
  2. Good First Issue 라벨: 초보자 친화적 이슈 표시
  3. Hacktoberfest 참여: 10월에 진행되는 오픈소스 축제
  4. SNS 홍보: Twitter, Reddit, Hacker News 등
  5. Show HN 게시: https://news.ycombinator.com/

기여자 응대하기

  1. 빠른 응답: 이슈와 PR에 24-48시간 내 응답
  2. 정중한 태도: 모든 기여에 감사 표시
  3. 명확한 피드백: 개선이 필요한 부분 구체적으로 설명
  4. 멘토링: 초보자에게는 친절한 가이드 제공

Semantic Versioning과 릴리스 프로세스¶

Semantic Versioning (SemVer)

버전 번호는 MAJOR.MINOR.PATCH 형식을 따릅니다.

v1.2.3
 │ │ │
 │ │ └─ PATCH: 버그 수정 (하위 호환)
 │ └─── MINOR: 새 기능 추가 (하위 호환)
 └───── MAJOR: 호환되지 않는 변경

예시

  • 1.0.0: 첫 정식 릴리스
  • 1.1.0: 새 기능 추가
  • 1.1.1: 버그 수정
  • 2.0.0: 호환되지 않는 변경 (Breaking Change)

Pre-release 버전

  • 1.0.0-alpha.1: 알파 버전
  • 1.0.0-beta.1: 베타 버전
  • 1.0.0-rc.1: Release Candidate

릴리스 프로세스

1단계: 변경 사항 정리

# CHANGELOG.md 업데이트
## [1.2.0] - 2025-02-17

### Added
- Korean translation support (#123)
- Dark mode theme (#125)

### Fixed
- Crash when input is empty (#124)

### Changed
- Updated dependencies

2단계: 버전 업데이트

# package.json (Node.js)
$ npm version minor  # 1.1.0 -> 1.2.0

# Python pyproject.toml
version = "1.2.0"

3단계: Tag 생성

$ git tag -a v1.2.0 -m "Release v1.2.0"
$ git push origin v1.2.0

4단계: GitHub Release 생성

GitHub UI 또는 CLI:

$ gh release create v1.2.0 \
  --title "v1.2.0" \
  --notes "$(cat CHANGELOG.md | sed -n '/## \[1.2.0\]/,/## \[1.1.0\]/p')"

자동 Release Notes

GitHub Actions로 자동화:

name: Release

on:
  push:
    tags:
      - 'v*'

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Create Release
        uses: softprops/action-gh-release@v1
        with:
          generate_release_notes: true

패키지 배포

# npm 배포
$ npm publish

# PyPI 배포
$ python -m build
$ twine upload dist/*

# Docker Hub 배포
$ docker build -t username/my-tool:1.2.0 .
$ docker push username/my-tool:1.2.0

베스트 프랙티스: 릴리스는 자주, 작은 단위로 진행하세요. 큰 변경을 모아서 릴리스하면 버그 추적이 어렵습니다.

절 요약¶

  • 오픈소스 프로젝트 시작 시 README, LICENSE, CONTRIBUTING, CODE_OF_CONDUCT가 필수입니다.
  • Issue Template과 PR Template로 구조화된 입력을 받을 수 있습니다.
  • 초기 기여자 확보를 위해 Good First Issue, Hacktoberfest 등을 활용합니다.
  • Semantic Versioning(MAJOR.MINOR.PATCH)으로 버전을 관리합니다.
  • GitHub Release와 패키지 배포로 릴리스를 완료합니다.

FAQ¶

Q1: Fork한 저장소가 원본보다 뒤처졌습니다. 어떻게 동기화하나요?

A1: git fetch upstream && git merge upstream/main으로 로컬에서 동기화한 후 git push origin main으로 Fork를 업데이트하세요. 또는 GitHub UI의 "Sync fork" 버튼을 사용할 수 있습니다.

Q2: PR을 만들었는데 CI 검사가 실패했습니다. 어떻게 하나요?

A2: PR 페이지에서 "Details"를 클릭하여 실패 원인을 확인하고, 로컬에서 수정 후 같은 브랜치에 커밋하여 push하면 CI가 자동으로 재실행됩니다.

Q3: 라이선스가 없는 GitHub 코드를 사용해도 되나요?

A3: 안 됩니다. 라이선스가 없으면 기본적으로 저작권법이 적용되어 저작권자의 명시적 허락 없이 사용, 수정, 배포가 금지됩니다.

Q4: MIT 프로젝트에 GPL 라이브러리를 사용할 수 있나요?

A4: GPL 라이브러리를 포함하면 전체 프로젝트가 GPL이 되어야 합니다. MIT를 유지하려면 GPL 라이브러리를 사용할 수 없습니다.

Q5: 오픈소스 프로젝트를 상업적으로 판매할 수 있나요?

A5: MIT, Apache 등 Permissive 라이선스는 가능하지만, GPL은 소스 코드를 공개해야 합니다. 라이선스를 반드시 확인하세요.

Q6: Good First Issue가 없는 프로젝트에는 어떻게 기여하나요?

A6: Issues 탭에서 최근에 생성된 간단한 버그를 찾거나, 문서 개선 (오타 수정, 예제 추가)을 제안하세요.

Q7: 리뷰어가 응답하지 않습니다. 어떻게 하나요?

A7: 1주일 정도 기다린 후 정중하게 ping(댓글로 @멘션) 해보세요. 프로젝트가 활발하지 않다면 다른 프로젝트를 고려하세요.

Q8: Semantic Versioning에서 0.x.x 버전은 무엇을 의미하나요?

A8: 초기 개발 단계로, 언제든 호환되지 않는 변경이 발생할 수 있습니다. 1.0.0이 첫 정식 릴리스입니다.

장 요약¶

이번 장에서는 오픈소스 생태계에 참여하는 방법을 종합적으로 배웠습니다. Fork 기반 워크플로우로 프로젝트에 기여하고, CONTRIBUTING.md와 Code of Conduct를 읽고, 첫 PR을 작성하며, 오픈소스 라이선스의 종류와 의무를 이해했습니다. 또한 자신의 오픈소스 프로젝트를 시작하고 운영하는 방법까지 익혔습니다.

오픈소스는 단순히 무료 코드가 아니라, 전 세계 개발자들이 협력하여 만드는 공동체입니다. 첫 기여는 작은 문서 수정일지라도, 그것이 오픈소스 여정의 시작입니다.
