실무

팀 협업 워크플로우¶

학습 목표 - Git Flow, GitHub Flow, Trunk-Based Development의 특징과 차이를 설명할 수 있다 - 팀의 상황에 맞는 브랜치 전략을 선택하고 도입할 수 있다 - 브랜치 보호 규칙과 CODEOWNERS를 설정하여 코드 품질을 보장할 수 있다 - 팀 커밋 규약을 정립하고 효과적인 코드 리뷰 문화를 구축할 수 있다 - 실무에서 발생하는 협업 문제를 워크플로우 개선으로 해결할 수 있다

Git Flow 워크플로우 (ch_11_s01)¶

도입¶

혼자서 개발할 때는 main 브랜치 하나면 충분합니다. 하지만 팀 규모가 커지고 여러 기능을 동시에 개발하며 정기적인 릴리스 일정을 관리해야 한다면 어떻게 해야 할까요? 이 절에서는 체계적인 릴리스 관리를 위한 Git Flow 워크플로우를 배웁니다.

핵심 개념 설명¶

Git Flow란 무엇인가¶

Git Flow는 Vincent Driessen이 2010년에 제안한 브랜치 전략으로, main, develop, feature, release, hotfix 다섯 종류의 브랜치를 체계적으로 운영하는 방식입니다.

릴리스 주기가 긴 프로젝트에 적합합니다.

Git Flow의 철학

Git Flow는 "명확한 역할 분리"를 핵심 철학으로 합니다. 각 브랜치 유형마다 명확한 책임이 있으며, 정해진 규칙에 따라서만 병합이 일어납니다.

팁: Git Flow는 복잡해 보이지만, 대규모 팀과 정기 릴리스 일정이 있는 프로젝트에서는 매우 효과적입니다. 모바일 앱, 패키지 소프트웨어, 엔터프라이즈 애플리케이션 등에 적합합니다.

Git Flow의 5가지 브랜치 역할¶

1. main 브랜치

프로덕션 배포 코드를 담는 브랜치입니다. main에 있는 모든 커밋은 프로덕션 배포 가능한 상태여야 합니다. 직접 커밋하지 않으며, release 또는 hotfix 브랜치에서만 병합됩니다.

$ git switch main
$ git log --oneline
# 출력:
# e3a4c21 (tag: v2.0.0) Release 2.0.0
# 7b2d891 (tag: v1.5.1) Hotfix: Critical security patch
# 9c1f342 (tag: v1.5.0) Release 1.5.0

2. develop 브랜치

다음 릴리스를 위한 개발 통합 브랜치입니다. 모든 feature 브랜치는 develop에서 분기하고 develop으로 병합됩니다. develop은 항상 최신 개발 상태를 반영합니다.

$ git switch develop
$ git log --oneline --graph -5
# 출력:
# *   2a3b4c5 Merge feature/user-profile
# |\
# | * 1d2e3f4 feat: Add user avatar upload
# | * 8e9f0a1 feat: Implement profile edit form
# |/
# * 5b6c7d8 Merge feature/search-optimization

3. feature 브랜치 (feature/기능명)

새로운 기능을 개발하는 브랜치입니다. develop에서 분기하여 개발 완료 후 develop으로 다시 병합됩니다. 개발자의 로컬에만 존재할 수도 있고, 협업을 위해 원격에 push할 수도 있습니다.

$ git switch develop
$ git switch -c feature/dark-mode
$ git push -u origin feature/dark-mode

# 작업 완료 후
$ git switch develop
$ git merge --no-ff feature/dark-mode
$ git branch -d feature/dark-mode

주의: --no-ff 옵션을 사용하면 Fast-Forward 병합이 가능한 상황에서도 명시적으로 병합 커밋을 생성합니다. 이렇게 하면 feature 브랜치의 존재를 히스토리에서 명확히 추적할 수 있습니다.

4. release 브랜치 (release/버전명)

릴리스 준비를 위한 브랜치입니다. develop에서 분기하여 버전 번호 변경, 문서 업데이트, 마이너 버그 수정 등을 수행합니다. 준비가 완료되면 main과 develop 양쪽에 모두 병합합니다.

$ git switch develop
$ git switch -c release/2.1.0

# 버전 번호 업데이트
$ echo "2.1.0" > VERSION
$ git add VERSION
$ git commit -m "chore: Bump version to 2.1.0"

# 릴리스 준비 완료 후 main에 병합
$ git switch main
$ git merge --no-ff release/2.1.0
$ git tag -a v2.1.0 -m "Release version 2.1.0"

# develop에도 병합 (릴리스 중 수정 사항 반영)
$ git switch develop
$ git merge --no-ff release/2.1.0

# release 브랜치 삭제
$ git branch -d release/2.1.0

5. hotfix 브랜치 (hotfix/이슈명)

프로덕션 긴급 수정을 위한 브랜치입니다. main에서 분기하여 버그를 수정하고, main과 develop(또는 현재 release) 양쪽에 병합합니다.

$ git switch main
$ git switch -c hotfix/security-vulnerability

# 긴급 수정
$ git add .
$ git commit -m "fix: Patch critical security vulnerability CVE-2024-1234"

# main에 병합 및 태그
$ git switch main
$ git merge --no-ff hotfix/security-vulnerability
$ git tag -a v2.1.1 -m "Hotfix 2.1.1: Security patch"

# develop에도 병합
$ git switch develop
$ git merge --no-ff hotfix/security-vulnerability

$ git branch -d hotfix/security-vulnerability

Git Flow 실습: 기능 개발부터 릴리스까지¶

실제 시나리오를 따라 Git Flow 전체 프로세스를 경험해봅시다.

시나리오: 쇼핑몰 프로젝트에 결제 기능과 배송 추적 기능을 추가하고 v1.2.0으로 릴리스합니다.

1단계: 초기 설정

$ mkdir shopping-mall-gitflow
$ cd shopping-mall-gitflow
$ git init
$ git commit --allow-empty -m "chore: Initial commit"

# develop 브랜치 생성
$ git switch -c develop
$ git push -u origin main develop

2단계: 결제 기능 개발 (feature 브랜치)

$ git switch develop
$ git switch -c feature/payment-integration

# 작업
$ echo "Payment gateway integration" > payment.txt
$ git add payment.txt
$ git commit -m "feat: Integrate Stripe payment gateway"

$ echo "Payment UI components" >> payment.txt
$ git add payment.txt
$ git commit -m "feat: Add payment form UI"

# develop에 병합
$ git switch develop
$ git merge --no-ff feature/payment-integration
$ git branch -d feature/payment-integration

3단계: 배송 추적 기능 개발 (병렬 작업)

$ git switch develop
$ git switch -c feature/shipment-tracking

$ echo "Tracking API integration" > tracking.txt
$ git add tracking.txt
$ git commit -m "feat: Add shipment tracking API"

$ git switch develop
$ git merge --no-ff feature/shipment-tracking
$ git branch -d feature/shipment-tracking

4단계: 릴리스 준비 (release 브랜치)

$ git switch develop
$ git switch -c release/1.2.0

# 버전 업데이트
$ echo "1.2.0" > VERSION
$ git add VERSION
$ git commit -m "chore: Bump version to 1.2.0"

# 문서 업데이트
$ echo "## Version 1.2.0\n- Payment integration\n- Shipment tracking" > CHANGELOG.md
$ git add CHANGELOG.md
$ git commit -m "docs: Update CHANGELOG for 1.2.0"

# main에 병합 및 태그
$ git switch main
$ git merge --no-ff release/1.2.0
$ git tag -a v1.2.0 -m "Release 1.2.0"

# develop에도 병합
$ git switch develop
$ git merge --no-ff release/1.2.0
$ git branch -d release/1.2.0

5단계: 긴급 버그 수정 (hotfix 브랜치)

$ git switch main
$ git switch -c hotfix/payment-crash

$ echo "Fix: Null pointer exception in payment" >> payment.txt
$ git add payment.txt
$ git commit -m "fix: Prevent crash when payment method is null"

# main에 병합 및 태그
$ git switch main
$ git merge --no-ff hotfix/payment-crash
$ git tag -a v1.2.1 -m "Hotfix 1.2.1"

# develop에도 병합
$ git switch develop
$ git merge --no-ff hotfix/payment-crash
$ git branch -d hotfix/payment-crash

베스트 프랙티스: Git Flow 도구(git flow 명령어)를 설치하면 이러한 브랜치 생성/병합을 자동화할 수 있습니다. 하지만 내부 동작을 이해하기 위해 먼저 수동으로 익히는 것을 권장합니다.

Git Flow의 장단점¶

장점

  1. 명확한 역할 분리: 각 브랜치 유형의 책임이 명확합니다.
  2. 릴리스 관리 용이: release 브랜치로 릴리스 준비와 개발을 분리할 수 있습니다.
  3. 긴급 수정 대응: hotfix 브랜치로 프로덕션 긴급 패치를 빠르게 적용할 수 있습니다.
  4. 히스토리 명확성: --no-ff 병합으로 feature 단위를 히스토리에서 추적할 수 있습니다.
  5. 대규모 팀 협업: 여러 팀이 독립적으로 feature를 개발할 수 있습니다.

단점

  1. 복잡도: 브랜치 5종류를 모두 관리하기 어려울 수 있습니다.
  2. 병합 오버헤드: main과 develop 양쪽 병합이 필요하여 번거롭습니다.
  3. 배포 주기 길어짐: release 브랜치 준비 단계로 배포가 지연될 수 있습니다.
  4. 충돌 가능성: develop이 오래 분기되면 병합 충돌이 잦아집니다.

Git Flow가 적합한 프로젝트 유형¶

  • 정기 릴리스 일정이 있는 프로젝트: 월간/분기별 릴리스 (모바일 앱, 패키지 소프트웨어)
  • 여러 버전을 동시 지원하는 프로젝트: v1.x, v2.x 동시 유지보수
  • 릴리스 전 QA 기간이 필요한 프로젝트: 금융, 의료 등 규제 산업
  • 대규모 개발 팀: 10명 이상의 개발자가 동시에 작업
  • 엔터프라이즈 환경: 명확한 릴리스 문서와 변경 관리가 필요한 경우

절 요약¶

  • Git Flow는 5종류 브랜치(main, develop, feature, release, hotfix)로 체계적 릴리스 관리를 합니다.
  • feature는 develop에서 분기하고, release는 develop에서 분기하여 main과 develop에 병합합니다.
  • hotfix는 main에서 분기하여 긴급 수정 후 main과 develop 양쪽에 병합합니다.
  • 명확한 역할 분리와 릴리스 관리가 장점이지만, 복잡도와 병합 오버헤드가 단점입니다.
  • 정기 릴리스와 대규모 팀 프로젝트에 적합합니다.

GitHub Flow 워크플로우 (ch_11_s02)¶

도입¶

Git Flow가 너무 복잡하다고 느껴진다면 GitHub Flow를 시도해보세요. GitHub 자체가 사용하는 이 워크플로우는 main 브랜치와 feature 브랜치만으로 빠르고 간결한 협업을 가능하게 합니다. 이 절에서는 웹 서비스와 SaaS 제품에 최적화된 GitHub Flow를 배웁니다.

핵심 개념 설명¶

GitHub Flow의 단순함: main + feature branch¶

GitHub Flow는 단 하나의 규칙을 강조합니다. "main 브랜치는 언제나 배포 가능한 상태여야 한다."

develop, release, hotfix 브랜치가 모두 사라지고, main과 feature 브랜치만 남습니다.

GitHub Flow의 핵심 원칙

  1. main은 항상 배포 가능한 상태를 유지합니다.
  2. 새 작업은 main에서 설명적 이름의 브랜치를 만듭니다.
  3. 로컬에서 작업하고 정기적으로 원격 브랜치에 push합니다.
  4. 피드백이나 도움이 필요하거나 머지 준비가 되면 PR을 만듭니다.
  5. PR이 리뷰되고 승인되면 main에 머지합니다.
  6. main에 머지되면 즉시 배포합니다.

팁: GitHub Flow의 "즉시 배포" 원칙은 CI/CD 파이프라인과 완벽하게 결합됩니다. main에 머지되는 순간 자동으로 테스트, 빌드, 배포가 실행됩니다.

GitHub Flow 6단계 프로세스¶

1단계: main에서 브랜치 생성

$ git switch main
$ git pull origin main
$ git switch -c add-user-authentication

브랜치 이름은 작업 내용을 명확히 설명해야 합니다. feature/, fix/, docs/ 같은 접두사를 붙이는 것도 좋습니다.

좋은 브랜치 이름 예시

add-user-authentication
fix-login-button-alignment
docs-update-api-reference
refactor-payment-module

나쁜 브랜치 이름 예시

temp
my-branch
test123
update

2단계: 커밋 추가

작업을 진행하며 의미 있는 단위로 커밋합니다.

$ git add src/auth/
$ git commit -m "feat: Add JWT token generation"

$ git add src/middleware/
$ git commit -m "feat: Add authentication middleware"

$ git add tests/auth/
$ git commit -m "test: Add auth middleware tests"

베스트 프랙티스: 작은 커밋을 자주 만드세요. "하나의 커밋은 하나의 목적만" 원칙을 따르면 나중에 문제가 생겼을 때 되돌리기 쉽습니다.

3단계: 원격에 정기적으로 push

$ git push -u origin add-user-authentication

원격에 push하면 세 가지 이점이 있습니다. 1. 백업: 로컬 컴퓨터 문제가 생겨도 작업 내용이 안전합니다. 2. 공유: 팀원이 내 작업을 확인하고 피드백을 줄 수 있습니다. 3. CI 검증: GitHub Actions가 자동으로 테스트를 실행합니다.

4단계: PR 생성

작업이 완료되었거나 초기 피드백이 필요하면 PR을 만듭니다.

$ gh pr create --title "Add user authentication with JWT" \
  --body "Implements #42. Adds JWT-based authentication middleware."

또는 GitHub 웹 UI에서 생성할 수 있습니다.

PR 설명 작성 요령

좋은 PR 설명은 다음을 포함합니다.

## Summary
JWT 기반 사용자 인증 시스템을 추가합니다.

## Changes
- JWT 토큰 생성/검증 로직 추가
- 인증 미들웨어 구현
- 로그인/로그아웃 API 엔드포인트 추가

## Related Issues
Closes #42

## Testing
- [ ] 단위 테스트 통과
- [ ] 통합 테스트 통과
- [ ] 로컬에서 수동 테스트 완료

## Screenshots
(스크린샷 첨부)

5단계: 리뷰 및 논의

팀원이 코드를 리뷰하고 피드백을 제공합니다.

# 리뷰어가 피드백 제공
# "auth.js:42 - 토큰 만료 시간을 환경 변수로 설정하면 어떨까요?"

# 개발자가 수정 후 push
$ git add src/auth/auth.js
$ git commit -m "refactor: Use env variable for token expiry"
$ git push

PR에 push하면 자동으로 업데이트되며, CI가 다시 실행됩니다.

주의: PR이 너무 크면 리뷰가 어렵습니다. 300줄 이하로 유지하고, 큰 작업은 여러 PR로 나누세요.

6단계: 테스트 통과 및 main에 머지

모든 리뷰가 승인되고 CI 테스트가 통과하면 main에 머지합니다.

$ gh pr merge --squash

머지 후 자동으로 배포 파이프라인이 실행됩니다.

# .github/workflows/deploy.yml
name: Deploy to Production
on:
  push:
    branches: [main]
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Deploy to production
        run: ./scripts/deploy.sh

GitHub Flow 실습¶

실제 시나리오로 GitHub Flow를 경험해봅시다.

시나리오: 블로그 프로젝트에 댓글 기능 추가

# 1. main에서 브랜치 생성
$ git switch main
$ git pull origin main
$ git switch -c add-comment-feature

# 2. 작업 및 커밋
$ mkdir -p src/components/Comment
$ echo "export const Comment = () => {}" > src/components/Comment/index.js
$ git add src/components/Comment
$ git commit -m "feat: Add Comment component skeleton"

$ echo "// API calls for comments" > src/api/comments.js
$ git add src/api/comments.js
$ git commit -m "feat: Add comment API functions"

$ git push -u origin add-comment-feature

# 3. PR 생성
$ gh pr create --title "Add comment feature" \
  --body "Implements user comments on blog posts.\n\nCloses #15"

# 4. CI 자동 실행 확인
$ gh pr checks

# 5. 리뷰 후 수정
$ echo "// Add error handling" >> src/api/comments.js
$ git commit -am "fix: Add error handling to comment API"
$ git push

# 6. 승인 후 머지
$ gh pr merge --squash --delete-branch

GitHub Flow의 장단점¶

장점

  1. 단순성: 브랜치 전략이 매우 간단하여 배우기 쉽습니다.
  2. 빠른 배포: PR 머지 즉시 배포하여 피드백 사이클이 짧습니다.
  3. 명확한 히스토리: main에 머지된 커밋은 모두 배포된 변경입니다.
  4. CI/CD 친화적: 자동화 파이프라인과 자연스럽게 통합됩니다.
  5. 작은 팀에 최적: 복잡한 브랜치 관리 없이 빠르게 협업합니다.

단점

  1. 릴리스 관리 부족: 정기 릴리스 일정이 필요하면 불편합니다.
  2. 프로덕션 보호: main이 곧 프로덕션이므로 실수 시 영향이 큽니다.
  3. 여러 버전 지원 어려움: 동시에 여러 버전을 유지보수하기 어렵습니다.
  4. 긴 QA 주기 부적합: 릴리스 전 긴 검증 기간이 필요하면 맞지 않습니다.

GitHub Flow가 적합한 프로젝트 유형¶

  • 웹 서비스/SaaS: 지속적 배포가 가능한 프로젝트
  • 내부 도구: 빠른 반복이 중요한 사내 시스템
  • 오픈소스 프로젝트: 커뮤니티 기여를 받는 라이브러리
  • 스타트업: 빠른 실험과 배포가 필요한 초기 단계 제품
  • 작은 팀(5명 이하): 간결한 프로세스로 효율성을 높이는 팀

베스트 프랙티스: GitHub 자체, Heroku, npm 등 많은 대형 서비스가 GitHub Flow를 성공적으로 사용하고 있습니다. 복잡도보다 속도가 중요하다면 GitHub Flow를 선택하세요.

절 요약¶

  • GitHub Flow는 main과 feature 브랜치만 사용하는 간결한 전략입니다.
  • main은 항상 배포 가능하며, PR 머지 즉시 배포합니다.
  • 6단계 프로세스: 브랜치 생성 → 커밋 → PR → 리뷰 → 테스트 → 머지 및 배포
  • 단순성과 빠른 배포가 장점이지만, 릴리스 관리가 약하고 프로덕션 보호가 필요합니다.
  • 웹 서비스, SaaS, 작은 팀에 적합합니다.

Trunk-Based Development (ch_11_s03)¶

도입¶

Google과 Meta 같은 초대형 조직은 수천 명의 개발자가 하루에 수백 번씩 main 브랜치에 통합합니다. 어떻게 가능할까요? 이 절에서는 빈번한 통합과 짧은 수명 브랜치를 강조하는 Trunk-Based Development를 배웁니다.

핵심 개념 설명¶

Trunk-Based Development의 원칙¶

Trunk-Based Development(트렁크 기반 개발)는 하나의 트렁크(main 브랜치)에 모든 개발자가 하루에 최소 한 번씩 통합하는 개발 방식입니다.

핵심 원칙

  1. 짧은 수명 브랜치: 브랜치는 최대 1-2일만 존재합니다.
  2. 빈번한 통합: 하루 최소 1회 main에 통합합니다.
  3. 작은 변경: 큰 기능을 작은 단위로 쪼개 점진적으로 통합합니다.
  4. 피처 플래그: 미완성 기능은 피처 플래그로 숨깁니다.
  5. 강력한 자동화: 모든 커밋이 자동으로 테스트됩니다.

짧은 수명의 브랜치와 빈번한 통합¶

전통적 방식 (긴 브랜치)

feature/new-dashboard (14일 동안 분기)
├── 60 commits
└── main에 머지 시 거대한 충돌 발생

Trunk-Based 방식 (짧은 브랜치)

feature/dashboard-api (1일)      → main 머지
feature/dashboard-ui (1일)       → main 머지
feature/dashboard-chart (1일)    → main 머지
feature/dashboard-filter (1일)   → main 머지

큰 기능을 작은 단위로 나누어 각각 독립적으로 main에 통합합니다. 통합 주기가 짧아 충돌이 적고, 문제 발생 시 영향 범위가 작습니다.

실습: 큰 기능을 작게 나누기

잘못된 예: 14일짜리 거대한 브랜치

$ git switch -c feature/complete-dashboard
# 14일 동안 개발...
# API, UI, 차트, 필터, 권한, 테스트 모두 한 브랜치에
$ git push  # 충돌 지옥 시작

올바른 예: 1-2일짜리 작은 브랜치

# 1일차: API 레이어만
$ git switch -c feature/dashboard-api-layer
$ git commit -m "feat: Add dashboard data API endpoints (hidden by feature flag)"
$ git push
$ gh pr create --title "Dashboard API layer" --body "Part 1/4: API only, no UI yet"
# 머지 후 삭제

# 2일차: UI 스켈레톤
$ git switch -c feature/dashboard-ui-skeleton
$ git commit -m "feat: Add dashboard page skeleton (hidden)"
$ git push
# 머지 후 삭제

# 3일차: 차트 컴포넌트
$ git switch -c feature/dashboard-chart-widget
$ git commit -m "feat: Add chart widget to dashboard"
$ git push
# 머지 후 삭제

# 4일차: 피처 플래그 제거 (기능 공개)
$ git switch -c enable/dashboard
$ git commit -m "feat: Enable dashboard for all users"
$ git push

베스트 프랙티스: 작은 PR은 리뷰가 빠르고 머지가 쉬우며 롤백도 간단합니다. "완성되지 않았어도 main에 머지할 수 있는가?"를 항상 자문하세요.

피처 플래그로 미완성 기능 관리¶

피처 플래그(Feature Flag)는 코드는 배포하되 기능은 숨기는 기법입니다.

// 환경 변수 또는 설정 파일
const featureFlags = {
  newDashboard: process.env.ENABLE_NEW_DASHBOARD === 'true',
  darkMode: process.env.ENABLE_DARK_MODE === 'true',
};

// 코드에서 사용
function Dashboard() {
  if (featureFlags.newDashboard) {
    return <NewDashboard />;  // 개발 중인 새 대시보드
  } else {
    return <OldDashboard />;  // 기존 대시보드
  }
}

피처 플래그의 장점

  1. 점진적 공개: 일부 사용자에게만 먼저 공개하여 테스트합니다 (카나리 배포).
  2. 안전한 롤백: 코드를 되돌리지 않고 플래그만 끄면 기능이 비활성화됩니다.
  3. A/B 테스트: 두 버전을 동시에 배포하여 효과를 측정합니다.
  4. 지속적 배포: 기능 완성 여부와 배포를 분리합니다.
// 점진적 롤아웃 예시
function isNewDashboardEnabled(user) {
  // 1% 사용자에게만 공개
  return user.id % 100 < 1;
}

// 특정 사용자 그룹에만 공개
function isNewDashboardEnabled(user) {
  return user.role === 'beta-tester' || user.email.endsWith('@company.com');
}

주의: 피처 플래그는 기술 부채가 될 수 있습니다. 기능이 완전히 안정화되면 플래그를 제거하고 코드를 정리하세요.

Google, Meta의 개발 사례¶

Google의 모노레포 + Trunk-Based

Google은 수십억 줄의 코드를 하나의 저장소(모노레포)에서 관리하며, 25,000명 이상의 개발자가 매일 main 브랜치에 통합합니다.

  • 모든 변경은 main 브랜치에 직접 커밋하거나 매우 짧은 브랜치(수 시간)를 사용합니다.
  • 자동화된 테스트가 수백만 개 실행되어 코드 품질을 보장합니다.
  • 피처 플래그로 미완성 기능을 숨깁니다.

Meta의 빠른 릴리스 사이클

Meta(Facebook)는 하루에 수백 번씩 프로덕션에 배포합니다.

  • 개발자는 작은 변경을 자주 main에 통합합니다.
  • 자동화된 카나리 배포로 1% 사용자에게 먼저 공개합니다.
  • 문제가 없으면 점진적으로 100% 사용자에게 확대합니다.

팁: 대규모 조직의 사례를 그대로 따라 할 필요는 없습니다. 하지만 "작은 변경, 빈번한 통합, 자동화된 테스트"의 핵심 원칙은 모든 규모의 팀에 유용합니다.

어떤 워크플로우를 선택할 것인가: 비교 가이드¶

기준 Git Flow GitHub Flow Trunk-Based
브랜치 복잡도 높음 (5종류) 낮음 (2종류) 매우 낮음 (1-2종류)
배포 빈도 주/월 단위 일 단위 하루 여러 번
브랜치 수명 수주~수개월 며칠~수주 수 시간~1일
릴리스 관리 매우 강함 약함 없음 (지속 배포)
팀 규모 대규모(10명+) 소규모(5명-) 초대규모(100명+)
CI/CD 필수도 선택 권장 필수
학습 곡선 가파름 완만함 중간 (문화 변화 필요)

선택 가이드

  • 정기 릴리스 일정이 있고 대규모 팀 → Git Flow
  • 빠른 배포와 간결함이 중요 → GitHub Flow
  • 초대규모 팀과 하루 수십 번 배포 → Trunk-Based Development

베스트 프랙티스: 워크플로우는 팀의 규모, 제품 특성, 배포 빈도에 맞춰 선택하고, 점진적으로 개선하세요. 완벽한 워크플로우는 없으며, 팀에 맞는 워크플로우가 최선입니다.

절 요약¶

  • Trunk-Based Development는 짧은 수명 브랜치(1-2일)와 빈번한 통합(하루 1회+)을 강조합니다.
  • 피처 플래그로 미완성 기능을 숨기고 지속적으로 배포합니다.
  • Google, Meta 등 초대규모 조직이 채택하여 하루 수백 번 배포합니다.
  • Git Flow는 릴리스 관리, GitHub Flow는 간결함, Trunk-Based는 통합 빈도에 최적화되었습니다.
  • 팀의 상황에 맞는 워크플로우를 선택하고 점진적으로 개선하세요.

브랜치 보호 규칙과 코드 소유권 (ch_11_s04)¶

도입¶

main 브랜치에 잘못된 코드가 직접 push되어 프로덕션이 중단된 경험이 있으신가요? 이 절에서는 브랜치 보호 규칙으로 코드 품질을 보장하고, CODEOWNERS로 전문가 리뷰를 자동화하는 방법을 배웁니다.

핵심 개념 설명¶

브랜치 보호 규칙 설정¶

브랜치 보호 규칙(Branch Protection Rules)은 특정 브랜치에 대한 push, force push, 삭제를 제한하고, 머지 조건을 강제하는 GitHub 설정입니다.

설정 위치

GitHub 저장소 → Settings → Branches → Branch protection rules → Add rule

주요 보호 규칙 옵션

1. Require a pull request before merging

main에 직접 push를 금지하고, 반드시 PR을 통해서만 머지하도록 강제합니다.

[✓] Require a pull request before merging
  [✓] Require approvals: 2  (최소 2명의 승인 필요)
  [✓] Dismiss stale pull request approvals when new commits are pushed
  [✓] Require review from Code Owners

실습: 필수 리뷰 승인 설정

1. Settings → Branches → Add rule
2. Branch name pattern: main
3. [✓] Require a pull request before merging
4. Required number of approvals before merging: 2
5. [✓] Dismiss stale approvals
6. Save changes

이제 main에 직접 push하려고 하면 오류가 발생합니다.

$ git push origin main
# 출력:
# remote: error: GH006: Protected branch update failed for refs/heads/main.
# remote: error: At least 2 approving reviews are required by reviewers with write access.

2. Require status checks to pass before merging

CI/CD 파이프라인의 모든 테스트가 통과해야만 머지를 허용합니다.

[✓] Require status checks to pass before merging
  [✓] Require branches to be up to date before merging
  Status checks that are required:
    - build
    - test
    - lint

실습: CI 통과 필수 설정

먼저 GitHub Actions 워크플로우를 만듭니다.

# .github/workflows/ci.yml
name: CI
on:
  pull_request:
    branches: [main]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run tests
        run: npm test
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run linter
        run: npm run lint

그 다음 브랜치 보호 규칙에 추가합니다.

Status checks that are required:
  - test
  - lint

이제 테스트나 린트가 실패하면 머지 버튼이 비활성화됩니다.

3. Require conversation resolution before merging

PR 코멘트가 모두 해결(resolved)되어야 머지할 수 있습니다.

[✓] Require conversation resolution before merging

4. Restrict who can push to matching branches

특정 사용자나 팀만 브랜치에 push할 수 있도록 제한합니다.

[✓] Restrict who can push to matching branches
  People, teams, or apps with push access:
    - @release-team
    - @admin-users

5. Do not allow bypassing the above settings

관리자를 포함한 모든 사용자가 보호 규칙을 따르도록 강제합니다.

[✓] Do not allow bypassing the above settings

주의: 이 옵션을 활성화하면 관리자도 긴급 상황에서 보호 규칙을 우회할 수 없습니다. 신중히 결정하세요.

CODEOWNERS 파일 작성¶

CODEOWNERS는 파일이나 디렉토리의 책임자를 지정하는 파일입니다. 지정된 파일이 변경되면 해당 소유자가 자동으로 리뷰어로 추가됩니다.

CODEOWNERS 파일 위치

.github/CODEOWNERS
또는
CODEOWNERS (저장소 루트)
또는
docs/CODEOWNERS

기본 문법

# 패턴         소유자 (GitHub 사용자명 또는 팀)
*.js           @frontend-team
/api/          @backend-team
/docs/         @docs-team @technical-writer
*.sql          @database-admin
package.json   @tech-lead

실습: CODEOWNERS 설정

# .github/CODEOWNERS

# 기본 소유자 (모든 파일)
*                      @default-reviewers

# 프론트엔드 코드
/src/components/       @frontend-team
/src/styles/           @ui-designer @frontend-team
*.tsx                  @frontend-team

# 백엔드 코드
/api/                  @backend-team
/database/migrations/  @backend-team @dba
*.sql                  @dba

# 인프라 설정
/infrastructure/       @devops-team
/.github/workflows/    @devops-team @tech-lead
Dockerfile             @devops-team

# 문서
/docs/                 @docs-team
README.md              @docs-team @product-manager

# 중요 설정 파일 (여러 승인 필요)
package.json           @tech-lead @senior-dev-1 @senior-dev-2
.env.example           @tech-lead @security-team

CODEOWNERS의 동작

  1. PR에서 /api/users.js 파일이 변경되면 @backend-team이 자동으로 리뷰어로 추가됩니다.
  2. 브랜치 보호 규칙에서 "Require review from Code Owners"를 활성화하면, 소유자 승인 없이는 머지할 수 없습니다.
# 예시: API 파일 수정
$ git switch -c update-user-api
$ echo "// New logic" >> api/users.js
$ git add api/users.js
$ git commit -m "feat: Update user API endpoint"
$ git push -u origin update-user-api
$ gh pr create

# 출력:
# Requested reviewers: @backend-team (automatically added by CODEOWNERS)

베스트 프랙티스: CODEOWNERS는 전문가 리뷰를 자동화합니다. 보안에 민감한 파일(*.env, secrets/)은 시니어 개발자나 보안팀을 소유자로 지정하세요.

Rulesets: 차세대 브랜치 보호¶

Rulesets는 브랜치 보호 규칙의 개선된 버전으로, 더 유연하고 강력한 보호 정책을 제공합니다.

Rulesets의 장점

  1. 여러 브랜치에 한 번에 적용: 패턴 매칭으로 release/*, hotfix/* 등에 동일 규칙 적용
  2. 조건부 규칙: 특정 조건에서만 규칙 활성화 (예: 프로덕션 환경)
  3. 우선순위: 여러 ruleset이 충돌할 때 우선순위 지정
  4. 상세한 감사 로그: 누가 규칙을 우회했는지 추적

설정 위치

GitHub 저장소 → Settings → Rules → Rulesets → New ruleset

실습: Ruleset 생성

1. Ruleset name: "Protect production branches"
2. Target branches:
   - Include by pattern: main, release/*, hotfix/*
3. Rules:
   - [✓] Require a pull request
   - [✓] Require status checks: [test, lint, build]
   - [✓] Block force pushes
   - [✓] Require signed commits
4. Bypass list:
   - @release-managers (긴급 상황용)
5. Create

팁: Rulesets는 Organization 수준에서도 설정하여 모든 저장소에 일관된 정책을 적용할 수 있습니다.

절 요약¶

  • 브랜치 보호 규칙은 필수 리뷰 승인, CI 통과, 대화 해결 등을 강제합니다.
  • CODEOWNERS로 파일별 전문가 리뷰를 자동화하고 코드 소유권을 명확히 합니다.
  • Rulesets는 여러 브랜치에 유연한 보호 정책을 적용하는 차세대 기능입니다.
  • 보호 규칙은 코드 품질을 보장하지만, 과도하면 개발 속도를 늦출 수 있으므로 균형이 중요합니다.

효과적인 팀 협업 패턴 (ch_11_s05)¶

도입¶

브랜치 전략과 보호 규칙을 설정했다면, 이제 팀 문화를 구축할 차례입니다. 이 절에서는 커밋 메시지 규약, 코드 리뷰 문화, PR 크기 가이드라인, 협업 도구 통합 등 효과적인 팀 협업 패턴을 배웁니다.

핵심 개념 설명¶

팀 커밋 메시지 규약 정립하기¶

일관된 커밋 메시지의 중요성

  1. 변경 내역 추적: 버그가 언제 도입되었는지 빠르게 파악합니다.
  2. 자동 Changelog: 커밋 메시지에서 릴리스 노트를 자동 생성합니다.
  3. 코드 리뷰 효율: 커밋 목적을 즉시 이해할 수 있습니다.
  4. 협업 명확성: 누가 왜 변경했는지 명확합니다.

Conventional Commits 규약

<type>(<scope>): <subject>

<body>

<footer>

Type (필수)

feat:     새로운 기능 추가
fix:      버그 수정
docs:     문서 수정
style:    코드 포맷팅 (기능 변경 없음)
refactor: 리팩터링 (기능 변경 없음)
test:     테스트 추가/수정
chore:    빌드, 설정 파일 수정
perf:     성능 개선
ci:       CI/CD 설정 수정

Scope (선택)

변경 범위를 명시합니다 (예: auth, api, ui).

Subject (필수)

50자 이내로 변경 내용을 요약합니다. 명령형 현재 시제로 작성합니다 (예: "Add" not "Added").

예시

# 좋은 커밋 메시지
$ git commit -m "feat(auth): Add JWT refresh token mechanism"
$ git commit -m "fix(api): Prevent null pointer exception in user endpoint"
$ git commit -m "docs(readme): Update installation instructions for Windows"

# 나쁜 커밋 메시지
$ git commit -m "update"
$ git commit -m "fixed bug"
$ git commit -m "asdfasdf"
$ git commit -m "WIP"

Breaking Changes 표시

하위 호환성을 깨는 변경은 BREAKING CHANGE:로 명시합니다.

$ git commit -m "feat(api): Change user ID format to UUID

BREAKING CHANGE: User IDs are now UUIDs instead of incremental integers.
All API clients must update their ID handling logic."

실습: Commitlint로 규약 강제하기

# 1. Commitlint 설치
$ npm install --save-dev @commitlint/cli @commitlint/config-conventional

# 2. commitlint.config.js 생성
$ echo "module.exports = {extends: ['@commitlint/config-conventional']}" > commitlint.config.js

# 3. Husky로 commit-msg hook 설정
$ npm install --save-dev husky
$ npx husky init
$ echo "npx --no -- commitlint --edit \$1" > .husky/commit-msg

# 4. 테스트
$ git commit -m "bad commit message"
# 출력:
# ⧗   input: bad commit message
# ✖   subject may not be empty [subject-empty]
# ✖   type may not be empty [type-empty]

베스트 프랙티스: 커밋 메시지 규약을 문서화하고 CONTRIBUTING.md에 포함하세요. 신규 팀원 온보딩 시 규약 교육을 진행하세요.

효과적인 코드 리뷰 문화 만들기¶

코드 리뷰의 목적

  1. 버그 발견: 배포 전 문제를 조기에 발견합니다.
  2. 지식 공유: 코드베이스에 대한 이해를 팀 전체로 확산합니다.
  3. 코드 품질: 일관된 스타일과 베스트 프랙티스를 유지합니다.
  4. 멘토링: 주니어 개발자가 시니어로부터 배웁니다.

효과적인 리뷰어가 되는 법

1. 빠르게 리뷰하기

PR이 올라온 후 24시간 이내에 리뷰하세요. 오래 방치된 PR은 충돌이 발생하고 컨텍스트를 잃습니다.

2. 건설적인 피드백 제공

# 나쁜 피드백
"이 코드는 이상해요."

# 좋은 피드백
"이 함수는 200줄로 너무 깁니다. `extractUserValidation()`과 `saveUserToDatabase()`로 분리하면 어떨까요? 단일 책임 원칙(SRP)을 따르면 테스트와 유지보수가 쉬워집니다."

3. 질문 vs 제안 vs 필수 변경 구분

# 질문 (이해를 위한)
**Q:** 왜 여기서 setTimeout을 사용했나요?

# 제안 (선택적)
**Suggestion:** `Promise.all()` 대신 `Promise.allSettled()`를 사용하면 일부 실패 시에도 결과를 받을 수 있습니다.

# 필수 변경 (반드시 수정)
**Required:** 이 엔드포인트에 인증 미들웨어가 없습니다. 인증을 추가해주세요.

4. 긍정적인 피드백도 제공

**Nice!** 이 에러 핸들링 로직이 정말 깔끔하네요. 다른 곳에서도 참고하겠습니다.

효과적인 PR 작성자가 되는 법

1. 작은 PR 만들기

이상적인 PR 크기: 200-300줄 이하
최대 크기: 500줄
500줄 초과: 여러 PR로 나누기

2. 설명이 풍부한 PR 설명

## Summary
사용자 프로필 편집 기능을 추가합니다.

## Changes
- 프로필 편집 폼 UI 추가
- 프로필 업데이트 API 엔드포인트 구현
- 이미지 업로드 기능 (AWS S3 연동)
- 단위 테스트 및 E2E 테스트 추가

## Screenshots
[Before/After 스크린샷]

## Testing
- [ ] 로컬 환경에서 수동 테스트 완료
- [ ] 단위 테스트 통과 (coverage 85%)
- [ ] E2E 테스트 통과

## Related Issues
Closes #123
Related to #124

## Migration Notes
데이터베이스 마이그레이션 필요:

$ npm run migrate:up

3. 리뷰 코멘트에 빠르게 응답

리뷰어가 피드백을 주면 24시간 이내에 응답하거나 수정하세요.

코드 리뷰 체크리스트

기능 (Functionality)

  • [ ] 요구 사항을 충족하는가?
  • [ ] 엣지 케이스를 처리하는가?
  • [ ] 에러 핸들링이 적절한가?

코드 품질 (Code Quality)

  • [ ] 함수가 단일 책임을 가지는가?
  • [ ] 변수명이 명확한가?
  • [ ] 중복 코드가 없는가?
  • [ ] 주석이 필요한 복잡한 로직에만 주석이 있는가?

테스트 (Testing)

  • [ ] 테스트가 충분한가?
  • [ ] 테스트가 실패 시나리오도 다루는가?
  • [ ] CI가 통과하는가?

보안 (Security)

  • [ ] SQL Injection, XSS 등 취약점이 없는가?
  • [ ] 민감 정보(API 키, 비밀번호)가 하드코딩되지 않았는가?
  • [ ] 인증/권한 검사가 있는가?

성능 (Performance)

  • [ ] 불필요한 API 호출이 없는가?
  • [ ] N+1 쿼리 문제가 없는가?
  • [ ] 대용량 데이터 처리를 고려했는가?

베스트 프랙티스: 리뷰는 협업이지 비판이 아닙니다. "당신은 틀렸어요" 대신 "우리 함께 더 나은 방법을 찾아봐요" 분위기를 만드세요.

PR 크기 가이드라인과 Stacked PRs¶

PR 크기가 중요한 이유

작은 PR:
- 리뷰 시간: 30분
- 머지 주기: 1일
- 충돌 확률: 낮음
- 버그 발견: 쉬움

큰 PR:
- 리뷰 시간: 4시간+
- 머지 주기: 1주일+
- 충돌 확률: 높음
- 버그 발견: 어려움

Stacked PRs (의존 PR)

큰 기능을 작은 PR로 나누되, 의존 관계가 있을 때 사용하는 기법입니다.

PR #101: [Part 1/3] Add user database schema
    ↓ depends on
PR #102: [Part 2/3] Add user API endpoints
    ↓ depends on
PR #103: [Part 3/3] Add user profile UI

실습: Stacked PRs 만들기

# 1. 첫 번째 PR (DB 스키마)
$ git switch -c part1-user-schema
$ git commit -m "feat: Add user database schema"
$ git push -u origin part1-user-schema
$ gh pr create --title "[Part 1/3] Add user database schema"

# 2. 두 번째 PR (API, part1에서 분기)
$ git switch -c part2-user-api
$ git commit -m "feat: Add user API endpoints"
$ git push -u origin part2-user-api
$ gh pr create --title "[Part 2/3] Add user API endpoints" \
  --body "Depends on #101"

# 3. 세 번째 PR (UI, part2에서 분기)
$ git switch -c part3-user-ui
$ git commit -m "feat: Add user profile UI"
$ git push -u origin part3-user-ui
$ gh pr create --title "[Part 3/3] Add user profile UI" \
  --body "Depends on #102"

Part 1이 머지되면, Part 2의 베이스를 main으로 업데이트합니다.

$ git switch part2-user-api
$ git rebase main
$ git push --force-with-lease

협업 도구 통합: Slack, Jira 연동¶

Slack 통합

GitHub과 Slack을 연동하여 PR, Issue, 배포 알림을 자동화합니다.

설정 방법

1. Slack에서 GitHub 앱 설치
2. /github subscribe owner/repo reviews,commits,releases

알림 예시

#dev-team 채널
🔔 New PR: Add user authentication (#42) by @johndoe
👀 Review requested from @janedoe

✅ PR merged: Add user authentication (#42)
🚀 Deployed to production

Jira 통합

커밋과 PR을 Jira 티켓과 자동 연결합니다.

설정 방법

1. GitHub에서 Jira 앱 설치
2. 커밋 메시지에 Jira 티켓 ID 포함

예시

$ git commit -m "feat(auth): Add SSO login [PROJ-123]"

Jira 티켓 PROJ-123에 자동으로 커밋 링크가 추가됩니다.

코드 리뷰 메트릭과 개선¶

측정할 메트릭

  1. 리뷰 소요 시간: PR 생성부터 첫 리뷰까지 시간
  2. 머지 주기: PR 생성부터 머지까지 시간
  3. PR 크기: 변경된 줄 수
  4. 리뷰 반복 횟수: 수정 요청 후 재리뷰 횟수

개선 방법

문제: 리뷰 시간이 3일 이상 걸림
해결: 리뷰 SLA 정책 수립 (24시간 이내 첫 리뷰)

문제: PR이 평균 800줄
해결: PR 크기 가이드라인 공유 및 린트 도구로 경고

문제: 리뷰 반복이 5회 이상
해결: PR 체크리스트 도입 및 Draft PR로 초기 피드백

팁: GitHub Insights나 외부 도구(LinearB, Waydev 등)로 메트릭을 시각화하고 팀 회고에서 논의하세요.

절 요약¶

  • Conventional Commits 규약으로 일관된 커밋 메시지를 작성하고 Commitlint로 자동 검증합니다.
  • 효과적인 코드 리뷰는 빠른 응답, 건설적 피드백, 긍정적 분위기가 핵심입니다.
  • PR은 200-300줄 이하로 유지하고, 큰 기능은 Stacked PRs로 나눕니다.
  • Slack, Jira 통합으로 협업 도구를 연결하고 메트릭으로 프로세스를 개선합니다.
  • 코드 리뷰는 협업이지 비판이 아니며, 팀 문화가 기술보다 중요합니다.

FAQ¶

Q1: 우리 팀은 5명인데 Git Flow가 너무 복잡합니다. 어떻게 해야 하나요?

A1: 5명 규모라면 GitHub Flow가 더 적합합니다. Git Flow는 대규모 팀과 정기 릴리스가 필요한 프로젝트를 위한 것입니다. GitHub Flow로 시작하여 팀이 커지면 점진적으로 복잡한 전략을 도입하세요.

Q2: 브랜치 보호 규칙을 설정했더니 개발 속도가 느려졌습니다.

A2: 보호 규칙과 속도의 균형이 중요합니다. 모든 브랜치에 동일한 규칙을 적용하지 말고, main은 엄격하게, develop은 유연하게 설정하세요. 또한 자동화(CI/CD, 자동 리뷰어 할당)로 수동 단계를 줄이세요.

Q3: CODEOWNERS 파일을 누가 관리해야 하나요?

A3: 기술 리드나 팀 리더가 초기 설정 후, 각 팀의 리드가 자신의 영역을 업데이트하도록 분산 관리하세요. CODEOWNERS 파일 자체에도 소유자를 지정하여 변경 시 리뷰를 받도록 하세요.

Q4: 커밋 메시지 규약을 강제하면 개발자들이 불만을 가질까요?

A4: 처음에는 불편할 수 있지만, 규약의 이점(자동 Changelog, 명확한 히스토리)을 교육하고 Commitlint로 자동 검증하면 빠르게 습관화됩니다. 팀 회고에서 규약의 효과를 공유하세요.

Q5: hotfix 브랜치와 feature 브랜치의 차이는 무엇인가요?

A5: hotfix는 프로덕션의 긴급 버그를 수정하기 위해 main에서 분기하며, 수정 후 main과 develop 양쪽에 머지합니다. feature는 새 기능 개발을 위해 develop에서 분기하고 develop으로 돌아갑니다. hotfix는 긴급성과 우선순위가 높습니다.

Q6: PR이 500줄이 넘는데 어떻게 나눠야 할지 모르겠습니다.

A6: 기능을 계층별로 나누세요. 1) 데이터베이스 스키마 → 2) API 레이어 → 3) 비즈니스 로직 → 4) UI 컴포넌트. 각 계층을 독립된 PR로 만들고, 미완성 부분은 피처 플래그로 숨기세요.

장 요약¶

이번 장에서는 팀 규모와 프로젝트 특성에 맞는 협업 워크플로우를 배웠습니다.

Git Flow는 5종류 브랜치로 체계적 릴리스 관리를 하며, 정기 릴리스와 대규모 팀에 적합합니다. GitHub Flow는 main과 feature 브랜치만으로 빠른 배포를 가능하게 하며, 웹 서비스와 작은 팀에 최적화되었습니다. Trunk-Based Development는 짧은 브랜치와 빈번한 통합으로 초대규모 조직에서 사용됩니다.

브랜치 보호 규칙은 필수 리뷰, CI 통과, 대화 해결을 강제하여 코드 품질을 보장합니다. CODEOWNERS 파일은 파일별 전문가 리뷰를 자동화하고, Rulesets는 여러 브랜치에 유연한 정책을 적용합니다.

효과적인 팀 협업은 기술보다 문화가 중요합니다. Conventional Commits로 일관된 커밋 메시지를 작성하고, 건설적인 코드 리뷰 문화를 만들며, 작은 PR로 리뷰 부담을 줄이세요. Slack, Jira 통합으로 협업 도구를 연결하고, 메트릭으로 프로세스를 지속적으로 개선하세요.

완벽한 워크플로우는 없으며, 팀에 맞는 워크플로우를 선택하고 점진적으로 개선하는 것이 최선입니다.
