04장: 원격 저장소와 협업의 시작

기초

원격 저장소와 협업의 시작¶

학습 목표 - 원격 저장소의 개념과 로컬 저장소와의 관계를 설명할 수 있다 - git clone과 git remote add로 원격 저장소를 연결할 수 있다 - SSH 키를 생성하고 GitHub에 등록하여 인증을 설정할 수 있다 - git push와 git pull로 변경 사항을 원격과 동기화할 수 있다 - git fetch와 git pull의 차이를 이해하고 상황에 맞게 사용할 수 있다

원격 저장소의 개념¶

도입¶

지금까지는 로컬 컴퓨터에서만 Git을 사용했습니다. 이제 코드를 인터넷상의 서버에 저장하고 팀원과 공유할 차례입니다. 원격 저장소(Remote Repository)는 협업의 시작점입니다.

핵심 개념 설명¶

로컬 저장소와 원격 저장소의 관계¶

원격 저장소(Remote Repository)는 네트워크를 통해 접근할 수 있는 Git 저장소로, 팀 협업과 백업을 위해 GitHub, GitLab 등의 호스팅 서비스에 위치합니다.

클라우드 드라이브(Google Drive, Dropbox)와 같습니다. 로컬 파일을 클라우드에 동기화하면 다른 기기에서 접근할 수 있고, 팀원과 공유할 수 있습니다.

origin: 기본 원격 저장소 별칭¶

origin은 원격 저장소의 기본 별칭입니다. Git은 URL 대신 짧은 별칭을 사용하여 원격 저장소를 참조합니다.

# 원격 저장소 목록 확인
$ git remote -v
origin  https://github.com/username/repo.git (fetch)
origin  https://github.com/username/repo.git (push)

여러 원격 저장소 관리하기¶

하나의 로컬 저장소는 여러 원격 저장소를 가질 수 있습니다.

# upstream 원격 저장소 추가 (오픈소스 기여 시)
$ git remote add upstream https://github.com/original/repo.git

# 원격 저장소 목록
$ git remote -v
origin    https://github.com/my-fork/repo.git (fetch)
origin    https://github.com/my-fork/repo.git (push)
upstream  https://github.com/original/repo.git (fetch)
upstream  https://github.com/original/repo.git (push)

git remote 명령어¶

# 원격 저장소 추가
$ git remote add <별칭> <URL>

# 원격 저장소 제거
$ git remote remove <별칭>

# 원격 저장소 이름 변경
$ git remote rename <기존이름> <새이름>

# 원격 저장소 URL 변경
$ git remote set-url origin <새URL>

팁: 원격 저장소 URL은 HTTPS와 SSH 두 가지 형식이 있습니다. SSH가 더 안전하고 편리합니다.

절 요약¶

  • 원격 저장소는 네트워크를 통해 접근하는 Git 저장소입니다
  • origin은 원격 저장소의 기본 별칭입니다
  • 하나의 로컬 저장소는 여러 원격 저장소를 가질 수 있습니다
  • git remote 명령으로 원격 저장소를 관리합니다

원격 저장소와 연결하기: clone과 remote¶

도입¶

원격 저장소와 연결하는 방법은 두 가지입니다. 기존 원격 저장소를 복제(clone)하거나, 로컬 저장소에 원격을 추가(remote add)하는 것입니다.

핵심 개념 설명¶

git clone: 원격 저장소 복제¶

git clone은 원격 저장소를 로컬로 완전히 복제하는 명령어입니다. 커밋 히스토리, 브랜치, 태그 모두 복사됩니다.

# HTTPS로 클론
$ git clone https://github.com/username/repo.git

# SSH로 클론
$ git clone git@github.com:username/repo.git

# 다른 디렉토리 이름으로 클론
$ git clone https://github.com/username/repo.git my-project

clone하면 자동으로 origin이 설정되고, main 브랜치가 체크아웃됩니다.

git remote add: 기존 로컬에 원격 연결¶

이미 로컬 저장소가 있고 나중에 원격 저장소를 만든 경우:

# GitHub에서 빈 저장소 생성 후
$ git remote add origin https://github.com/username/new-repo.git

# 처음 push (업스트림 설정)
$ git push -u origin main

HTTPS vs SSH 인증 방식 비교¶

구분 HTTPS SSH
설정 간단 키 생성 필요
인증 매번 토큰/비밀번호 자동 (키 기반)
보안 토큰 관리 필요 개인키 보호 필수
방화벽 대부분 허용 차단될 수 있음
권장 초보자 숙련자, 빈번한 push

SSH 키 생성 및 GitHub에 등록하기¶

Windows/macOS/Linux 공통:

# 1. SSH 키 생성
$ ssh-keygen -t ed25519 -C "your_email@example.com"

# 2. 엔터 3번 (기본 경로, 비밀번호 없음으로 설정)
# 또는 비밀번호 설정 (더 안전)

# 3. 공개키 내용 복사
$ cat ~/.ssh/id_ed25519.pub

GitHub에 등록: 1. GitHub → Settings → SSH and GPG keys 2. New SSH key 클릭 3. 복사한 공개키 붙여넣기 4. Add SSH key 클릭

연결 테스트:

$ ssh -T git@github.com
Hi username! You've successfully authenticated...

주의: 개인키(id_ed25519)는 절대 공유하지 마세요. 공개키(.pub)만 GitHub에 등록합니다.

베스트 프랙티스: SSH 키에 비밀번호를 설정하고, ssh-agent를 사용하여 세션 동안 비밀번호를 기억하게 하세요.

절 요약¶

  • git clone은 원격 저장소를 로컬로 복제합니다
  • git remote add는 기존 로컬에 원격을 추가합니다
  • HTTPS는 간단하지만 매번 인증이 필요하고, SSH는 설정이 복잡하지만 편리합니다
  • SSH 키는 ssh-keygen으로 생성하고 공개키를 GitHub에 등록합니다

변경 사항 공유하기: push와 pull¶

도입¶

로컬에서 작업한 커밋을 팀원과 공유하려면 원격으로 전송(push)해야 합니다. 반대로 팀원의 변경 사항을 받아오려면 pull을 사용합니다.

핵심 개념 설명¶

git push: 로컬 커밋을 원격에 전송¶

# 현재 브랜치를 origin의 같은 이름 브랜치로 push
$ git push origin main

# 모든 브랜치 push
$ git push --all origin

git push -u: 업스트림 브랜치 설정¶

# 처음 push할 때 -u 옵션으로 업스트림 설정
$ git push -u origin main

# 이후로는 git push만으로 가능
$ git push

-u(또는 --set-upstream) 옵션은 로컬 브랜치와 원격 브랜치를 추적 관계로 설정합니다.

git pull: 원격 변경 사항 가져오기¶

# origin/main의 변경을 가져와 현재 브랜치에 병합
$ git pull origin main

# 업스트림이 설정되어 있다면
$ git pull

git pull은 git fetch + git merge를 한 번에 수행합니다.

추적 브랜치(Tracking Branch)의 역할¶

추적 브랜치(Tracking Branch)는 로컬 브랜치가 특정 원격 브랜치를 추적하도록 설정한 것입니다.

# 현재 브랜치의 추적 정보 확인
$ git branch -vv
* main                7f3a9e8 [origin/main] feat: 로그인 추가
  feature/dashboard   5d2c1a3 feat: 대시보드 작성

[origin/main]은 로컬 main이 origin/main을 추적 중임을 나타냅니다.

push 거부 상황과 해결 방법¶

팀원이 먼저 push한 경우, 내 push가 거부됩니다.

$ git push
To https://github.com/username/repo.git
 ! [rejected]        main -> main (fetch first)
error: failed to push some refs to 'https://github.com/username/repo.git'

해결 방법:

# 1. 원격 변경 사항 먼저 가져오기
$ git pull

# 2. 충돌이 있다면 해결

# 3. 다시 push
$ git push

주의: git push --force는 팀원의 커밋을 덮어쓸 수 있어 매우 위험합니다. 절대 공유 브랜치에서 사용하지 마세요.

절 요약¶

  • git push는 로컬 커밋을 원격에 전송합니다
  • git push -u는 업스트림 브랜치를 설정하여 이후 push를 간편하게 합니다
  • git pull은 원격 변경을 가져와 병합합니다
  • push 거부 시 먼저 pull로 원격 변경을 받아온 후 다시 push하세요

fetch와 pull의 차이¶

도입¶

많은 초보자가 fetch와 pull을 헷갈려합니다. 두 명령의 차이를 이해하면 더 안전하게 협업할 수 있습니다.

핵심 개념 설명¶

git fetch: 원격 변경 사항만 다운로드¶

git fetch는 원격 저장소의 변경 사항을 로컬의 원격 추적 브랜치에만 다운로드하고, 현재 작업 브랜치에는 병합하지 않습니다.

온라인 쇼핑몰에서 상품을 '장바구니에 담기'하는 것과 같습니다. 장바구니에 담았다고 구매(merge)한 것은 아니며, 확인 후 구매 여부를 결정할 수 있습니다.

# 원격 변경 사항 다운로드
$ git fetch origin

# 원격 추적 브랜치 확인
$ git branch -r
  origin/main
  origin/feature/dashboard

원격 추적 브랜치(origin/main)의 개념¶

원격 추적 브랜치(Remote Tracking Branch)는 원격 저장소의 브랜치 상태를 로컬에 캐시한 읽기 전용 브랜치로, origin/main과 같은 형태로 표현됩니다.

해외 뉴스의 국내 속보와 같습니다. 실시간 해외 상황(원격 브랜치)을 직접 보지 않더라도, 속보(원격 추적 브랜치)를 통해 마지막으로 확인한 시점의 상태를 알 수 있습니다.

fetch 후 merge vs pull의 차이¶

# 방법 1: fetch 후 수동 merge (안전)
$ git fetch origin
$ git log --oneline origin/main  # 원격 변경 사항 확인
$ git merge origin/main

# 방법 2: pull (fetch + merge를 한 번에)
$ git pull

언제 fetch를 사용하나요? - 원격 변경을 먼저 확인하고 싶을 때 - 병합하기 전에 코드 리뷰를 하고 싶을 때 - 여러 브랜치의 변경을 한 번에 가져오고 싶을 때

안전한 업데이트를 위한 fetch-merge 패턴¶

# 1. 원격 변경 가져오기
$ git fetch origin

# 2. 로컬과 원격의 차이 확인
$ git log --oneline --graph --all

# 3. 변경 내용 확인 (선택)
$ git diff main origin/main

# 4. 안전하다고 판단되면 병합
$ git merge origin/main

베스트 프랙티스: 중요한 프로젝트에서는 fetch-merge 패턴을 사용하여 예상치 못한 병합을 방지하세요.

절 요약¶

  • git fetch는 원격 변경을 다운로드만 하고 병합하지 않습니다
  • origin/main은 원격 추적 브랜치로 읽기 전용입니다
  • git pull = git fetch + git merge
  • 안전한 업데이트를 위해서는 fetch 후 수동 merge를 권장합니다

원격 브랜치 관리¶

도입¶

원격 저장소에도 여러 브랜치가 있을 수 있습니다. 원격 브랜치를 확인하고 추적하며 삭제하는 방법을 배웁니다.

핵심 개념 설명¶

원격 브랜치 목록 확인: git branch -r¶

# 원격 브랜치만 표시
$ git branch -r
  origin/main
  origin/feature/login
  origin/develop

# 로컬 + 원격 모든 브랜치 표시
$ git branch -a
* main
  feature/dashboard
  remotes/origin/main
  remotes/origin/feature/login

원격 브랜치 추적하기¶

# 원격 브랜치를 로컬로 체크아웃 (자동 추적 설정)
$ git switch feature/login

# 또는 명시적으로 추적 설정
$ git switch -c feature/login origin/feature/login

원격 브랜치 삭제¶

# 로컬 브랜치 삭제
$ git branch -d feature/login

# 원격 브랜치 삭제
$ git push origin --delete feature/login

오래된 원격 추적 브랜치 정리: git remote prune¶

원격에서 삭제된 브랜치의 로컬 추적 정보는 자동으로 제거되지 않습니다.

# 오래된 원격 추적 브랜치 확인 및 제거
$ git remote prune origin

# fetch와 동시에 정리
$ git fetch --prune

팁: git config --global fetch.prune true로 설정하면 fetch 시 자동으로 정리됩니다.

절 요약¶

  • git branch -r로 원격 브랜치 목록을 확인합니다
  • git switch <원격브랜치명>으로 원격 브랜치를 추적합니다
  • git push origin --delete <브랜치>로 원격 브랜치를 삭제합니다
  • git remote prune으로 오래된 원격 추적 브랜치를 정리합니다

FAQ¶

Q1: git pull과 git fetch + git merge의 실질적 차이는 무엇인가요?

A1: 결과는 같지만 안전성이 다릅니다. git pull은 자동으로 병합하여 예상치 못한 충돌이 발생할 수 있습니다. fetch + merge는 원격 변경을 먼저 확인한 후 병합 여부를 결정할 수 있어 더 안전합니다.

Q2: SSH 키 비밀번호를 매번 입력하는 것이 번거로운데 해결 방법이 있나요?

A2: ssh-agent를 사용하세요. macOS는 ssh-add --apple-use-keychain ~/.ssh/id_ed25519, Linux/Windows는 eval "$(ssh-agent -s)" 후 ssh-add ~/.ssh/id_ed25519로 세션 동안 비밀번호를 기억하게 할 수 있습니다.

Q3: 실수로 push --force를 했는데 팀원의 커밋이 사라졌습니다. 복구할 수 있나요?

A3: 팀원의 로컬에 커밋이 남아있다면 다시 push할 수 있습니다. 원격 저장소에서 완전히 사라졌다면 GitHub Enterprise의 경우 관리자에게 복구를 요청할 수 있습니다. 교훈: 절대 공유 브랜치에 --force를 사용하지 마세요.

Q4: origin/main과 main의 차이가 무엇인가요?

A4: main은 로컬 브랜치이고, origin/main은 원격 추적 브랜치입니다. origin/main은 마지막 fetch/pull 시점의 원격 상태를 나타내며 읽기 전용입니다. 직접 커밋할 수 없고, fetch나 pull로만 업데이트됩니다.

Q5: git pull --rebase는 언제 사용하나요?

A5: git pull --rebase는 병합 커밋 없이 선형 히스토리를 유지하고 싶을 때 사용합니다. 원격 변경을 가져와 그 위에 내 커밋을 재적용합니다. 팀이 선형 히스토리를 선호한다면 유용하지만, rebase의 위험성을 이해하고 사용해야 합니다 (7장에서 자세히 배움).

장 요약¶

이번 장에서는 로컬을 넘어 원격 저장소와 연결하는 방법을 배웠습니다. git clone과 git remote add로 연결하고, SSH 키를 설정하여 안전한 인증을 구축했습니다. git push와 git pull로 변경 사항을 동기화하고, git fetch와 git pull의 차이를 이해했습니다. 이제 팀원과 코드를 공유하고 협업할 준비가 되었습니다.
