14장: 엔터프라이즈 Git/GitHub 활용

실무

엔터프라이즈 Git/GitHub 활용¶

학습 목표 - GitHub Organization과 Team을 구성하여 기업 수준의 접근 제어를 설정할 수 있다 - SAML SSO와 SCIM으로 기업 인증 시스템을 GitHub에 통합할 수 있다 - 감사 로그를 활용하여 저장소 활동을 추적하고 규정 준수를 입증할 수 있다 - 다른 플랫폼에서 GitHub으로의 마이그레이션을 계획하고 실행할 수 있다 - GitHub Copilot 등 AI 도구를 팀 개발 프로세스에 안전하게 통합할 수 있다

GitHub Enterprise와 조직 관리 (ch_14_s01)¶

도입¶

개인 프로젝트나 소규모 팀에서는 GitHub Free나 Pro 플랜으로 충분하지만, 수십 명에서 수백 명의 개발자가 협업하는 기업 환경에서는 고급 보안, 세밀한 권한 관리, 규정 준수, 전담 지원이 필수입니다. 이 절에서는 GitHub Enterprise의 특징과 Organization 구조 설계, Team 기반 권한 관리를 배웁니다.

핵심 개념 설명¶

GitHub Free/Pro/Team/Enterprise 비교¶

GitHub는 네 가지 플랜을 제공하며, 각 플랜은 대상 규모와 필요 기능에 따라 구분됩니다.

플랜 대상 주요 기능 가격
GitHub Free 개인, 소규모 오픈소스 무제한 public/private 저장소, 기본 CI/CD(월 2,000분), 코드 스캔(public만) 무료
GitHub Pro 개인 개발자 Advanced Insights, Protected Branches, Code Owners, Pages 고급 기능 월 $4
GitHub Team 소규모 팀(최대 수십 명) Team 액세스 제어, Draft PR, Required Reviews, Team Discussions 사용자당 월 $4
GitHub Enterprise 대기업(수십~수백 명) SAML SSO, SCIM 프로비저닝, 감사 로그, 99.95% SLA, 전담 지원, Advanced Security 사용자당 월 $21+

베스트 프랙티스: 50명 이상의 개발자가 있거나, 금융/의료 등 규정 준수가 중요한 산업이라면 GitHub Enterprise를 권장합니다. SAML SSO와 감사 로그는 보안 감사에 필수입니다.

GitHub Organization 생성과 구조 설계¶

Organization(조직)은 개인 계정이 아닌 법인이나 팀을 대표하는 GitHub 계정 유형입니다. 여러 사용자와 저장소를 그룹으로 관리하고, Team 단위로 권한을 부여합니다.

Organization 생성하기

웹 UI에서 생성:

# 1. GitHub 우측 상단 프로필 클릭 → Settings
# 2. 좌측 메뉴 → Organizations → New organization
# 3. 플랜 선택 (Free/Team/Enterprise)
# 4. Organization 이름, 청구 이메일, 소속 입력

GitHub CLI로 생성(Enterprise 플랜 필요):

$ gh api \
  --method POST \
  -H "Accept: application/vnd.github+json" \
  /admin/organizations \
  -f login='acme-corp' \
  -f admin='admin-user'
# 출력: Organization 생성 완료 (Enterprise Admin 권한 필요)

Organization 구조 설계 원칙

  1. 팀 기반 분리: 제품팀, 플랫폼팀, 인프라팀 등 조직도 반영
  2. 저장소 명명 규칙: {팀}-{서비스} (예: frontend-web-app, backend-api-server)
  3. 권한 최소 원칙: 각 팀은 자신의 저장소에만 Write 권한, 다른 팀 저장소는 Read
  4. 외부 협력자 관리: 외부 컨설턴트는 Organization 멤버가 아닌 Outside Collaborator로 초대

조직 구조 예시

Organization: Acme Corp
│
├── Team: Frontend
│   ├── Members: alice, bob, charlie (3명)
│   ├── Repositories
│   │   ├── frontend-web-app (Admin)
│   │   ├── frontend-mobile-app (Admin)
│   │   └── backend-api-server (Read) ← 다른 팀 저장소는 읽기만
│   └── Permissions: 팀 저장소 Admin, 타 저장소 Read
│
├── Team: Backend
│   ├── Members: david, emma (2명)
│   ├── Repositories
│   │   ├── backend-api-server (Admin)
│   │   ├── backend-worker (Admin)
│   │   └── frontend-web-app (Read)
│   └── Permissions: 팀 저장소 Admin, 타 저장소 Read
│
├── Team: DevOps
│   ├── Members: frank (1명)
│   ├── Repositories: 모든 저장소 (Admin) ← 배포 관리
│   └── Permissions: Organization Admin
│
└── Outside Collaborators
    └── consultant-user (특정 저장소만 Write)

Team 구성과 저장소 권한 관리¶

Team 생성하기

# 웹 UI: Organization → Teams → New team
# Team 이름: Frontend
# Description: Frontend 개발팀
# Privacy: Visible (조직 내 공개) or Secret (비공개)

GitHub CLI:

$ gh api \
  --method POST \
  -H "Accept: application/vnd.github+json" \
  /orgs/acme-corp/teams \
  -f name='Frontend' \
  -f description='Frontend 개발팀' \
  -f privacy='closed'
# 출력: Team 생성 완료

Team 멤버 추가

# 웹 UI: Teams → Frontend → Members → Add a member

# CLI:
$ gh api \
  --method PUT \
  /orgs/acme-corp/teams/frontend/memberships/alice
# 출력: alice를 Frontend 팀에 추가 완료

저장소에 Team 권한 부여

저장소 권한 레벨:

권한 설명 가능 작업
Read 읽기 전용 코드 조회, 이슈 읽기, Clone
Triage 이슈 관리 Read + 이슈/PR 관리(라벨, 할당, 닫기)
Write 읽기/쓰기 Triage + Push, PR 생성, 일부 Settings
Maintain 유지 관리 Write + Settings 일부, 릴리스 관리
Admin 관리자 모든 권한 + 팀 관리, 저장소 삭제, Webhooks 
# Frontend 팀에 frontend-web-app 저장소 Admin 권한 부여
$ gh api \
  --method PUT \
  /orgs/acme-corp/teams/frontend/repos/acme-corp/frontend-web-app \
  -f permission='admin'
# 출력: 권한 부여 완료

주의: Admin 권한은 저장소 삭제, Webhooks 설정 등 강력한 권한을 포함하므로 신중하게 부여해야 합니다. 일반 개발 팀에는 Write 권한이 적절합니다.

Organization 수준 설정: 보안 정책, 기본 설정¶

Organization Settings에서 전체 저장소에 적용할 기본 정책을 설정합니다.

보안 정책(Security & analysis)

Settings → Security & analysis
  • Dependency graph: 의존성 자동 추적 (활성화 권장)
  • Dependabot alerts: 취약한 의존성 알림 (활성화 권장)
  • Dependabot security updates: 자동 보안 패치 PR (활성화 권장)
  • Secret scanning: 토큰/비밀번호 실수 커밋 탐지 (Enterprise만, 활성화 필수)
  • Code scanning: 코드 보안 취약점 자동 스캔 (Advanced Security 필요)

기본 저장소 설정(Repository defaults)

Settings → Repository defaults
  • 기본 브랜치: main (새 저장소 생성 시 기본값)
  • Repository creation: 멤버가 저장소 생성 가능 여부
  • Repository forking: 외부 Fork 허용 여부
  • Repository visibility: Public/Private 기본값

멤버 권한(Member privileges)

Settings → Member privileges
  • Base permissions: Organization 멤버의 기본 권한 (Read/Write/None)
  • Repository creation: 멤버가 저장소 생성 가능 여부
  • Pages creation: 멤버가 GitHub Pages 배포 가능 여부
  • Team creation: 멤버가 팀 생성 가능 여부

베스트 프랙티스: Base permissions는 "Read"로 설정하고, 필요한 저장소에만 Team을 통해 Write 권한을 부여하는 것이 안전합니다. 이를 "Principle of Least Privilege(최소 권한 원칙)"라고 합니다.

Enterprise Managed Users(EMU)¶

EMU는 GitHub Enterprise Cloud의 고급 기능으로, 모든 사용자 계정을 기업의 ID 공급자(IdP)에서 완전히 관리합니다.

기존 방식 vs EMU

항목 기존 Organization EMU
계정 소유자 개인 GitHub 계정 기업 (계정명: user_corp)
인증 개인 비밀번호 또는 SSO 기업 IdP만 (SSO 필수)
외부 협업 자유롭게 가능 제한적 (허용 목록 필요)
계정 라이프사이클 개인이 직접 관리 SCIM으로 자동 관리
적용 대상 일반적인 조직 금융/공공기관 등 엄격한 규제 환경

EMU 계정 형식:

일반: alice
EMU: alice_acme-corp

주의: EMU는 도입 후 기존 계정으로 되돌릴 수 없으므로, 도입 전 충분한 검토가 필요합니다.

절 요약¶

  • GitHub Enterprise는 SAML SSO, 감사 로그, 고급 보안, 전담 지원을 제공하는 대기업용 플랜입니다.

  • Team을 통해 저장소별 세밀한 권한(Read/Write/Admin)을 부여합니다.

인증과 접근 제어 (ch_14_s02)¶

도입¶

기업 환경에서는 수백 명의 직원이 GitHub에 접근하며, 퇴사자 계정 관리, 비밀번호 정책, 다중 인증이 필수입니다. 이 절에서는 SAML SSO로 기업 ID 공급자를 GitHub과 통합하고, SCIM으로 사용자를 자동 관리하며, Personal Access Token(PAT)으로 안전한 API 접근을 설정하는 방법을 배웁니다.

핵심 개념 설명¶

SAML SSO 통합: 기업 ID 공급자 연동¶

SAML SSO(Single Sign-On)는 하나의 인증 정보로 여러 서비스에 로그인할 수 있게 하는 기업 인증 표준입니다. 회사 이메일과 비밀번호로 GitHub, Slack, Jira 등 모든 서비스에 한 번에 로그인할 수 있습니다.

SAML SSO의 장점

  1. 단일 인증: 하나의 회사 계정으로 모든 서비스 접근
  2. 중앙 관리: IT 팀이 한곳에서 모든 계정 관리
  3. 보안 강화: 강력한 비밀번호 정책, 다중 인증(MFA) 강제 적용
  4. 빠른 퇴사 처리: IdP에서 계정 비활성화 시 모든 서비스 접근 차단

지원 ID 공급자(IdP)

  • Okta: 가장 널리 사용되는 SSO 솔루션
  • Azure Active Directory: Microsoft 기업 고객
  • OneLogin: 중소기업 친화적
  • Google Workspace: Google 기반 조직
  • PingIdentity: 대기업용 엔터프라이즈 솔루션

SAML SSO 설정 절차

# 1. Organization Settings → Authentication security
# 2. "Enable SAML authentication" 클릭
# 3. IdP에서 제공한 정보 입력
#    - Sign on URL: https://idp.example.com/sso/saml
#    - Issuer: https://idp.example.com
#    - Public certificate: (IdP에서 다운로드한 인증서 업로드)
# 4. "Test SAML configuration" 클릭하여 테스트
# 5. 정상 동작 확인 후 "Require SAML SSO authentication" 활성화

SAML 인증 흐름

1. 사용자가 GitHub Organization 접근 시도
   ↓
2. GitHub가 사용자를 IdP 로그인 페이지로 리디렉션
   ↓
3. 사용자가 회사 이메일/비밀번호 입력 (+ MFA)
   ↓
4. IdP가 인증 성공 시 SAML 응답(Assertion) 발행
   ↓
5. GitHub가 SAML 응답 검증 후 접근 허용
   ↓
6. 사용자가 GitHub Organization 접근 성공

SAML Assertion 예시

<saml:Assertion xmlns:saml="urn:oasis:names:tc:SAML:2.0:assertion">
  <saml:Subject>
    <saml:NameID>alice@acme-corp.com</saml:NameID>
  </saml:Subject>
  <saml:AttributeStatement>
    <saml:Attribute Name="email">
      <saml:AttributeValue>alice@acme-corp.com</saml:AttributeValue>
    </saml:Attribute>
    <saml:Attribute Name="login">
      <saml:AttributeValue>alice</saml:AttributeValue>
    </saml:Attribute>
  </saml:AttributeStatement>
</saml:Assertion>

팁: SAML 설정 중 "Test SAML configuration" 단계에서 반드시 여러 계정으로 테스트하세요. 설정 실수로 전체 조직이 접근 불가능해질 수 있으므로, 최소 2명의 Admin 계정으로 테스트 후 활성화하는 것이 안전합니다.

SCIM 프로비저닝: 사용자 자동 관리¶

SCIM(System for Cross-domain Identity Management)은 사용자 계정의 생성, 수정, 삭제를 IdP와 GitHub 간 자동 동기화하는 표준입니다.

SCIM의 장점

  1. 자동 온보딩: 신규 입사자가 IdP에 등록되면 GitHub Organization에 자동 추가
  2. 자동 오프보딩: 퇴사자가 IdP에서 비활성화되면 GitHub 접근 즉시 차단
  3. 정보 동기화: 이름, 이메일, 팀 변경 시 자동 반영
  4. 수동 작업 제거: IT 팀의 반복적인 계정 관리 작업 자동화

SCIM 설정 절차

# 1. Organization Settings → Authentication security → SCIM
# 2. "Enable SCIM" 클릭
# 3. Personal Access Token 생성 (admin:org, read:user 권한 필요)
# 4. IdP(Okta/Azure AD)에서 GitHub App 추가
# 5. IdP에 다음 정보 입력
#    - SCIM Base URL: https://api.github.com/scim/v2/organizations/{org}
#    - Access Token: (3번에서 생성한 PAT)
# 6. IdP에서 사용자/그룹 할당
# 7. 동기화 테스트 실행

SCIM 동기화 이벤트

IdP 이벤트 GitHub 동작
사용자 생성 Organization에 멤버 초대 메일 발송
사용자 정보 수정 GitHub 프로필 업데이트
사용자 비활성화 Organization 멤버십 일시 중단
사용자 삭제 Organization에서 제거
그룹 생성 GitHub Team 생성
그룹 멤버십 변경 Team 멤버 추가/제거

SCIM API 요청 예시

사용자 생성:

POST /scim/v2/organizations/{org}/Users
Content-Type: application/scim+json

{
  "schemas": ["urn:ietf:params:scim:schemas:core:2.0:User"],
  "userName": "alice",
  "name": {
    "givenName": "Alice",
    "familyName": "Smith"
  },
  "emails": [
    {
      "value": "alice@acme-corp.com",
      "primary": true
    }
  ]
}

# 출력: 사용자 생성 완료, GitHub 초대 메일 발송

베스트 프랙티스: SCIM 설정 후 주기적으로 IdP와 GitHub의 멤버 목록을 비교하여 동기화 상태를 확인하세요. Audit Log에서 SCIM 이벤트를 모니터링할 수 있습니다.

Personal Access Token(PAT) 관리¶

PAT는 Git 명령어나 API 호출 시 비밀번호 대신 사용하는 토큰으로, 세밀한 권한 제어가 가능합니다.

Classic PAT vs Fine-grained PAT

항목 Classic PAT Fine-grained PAT
도입 시기 기존부터 존재 2022년 10월 도입
권한 범위 전체 계정 또는 전체 Organization 저장소별, Organization별 세밀한 권한

Fine-grained PAT 생성 (권장)

# 1. GitHub 우측 상단 프로필 → Settings
# 2. Developer settings → Personal access tokens → Fine-grained tokens
# 3. "Generate new token" 클릭
# 4. 토큰 설정
#    - Token name: "CI/CD Pipeline Token"
#    - Expiration: 90 days (최대 1년)
#    - Repository access: "Only select repositories" → 선택
#    - Permissions:
#        - Repository permissions:
#            - Contents: Read and write (코드 push)
#            - Metadata: Read-only (자동 포함)
#        - Organization permissions:
#            - Members: Read-only (팀원 목록 조회)
# 5. "Generate token" 클릭
# 6. 생성된 토큰 복사 (다시 볼 수 없으므로 안전한 곳에 저장)

생성된 토큰 사용:

# Git 인증
$ git clone https://ghp_xxxxxxxxxxxxxxxxxxxx@github.com/acme-corp/api-server.git
# 또는
$ git remote set-url origin https://ghp_xxxxxxxxxxxxxxxxxxxx@github.com/acme-corp/api-server.git

# API 호출
$ curl -H "Authorization: token ghp_xxxxxxxxxxxxxxxxxxxx" \
  https://api.github.com/user
# 출력: {"login":"alice",...}

주의: PAT는 비밀번호와 동일한 권한을 가지므로 절대 코드에 포함하거나 공개 저장소에 커밋하면 안 됩니다. GitHub Secrets, 환경 변수, 비밀 관리 도구(HashiCorp Vault)에 저장하세요.

IP 허용 목록(IP Allow List) 설정¶

IP Allow List는 특정 IP 주소 또는 범위에서만 GitHub Organization 접근을 허용하는 보안 기능입니다(Enterprise 플랜 필요).

설정 절차

# 1. Organization Settings → Authentication security → IP allow list
# 2. "Enable IP allow list" 클릭
# 3. "Add IP address" 클릭
#    - IP address or range: 203.0.113.0/24 (회사 사무실 IP 대역)
#    - Description: "본사 사무실"
# 4. VPN IP, 재택 IP 등 필요한 IP 모두 추가
# 5. "Allow access via GitHub Apps" 활성화 (CI/CD 도구 허용)

CIDR 표기법 예시

단일 IP: 203.0.113.50/32
C 클래스 대역: 203.0.113.0/24 (203.0.113.0 ~ 203.0.113.255)
B 클래스 대역: 172.16.0.0/16 (172.16.0.0 ~ 172.16.255.255)

주의: IP Allow List 활성화 전에 반드시 본인의 IP를 먼저 추가하세요. 그렇지 않으면 자신도 Organization에 접근할 수 없게 됩니다. VPN, 재택근무, 외부 협력자 IP를 모두 고려해야 합니다.

절 요약¶

  • SAML SSO는 기업 IdP와 GitHub을 연동하여 단일 인증으로 접근을 관리합니다.
  • SCIM은 사용자 계정의 생성, 수정, 삭제를 IdP와 자동 동기화합니다.
  • Fine-grained PAT는 저장소별 세밀한 권한 제어가 가능한 현대적인 인증 토큰입니다.
  • IP Allow List는 특정 IP에서만 Organization 접근을 허용하여 보안을 강화합니다.

감사와 규정 준수 (ch_14_s03)¶

도입¶

금융, 의료, 공공기관 등 규제가 엄격한 산업에서는 "누가, 언제, 무엇을, 왜" 변경했는지 모든 활동을 추적하고 증명해야 합니다. 이 절에서는 감사 로그(Audit Log)로 Organization 활동을 추적하고, 규정 준수(Compliance) 보고서를 생성하며, 데이터 거주지(Data Residency) 설정을 배웁니다.

핵심 개념 설명¶

감사 로그: 누가 무엇을 했는가 추적¶

감사 로그(Audit Log)는 Organization 또는 Enterprise 수준에서 발생한 모든 중요 이벤트를 기록하는 로그입니다(Enterprise 플랜 필요).

기록되는 주요 이벤트

카테고리 이벤트 예시
저장소 repo.create, repo.destroy, repo.rename, repo.transfer
멤버 org.add_member, org.remove_member, org.update_member
팀 team.create, team.add_member, team.remove_member
권한 repo.access, repo.add_collaborator, repo.remove_collaborator
브랜치 보호 protected_branch.create, protected_branch.destroy
보안 org.enable_saml, org.disable_saml, two_factor_authentication.enabled
Webhooks hook.create, hook.config_changed, hook.destroy

웹 UI에서 감사 로그 확인

# 1. Organization Settings → Audit log
# 2. 필터 옵션:
#    - Action: 특정 이벤트 유형 선택 (예: repo.create)
#    - Actor: 특정 사용자 선택
#    - Repository: 특정 저장소 선택
#    - Time range: 날짜 범위 선택 (최대 180일)
# 3. "Export" 버튼으로 CSV/JSON 다운로드

감사 로그 쿼리 문법

GitHub 감사 로그는 강력한 검색 문법을 지원합니다.

# 특정 사용자의 모든 활동
actor:alice

# 저장소 삭제 이벤트만 조회
action:repo.destroy

# 특정 저장소 관련 이벤트
repo:acme-corp/api-server

# 특정 기간
created:2025-01-01..2025-01-31

# 조합 쿼리: alice가 1월에 삭제한 저장소
actor:alice action:repo.destroy created:2025-01-01..2025-01-31

감사 로그 항목 예시

{
  "actor": "alice",
  "action": "repo.destroy",
  "created_at": "2025-02-17T14:30:00Z",
  "repo": "acme-corp/old-project",
  "org": "acme-corp",
  "@timestamp": 1708180200000,
  "actor_ip": "203.0.113.50",
  "user_agent": "Mozilla/5.0 (Windows NT 10.0; Win64; x64)"
}

베스트 프랙티스: 감사 로그는 180일 후 자동 삭제됩니다. 장기 보관이 필요하면 주기적으로 내보내기(Export)하여 SIEM(Security Information and Event Management) 시스템에 저장하세요.

감사 로그 스트리밍(Audit Log Streaming)

Enterprise Cloud 플랜에서는 감사 로그를 실시간으로 외부 시스템에 전송할 수 있습니다.

지원 대상:

  • Amazon S3: AWS S3 버킷에 JSON 파일 저장
  • Azure Event Hubs: Azure 이벤트 스트림
  • Splunk: Splunk HTTP Event Collector
  • Google Cloud Storage: GCS 버킷

설정 절차(Amazon S3 예시):

# 1. Enterprise Settings → Audit log → Streaming
# 2. "New stream" → "Amazon S3"
# 3. AWS 정보 입력:
#    - Bucket: audit-logs-acme-corp
#    - Region: us-east-1
#    - AWS Access Key ID: AKIA...
#    - AWS Secret Access Key: (비밀번호)
# 4. "Test connection" → "Save"

스트리밍된 로그 파일 예시:

s3://audit-logs-acme-corp/2025/02/17/14/audit-log-1708180200.json
s3://audit-logs-acme-corp/2025/02/17/15/audit-log-1708183800.json
...

팁: 감사 로그 스트리밍을 SIEM 도구(Splunk, Datadog, ELK Stack)와 연동하면 이상 행동 탐지, 대시보드 시각화, 자동 알림 설정이 가능합니다.

GitHub Enterprise는 주요 보안 및 규정 준수 인증을 획득하여, 고객이 자체 감사를 통과하는 데 필요한 증빙을 제공합니다.

인증/규정 설명 대상 산업

  • [ ] 2FA(이중 인증) 모든 멤버 필수 설정
  • [ ] 브랜치 보호 규칙 설정 (main 브랜치 직접 push 금지)

데이터 거주지(Data Residency) 설정¶

데이터 거주지(Data Residency)는 데이터가 물리적으로 저장되는 지리적 위치를 의미하며, EU GDPR 등 특정 규정은 데이터가 해당 지역 내에 보관되도록 요구합니다.

GitHub 데이터 센터 위치

  • GitHub.com 기본: 미국 동부/서부
  • GitHub Enterprise Server: 고객 자체 데이터 센터 또는 클라우드
  • GitHub AE (Azure Edition): Azure 리전 선택 가능 (EU, Asia 등)

데이터 거주지 요구사항 준수 방법

  1. GitHub Enterprise Server (자체 호스팅)
  2. 고객 데이터 센터 또는 원하는 클라우드 리전에 직접 설치

  3. 완전한 데이터 통제권

  4. GitHub AE (Azure Edition)

  5. Azure 리전 선택 (예: EU West, Asia Pacific)
  6. Microsoft Azure의 규정 준수 인증 활용

팁: GDPR 준수가 필수인 EU 고객은 GitHub Enterprise Server를 EU 리전에 설치하거나, GitHub AE를 EU 리전에 배포하는 것이 권장됩니다.

저장소 정책과 규칙 일괄 적용¶

Rulesets (Beta)는 Organization 또는 Enterprise 수준에서 여러 저장소에 일괄 정책을 적용하는 차세대 브랜치 보호 기능입니다.

기존 브랜치 보호 규칙 vs Rulesets:

항목 브랜치 보호 규칙 Rulesets
적용 범위 저장소별 개별 설정 Organization/Enterprise 일괄
상속 불가능 가능
패턴 매칭 브랜치명만 브랜치명 + 태그명 + 파일 경로
우선순위 없음 여러 Ruleset 충돌 시 우선순위 지정

Ruleset 생성 예시

# 1. Organization Settings → Repository rulesets → New ruleset
# 2. Ruleset 설정:
#    - Name: "Production Branches Protection"
#    - Enforcement: Active (강제 적용)
#    - Target: All repositories (모든 저장소)
#    - Branch naming pattern: main, release/*
# 3. Rules:
#    - ✅ Require pull request before merging
#    - ✅ Require approvals (2명)
#    - ✅ Require status checks to pass (CI 통과 필수)
#    - ✅ Block force pushes
#    - ✅ Require linear history
# 4. "Create ruleset"

베스트 프랙티스: Rulesets를 단계적으로 도입하세요. 처음에는 "Evaluate" 모드로 설정하여 위반 사항을 모니터링하고, 팀이 적응한 후 "Active" 모드로 전환하세요.

절 요약¶

  • 감사 로그는 Organization의 모든 중요 활동을 180일간 기록하며, API 또는 스트리밍으로 외부 시스템에 전송할 수 있습니다.
  • GitHub는 SOC 2, ISO 27001, FedRAMP, GDPR 등 주요 규정 준수 인증을 획득했습니다.
  • 데이터 거주지 요구사항은 GitHub Enterprise Server 또는 GitHub AE로 충족할 수 있습니다.
  • Rulesets는 Organization 수준에서 여러 저장소에 일괄 정책을 적용하는 현대적인 방법입니다.

마이그레이션 전략 (ch_14_s04)¶

도입¶

기업이 SVN, GitLab, Bitbucket 등 기존 버전 관리 시스템에서 GitHub으로 전환하는 것은 단순한 코드 이동이 아닙니다. 커밋 히스토리, 이슈, Pull Request, CI/CD 파이프라인, 팀원 교육까지 포함하는 종합적인 프로젝트입니다. 이 절에서는 성공적인 마이그레이션을 위한 계획 수립, 도구 선택, 실행, 검증 전략을 배웁니다.

핵심 개념 설명¶

SVN에서 Git으로 마이그레이션¶

SVN(Subversion)은 중앙 집중식 버전 관리 시스템으로, 많은 기업이 여전히 사용하고 있습니다. Git으로 마이그레이션하면 브랜치 비용 감소, 오프라인 작업, 분산 협업 등의 이점을 얻을 수 있습니다.

SVN과 Git의 개념 차이

SVN 개념 Git 개념 설명
trunk main/master 주 개발 브랜치
branches/feature-x feature-x 브랜치 기능 개발 브랜치
tags/v1.0 v1.0 태그 릴리스 버전 표시
revision 1234 커밋 해시 a3f5c2... 버전 식별자
svn update git pull 최신 변경 사항 가져오기
svn commit git commit + git push 변경 사항 저장

git-svn으로 마이그레이션

Git은 git svn 명령어로 SVN 저장소를 Git으로 변환할 수 있습니다.

# 1. SVN 저장소 클론 (히스토리 포함)
$ git svn clone http://svn.example.com/project \
  --stdlayout \
  --authors-file=authors.txt \
  my-project-git

# --stdlayout: 표준 SVN 구조(trunk, branches, tags) 인식
# --authors-file: SVN 사용자명 → Git 이메일 매핑 파일

authors.txt 파일 예시

alice = Alice Smith <alice@acme-corp.com>
bob = Bob Johnson <bob@acme-corp.com>
charlie = Charlie Lee <charlie@acme-corp.com>

# 2. SVN 브랜치를 Git 브랜치로 변환
$ cd my-project-git
$ git branch -r
# 출력:
#   origin/trunk
#   origin/branches/feature-x
#   origin/tags/v1.0

# trunk를 main으로 변경
$ git checkout -b main origin/trunk

# SVN 브랜치를 Git 브랜치로 변환
$ for branch in $(git branch -r | grep 'origin/branches/'); do
    git branch ${branch#origin/branches/} $branch
  done

# SVN 태그를 Git 태그로 변환
$ for tag in $(git branch -r | grep 'origin/tags/'); do
    git tag ${tag#origin/tags/} $tag
  done

# 3. SVN 메타데이터 제거 (선택 사항)
$ git filter-branch --prune-empty --subdirectory-filter trunk -- --all

# 4. GitHub에 푸시
$ git remote add origin https://github.com/acme-corp/my-project.git
$ git push -u origin main --tags

주의: SVN 히스토리가 수년치 있는 대형 저장소는 git svn clone에 수 시간이 걸릴 수 있습니다. 테스트 마이그레이션으로 소요 시간을 미리 측정하세요.

GitLab/Bitbucket에서 GitHub으로 마이그레이션¶

GitLab, Bitbucket에서 GitHub으로 전환 시에는 Git 히스토리뿐만 아니라 이슈, Merge Request, Wiki까지 마이그레이션할 수 있습니다.

GitHub Importer (웹 UI)

간단한 마이그레이션에 적합:

# 1. GitHub에서 New repository 생성
# 2. "Import code" 클릭
# 3. Old repository URL 입력:
#    - GitLab: https://gitlab.com/acme/project
#    - Bitbucket: https://bitbucket.org/acme/project
# 4. 인증 정보 입력 (Personal Access Token)
# 5. "Begin import" 클릭
# 6. 진행 상황 모니터링

GitHub CLI (gh)

# GitLab에서 마이그레이션
$ gh repo create acme-corp/my-project --public
$ git clone --mirror https://gitlab.com/acme/project.git
$ cd project.git
$ git push --mirror https://github.com/acme-corp/my-project.git

이슈/PR 마이그레이션 도구

GitHub Importer는 Git 히스토리만 가져오므로, 이슈/MR은 별도 도구가 필요합니다.

GitHub Enterprise Importer (GEI)¶

GEI는 GitHub Enterprise Cloud 고객을 위한 공식 마이그레이션 도구로, 대규모 조직 단위 마이그레이션을 지원합니다.

GEI 특징

  • Organization 단위 마이그레이션: 수백 개 저장소 일괄 처리
  • 메타데이터 보존: 이슈, PR, 릴리스, Wiki, 프로젝트 보드
  • Team/멤버십 마이그레이션: Organization 구조 유지
  • 대용량 저장소 지원: Git LFS, 대형 바이너리 처리
  • 점진적 마이그레이션: 저장소별 순차 전환 가능

마이그레이션 계획 수립과 체크리스트¶

성공적인 마이그레이션은 철저한 사전 계획에서 시작됩니다.

마이그레이션 6단계

1. 현황 분석 (2주)
   └─ 저장소 수, 크기, 활성 브랜치, 팀 구조, CI/CD 파이프라인 파악

2. 도구 선택 (1주)
   └─ SVN: git-svn, GitLab/Bitbucket: GitHub Importer/GEI 선택

3. 테스트 마이그레이션 (2주)
   └─ 대표 저장소 3~5개 선정 → 테스트 Organization에 마이그레이션

4. 데이터 검증 (1주)
   └─ 커밋 수, 브랜치, 태그, 이슈 수 원본과 비교 검증

5. 본 마이그레이션 (1일~1주, 저장소 수에 따라)
   └─ 팀별 순차 마이그레이션 또는 전체 일괄 마이그레이션

6. 전환 완료 및 교육 (2주)
   └─ 구 시스템 읽기 전용 전환 → 팀원 교육 → 모니터링

GitHub Copilot과 AI 통합 (ch_14_s05)¶

도입¶

AI 기술의 발전으로 GitHub Copilot을 비롯한 AI 코드 보조 도구가 개발자의 생산성을 획기적으로 높이고 있습니다. 하지만 AI 생성 코드의 보안, 라이선스, 품질 문제도 함께 고려해야 합니다. 이 절에서는 GitHub Copilot의 기능, 팀 도입 전략, 보안 고려사항, AI 코드 리뷰 활용법을 배웁니다.

AI 도구의 보안과 라이선스 고려사항¶

AI 생성 코드는 편리하지만, 보안과 법적 위험도 있습니다.

보안 고려사항

  1. 민감 정보 유출
  2. Copilot은 공개 GitHub 저장소로 학습하므로, 생성된 코드에 실제 API 키나 비밀번호가 포함될 수 있습니다.

  3. 해결책: Secret Scanning 활성화, 코드 리뷰 필수

  4. 취약한 코드 생성

  5. AI가 SQL Injection, XSS 등 보안 취약점이 있는 코드를 생성할 수 있습니다.

  6. 해결책: GitHub Advanced Security의 Code Scanning과 함께 사용

  7. 오래된 패턴 사용

  8. 학습 데이터에 deprecated 함수나 취약한 라이브러리 버전이 포함될 수 있습니다.
  9. 해결책: Dependabot으로 의존성 자동 업데이트

FAQ¶

Q1: GitHub Free와 GitHub Enterprise의 주요 차이점은 무엇인가요?

A1: GitHub Enterprise는 SAML SSO, SCIM 프로비저닝, 감사 로그, 99.95% SLA, 전담 기술 지원, Advanced Security(Code Scanning, Secret Scanning), IP Allow List 등 기업 보안과 규정 준수 기능을 제공합니다. Free는 기본 CI/CD와 코드 호스팅만 지원하며, 50명 이상 조직이나 금융/의료 등 규제 산업에는 Enterprise가 필수입니다.

Q2: Organization에 몇 명까지 멤버를 추가할 수 있나요?

A2: GitHub Enterprise 플랜은 사용자 수 제한이 없습니다. Free 플랜도 무제한 멤버를 추가할 수 있지만, Team 기반 권한 관리, SAML SSO, 감사 로그 등 고급 기능은 사용할 수 없습니다. Team 플랜은 소규모 팀(수십 명)에 적합하며, 수백 명 이상은 Enterprise를 권장합니다.

Q3: SAML SSO 설정 시 실수로 모든 멤버가 접근 불가능해지면 어떻게 하나요?

A3: GitHub는 "SSO Recovery Codes"를 제공합니다. SAML 활성화 시 복구 코드를 다운로드받아 안전한 곳에 보관하세요. 만약 IdP 장애로 접근이 불가능하면, Organization Owner가 복구 코드로 로그인하여 SAML을 일시 비활성화할 수 있습니다. 또한 최소 2명 이상의 Owner를 두어 단일 장애점을 방지하는 것이 권장됩니다.

Q4: 감사 로그는 얼마나 오래 보관되나요?

A4: GitHub 감사 로그는 기본 180일간 보관됩니다. 이후 자동 삭제되므로, 장기 보관이 필요한 기업은 주기적으로 Export하거나 Audit Log Streaming을 SIEM(Splunk, Datadog 등)으로 연동하여 외부에 저장해야 합니다. 금융/의료 등 규제 산업은 보통 7년 이상 보관이 요구됩니다.

Q5: SVN 저장소가 매우 크고 오래되었는데, 마이그레이션 시간이 얼마나 걸리나요?

A5: git svn clone은 SVN 히스토리를 처음부터 끝까지 순회하므로, 10년 이상 히스토리가 있고 수만 개 커밋이 있는 저장소는 수 시간에서 하루 이상 걸릴 수 있습니다. 테스트 마이그레이션으로 소요 시간을 미리 측정하고, 히스토리가 꼭 필요 없다면 최근 N년치만 마이그레이션하거나, 히스토리 없이 최신 코드만 GitHub에 커밋하는 방법도 고려할 수 있습니다.

Q6: GitHub Copilot이 생성한 코드에 버그가 있으면 누가 책임지나요?

A6: GitHub 이용약관에 따르면 AI 생성 코드의 책임은 전적으로 사용자에게 있습니다. Copilot은 "제안 도구"일 뿐이며, 생성된 코드의 정확성, 보안, 라이선스 준수는 개발자가 검증해야 합니다. 따라서 AI 생성 코드도 사람의 코드 리뷰, 테스트, 보안 스캔을 거쳐야 하며, 중요한 로직은 여러 개발자가 교차 검토하는 것이 안전합니다.

장 요약¶

엔터프라이즈 환경에서 Git과 GitHub를 운영하는 것은 개인 프로젝트와 차원이 다른 도전 과제입니다. 이번 장에서는 GitHub Organization과 Team으로 수백 명의 개발자를 체계적으로 관리하고, SAML SSO와 SCIM으로 기업 ID 시스템과 통합하며, 감사 로그로 모든 활동을 추적하고 규정 준수를 입증하는 방법을 배웠습니다. 또한 SVN, GitLab, Bitbucket에서 GitHub으로의 마이그레이션 전략과, GitHub Copilot 등 AI 도구를 안전하게 도입하여 개발 생산성을 높이는 방법을 익혔습니다.
