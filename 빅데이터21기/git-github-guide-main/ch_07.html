07장: 고급 Git 기법

중급

고급 Git 기법¶

학습 목표 - git rebase로 커밋 히스토리를 깔끔하게 정리하고 리베이스의 위험을 인지할 수 있다 - git cherry-pick으로 특정 커밋을 다른 브랜치에 적용할 수 있다 - git stash로 진행 중인 작업을 임시 저장하고 복원할 수 있다 - git reflog로 실수를 복구하고 삭제된 작업을 되살릴 수 있다 - git bisect로 버그를 도입한 커밋을 효율적으로 찾을 수 있다

히스토리 재작성: rebase (ch_07_s01)¶

도입¶

지금까지 브랜치를 병합할 때는 주로 git merge를 사용했습니다. 하지만 merge는 병합 커밋을 생성하여 히스토리가 복잡해질 수 있습니다. 이 절에서는 커밋 히스토리를 선형으로 깔끔하게 정리하는 git rebase를 배웁니다. 강력한 만큼 위험한 도구이므로, 언제 사용하고 언제 피해야 하는지도 함께 배웁니다.

핵심 개념 설명¶

rebase의 동작 원리: 커밋 재적용¶

Rebase는 브랜치의 기준점(Base)을 변경하여 커밋을 다른 베이스 위에 다시 적용(Re-apply)하는 작업입니다.

블로그 글을 수정하는 것과 같습니다. 어제 쓴 글의 내용을 오늘 새로운 정보 위에 다시 작성하면, 처음부터 오늘 정보를 바탕으로 쓴 것처럼 깔끔해집니다. 하지만 이미 다른 사람이 어제 글을 인용했다면 혼란이 생깁니다.

merge와 rebase 비교

초기 상태:
main:     A --- B --- C
                       \
feature:                D --- E

merge 결과:
main:     A --- B --- C ----------- M
                       \           /
feature:                D --- E ---

rebase 결과:
main:     A --- B --- C --- D' --- E'

merge는 병합 커밋 M을 만들고 히스토리가 두 갈래로 표시됩니다. rebase는 D와 E를 C 위에 다시 적용하여 선형 히스토리를 만듭니다. D'와 E'는 D와 E의 변경 내용은 같지만 새로운 커밋 해시를 가집니다.

주의: rebase는 커밋 해시를 변경하므로 이미 공유된 커밋에는 사용하면 안 됩니다.

브랜치를 최신 상태로 갱신하기¶

사용 시나리오

feature 브랜치에서 작업 중인데, main 브랜치에 새로운 커밋이 추가되었습니다. feature를 main의 최신 상태로 갱신하고 싶습니다.

merge 방식

$ git switch feature/login
$ git merge main
# 출력: Merge branch 'main' into feature/login

이 방식은 병합 커밋을 만들어 히스토리가 복잡해집니다.

rebase 방식

$ git switch feature/login
$ git rebase main
# 출력:
# Successfully rebased and updated refs/heads/feature/login.

feature/login의 커밋들이 main의 최신 커밋 위에 재적용됩니다. 히스토리가 선형으로 유지되어 깔끔합니다.

실습 예제

# main 브랜치의 최신 변경 가져오기
$ git switch main
$ git pull

# feature 브랜치로 전환
$ git switch feature/login

# main의 최신 커밋 위에 feature 커밋 재적용
$ git rebase main
# 출력:
# First, rewinding head to replay your work on top of it...
# Applying: feat: 로그인 폼 추가
# Applying: feat: 로그인 API 연동

충돌 발생 시

$ git rebase main
# 출력:
# CONFLICT (content): Merge conflict in src/login.js
# error: could not apply abc1234... feat: 로그인 폼 추가
# Resolve all conflicts manually, mark them as resolved with
# "git add/rm <conflicted_files>", then run "git rebase --continue".

충돌 해결 단계

  1. 충돌 파일 수정
  2. git add <파일명>으로 해결 표시
  3. git rebase --continue로 진행
  4. 모든 커밋이 재적용될 때까지 반복

rebase 중단

$ git rebase --abort
# rebase를 취소하고 이전 상태로 되돌립니다

베스트 프랙티스: 작업 브랜치를 push하기 전에 main을 rebase하면 PR에서 충돌을 미리 방지할 수 있습니다.

인터랙티브 리베이스: 커밋 편집, 합치기, 삭제¶

인터랙티브 리베이스(Interactive Rebase)는 과거 커밋을 자유롭게 수정할 수 있는 강력한 기능입니다.

사용 시나리오

  • 여러 작은 커밋을 하나로 합치기(Squash)
  • 커밋 메시지 수정(Reword)
  • 커밋 순서 변경
  • 불필요한 커밋 삭제(Drop)
  • 커밋 분리(Edit)

기본 사용법

# 최근 3개 커밋을 편집
$ git rebase -i HEAD~3

에디터가 열리면 다음과 같은 화면이 나타납니다.

pick abc1234 feat: 로그인 폼 추가
pick def5678 fix: 오타 수정
pick ghi9012 feat: 로그인 검증 추가

# Commands:
# p, pick <commit> = use commit
# r, reword <commit> = use commit, but edit the commit message
# e, edit <commit> = use commit, but stop for amending
# s, squash <commit> = use commit, but meld into previous commit
# f, fixup <commit> = like "squash", but discard this commit's log message
# d, drop <commit> = remove commit

커밋 합치기 (Squash)

pick abc1234 feat: 로그인 폼 추가
squash def5678 fix: 오타 수정
pick ghi9012 feat: 로그인 검증 추가

저장 후 에디터가 다시 열리면 합쳐진 커밋의 메시지를 작성합니다.

feat: 로그인 폼 추가

로그인 폼 컴포넌트를 구현했습니다.
오타를 수정했습니다.

커밋 메시지 수정 (Reword)

pick abc1234 feat: 로그인 폼 추가
reword def5678 fix: 오타 수정
pick ghi9012 feat: 로그인 검증 추가

저장 후 해당 커밋의 메시지를 수정할 수 있는 에디터가 열립니다.

커밋 삭제 (Drop)

pick abc1234 feat: 로그인 폼 추가
drop def5678 fix: 오타 수정
pick ghi9012 feat: 로그인 검증 추가

또는 해당 줄을 삭제하는 것으로도 같은 효과를 얻습니다.

실전 예제: PR 전 커밋 정리

# 작업 중 만든 5개의 자잘한 커밋을 2개로 정리
$ git log --oneline
abc1234 feat: 로그인 검증 추가
def5678 fix: 로그인 검증 오타
ghi9012 feat: 로그인 폼 추가
jkl3456 WIP: 로그인 폼 작업 중
mno7890 fix: 로그인 폼 스타일 수정

$ git rebase -i HEAD~5

에디터에서:

pick mno7890 fix: 로그인 폼 스타일 수정
squash jkl3456 WIP: 로그인 폼 작업 중
squash ghi9012 feat: 로그인 폼 추가
pick def5678 fix: 로그인 검증 오타
squash abc1234 feat: 로그인 검증 추가

결과적으로 깔끔한 2개의 커밋으로 정리됩니다.

feat: 로그인 폼 컴포넌트 구현
feat: 로그인 입력 검증 추가

팁: git rebase -i는 push 전 로컬 커밋을 정리하는 데 매우 유용합니다. 리뷰어가 깔끔한 히스토리를 볼 수 있습니다.

리베이스 vs 머지: 언제 무엇을 쓸 것인가¶

merge를 사용해야 할 때

  • 공유 브랜치(main, develop)에 병합할 때
  • 협업 중인 브랜치를 업데이트할 때
  • 병합 시점을 명확히 기록하고 싶을 때
  • 안전하게 작업하고 싶을 때 (rebase는 히스토리 재작성)

rebase를 사용해야 할 때

  • 로컬 작업 브랜치를 최신 main으로 갱신할 때
  • PR 전 커밋 히스토리를 정리할 때
  • 선형 히스토리를 선호할 때
  • 아직 push하지 않은 커밋을 수정할 때

비교 테이블

구분 merge rebase
히스토리 브랜치 구조 유지 선형 히스토리
커밋 해시 변경 없음 변경됨
안전성 안전 (히스토리 보존) 위험 (히스토리 재작성)
충돌 해결 한 번만 커밋마다 가능
공유 브랜치 안전 위험
사용 난이도 쉬움 어려움

베스트 프랙티스: "로컬 작업은 rebase, 공유 작업은 merge"를 원칙으로 삼으세요.

리베이스의 황금 규칙: 공유 브랜치에서 하지 말 것¶

황금 규칙: 이미 push하여 다른 사람과 공유한 커밋을 rebase하지 마세요.

왜 위험한가

rebase는 커밋 해시를 변경합니다. 다른 팀원이 이미 해당 커밋을 기반으로 작업했다면, rebase 후 히스토리가 달라져 충돌과 혼란이 발생합니다.

위험한 예시

# main 브랜치를 rebase (절대 금지!)
$ git switch main
$ git rebase feature/new-feature  # 위험!
$ git push --force  # 팀원의 작업이 날아갑니다!

안전한 예시

# 로컬 작업 브랜치만 rebase (안전)
$ git switch feature/my-work
$ git rebase main  # 안전 (아직 push 안 함)
$ git push  # 첫 push라면 안전

이미 push한 브랜치를 rebase한 경우

$ git rebase main
$ git push
# 출력: error: failed to push some refs
# hint: Updates were rejected because the tip of your current branch is behind

force push가 필요합니다.

$ git push --force-with-lease

하지만 다른 팀원이 같은 브랜치에서 작업 중이라면 문제가 발생합니다.

경고: main, develop 등 공유 브랜치에 force push는 절대 금지입니다. 브랜치 보호 규칙으로 차단하세요.

절 요약¶

  • rebase는 커밋을 다른 베이스 위에 재적용하여 선형 히스토리를 만듭니다.
  • 인터랙티브 rebase로 커밋을 합치고, 메시지를 수정하고, 순서를 변경할 수 있습니다.
  • 로컬 작업 브랜치에서는 rebase, 공유 브랜치에서는 merge를 사용합니다.
  • 이미 push한 커밋을 rebase하면 팀원에게 문제가 발생하므로 주의해야 합니다.

특정 커밋 가져오기: cherry-pick (ch_07_s02)¶

도입¶

브랜치 A에 있는 특정 커밋 하나만 브랜치 B로 가져오고 싶다면 어떻게 할까요? merge는 모든 커밋을 가져오고, rebase는 브랜치 전체를 재작성합니다. 이럴 때 필요한 것이 git cherry-pick입니다. 뷔페에서 원하는 음식만 골라 담듯이, 필요한 커밋만 선택적으로 가져올 수 있습니다.

핵심 개념 설명¶

cherry-pick의 개념과 사용 시나리오¶

Cherry-pick은 다른 브랜치의 특정 커밋을 현재 브랜치에 복사하여 적용하는 명령어입니다.

뷔페에서 원하는 음식만 골라 담는 것과 같습니다. 전체 메뉴(브랜치)를 다 가져오는 것이 아니라, 필요한 요리(커밋)만 선택합니다.

사용 시나리오

  1. 핫픽스 적용: 버그 수정 커밋을 여러 릴리스 브랜치에 적용
  2. 특정 기능만 가져오기: 실험 브랜치에서 성공한 기능만 main에 적용
  3. 실수 복구: 잘못 병합한 브랜치에서 특정 커밋만 구출
  4. 커밋 이동: 잘못된 브랜치에 커밋한 것을 올바른 브랜치로 이동

주의: cherry-pick은 커밋을 '복사'하는 것이며, 원본 커밋은 그대로 남아있습니다.

단일 커밋 cherry-pick¶

기본 사용법

# 현재 브랜치에 특정 커밋을 적용
$ git cherry-pick <커밋해시>

실습 예제

# 현재 상황 확인
$ git log --oneline --all --graph
* abc1234 (hotfix) fix: 로그인 오류 수정
* def5678 (main) feat: 대시보드 추가
* ghi9012 feat: 사용자 관리 추가

# main 브랜치로 전환
$ git switch main

# hotfix 브랜치의 수정 사항만 가져오기
$ git cherry-pick abc1234
# 출력:
# [main jkl3456] fix: 로그인 오류 수정
#  Date: ...
#  1 file changed, 2 insertions(+), 1 deletion(-)

# 결과 확인
$ git log --oneline
jkl3456 (HEAD -> main) fix: 로그인 오류 수정
def5678 feat: 대시보드 추가
ghi9012 feat: 사용자 관리 추가

abc1234와 jkl3456는 내용은 같지만 해시가 다른 별개의 커밋입니다.

실전 시나리오: 릴리스 브랜치에 핫픽스 적용

# production에서 버그 발견
$ git switch main
$ git switch -c hotfix/login-bug

# 버그 수정
$ git add src/login.js
$ git commit -m "fix: 로그인 시 null 체크 추가"

# main에 병합
$ git switch main
$ git merge hotfix/login-bug

# 이전 릴리스 브랜치에도 적용
$ git switch release/v1.0
$ git cherry-pick <핫픽스커밋해시>

$ git switch release/v2.0
$ git cherry-pick <핫픽스커밋해시>

팁: 커밋 해시는 git log --oneline이나 GitHub 웹 UI에서 확인할 수 있습니다.

여러 커밋 cherry-pick¶

연속된 범위 cherry-pick

# A부터 B까지의 모든 커밋 적용 (A는 제외, B는 포함)
$ git cherry-pick A..B

# A부터 B까지 모두 포함
$ git cherry-pick A^..B

불연속 커밋 여러 개

$ git cherry-pick abc1234 def5678 ghi9012

예제

# feature 브랜치의 3개 커밋을 main에 적용
$ git log --oneline feature
abc1234 feat: 검색 기능 추가
def5678 feat: 정렬 기능 추가
ghi9012 feat: 필터 기능 추가

$ git switch main
$ git cherry-pick def5678 ghi9012
# 검색 기능은 제외하고 정렬, 필터만 가져옴

베스트 프랙티스: 커밋 해시 대신 브랜치 이름을 사용할 수도 있습니다. git cherry-pick feature~2 (feature의 2번째 이전 커밋)

cherry-pick 충돌 해결¶

충돌 발생 시

$ git cherry-pick abc1234
# 출력:
# error: could not apply abc1234... feat: 검색 기능 추가
# CONFLICT (content): Merge conflict in src/search.js
# hint: after resolving the conflicts, mark the corrected paths
# hint: with 'git add <paths>' or 'git rm <paths>'
# hint: and commit the result with 'git commit'

해결 단계

# 1. 충돌 파일 수정
$ code src/search.js

# 2. 해결 표시
$ git add src/search.js

# 3. cherry-pick 계속 진행
$ git cherry-pick --continue

cherry-pick 중단

$ git cherry-pick --abort
# cherry-pick을 취소하고 이전 상태로 되돌립니다

커밋 메시지 수정

# cherry-pick 시 커밋 메시지를 편집하려면
$ git cherry-pick abc1234 --edit

팁: 충돌이 자주 발생한다면 해당 커밋이 의존성이 많은 것입니다. 범위를 넓혀 여러 커밋을 함께 가져오는 것을 고려하세요.

cherry-pick 주의사항: 중복 커밋¶

중복 커밋 문제

cherry-pick한 커밋을 나중에 merge하면 같은 변경이 두 번 적용될 수 있습니다.

# 상황: hotfix 커밋을 cherry-pick
$ git switch main
$ git cherry-pick abc1234

# 나중에 hotfix 브랜치 전체를 merge
$ git merge hotfix
# abc1234의 내용이 중복됩니다

Git은 대부분 중복을 감지하지만, 복잡한 경우 충돌이나 중복 적용이 발생할 수 있습니다.

해결 방법

  1. cherry-pick 후 원본 브랜치 삭제: 중복 방지
  2. merge --strategy-option theirs: 충돌 시 한쪽 우선
  3. 문서화: 어떤 커밋을 cherry-pick했는지 기록

좋은 커밋 메시지 예시

fix: 로그인 시 null 체크 추가 (cherry-picked from abc1234)

원본 커밋: abc1234 (hotfix/login-bug)
릴리스 v1.0, v2.0에도 적용됨

베스트 프랙티스: cherry-pick은 임시방편입니다. 자주 필요하다면 브랜치 전략을 재검토하세요.

절 요약¶

  • cherry-pick은 다른 브랜치의 특정 커밋을 현재 브랜치에 복사합니다.
  • 핫픽스를 여러 브랜치에 적용하거나 특정 기능만 선택적으로 가져올 때 유용합니다.
  • 충돌 발생 시 수정 후 --continue로 진행하거나 --abort로 중단합니다.
  • 중복 커밋을 방지하기 위해 cherry-pick 이력을 문서화하는 것이 좋습니다.

작업 임시 저장: stash (ch_07_s03)¶

도입¶

작업 중인데 갑자기 긴급한 버그를 수정해야 한다면 어떻게 할까요? 현재 변경 사항을 커밋하기는 아직 이르고, 그렇다고 버리기는 아깝습니다. 이럴 때 git stash로 작업을 임시 저장하고 나중에 다시 꺼내 쓸 수 있습니다. 책상 위 서류를 급히 서랍에 넣어두고 다른 일을 처리하는 것과 같습니다.

핵심 개념 설명¶

stash의 개념: 임시 서랍 비유¶

Stash는 현재 작업 디렉토리와 스테이징 영역의 변경 사항을 임시로 저장하고, 작업 디렉토리를 깨끗한 상태로 만드는 명령어입니다.

책상에서 작업 중인 서류를 급히 서랍에 넣어두는 것과 같습니다. 급한 일을 처리한 후 서랍에서 서류를 꺼내 다시 작업을 이어갈 수 있습니다.

사용 시나리오

  1. 긴급 작업 처리: 작업 중 핫픽스를 해야 할 때
  2. 브랜치 전환: 커밋하기 애매한 상태에서 다른 브랜치로 이동
  3. 실험적 변경: 변경을 임시 보관하고 다른 방법 시도
  4. 충돌 회피: pull 전 로컬 변경을 임시 저장

팁: stash는 커밋 히스토리에 기록되지 않는 임시 저장소입니다.

git stash push: 변경 사항 임시 저장¶

기본 사용법

$ git stash
# 또는
$ git stash push

실습 예제

# 현재 작업 중
$ git status
# 출력:
# Changes not staged for commit:
#   modified:   src/feature.js
#
# Untracked files:
#   src/new-file.js

# 긴급 작업을 위해 임시 저장
$ git stash push -m "기능 A 작업 중"
# 출력:
# Saved working directory and index state On main: 기능 A 작업 중

# 작업 디렉토리가 깨끗해짐
$ git status
# 출력:
# On branch main
# nothing to commit, working tree clean

# 긴급 작업 진행
$ git switch hotfix
# 버그 수정...
$ git add .
$ git commit -m "fix: 긴급 버그 수정"
$ git switch main

# 이전 작업 복원
$ git stash pop
# 출력:
# On branch main
# Changes not staged for commit:
#   modified:   src/feature.js

메시지와 함께 저장

$ git stash push -m "사용자 인증 기능 작업 중"

메시지를 추가하면 나중에 어떤 작업인지 쉽게 파악할 수 있습니다.

특정 파일만 stash

$ git stash push -m "로그인 폼만 임시 저장" src/login.js

Untracked 파일 포함

$ git stash push -u
# 또는
$ git stash push --include-untracked

기본적으로 stash는 추적 중인 파일만 저장합니다. -u 옵션으로 새 파일도 포함할 수 있습니다.

베스트 프랙티스: stash에 의미 있는 메시지를 추가하세요. 나중에 어떤 작업인지 기억하기 어렵습니다.

git stash pop vs apply: 복원 방법¶

pop: 적용 후 삭제

$ git stash pop

stash를 적용하고 stash 목록에서 제거합니다. 한 번만 사용할 작업에 적합합니다.

apply: 적용하지만 유지

$ git stash apply

stash를 적용하지만 stash 목록에 남겨둡니다. 여러 브랜치에 같은 변경을 적용하거나, 백업 목적으로 유용합니다.

차이 비교

# stash 목록 확인
$ git stash list
stash@{0}: On main: 기능 A 작업 중
stash@{1}: On main: 기능 B 작업 중

# pop 사용
$ git stash pop
# stash@{0}이 적용되고 목록에서 사라짐

$ git stash list
stash@{0}: On main: 기능 B 작업 중  # 번호가 당겨짐

# apply 사용
$ git stash apply
# stash@{0}이 적용되지만 목록에 남음

$ git stash list
stash@{0}: On main: 기능 B 작업 중  # 그대로 유지

충돌 발생 시

$ git stash pop
# 출력:
# CONFLICT (content): Merge conflict in src/feature.js
# The stash entry is kept in case you need it again.

pop 시 충돌이 발생하면 stash가 자동으로 보존됩니다. 충돌을 해결한 후 수동으로 삭제해야 합니다.

# 충돌 해결 후
$ git add src/feature.js
$ git stash drop  # 수동으로 삭제

팁: 안전을 위해 처음에는 apply를 사용하고, 확인 후 drop으로 삭제하는 것이 좋습니다.

git stash list와 특정 stash 적용¶

stash 목록 확인

$ git stash list
stash@{0}: On main: 기능 A 작업 중
stash@{1}: WIP on feature: abc1234 커밋 메시지
stash@{2}: On main: 기능 C 작업 중

stash@{n} 형식으로 각 stash에 인덱스가 부여됩니다. 0이 가장 최근, 숫자가 클수록 오래된 stash입니다.

특정 stash 적용

# 두 번째 stash 적용 (apply)
$ git stash apply stash@{1}

# 세 번째 stash 적용 후 삭제 (pop)
$ git stash pop stash@{2}

stash 내용 미리 보기

# stash@{0}의 변경 사항 확인
$ git stash show stash@{0}
# 출력: (파일별 변경 통계)
#  src/feature.js | 10 +++++-----
#  1 file changed, 5 insertions(+), 5 deletions(-)

# 상세 diff 보기
$ git stash show -p stash@{0}

특정 stash 삭제

$ git stash drop stash@{1}

모든 stash 삭제

$ git stash clear
# 경고: 복구할 수 없습니다!

주의: git stash clear는 모든 stash를 영구 삭제합니다. 신중히 사용하세요.

git stash branch: stash에서 브랜치 만들기¶

stash에서 새 브랜치 생성

stash를 적용하다가 충돌이 자주 발생한다면, stash를 새 브랜치로 만드는 것이 편리합니다.

$ git stash branch <새브랜치명> stash@{n}

이 명령은: 1. stash가 생성된 커밋에서 새 브랜치 생성 2. stash 내용 적용 3. 성공 시 stash 삭제

예제

# stash 생성
$ git stash push -m "실험적 기능"

# 다른 작업...

# stash를 새 브랜치로 전환
$ git stash branch feature/experiment stash@{0}
# 출력:
# Switched to a new branch 'feature/experiment'
# On branch feature/experiment
# Changes not staged for commit:
#   modified:   src/experiment.js
# Dropped stash@{0} (abc1234...)

충돌 없이 깔끔하게 작업을 이어갈 수 있습니다.

베스트 프랙티스: stash가 오래되어 충돌 가능성이 높다면 stash branch로 독립 브랜치를 만들어 작업하세요.

절 요약¶

  • stash는 작업 디렉토리의 변경 사항을 임시 저장하는 명령어입니다.
  • git stash push로 저장하고 git stash pop 또는 apply로 복원합니다.
  • pop은 적용 후 삭제, apply는 적용 후 유지합니다.
  • git stash list로 목록을 확인하고 특정 stash를 선택적으로 적용할 수 있습니다.
  • git stash branch로 stash를 새 브랜치로 전환할 수 있습니다.

실수 복구의 최후 보루: reflog (ch_07_s04)¶

도입¶

git reset --hard로 중요한 커밋을 날려버렸다면 어떻게 할까요? 브랜치를 실수로 삭제했거나 rebase가 잘못되어 작업이 사라졌다면? Git에는 이런 절망적인 상황을 구해주는 안전망이 있습니다. 바로 git reflog입니다. 웹 브라우저의 방문 기록처럼, Git은 모든 HEAD 이동을 기록하여 실수를 복구할 수 있게 합니다.

핵심 개념 설명¶

reflog란: Git의 안전망¶

Reflog(Reference Log)는 HEAD와 브랜치 참조의 모든 이동 히스토리를 시간순으로 기록하는 로컬 로그입니다.

웹 브라우저의 방문 기록(History)과 같습니다. 실수로 탭을 닫아도 방문 기록에서 다시 찾아갈 수 있듯이, reflog에서 이전 상태를 찾아 복구할 수 있습니다.

reflog의 특징

  • 로컬 전용: 원격 저장소에는 전송되지 않음
  • 시간 제한: 기본 90일 후 자동 삭제 (설정 변경 가능)
  • 모든 이동 기록: commit, reset, rebase, checkout 등 모든 HEAD 이동

기본 사용법

$ git reflog
abc1234 (HEAD -> main) HEAD@{0}: commit: feat: 사용자 프로필 추가
def5678 HEAD@{1}: reset: moving to HEAD~1
ghi9012 HEAD@{2}: commit: fix: 버그 수정
jkl3456 HEAD@{3}: checkout: moving from feature to main
mno7890 HEAD@{4}: commit: feat: 로그인 기능 추가

각 항목은 HEAD@{n} 형식으로 참조할 수 있습니다.

팁: reflog는 "시간 여행 지도"입니다. 과거 어느 시점으로든 돌아갈 수 있습니다.

git reflog로 히스토리 추적하기¶

상세 로그 보기

$ git reflog show --all
# 모든 브랜치의 reflog 표시

날짜 포함 보기

$ git reflog show --date=relative
# 출력:
# abc1234 HEAD@{2 hours ago}: commit: feat: 프로필 추가
# def5678 HEAD@{1 day ago}: reset: moving to HEAD~1

특정 브랜치 reflog

$ git reflog show feature/login
# feature/login 브랜치의 reflog만 표시

특정 시점의 상태 확인

# 3번째 이전 상태의 diff 보기
$ git diff HEAD@{3}

# 5번째 이전 상태에서 특정 파일 보기
$ git show HEAD@{5}:src/app.js

베스트 프랙티스: 중요한 작업 전 git reflog로 현재 상태를 기록해두세요. 복구 시 참고점이 됩니다.

잘못된 reset/rebase 되돌리기¶

시나리오: reset --hard로 커밋 날림

# 현재 커밋 히스토리
$ git log --oneline
abc1234 (HEAD -> main) feat: 중요한 기능 완성
def5678 feat: 기능 작업 중
ghi9012 feat: 초기 구현

# 실수로 2개 커밋 삭제
$ git reset --hard HEAD~2
# 출력: HEAD is now at ghi9012 feat: 초기 구현

$ git log --oneline
ghi9012 (HEAD -> main) feat: 초기 구현
# abc1234, def5678이 사라짐!

reflog로 복구

# reflog 확인
$ git reflog
ghi9012 (HEAD -> main) HEAD@{0}: reset: moving to HEAD~2
abc1234 HEAD@{1}: commit: feat: 중요한 기능 완성
def5678 HEAD@{2}: commit: feat: 기능 작업 중

# 삭제 전 상태로 복구
$ git reset --hard HEAD@{1}
# 출력: HEAD is now at abc1234 feat: 중요한 기능 완성

$ git log --oneline
abc1234 (HEAD -> main) feat: 중요한 기능 완성
def5678 feat: 기능 작업 중
ghi9012 feat: 초기 구현
# 복구 완료!

시나리오: rebase 실패 복구

# rebase 시도
$ git rebase main
# 출력: 충돌 발생, 여러 번 해결 시도 후 엉망이 됨

# reflog로 rebase 전 상태 찾기
$ git reflog
abc1234 HEAD@{0}: rebase: feat: 기능 B
def5678 HEAD@{1}: rebase: feat: 기능 A
ghi9012 HEAD@{2}: rebase (start): checkout main
jkl3456 HEAD@{3}: commit: feat: 기능 B  # rebase 시작 전

# rebase 시작 전으로 복구
$ git reset --hard HEAD@{3}

주의: reset --hard는 작업 디렉토리의 변경 사항도 삭제합니다. 저장되지 않은 작업은 사라집니다.

삭제된 브랜치 복구하기¶

시나리오: 브랜치 실수로 삭제

# feature 브랜치 삭제
$ git branch -D feature/important
# 출력: Deleted branch feature/important (was abc1234).

# 앗, 병합하지 않은 브랜치였다!

reflog로 복구

# reflog에서 브랜치 마지막 커밋 찾기
$ git reflog show --all | grep important
abc1234 HEAD@{10}: commit: feat: 중요한 기능 완성 (on feature/important)

# 해당 커밋에서 브랜치 재생성
$ git switch -c feature/important abc1234
# 또는
$ git branch feature/important abc1234

# 복구 완료!
$ git log --oneline feature/important
abc1234 (HEAD -> feature/important) feat: 중요한 기능 완성

다른 방법: fsck로 떠도는 커밋 찾기

# 어떤 브랜치에도 속하지 않는 커밋 찾기
$ git fsck --lost-found
# dangling commit abc1234

$ git show abc1234
# 내용 확인 후 브랜치 생성
$ git branch recovered abc1234

팁: 브랜치를 삭제하기 전 git log 로 확인하고, 의심스러우면 GitHub에 백업 push를 권장합니다.

reflog의 유효 기간과 한계¶

reflog 보관 기간

기본 설정: - 도달 가능한 항목(reachable): 90일 - 도달 불가능한 항목(unreachable): 30일

# 설정 확인
$ git config gc.reflogExpire
# 출력: 90 (일 단위)

$ git config gc.reflogExpireUnreachable
# 출력: 30

# 설정 변경 (365일로)
$ git config gc.reflogExpire 365

수동 가비지 컬렉션

# Git 저장소 정리 (오래된 reflog 삭제 포함)
$ git gc
# 출력: Counting objects: ..., done.

# 즉시 만료된 reflog 삭제
$ git reflog expire --expire=now --all

reflog의 한계

  1. 로컬 전용: 원격 저장소에는 없음
  2. 시간 제한: 기본 30~90일 후 삭제
  3. 저장되지 않은 변경: 커밋하지 않은 작업은 reflog에 없음
  4. 다른 컴퓨터: clone한 저장소에는 reflog가 비어있음

완전한 데이터 손실 시나리오

# 파일 수정 후 커밋하지 않음
$ echo "중요한 코드" > important.js

# reset --hard로 날림
$ git reset --hard HEAD
# important.js의 변경 사항 영구 손실 (reflog에도 없음)

경고: reflog는 커밋된 작업만 복구할 수 있습니다. 커밋하지 않은 변경은 reset --hard로 날아가면 복구 불가능합니다.

절 요약¶

  • reflog는 HEAD 이동 히스토리를 기록하는 Git의 안전망입니다.
  • git reflog로 과거 상태를 확인하고 git reset --hard HEAD@{n}으로 복구합니다.
  • 잘못된 reset, rebase, 브랜치 삭제 등을 복구할 수 있습니다.
  • reflog는 로컬 전용이며 기본 90일 후 자동 삭제됩니다.
  • 커밋하지 않은 변경은 reflog로도 복구할 수 없으므로 자주 커밋하는 습관이 중요합니다.

버그 찾기와 버전 표시: bisect와 tag (ch_07_s05)¶

도입¶

프로젝트가 커지면 언제부터 버그가 생겼는지 찾기 어렵습니다. 100개의 커밋을 하나씩 확인하는 것은 비효율적입니다. 이럴 때 git bisect는 이진 탐색 알고리즘으로 버그 커밋을 빠르게 찾아줍니다. 또한 릴리스 버전을 git tag로 표시하여 특정 시점을 명확히 기록하는 방법도 배웁니다.

핵심 개념 설명¶

git bisect: 이진 탐색으로 버그 커밋 찾기¶

Bisect는 이진 탐색 알고리즘을 이용하여 버그를 도입한 커밋을 효율적으로 찾아내는 디버깅 도구입니다.

사전에서 단어를 찾는 것과 같습니다. 중간 페이지를 펼쳐 찾는 단어보다 앞인지 뒤인지 확인하고, 범위를 반씩 줄여나가면 빠르게 찾을 수 있습니다.

효율성

  • 선형 탐색: 100개 커밋이면 평균 50번 테스트
  • 이진 탐색: 100개 커밋이면 최대 7번 테스트 (log₂100 ≈ 6.6)

사용 시나리오

  1. 이전 릴리스에는 없던 버그가 현재 버전에 존재
  2. 버그가 언제 도입되었는지 모름
  3. 수많은 커밋 중 문제 커밋을 찾아야 함

팁: bisect는 회귀 테스트(Regression Testing)에 매우 유용합니다.

bisect 실습: good/bad 마킹¶

기본 워크플로우

# 1. bisect 시작
$ git bisect start

# 2. 현재 버전이 나쁨(버그 있음)을 표시
$ git bisect bad

# 3. 버그가 없던 과거 커밋 표시 (예: v1.0 태그)
$ git bisect good v1.0
# 출력:
# Bisecting: 50 revisions left to test after this (roughly 6 steps)
# [abc1234] 중간 커밋으로 자동 이동

테스트 및 마킹 반복

# 현재 커밋에서 버그 테스트 (수동 또는 자동)
$ npm test

# 버그 있음
$ git bisect bad
# 출력:
# Bisecting: 25 revisions left to test after this (roughly 5 steps)

# 또는 버그 없음
$ git bisect good
# 출력:
# Bisecting: 12 revisions left to test after this (roughly 4 steps)

결과 확인

# 반복 후 최종 결과
$ git bisect bad
# 출력:
# abc1234 is the first bad commit
# commit abc1234
# Author: John Doe <john@example.com>
# Date:   Mon Jan 15 14:23:45 2024 +0900
#
#     refactor: 사용자 인증 로직 변경

bisect 종료

$ git bisect reset
# 출력: Previous HEAD position was abc1234... refactor: ...
# Switched to branch 'main'

실전 예제

# 시나리오: 로그인이 어느 시점부터 작동하지 않음

$ git bisect start
$ git bisect bad  # 현재 로그인 안 됨
$ git bisect good v2.0  # v2.0에서는 로그인 됨

# 자동으로 중간 커밋으로 이동
$ npm run dev
# 브라우저에서 로그인 테스트

# 로그인 실패
$ git bisect bad

# 다시 중간 커밋으로 이동
$ npm run dev
# 로그인 성공
$ git bisect good

# 반복...

# 최종 결과
# def5678 is the first bad commit
# commit def5678
#     refactor: JWT 검증 로직 수정

$ git bisect reset
$ git show def5678  # 버그 커밋 확인

베스트 프랙티스: 각 커밋이 독립적으로 실행 가능하도록 작은 단위로 커밋하면 bisect가 효과적입니다.

자동화된 bisect¶

테스트 스크립트로 자동화

수동 테스트 대신 스크립트를 실행하여 자동으로 good/bad를 판단할 수 있습니다.

$ git bisect start
$ git bisect bad
$ git bisect good v1.0

# 테스트 스크립트 실행 (종료 코드 0: good, 1-127(125 제외): bad)
$ git bisect run npm test

Git은 각 커밋에서 npm test를 실행하고 결과에 따라 자동으로 good/bad를 마킹합니다.

커스텀 테스트 스크립트

#!/bin/bash
# test-login.sh

npm run build
npm run dev &
sleep 5

# 로그인 테스트 (예: curl로 API 호출)
response=$(curl -s -X POST http://localhost:3000/login \
  -d '{"username":"test","password":"test"}')

if echo "$response" | grep -q "token"; then
  exit 0  # good
else
  exit 1  # bad
fi

$ chmod +x test-login.sh
$ git bisect run ./test-login.sh

팁: 자동화 스크립트는 빠르고 신뢰할 수 있어야 합니다. 각 커밋마다 실행되므로 시간이 오래 걸리면 전체 bisect가 느려집니다.

git tag: 릴리스 버전 표시¶

Tag는 특정 커밋에 이름표를 붙여 중요한 시점(릴리스, 마일스톤)을 표시하는 기능입니다.

Lightweight Tag vs Annotated Tag

구분 Lightweight Tag Annotated Tag
생성 git tag v1.0 git tag -a v1.0 -m "메시지"
메타데이터 커밋 참조만 작성자, 날짜, 메시지, GPG 서명
사용 용도 임시 마커 공식 릴리스
권장 여부 비권장 권장

Annotated Tag 생성

$ git tag -a v1.0.0 -m "첫 번째 안정 릴리스"
# 출력: (없음, 조용히 생성됨)

# 태그 확인
$ git tag
v1.0.0

# 태그 상세 정보
$ git show v1.0.0
# 출력:
# tag v1.0.0
# Tagger: John Doe <john@example.com>
# Date:   Mon Jan 15 14:00:00 2024 +0900
#
# 첫 번째 안정 릴리스
#
# commit abc1234...

과거 커밋에 태그 추가

$ git tag -a v0.9.0 def5678 -m "베타 릴리스"

태그 목록

# 모든 태그
$ git tag
v0.9.0
v1.0.0
v1.1.0

# 패턴 필터
$ git tag -l "v1.*"
v1.0.0
v1.1.0

태그 push

# 단일 태그 push
$ git push origin v1.0.0

# 모든 태그 push
$ git push origin --tags

태그로 체크아웃

$ git checkout v1.0.0
# 출력: You are in 'detached HEAD' state...

태그는 특정 커밋을 가리키므로 브랜치가 아닙니다. 수정이 필요하면 새 브랜치를 만드세요.

$ git checkout -b hotfix/v1.0.1 v1.0.0

태그 삭제

# 로컬 태그 삭제
$ git tag -d v0.9.0

# 원격 태그 삭제
$ git push origin --delete v0.9.0

베스트 프랙티스: 모든 공식 릴리스에 Annotated Tag를 사용하고, Lightweight Tag는 개인적인 임시 마커로만 사용하세요.

Semantic Versioning(SemVer)과 태그¶

Semantic Versioning (SemVer)

버전 번호를 MAJOR.MINOR.PATCH 형식으로 관리하는 표준입니다.

v1.2.3
│ │ │
│ │ └─ PATCH: 버그 수정 (하위 호환)
│ └─── MINOR: 기능 추가 (하위 호환)
└───── MAJOR: 호환성 깨지는 변경

버전 증가 규칙

  1. MAJOR: API 변경으로 하위 호환성이 깨질 때
  2. v1.5.3 → v2.0.0
  3. MINOR: 하위 호환되는 새 기능 추가
  4. v1.5.3 → v1.6.0
  5. PATCH: 하위 호환되는 버그 수정
  6. v1.5.3 → v1.5.4

Pre-release 버전

v1.0.0-alpha
v1.0.0-beta.1
v1.0.0-rc.2

예제

# 첫 릴리스
$ git tag -a v1.0.0 -m "첫 번째 안정 릴리스"

# 버그 수정
$ git tag -a v1.0.1 -m "로그인 버그 수정"

# 새 기능 추가
$ git tag -a v1.1.0 -m "사용자 프로필 기능 추가"

# 호환성 깨지는 변경
$ git tag -a v2.0.0 -m "API v2로 전환 (Breaking Change)"

GitHub Release와 연동

# 태그 생성 및 push
$ git tag -a v1.2.0 -m "릴리스 1.2.0"
$ git push origin v1.2.0

# GitHub에서 Release 생성
$ gh release create v1.2.0 --title "v1.2.0" --notes "버그 수정 및 성능 개선"

베스트 프랙티스: 프로젝트 초기부터 SemVer를 도입하고 일관되게 적용하세요. 사용자가 버전 번호만으로 호환성을 판단할 수 있습니다.

절 요약¶

  • git bisect는 이진 탐색으로 버그를 도입한 커밋을 빠르게 찾습니다.
  • good/bad 마킹을 반복하며 범위를 좁혀나가고, 테스트 스크립트로 자동화할 수 있습니다.
  • git tag로 릴리스 버전을 표시하며, Annotated Tag가 권장됩니다.
  • Semantic Versioning(MAJOR.MINOR.PATCH)으로 일관된 버전 관리를 할 수 있습니다.

FAQ¶

Q1: rebase 중 충돌이 너무 많이 발생합니다. 어떻게 해야 하나요?

A1: git rebase --abort로 중단하고 git merge를 사용하세요. 충돌이 많다면 브랜치가 오래되어 main과 차이가 크다는 뜻입니다. 향후에는 자주 main을 rebase하거나 merge하여 차이를 줄이세요. 또는 인터랙티브 rebase로 커밋을 squash하여 충돌 횟수를 줄일 수 있습니다.

Q2: cherry-pick한 커밋을 나중에 merge하면 문제가 생기나요?

A2: Git은 대부분 중복을 자동으로 처리하지만, 복잡한 경우 충돌이 발생할 수 있습니다. cherry-pick한 커밋을 커밋 메시지에 기록하고 (cherry-picked from abc1234), 가능하면 원본 브랜치를 빨리 병합하거나 삭제하여 중복을 방지하세요.

Q3: stash를 오래 보관해도 되나요?

A3: stash는 임시 저장소이므로 오래 보관하지 마세요. reflog처럼 시간 제한은 없지만, 프로젝트 상태가 변하면 충돌 가능성이 높아집니다. stash가 1주일 이상 필요하다면 브랜치를 만들어 커밋하는 것이 안전합니다.

Q4: reflog로 복구할 수 없는 경우는 언제인가요?

A4: 1) 커밋하지 않은 변경은 reflog에 기록되지 않습니다. 2) 90일(기본값)이 지나면 자동 삭제됩니다. 3) git gc 후 도달 불가능한 커밋은 30일 후 삭제됩니다. 4) 다른 컴퓨터에서 clone한 저장소에는 reflog가 없습니다. 중요한 작업은 반드시 커밋하고 원격에 push하세요.

Q5: bisect 중 빌드가 실패하는 커밋이 나오면 어떻게 하나요?

A5: git bisect skip으로 해당 커밋을 건너뛸 수 있습니다. Git이 다른 커밋을 선택합니다. 하지만 skip이 많으면 bisect 효율이 떨어지므로, 각 커밋이 빌드 가능하도록 작은 단위로 커밋하는 습관이 중요합니다.

Q6: 태그와 브랜치의 차이는 무엇인가요?

A6: 브랜치는 커밋이 추가되면 자동으로 이동하는 '움직이는 포인터'입니다. 태그는 특정 커밋을 가리키는 '고정된 포인터'로 이동하지 않습니다. 브랜치는 개발 작업용, 태그는 릴리스 마킹용입니다.

Q7: 로컬에서 인터랙티브 rebase한 후 push가 거부됩니다.

A7: rebase는 커밋 해시를 변경하므로 이미 push한 브랜치를 rebase하면 히스토리가 달라져 push가 거부됩니다. git push --force-with-lease를 사용할 수 있지만, 다른 팀원이 같은 브랜치를 사용 중이라면 문제가 발생합니다. 로컬에서만 작업한 브랜치에만 rebase를 사용하세요.

Q8: reflog에서 찾은 커밋을 영구 보존하려면?

A8: reflog에서 복구한 커밋을 브랜치나 태그로 만들어 보존하세요.

$ git branch recovered-work HEAD@{5}
$ git tag important-commit abc1234

브랜치나 태그로 참조되면 가비지 컬렉션에서 삭제되지 않습니다.

장 요약¶

이번 장에서는 Git의 강력한 고급 명령어들을 마스터했습니다. git rebase로 커밋 히스토리를 깔끔하게 정리하고 선형으로 만드는 방법을 배웠으며, 인터랙티브 rebase로 커밋을 합치고 메시지를 수정하는 기술을 익혔습니다. 하지만 공유 브랜치에서는 rebase가 위험하다는 황금 규칙도 반드시 기억해야 합니다. git cherry-pick으로 필요한 커밋만 선택적으로 다른 브랜치에 적용하는 방법을 배웠고, 핫픽스를 여러 릴리스 브랜치에 적용하는 실무 시나리오를 연습했습니다. git stash로 작업 중인 변경을 임시 저장하고 나중에 복원하는 기술을 익혔으며, pop과 apply의 차이를 이해했습니다. git reflog는 Git의 안전망으로, 잘못된 reset이나 삭제된 브랜치를 복구하는 강력한 도구임을 배웠습니다. git bisect로 이진 탐색 알고리즘을 활용하여 버그를 도입한 커밋을 효율적으로 찾아내는 디버깅 기법을 마스터했습니다. 마지막으로 git tag로 릴리스 버전을 표시하고 Semantic Versioning을 적용하여 일관된 버전 관리를 하는 방법을 배웠습니다.
