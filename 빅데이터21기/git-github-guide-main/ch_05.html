05장: GitHub 시작하기

기초

GitHub 시작하기¶

학습 목표 - GitHub 계정을 생성하고 2FA를 설정하여 보안을 강화할 수 있다 - GitHub CLI(gh)를 설치하고 인증하여 터미널에서 GitHub를 사용할 수 있다 - 목적에 맞는 저장소를 생성하고 README, LICENSE를 올바르게 설정할 수 있다 - Issue와 Label을 활용하여 작업을 체계적으로 관리할 수 있다 - GitHub Flavored Markdown으로 풍부한 문서를 작성할 수 있다

GitHub 계정과 프로필 설정 (ch_05_s01)¶

도입¶

Git이 버전 관리 도구라면, GitHub는 Git 저장소를 호스팅하고 협업을 지원하는 플랫폼입니다. 전 세계 수천만 개발자가 GitHub에서 코드를 공유하고, 프로젝트를 관리하며, 오픈소스에 기여합니다. 이 절에서는 GitHub 계정을 만들고, 보안을 강화하며, 프로필을 커스터마이징하는 방법을 배웁니다.

핵심 개념 설명¶

GitHub란 무엇인가¶

GitHub는 Git 저장소를 클라우드에 호스팅하고, 협업 도구(Issue, Pull Request, Projects 등)를 제공하는 웹 기반 플랫폼입니다.

GitHub의 핵심 역할

  1. 원격 저장소 호스팅: 로컬 Git 저장소를 클라우드에 백업하고 공유합니다.
  2. 협업 플랫폼: Pull Request, Code Review, Issue Tracking 등 팀 협업 도구를 제공합니다.
  3. 포트폴리오: 개발자의 활동 기록이 프로필에 표시되어 이력서 역할을 합니다.
  4. 오픈소스 중심지: 수백만 개의 오픈소스 프로젝트가 GitHub에서 운영됩니다.

팁: GitHub는 Git을 기반으로 하지만, Git 없이도 웹 UI에서 파일 편집, 커밋, PR 생성 등 대부분의 작업을 수행할 수 있습니다.

GitHub 가입과 2FA(이중 인증) 설정¶

1단계: 계정 생성

https://github.com/signup에 접속하여 무료 계정을 생성합니다.

  • 사용자명(Username): 다른 사람에게 보이는 공개 ID입니다. 신중하게 선택하세요. (나중에 변경 가능하지만 링크가 깨질 수 있습니다.)
  • 이메일: 알림을 받을 주요 이메일 주소입니다.
  • 비밀번호: 최소 15자 또는 8자 + 숫자 + 특수문자 조합을 권장합니다.

2단계: 이메일 인증

가입 시 입력한 이메일로 인증 메일이 전송됩니다. 이메일 내 링크를 클릭하여 계정을 활성화합니다.

3단계: 2FA(이중 인증) 설정

보안을 위해 반드시 2FA를 설정하세요. 비밀번호만으로는 계정 탈취 위험이 있습니다.

Settings → Password and authentication → Enable two-factor authentication

2FA 방식 선택

  1. Authenticator 앱 (권장): Google Authenticator, Authy, 1Password 등
  2. SMS: 휴대폰 문자 메시지 (덜 안전함)

Recovery Codes 저장

2FA 설정 시 제공되는 Recovery Codes는 안전한 곳에 백업하세요. 인증 앱을 분실하면 이 코드가 유일한 복구 수단입니다.

# Recovery Codes 예시 (실제로는 16자리 코드 10개가 생성됨)
12ab-34cd-56ef-78gh
90ij-12kl-34mn-56op
...

주의: Recovery Codes를 스크린샷으로 찍어 클라우드에 저장하지 마세요. 암호화된 비밀번호 관리자(1Password, Bitwarden)나 오프라인 저장소에 보관하세요.

베스트 프랙티스: 2FA는 선택이 아닌 필수입니다. 2024년부터 GitHub는 많은 계정에 2FA를 강제하고 있습니다. 미리 설정하세요.

프로필 페이지 커스터마이징¶

GitHub 프로필은 당신의 개발 이력을 보여주는 명함입니다.

프로필 정보 수정

Settings → Public profile
  • Name: 실명 또는 활동명
  • Bio: 간단한 자기소개 (최대 160자)
  • Company: 소속 회사/조직
  • Location: 위치 (선택)
  • Website: 개인 블로그, 포트폴리오 사이트
  • Social accounts: Twitter, LinkedIn 등 연결

프로필 이미지 설정

프로필 이미지가 없으면 기본 아바타가 표시됩니다. 전문적인 이미지나 개성 있는 아바타를 설정하세요.

프로필 README.md 만들기¶

GitHub의 특별한 기능: 자신의 사용자명과 동일한 이름의 저장소를 만들면, 그 저장소의 README.md 파일이 프로필 페이지에 표시됩니다.

프로필 저장소 생성

사용자명이 johndoe라면:

# 웹 UI에서 저장소 생성
1. New repository 클릭
2. Repository name: johndoe (자신의 사용자명과 동일하게)
3. "This is a special repository..." 메시지 확인
4. Public으로 설정
5. Add a README file 체크
6. Create repository

GitHub CLI(gh) 설치 및 인증¶

GitHub CLI(gh)는 터미널에서 GitHub 기능을 사용할 수 있게 해주는 공식 명령줄 도구로, Issue, PR, Actions 등을 CLI로 관리할 수 있습니다.

은행 앱과 같습니다. 은행 창구(웹 UI)에 가지 않아도 앱(CLI)에서 대부분의 업무를 처리할 수 있습니다.

설치 방법

# macOS (Homebrew)
$ brew install gh

# Windows (winget)
$ winget install GitHub.cli

# Windows (Scoop)
$ scoop install gh

# Linux (Debian/Ubuntu)
$ sudo apt install gh

# Linux (Fedora/RHEL)
$ sudo dnf install gh

설치 확인

$ gh --version
# 출력: gh version 2.40.1 (2024-01-15)

인증

$ gh auth login

인터랙티브 프롬프트가 나타납니다:

? What account do you want to log into? GitHub.com
? What is your preferred protocol for Git operations? HTTPS
? Authenticate Git with your GitHub credentials? Yes
? How would you like to authenticate GitHub CLI? Login with a web browser

! First copy your one-time code: ABCD-1234
Press Enter to open github.com in your browser...

웹 브라우저가 열리면 코드를 입력하고 인증을 완료합니다.

인증 확인

$ gh auth status
# 출력:
# github.com
#   ✓ Logged in to github.com as johndoe
#   ✓ Git operations for github.com configured to use https protocol.
#   ✓ Token: *******************

주요 gh 명령어

# 저장소 생성
$ gh repo create my-project --public --clone

# Issue 목록 보기
$ gh issue list

# Issue 생성
$ gh issue create --title "버그: 로그인 에러" --body "상세 설명..."

# PR 생성
$ gh pr create --title "기능: 로그인 추가" --body "PR 설명..."

# PR 목록
$ gh pr list

# PR 상태 확인
$ gh pr status

베스트 프랙티스: 웹 UI와 CLI를 병행하세요. 단순 조회는 웹이 편하고, 반복 작업은 CLI가 효율적입니다.

절 요약¶

  • GitHub는 Git 저장소를 호스팅하고 협업을 지원하는 플랫폼입니다.
  • 계정 생성 후 2FA(이중 인증)를 반드시 설정하여 보안을 강화합니다.
  • 프로필 README(사용자명과 동일한 저장소)로 개인 페이지를 커스터마이징할 수 있습니다.
  • GitHub CLI(gh)를 설치하면 터미널에서 GitHub를 효율적으로 사용할 수 있습니다.

저장소 생성과 관리 (ch_05_s02)¶

도입¶

GitHub 저장소는 프로젝트의 집입니다. 코드, 문서, 이슈, PR 모두 저장소 안에서 관리됩니다. 이 절에서는 저장소를 생성하고, README와 LICENSE를 올바르게 설정하며, Public/Private를 선택하는 방법을 배웁니다.

핵심 개념 설명¶

새 저장소 만들기: 웹 UI와 gh CLI¶

방법 1: 웹 UI에서 생성

GitHub 우측 상단의 + 아이콘을 클릭하고 New repository를 선택합니다.

저장소 생성 옵션

옵션 설명 권장 설정
Repository name 저장소 이름 (URL에 사용됨) 소문자, 하이픈 사용 (예: my-awesome-project)
Description 저장소 설명 (한 줄 요약) 명확하고 간결하게
Public / Private 공개 여부 오픈소스는 Public, 회사 프로젝트는 Private
Initialize this repository with 초기 파일 설정 -
Add a README file README.md 자동 생성 체크 권장
Add .gitignore .gitignore 템플릿 선택 프로젝트 언어 선택 (예: Python, Node)
Choose a license 라이선스 파일 추가 오픈소스는 MIT 권장

생성 완료 후

# 로컬에 클론
$ git clone https://github.com/johndoe/my-awesome-project.git
$ cd my-awesome-project

방법 2: gh CLI로 생성

# Public 저장소 생성 및 클론
$ gh repo create my-awesome-project --public --clone

# Private 저장소 생성
$ gh repo create my-secret-project --private

# 현재 디렉토리를 GitHub 저장소로 생성
$ cd existing-project
$ gh repo create --source=. --public

팁: 기존 로컬 프로젝트를 GitHub에 올리려면 gh repo create --source=. 옵션이 가장 편리합니다.

README.md 작성 모범 사례¶

README.md는 프로젝트의 첫인상입니다. 방문자가 가장 먼저 보는 문서이므로 명확하고 친절하게 작성해야 합니다.

좋은 README 구조

라이선스 파일 선택 가이드¶

라이선스(License)는 다른 사람이 내 코드를 어떻게 사용할 수 있는지 명시하는 법적 문서입니다.

왜 라이선스가 중요한가

라이선스가 없는 코드는 기본적으로 저작권법의 보호를 받아 타인의 사용이 제한됩니다. 오픈소스로 공개하려면 명시적인 라이선스가 필요합니다.

주요 오픈소스 라이선스 비교

라이선스 자유도 주요 의무사항 권장 대상
MIT 매우 높음 출처 표시만 필요 대부분의 프로젝트, 기업 친화적
Apache 2.0 높음 출처 표시 + 변경 사항 명시 + 특허 보호 기업용, 특허 우려가 있는 프로젝트
GPL v3 중간 수정본도 동일 라이선스로 공개 필수 (Copyleft) 강한 오픈소스 정신, 상업 소프트웨어와 분리
BSD 3-Clause 매우 높음 출처 표시 + 작성자 이름 무단 사용 금지 학술 프로젝트
Unlicense 최고 의무사항 없음 (완전 공개 도메인) 제약 없이 자유롭게 공유

라이선스 선택 가이드

나의 코드를 어떻게 사용하길 원하는가?

  ├─ 최대한 자유롭게 (기업도 상업적으로 사용 가능)
  │   → MIT, Apache 2.0
  │
  ├─ 수정본도 오픈소스로 공개되길 원함
  │   → GPL v3
  │
  └─ 아무 제약 없이 완전 공개
      → Unlicense

라이선스 추가 방법

웹 UI에서 저장소 생성 시 Choose a license 선택하거나, 기존 저장소에 추가:

# GitHub에서 LICENSE 템플릿 추가
1. Add file → Create new file
2. 파일명: LICENSE
3. "Choose a license template" 버튼 클릭
4. MIT License 선택
5. Commit

주의: 라이선스는 한 번 공개하면 되돌리기 어렵습니다. 특히 GPL 같은 Copyleft 라이선스는 신중히 선택하세요.

Public vs Private 저장소¶

Public 저장소

  • 장점: 누구나 볼 수 있음, 무료 무제한, GitHub 검색 노출, 포트폴리오 효과
  • 단점: 코드가 공개됨, 민감 정보 유출 위험
  • 적합한 경우: 오픈소스 프로젝트, 포트폴리오, 학습용 프로젝트

Private 저장소

  • 장점: 본인과 초대한 협업자만 접근, 비공개 개발 가능
  • 단점: GitHub Free는 협업자 제한, 검색 노출 안 됨
  • 적합한 경우: 회사 프로젝트, 비공개 실험, 개인 백업

GitHub Free 플랜 제한

기능 Public Private
저장소 개수 무제한 무제한
협업자 수 무제한 무제한 (2023년부터 제한 해제)
GitHub Actions 월 2,000분 월 2,000분
스토리지 500MB LFS 500MB LFS

저장소 설정: Description, Topics, Website¶

Description (저장소 설명)

저장소 상단에 표시되는 한 줄 요약입니다.

Settings → General → Repository name 아래 Description 입력

예시: "AI 기반 텍스트 감정 분석 라이브러리 - Node.js와 Python 지원"

Topics (주제 태그)

저장소를 분류하는 키워드입니다. GitHub 검색과 Explore에 노출됩니다.

저장소 메인 페이지 → About 섹션 → ⚙️ 아이콘 → Topics 추가

예시 Topics: javascript, sentiment-analysis, machine-learning, nlp

Website (프로젝트 홈페이지)

프로젝트 문서, 데모, 블로그 링크를 설정합니다.

저장소 메인 페이지 → About 섹션 → ⚙️ 아이콘 → Website 입력

베스트 프랙티스: Description과 Topics를 잘 설정하면 GitHub 검색에 노출되어 프로젝트 발견 가능성이 높아집니다.

절 요약¶

  • 웹 UI 또는 gh CLI로 저장소를 생성할 수 있습니다.
  • README.md는 프로젝트의 첫인상이므로 설치, 사용법, 라이선스를 명확히 작성합니다.
  • 라이선스는 코드 사용 방식을 명시하며, MIT가 가장 범용적입니다.
  • Public은 오픈소스와 포트폴리오, Private는 비공개 프로젝트에 적합합니다.

GitHub UI 완전 정복 (ch_05_s03)¶

도입¶

GitHub 웹 인터페이스는 강력한 기능을 제공합니다. 코드 탐색, 커밋 히스토리 검색, 실시간 편집, 통계 확인 등 모든 것을 브라우저에서 할 수 있습니다. 이 절에서는 GitHub UI의 핵심 탭과 기능을 마스터합니다.

핵심 개념 설명¶

GitHub 저장소 페이지 구조¶

Code 탭: 파일 탐색과 편집¶

파일 트리 탐색

왼쪽 사이드바에 디렉토리 구조가 표시됩니다. 파일을 클릭하면 내용이 오른쪽에 렌더링됩니다.

코드 검색

/ (슬래시) 키 → 파일명 검색

또는 저장소 내 코드 검색:

검색창 → This repository → 검색어 입력

웹 기반 VS Code (github.dev)

파일 탐색 중 . 키를 누르면 웹 기반 VS Code 에디터가 열립니다.

github.com/user/repo → . 키
→ github.dev/user/repo (URL 자동 변경)

주요 기능

  • 파일 편집 (구문 강조 지원)
  • 터미널 사용 가능
  • 익스텐션 설치 가능
  • 편집 후 커밋 또는 PR 생성

팁: github.dev는 가벼운 수정에 적합합니다. 빌드나 테스트는 불가능하며, 편집과 커밋만 가능합니다.

파일 직접 편집

파일 페이지에서 연필 아이콘을 클릭하면 편집 모드로 전환됩니다.

1. 파일 열기
2. 연필 아이콘 (Edit this file) 클릭
3. 내용 수정
4. 하단에 커밋 메시지 입력
5. "Commit changes" 버튼 클릭

주의: 웹에서 직접 편집하면 기본 브랜치(main)에 바로 커밋됩니다. 협업 프로젝트에서는 브랜치를 만들거나 PR을 생성하는 것이 안전합니다.

커밋 히스토리와 Blame 보기¶

커밋 히스토리

Code 탭 → 우측 상단 "XX commits" 클릭

# 커밋 목록이 시간 역순으로 표시됨
커밋 메시지, 작성자, 날짜, 커밋 해시

각 커밋을 클릭하면 변경 사항(diff)을 볼 수 있습니다.

Blame (누가 각 줄을 작성했는가)

파일 페이지에서 "Blame" 버튼 클릭

1. 파일 열기
2. 우측 상단 "Blame" 클릭
3. 각 줄 왼쪽에 작성자, 커밋 해시, 날짜 표시

Blame 활용 사례

  • 특정 코드를 누가 작성했는지 확인 (문의 대상 파악)
  • 버그가 언제 도입되었는지 추적
  • 코드 변경 이유를 커밋 메시지에서 확인

팁: Blame에서 커밋 해시를 클릭하면 해당 커밋의 전체 변경 사항을 볼 수 있습니다.

Settings 탭 핵심 설정¶

저장소 관리자(Admin 권한)만 Settings 탭에 접근할 수 있습니다.

General (일반 설정)

  • Repository name: 저장소 이름 변경 (URL 변경됨, 주의)
  • Default branch: 기본 브랜치 설정 (main, master 등)
  • Danger Zone: 저장소 가시성 변경(Public ↔ Private), 저장소 삭제

Branches (브랜치 보호)

Settings → Branches → Branch protection rules → Add rule

보호할 브랜치(예: main) 설정:

  • Require a pull request before merging: 직접 push 금지, PR 필수
  • Require approvals: 리뷰 승인 필수 (1명 이상)
  • Require status checks to pass: CI 통과 필수
  • Include administrators: 관리자도 규칙 적용

베스트 프랙티스: main 브랜치는 반드시 보호 규칙을 설정하여 실수로 직접 push되지 않도록 하세요.

Collaborators (협업자 관리)

Settings → Collaborators → Add people

GitHub 사용자명 또는 이메일로 협업자를 초대합니다.

권한 레벨

  • Read: 코드 읽기, Issue/PR 생성 가능
  • Write: 코드 push, 브랜치 생성, Issue/PR 관리
  • Admin: 모든 권한 (Settings 접근 포함)

Actions (GitHub Actions 설정)

Settings → Actions → General
  • Actions permissions: 워크플로우 실행 허용 여부
  • Workflow permissions: GITHUB_TOKEN 권한 설정

Insights 탭: 저장소 통계¶

Contributors (기여자)

누가 얼마나 커밋했는지 그래프로 표시합니다.

Commits (커밋 빈도)

시간대별 커밋 활동을 시각화합니다.

Code frequency (코드 추가/삭제)

매주 얼마나 많은 줄이 추가/삭제되었는지 표시합니다.

Network (브랜치 네트워크 그래프)

모든 브랜치와 Fork의 관계를 시각화합니다. 복잡한 협업 프로젝트의 흐름을 한눈에 파악할 수 있습니다.

팁: Insights는 프로젝트 활성도를 확인하거나 기여 보고서를 작성할 때 유용합니다.

절 요약¶

  • GitHub 저장소는 Code, Issues, Pull requests, Actions 등 여러 탭으로 구성됩니다.
  • Code 탭에서 파일 탐색, 편집, Blame 확인이 가능합니다.
  • . 키를 누르면 웹 기반 VS Code(github.dev)가 열립니다.
  • Settings에서 브랜치 보호, 협업자 관리, Actions 설정을 할 수 있습니다.
  • Insights 탭은 기여자, 커밋 빈도, 코드 변화를 시각화합니다.

Issues로 작업 관리하기 (ch_05_s04)¶

도입¶

Issue는 GitHub의 작업 관리 도구입니다. 버그 보고, 기능 요청, 할 일 목록, 토론 등 모든 작업을 Issue로 추적할 수 있습니다. 이 절에서는 Issue를 생성하고, Label로 분류하며, Milestone과 연결하고, 커밋과 자동으로 연동하는 방법을 배웁니다.

핵심 개념 설명¶

Issue란 무엇인가¶

Issue(이슈)는 버그 보고, 기능 요청, 작업 항목 등을 추적하고 관리하는 GitHub의 작업 관리 도구로, Label, Assignee, Milestone 등으로 체계적으로 분류할 수 있습니다.

회사의 업무 요청서와 같습니다. 누가 요청했는지, 무엇을 해야 하는지, 얼마나 급한지, 누가 담당하는지를 기록하고 추적합니다.

Issue의 주요 용도

  1. 버그 보고: 발견된 버그를 상세히 기록
  2. 기능 요청: 새로운 기능 아이디어 제안
  3. 작업 추적: 개발 태스크를 Issue로 분해하여 관리
  4. 문서 개선: 문서 오타, 누락 사항 보고
  5. 질문과 토론: 프로젝트 관련 질문 (Discussions와 구분)

팁: Issue는 버그 리포트용으로만 사용한다고 생각하지만, 기능 요청, 토론, 문서 작업 등 모든 종류의 작업 추적에 활용됩니다.

Issue 생성과 작성 요령¶

Issue 생성

Issues 탭 → New issue → 제목과 본문 작성 → Submit new issue

좋은 Issue 작성법

Issue는 명확하고 재현 가능하게 작성해야 합니다.

버그 보고 템플릿

### 문제 설명
로그인 버튼 클릭 시 404 에러가 발생합니다.

### 재현 방법
1. https://example.com/login 접속
2. 올바른 ID/비밀번호 입력
3. "로그인" 버튼 클릭

### 예상 동작
대시보드 페이지(/)로 리디렉션

### 실제 동작
404 Not Found 페이지 표시

### 환경
- 브라우저: Chrome 121.0.6167.85
- OS: Windows 11
- 계정: test@example.com

### 스크린샷
![404 에러](https://example.com/screenshot.png)

### 추가 정보
개발자 도구 콘솔에 `POST /api/login 404` 에러 표시됨

기능 요청 템플릿

### 기능 설명
다크 모드 지원을 추가해주세요.

### 동기
현재는 라이트 모드만 지원하여 야간 사용 시 눈이 부십니다.

### 제안하는 해결책
Settings 페이지에 "다크 모드" 토글 스위치 추가

### 대안
시스템 테마 설정 자동 감지 (prefers-color-scheme)

### 우선순위
Medium (필수는 아니지만 UX 개선에 도움)

베스트 프랙티스: Issue 템플릿을 설정하면 작성자가 필수 정보를 누락하지 않도록 안내할 수 있습니다. .github/ISSUE_TEMPLATE/ 디렉토리에 템플릿을 추가하세요.

Label로 이슈 분류하기¶

Label(레이블)은 Issue를 카테고리별로 분류하는 태그입니다.

기본 Label

GitHub는 새 저장소에 기본 Label을 자동 생성합니다:

Label 색상 용도
bug 빨강(Red) 버그 보고
documentation 파랑(Blue) 문서 작업
duplicate 회색(Gray) 중복 이슈
enhancement 초록(Green) 기능 개선
good first issue 보라(Purple) 초보자 친화적 이슈
help wanted 초록(Green) 외부 도움 필요
invalid 회색(Gray) 유효하지 않은 이슈
question 보라(Purple) 질문
wontfix 회색(Gray) 수정하지 않을 이슈

Label 추가

Issue 작성 시 또는 작성 후 우측 사이드바에서:

Labels → (설정 아이콘) → 해당 Label 체크

커스텀 Label 생성

Issues 탭 → Labels → New label

예시:

  • priority: high (빨강)
  • priority: low (회색)
  • status: in progress (노랑)
  • area: frontend (파랑)
  • area: backend (초록)

베스트 프랙티스: Label을 체계적으로 관리하세요. 우선순위(priority), 상태(status), 영역(area), 유형(type) 등 일관된 네이밍 규칙을 사용하면 Issue 필터링이 쉬워집니다.

Assignee와 Milestone 지정¶

Assignee (담당자)

누가 이 Issue를 처리할지 지정합니다.

Issue 페이지 우측 사이드바 → Assignees → 사용자 선택

자신을 담당자로 지정:

"assign yourself" 클릭

Milestone (마일스톤)

여러 Issue를 묶어 릴리스 목표를 관리합니다.

Milestone 생성

Issues 탭 → Milestones → New milestone

예시:

  • Title: v1.0 Release
  • Due date: 2025-03-31
  • Description: 첫 번째 안정 버전 릴리스

Issue를 Milestone에 연결

Issue 페이지 우측 사이드바 → Milestone → 선택

Milestone 페이지에서 진행률을 확인할 수 있습니다:

5 Open / 12 Closed (70% complete)

팁: Milestone은 Sprint나 릴리스 주기에 맞춰 설정하세요. 예: Sprint 2024-W08, v2.0 Beta

Issue Template 설정¶

Issue 작성자가 필수 정보를 빠뜨리지 않도록 템플릿을 제공합니다.

템플릿 파일 생성

```.github/ISSUE_TEMPLATE/bug_report.md

내용:

```markdown
---
name: 버그 보고
about: 버그를 발견하셨나요? 자세히 알려주세요.
title: "[BUG] "
labels: bug
assignees: ''
---

### 문제 설명
간결하고 명확하게 버그를 설명하세요.

### 재현 방법
1. '...'로 이동
2. '...'를 클릭
3. '...'까지 스크롤
4. 에러 발생

### 예상 동작
무엇이 일어나야 하는지 설명하세요.

### 스크린샷
가능하다면 스크린샷을 첨부하세요.

### 환경
- OS: [예: Windows 11]
- 브라우저: [예: Chrome 121]
- 버전: [예: 1.2.3]

템플릿이 있으면 Issue 생성 시 선택할 수 있습니다:

New issue → Get started (버그 보고 템플릿)

이슈와 커밋 연결하기 (Closes #123)¶

커밋 메시지에 키워드를 포함하면 Issue를 자동으로 닫을 수 있습니다.

자동 닫기 키워드

  • Closes #123
  • Fixes #123
  • Resolves #123

사용 예시

$ git commit -m "fix: 로그인 버튼 404 에러 수정

사용자 인증 API 엔드포인트를 /api/auth/login으로 변경

Fixes #123"

이 커밋이 기본 브랜치(main)에 머지되면 Issue #123이 자동으로 닫힙니다.

PR에서도 사용 가능

PR 설명에 Closes #123을 포함하면, PR이 머지될 때 Issue가 자동으로 닫힙니다.

베스트 프랙티스: 모든 버그 수정 커밋에 Fixes #이슈번호를 포함하여 Issue와 커밋을 연결하세요. 나중에 어떤 커밋으로 버그가 수정되었는지 추적하기 쉽습니다.

절 요약¶

  • Issue는 버그, 기능 요청, 작업 항목을 추적하는 도구입니다.
  • 명확한 재현 방법과 환경 정보를 포함하여 Issue를 작성합니다.
  • Label로 Issue를 분류하고, Assignee로 담당자를 지정합니다.
  • Milestone으로 여러 Issue를 릴리스 목표에 묶어 관리합니다.
  • 커밋 메시지에 Fixes #123을 포함하면 Issue가 자동으로 닫힙니다.

Markdown 마스터하기 (ch_05_s05)¶

도입¶

GitHub는 README, Issue, PR, Comments 등 모든 곳에서 Markdown을 사용합니다. 특히 GitHub Flavored Markdown(GFM)은 표준 Markdown에 작업 목록, 테이블, 자동 링크 등 유용한 기능을 추가했습니다. 이 절에서는 GFM 문법을 마스터하여 풍부한 문서를 작성하는 방법을 배웁니다.

GitHub Flavored Markdown(GFM) 확장 기능¶

GFM은 표준 Markdown에 GitHub 특화 기능을 추가한 확장 문법입니다.

작업 목록 (Task List)

- [x] 완료된 작업
- [ ] 미완료 작업
- [ ] 앞으로 할 작업

렌더링: - [x] 완료된 작업 - [ ] 미완료 작업 - [ ] 앞으로 할 작업

Issue나 PR에서 작업 목록을 사용하면 진행률이 자동 표시됩니다:

[●●●○○○○○○○] 3 of 10 tasks completed

테이블

| 헤더 1 | 헤더 2 | 헤더 3 |
|--------|--------|--------|
| 셀 1   | 셀 2   | 셀 3   |
| 셀 4   | 셀 5   | 셀 6   |

정렬 지정

| 왼쪽 정렬 | 가운데 정렬 | 오른쪽 정렬 |
|:---------|:----------:|----------:|
| 왼쪽     | 가운데      | 오른쪽     |

자동 링크

GitHub는 특정 패턴을 자동으로 링크로 변환합니다:

Issue #123 → (이슈 링크)
@username → (사용자 프로필 링크)
SHA 커밋 해시 → (커밋 링크)

취소선

~~이 텍스트는 취소선~~

렌더링: ~~이 텍스트는 취소선~~

이모지

:+1: :tada: :rocket: :sparkles:

렌더링: :+1: :tada: :rocket: :sparkles: (실제로는 각각의 이모지로 표시됨)

전체 이모지 목록: https://github.com/ikatyang/emoji-cheat-sheet

주의: 본 교재에서는 이모지 사용이 금지되어 있습니다(book_config의 emoji_forbidden: true). 위 예시는 설명 목적이며, 실제 산출물에는 사용하지 않습니다.

#### Mermaid 다이어그램 삽입

GitHub는 Mermaid 다이어그램을 자동으로 렌더링합니다.

**플로우차트**

````markdown
```mermaid
graph LR
    A[시작] --> B{조건}
    B -->|Yes| C[실행]
    B -->|No| D[종료]
    C --> D

**시퀀스 다이어그램**

````markdown
```mermaid
sequenceDiagram
    participant 사용자
    participant 서버
    사용자->>서버: 로그인 요청
    서버-->>사용자: 토큰 반환
    사용자->>서버: 데이터 요청 (토큰 포함)
    서버-->>사용자: 데이터 반환
```

간트 차트

```mermaid
gantt
    title 프로젝트 일정
    dateFormat  YYYY-MM-DD
    section 설계
    요구사항 분석    :a1, 2025-01-01, 7d
    UI 디자인       :a2, after a1, 5d
    section 개발
    백엔드 개발     :b1, after a2, 14d
    프론트엔드 개발  :b2, after a2, 14d
```

ER 다이어그램

```mermaid
erDiagram
    USER ||--o{ ORDER : places
    USER {
        int id
        string name
        string email
    }
    ORDER {
        int id
        date created_at
        string status
    }
```

팁: Mermaid는 복잡한 다이어그램 도구(Draw.io, Lucidchart) 없이도 텍스트만으로 다이어그램을 그릴 수 있어 버전 관리가 쉽습니다.

코드 블록과 구문 강조¶

GitHub는 150개 이상의 언어에 대해 구문 강조(Syntax Highlighting)를 지원합니다.

구문 강조 지정

```python
def factorial(n):
    return 1 if n == 0 else n * factorial(n-1)
```

주요 언어 키워드

  • python, javascript, java, c, cpp, ruby, go, rust
  • bash, shell, powershell
  • json, yaml, xml, markdown
  • sql, html, css, scss

diff 형식

변경 사항을 강조:

```diff
- old line
+ new line
  unchanged line
```

파일명 표시 (코드 주석)

```python
# file: app.py
def main():
    print("Hello")
```

베스트 프랙티스: 코드 예제에는 항상 언어 키워드를 지정하여 구문 강조를 활성화하세요. 가독성이 크게 향상됩니다.

절 요약¶

  • GitHub는 모든 문서에 Markdown을 사용합니다.
  • GFM은 작업 목록, 테이블, 자동 링크, 취소선 등 확장 기능을 제공합니다.
  • Mermaid를 사용하면 텍스트로 다이어그램을 작성할 수 있습니다.
  • 코드 블록에 언어를 지정하면 구문 강조가 적용됩니다.
  • Details 태그로 긴 내용을 접어서 문서를 깔끔하게 유지할 수 있습니다.

FAQ¶

Q1: Private 저장소는 무료로 몇 개까지 만들 수 있나요?

A1: GitHub Free 플랜에서 Private 저장소는 무제한으로 만들 수 있습니다. 2023년부터 협업자 수 제한도 해제되었습니다. 단, GitHub Actions 무료 사용량(월 2,000분)과 스토리지(500MB LFS)는 Public과 동일하게 적용됩니다.

Q2: README에 배지(Badge)를 추가하려면 어떻게 하나요?

A2: shields.io에서 배지를 생성할 수 있습니다. 예시:

![Build Status](https://img.shields.io/github/actions/workflow/status/user/repo/ci.yml)
![License](https://img.shields.io/github/license/user/repo)
![Stars](https://img.shields.io/github/stars/user/repo?style=social)

동적 배지는 GitHub Actions, 라이선스, 버전, 다운로드 수 등을 실시간으로 반영합니다.

Q3: Issue에서 코드 블록을 예쁘게 표시하려면?

A3: 백틱 3개로 감싸고 언어를 지정하세요:

```python
def hello():
    print("Hello, World!")
```

렌더링 시 Python 구문 강조가 적용됩니다.

Q4: github.dev와 GitHub Codespaces의 차이는 무엇인가요?

A4: github.dev는 웹 기반 VS Code로 파일 편집만 가능합니다(빌드, 실행 불가). GitHub Codespaces는 클라우드 VM에서 전체 개발 환경을 제공하여 빌드, 테스트, 디버깅이 가능합니다. Codespaces는 유료이며, 무료 플랜은 월 60시간까지 사용 가능합니다.

Q5: Issue를 삭제할 수 있나요?

A5: Issue는 삭제할 수 없고, 닫기(Close)만 가능합니다. 잘못 생성된 Issue는 "invalid" 또는 "wontfix" Label을 붙이고 닫으세요. 저장소 관리자는 Issue를 숨길 수 있지만(spam 차단용), 완전 삭제는 불가능합니다. Issue 번호는 재사용되지 않습니다.
