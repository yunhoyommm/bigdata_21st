10장: Git 보안과 관리

중급

Git 보안과 관리¶

학습 목표 - GPG/SSH 키로 커밋을 서명하고 검증할 수 있다 - GitHub Secrets와 Secret Scanning으로 민감 데이터를 안전하게 관리할 수 있다 - Dependabot을 설정하여 의존성 보안을 자동화할 수 있다 - git filter-repo로 히스토리에서 민감 데이터를 완전히 제거할 수 있다 - Git LFS를 설정하여 대용량 파일을 효율적으로 관리할 수 있다

커밋 서명과 검증¶

도입¶

GitHub에서 커밋을 보면 작성자 이름과 이메일이 표시됩니다. 하지만 이 정보는 쉽게 위조할 수 있습니다. git config로 아무 이름이나 설정할 수 있기 때문입니다. 커밋의 작성자를 암호학적으로 증명하려면 커밋 서명(Commit Signing)이 필요합니다.

핵심 개념 설명¶

왜 커밋 서명이 필요한가: 신뢰의 문제¶

커밋 서명은 커밋이 정말 본인이 작성한 것임을 암호학적으로 증명하는 기능입니다. GPG 또는 SSH 키로 서명하면 GitHub에서 "Verified" 배지를 표시합니다.

공증 서류와 같습니다. 일반 서명은 누구나 흉내 낼 수 있지만, 공증(GPG 서명)은 제3자가 신원을 보증하므로 위조가 어렵습니다.

위조 가능한 커밋 예시:

# 누구나 다른 사람의 이름으로 커밋 가능
$ git config user.name "Linus Torvalds"
$ git config user.email "torvalds@linux-foundation.org"
$ git commit -m "Add backdoor"  # 이 커밋은 Linus가 한 것처럼 보임

이런 위조를 막기 위해 커밋 서명이 필요합니다.

커밋 서명이 중요한 시나리오: - 오픈소스 프로젝트의 보안 패치 - 금융, 의료 등 민감한 산업의 코드 - 규정 준수(Compliance) 요구사항 - 공급망 보안(Supply Chain Security)

GPG 키 생성과 GitHub 등록¶

GPG(GNU Privacy Guard)는 암호화와 전자 서명을 위한 오픈소스 도구입니다.

1. GPG 설치 확인:

# GPG 버전 확인
$ gpg --version
gpg (GnuPG) 2.4.0

Windows: gpg4win 설치 macOS: brew install gnupg Linux: 대부분 기본 설치됨

2. GPG 키 생성:

$ gpg --full-generate-key

대화형 프롬프트:

키 종류를 선택하세요:
(1) RSA and RSA (기본값)
→ 1 입력 또는 Enter

키 크기를 입력하세요 (1024-4096):
→ 4096 입력 (최대 보안)

키 유효 기간:
→ 2y (2년) 또는 0 (무기한)

실명:
→ Kim Developer

이메일:
→ kim@example.com (GitHub 이메일과 동일해야 함)

코멘트:
→ (비워두기 또는 "GitHub signing key")

비밀번호:
→ 강력한 비밀번호 입력 (커밋 서명 시마다 필요)

3. GPG 키 ID 확인:

$ gpg --list-secret-keys --keyid-format=long

/home/user/.gnupg/secring.gpg
------------------------------------
sec   rsa4096/3AA5C34371567BD2 2025-01-01 [SC] [expires: 2027-01-01]
      0123456789ABCDEF0123456789ABCDEF01234567
uid                 [ultimate] Kim Developer <kim@example.com>
ssb   rsa4096/42B317FD4BA89E7A 2025-01-01 [E] [expires: 2027-01-01]

키 ID는 3AA5C34371567BD2 부분입니다.

4. 공개키 내보내기:

$ gpg --armor --export 3AA5C34371567BD2
-----BEGIN PGP PUBLIC KEY BLOCK-----

mQINBGX...
(긴 문자열)
...qwerty
-----END PGP PUBLIC KEY BLOCK-----

이 내용을 복사합니다.

5. GitHub에 공개키 등록:

  1. GitHub → Settings → SSH and GPG keys → New GPG key
  2. 복사한 공개키 붙여넣기
  3. Add GPG key

커밋 서명 설정: git config commit.gpgsign¶

Git에 GPG 키 설정:

# 전역 설정 (모든 저장소에 적용)
$ git config --global user.signingkey 3AA5C34371567BD2

# 자동 서명 활성화
$ git config --global commit.gpgsign true

# Tag 서명도 자동화
$ git config --global tag.gpgsign true

서명된 커밋 만들기:

# 자동 서명 활성화 후
$ git commit -m "feat: 사용자 인증 기능 추가"

# 수동 서명 (자동 서명 비활성화 시)
$ git commit -S -m "feat: 사용자 인증 기능 추가"

GPG 비밀번호를 입력하면 커밋이 서명됩니다.

서명 확인:

$ git log --show-signature -1

commit 7f3a9e8c1d2b4f5e (HEAD -> main)
gpg: Signature made Mon Jan 15 14:30:25 2025 KST
gpg:                using RSA key 3AA5C34371567BD2
gpg: Good signature from "Kim Developer <kim@example.com>" [ultimate]
Author: Kim Developer <kim@example.com>
Date:   Mon Jan 15 14:30:25 2025 +0900

    feat: 사용자 인증 기능 추가

GitHub에서는 이 커밋에 "Verified" 배지가 표시됩니다.

팁: GPG 비밀번호를 매번 입력하기 번거롭다면 gpg-agent를 설정하여 일정 시간 캐시할 수 있습니다.

gpg-agent 캐시 설정:

~/.gnupg/gpg-agent.conf:

default-cache-ttl 3600
max-cache-ttl 7200

$ gpg-connect-agent reloadagent /bye

서명된 커밋과 태그 만들기¶

서명된 Tag:

Tag도 커밋처럼 서명할 수 있습니다.

# 서명된 Annotated Tag 생성
$ git tag -s v1.0.0 -m "Release v1.0.0"

# Tag 서명 확인
$ git tag -v v1.0.0
gpg: Signature made ...
gpg: Good signature from "Kim Developer <kim@example.com>" [ultimate]

GitHub Release를 만들 때 서명된 Tag를 사용하면 릴리스의 신뢰성이 높아집니다.

SSH 키를 이용한 커밋 서명 (Git 2.34+)¶

Git 2.34부터는 SSH 키로도 커밋을 서명할 수 있습니다. GPG보다 설정이 간단합니다.

1. SSH 키 생성 (이미 있다면 생략):

$ ssh-keygen -t ed25519 -C "kim@example.com"

2. 공개키 GitHub에 등록:

Settings → SSH and GPG keys → New SSH key → Signing Key 타입 선택

~/.ssh/id_ed25519.pub 내용 복사하여 등록

3. Git 설정:

# SSH 서명 활성화
$ git config --global gpg.format ssh

# SSH 공개키 지정
$ git config --global user.signingkey ~/.ssh/id_ed25519.pub

# 자동 서명
$ git config --global commit.gpgsign true

4. 허용된 서명자 설정:

.git/allowed_signers (또는 전역):

kim@example.com ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIB...

$ git config --global gpg.ssh.allowedSignersFile ~/.ssh/allowed_signers

이제 커밋하면 SSH 키로 자동 서명됩니다.

베스트 프랙티스: SSH 키 서명은 GPG보다 설정이 간단하지만, GPG는 더 광범위하게 지원됩니다. 팀 정책에 따라 선택하세요.

절 요약¶

  • 커밋 서명은 커밋 작성자의 신원을 암호학적으로 증명합니다
  • GPG 키를 생성하고 GitHub에 등록하여 "Verified" 배지를 받을 수 있습니다
  • git config commit.gpgsign true로 자동 서명을 활성화합니다
  • Git 2.34+에서는 SSH 키로도 커밋을 서명할 수 있습니다
  • 서명된 Tag는 릴리스의 신뢰성을 보장합니다

Secret 관리와 민감 데이터 보호¶

도입¶

소프트웨어 개발 중 API 키, 데이터베이스 비밀번호, 인증 토큰 등 민감한 정보를 다루게 됩니다. 이런 비밀(Secrets)을 실수로 Git에 커밋하면 공개 저장소에 노출되어 심각한 보안 사고로 이어질 수 있습니다. GitHub는 여러 계층의 보안 기능을 제공하여 이를 방지합니다.

핵심 개념 설명¶

절대 커밋하면 안 되는 것들: API 키, 비밀번호, 토큰¶

커밋하면 안 되는 민감 데이터:

  1. 인증 정보:
  2. API 키 (AWS, Google Cloud, OpenAI 등)
  3. OAuth 토큰
  4. 데이터베이스 비밀번호

  5. SSH 개인키

  6. 설정 파일:

  7. .env (환경 변수)
  8. secrets.json
  9. credentials.yaml

  10. config/production.yml

  11. 인증서와 키:

  12. SSL/TLS 개인키 (.key, .pem)

  13. 서명 인증서

  14. 기타 민감 정보:

  15. 내부 IP 주소, 호스트명
  16. 개인 식별 정보 (PII)
  17. 라이선스 키

실수로 커밋 시 발생 가능한 문제: - 무단 API 사용 및 과금 - 데이터베이스 침해 - 서비스 거부 공격(DDoS) - 데이터 유출

주의: 한 번이라도 Git 히스토리에 커밋되면, 나중에 삭제해도 히스토리에 남습니다. 완전히 제거하려면 히스토리 재작성이 필요합니다.

.gitignore와 .env 파일 관리¶

.gitignore 설정:

프로젝트 시작 시 .gitignore에 민감한 파일 패턴을 추가하세요.

# 환경 변수
.env
.env.local
.env.production

# 설정 파일
config/secrets.yml
credentials.json
secrets.json

# 인증서
*.pem
*.key
*.crt
*.p12

# 클라우드 제공자
.aws/credentials
.gcp/keyfile.json

# IDE 설정 (민감 정보 포함 가능)
.vscode/settings.json
.idea/workspace.xml

환경 변수 사용 패턴:

민감한 정보는 코드에 직접 작성하지 말고 환경 변수로 분리합니다.

나쁜 예:

# ❌ 코드에 API 키 하드코딩
api_key = "sk-1234567890abcdef"

좋은 예:

# ✅ 환경 변수 사용
import os
api_key = os.getenv("OPENAI_API_KEY")

.env 파일:

OPENAI_API_KEY=sk-1234567890abcdef
DATABASE_URL=postgresql://user:password@localhost/mydb
AWS_ACCESS_KEY_ID=AKIAIOSFODNN7EXAMPLE

.env.example (저장소에 커밋):

OPENAI_API_KEY=your_api_key_here
DATABASE_URL=postgresql://user:password@localhost/mydb
AWS_ACCESS_KEY_ID=your_access_key_here

팀원은 .env.example을 .env로 복사하여 자신의 값을 입력합니다.

$ cp .env.example .env
# .env 파일 편집하여 실제 키 입력

GitHub Secrets: 안전한 비밀 저장¶

GitHub Secrets는 GitHub Actions 워크플로우에서 사용할 비밀을 안전하게 저장하는 기능입니다. 암호화되어 저장되며, 로그에도 마스킹됩니다.

Repository Secret 추가:

  1. 저장소 → Settings → Secrets and variables → Actions
  2. New repository secret
  3. Name: OPENAI_API_KEY
  4. Secret: sk-1234567890abcdef
  5. Add secret

워크플로우에서 사용:

.github/workflows/deploy.yml:

name: Deploy

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Deploy to Production
        env:
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
          DATABASE_URL: ${{ secrets.DATABASE_URL }}
        run: |
          echo "Deploying with API key..."
          # 배포 스크립트 실행

Secret은 ${{ secrets.SECRET_NAME }} 형식으로 접근합니다.

Secret 보안 기능: - AES-256 암호화 저장 - 로그에서 자동 마스킹 (***) - Pull Request에서는 기본적으로 접근 불가 (설정 필요)

Secret 범위: - Repository secrets: 해당 저장소에서만 사용 - Organization secrets: 조직의 여러 저장소에서 공유 - Environment secrets: 특정 환경(staging, production)에서만 사용

베스트 프랙티스: 프로덕션 Secret은 Environment secrets로 분리하고, 배포 승인(approval) 설정을 추가하세요.

Secret Scanning: 실수로 커밋된 비밀 탐지¶

Secret Scanning은 GitHub가 저장소를 자동으로 스캔하여 알려진 Secret 패턴을 탐지하는 기능입니다.

지원하는 Secret 타입 (200개 이상): - AWS 액세스 키 - Google Cloud API 키 - GitHub Personal Access Token - Stripe API 키 - Slack 토큰 - Azure 연결 문자열 - 데이터베이스 연결 문자열

활성화:

Public 저장소는 기본 활성화됩니다. Private 저장소는 GitHub Advanced Security 라이선스 필요 (Enterprise 플랜).

Settings → Code security and analysis → Secret scanning → Enable

탐지 시 동작:

  1. GitHub가 커밋에서 Secret 패턴 발견
  2. Security 탭에 Alert 생성
  3. 저장소 관리자에게 이메일 알림
  4. 파트너 서비스 (AWS, Google 등)에 자동 알림 (선택)

Alert 처리:

  1. Security → Secret scanning alerts
  2. Alert 확인
  3. 조치:
  4. Secret 즉시 무효화/재발급
  5. 히스토리에서 완전 제거 (아래 섹션 참조)
  6. Alert를 "Resolved"로 표시

주의: Secret Scanning이 감지하지 못하는 Secret도 있습니다. 완벽하게 의존하지 말고 .gitignore 설정을 철저히 하세요.

Push Protection으로 비밀 유출 사전 방지¶

Push Protection은 Secret이 포함된 커밋을 push하려 할 때 사전에 차단하는 기능입니다.

활성화:

Settings → Code security and analysis → Push protection → Enable

동작 방식:

$ git push origin main

remote: error: Secret detected in commit 7f3a9e8c1d2b4f5e
remote:
remote: GitHub found the following secrets:
remote: - AWS Access Key ID (AKIAIOSFODNN7EXAMPLE)
remote:
remote: To push anyway, run:
remote: git push --no-verify
remote:
remote: Or remove the secret from your commit history and try again.
To github.com:user/repo.git
 ! [remote rejected] main -> main (push declined due to secret)
error: failed to push some refs to 'github.com:user/repo.git'

우회 옵션:

정당한 이유가 있을 때 (예: 테스트용 더미 키) 우회할 수 있습니다:

$ git push --no-verify

하지만 대부분의 경우 Secret을 제거하고 push해야 합니다.

베스트 프랙티스: Push Protection을 활성화하고, 팀원에게 .gitignore 설정과 환경 변수 사용을 교육하세요.

절 요약¶

  • API 키, 비밀번호, 토큰은 절대 Git에 커밋하지 않습니다
  • .gitignore에 .env, secrets.json 등을 추가하여 실수 방지
  • GitHub Secrets에 민감 정보를 암호화 저장하고 Actions에서 사용
  • Secret Scanning이 커밋된 Secret을 자동 탐지하여 알립니다
  • Push Protection으로 Secret 포함 커밋을 push 전에 차단합니다

Dependabot과 의존성 보안¶

도입¶

현대 소프트웨어는 수십, 수백 개의 외부 라이브러리에 의존합니다. 이런 의존성(dependency)에서 보안 취약점이 발견되면 프로젝트도 위험해집니다. 수동으로 모든 의존성을 추적하는 것은 불가능합니다. Dependabot은 이를 자동화하는 GitHub의 기능입니다.

핵심 개념 설명¶

Dependabot Alerts: 취약한 의존성 알림¶

Dependabot Alerts는 프로젝트의 의존성에서 알려진 보안 취약점(CVE)을 자동으로 탐지하고 알려줍니다.

건물의 자동 보안 점검 시스템과 같습니다. 정기적으로 건물(프로젝트)의 잠금장치(의존성)를 검사하고, 취약한 부분이 발견되면 자동으로 수리 요청서(PR)를 발행합니다.

지원하는 생태계: - JavaScript (npm, yarn) - Python (pip) - Ruby (bundler) - Java (maven, gradle) - .NET (nuget) - PHP (composer) - Go (go modules) - Rust (cargo) - Docker

활성화:

Settings → Code security and analysis → Dependabot alerts → Enable

Alert 확인:

Security → Dependabot alerts

예시:

lodash
Severity: High

Dependabot Security Updates: 자동 패치 PR¶

Dependabot Security Updates는 취약점이 발견되면 자동으로 패치 PR을 생성합니다.

활성화:

Settings → Code security and analysis → Dependabot security updates → Enable

동작 흐름:

  1. Dependabot이 취약점 발견 (예: lodash < 4.17.21)
  2. 패치 버전 확인 (4.17.21)
  3. 자동으로 PR 생성:
  4. 제목: Bump lodash from 4.17.20 to 4.17.21
  5. 본문: 취약점 설명, Changelog 링크

  6. package.json 또는 requirements.txt 업데이트

  7. CI 테스트 자동 실행

  8. 팀원이 리뷰 후 머지

PR 예시:

## Bumps lodash from 4.17.20 to 4.17.21

### Release notes
- Fix: Prototype Pollution vulnerability

### Commits
- `abc1234` Release 4.17.21
- `def5678` Fix prototype pollution

**Dependabot compatibility score**: 98/100

팁: Dependabot PR은 CI 테스트가 통과하고 호환성 점수가 높으면 안심하고 머지할 수 있습니다.

Dependabot Version Updates: 버전 최신화 자동화¶

Dependabot Version Updates는 보안 취약점과 관계없이 정기적으로 의존성을 최신 버전으로 업데이트하는 PR을 생성합니다.

.github/dependabot.yml 설정:

version: 2
updates:
  # JavaScript 의존성
  - package-ecosystem: "npm"
    directory: "/"
    schedule:
      interval: "weekly"
      day: "monday"
      time: "09:00"
      timezone: "Asia/Seoul"
    open-pull-requests-limit: 5
    reviewers:
      - "frontend-team"
    labels:
      - "dependencies"
      - "javascript"
    commit-message:
      prefix: "chore(deps)"

  # Python 의존성
  - package-ecosystem: "pip"
    directory: "/backend"
    schedule:
      interval: "weekly"
    versioning-strategy: "increase-if-necessary"
    ignore:
      - dependency-name: "django"
        update-types: ["version-update:semver-major"]

  # Docker
  - package-ecosystem: "docker"
    directory: "/"
    schedule:
      interval: "weekly"

  # GitHub Actions
  - package-ecosystem: "github-actions"
    directory: "/"
    schedule:
      interval: "monthly"

주요 설정:

  • schedule.interval: daily, weekly, monthly
  • open-pull-requests-limit: 동시 열린 PR 개수 제한
  • reviewers: 자동 리뷰어 지정
  • labels: PR에 자동 라벨 추가
  • ignore: 특정 의존성 또는 버전 타입 무시
  • versioning-strategy: 업데이트 전략

versioning-strategy 옵션: - auto: Dependabot이 자동 결정 (기본값) - increase: 항상 최신 버전으로 업데이트 - increase-if-necessary: 취약점 수정 시에만 업데이트 - lockfile-only: lockfile만 업데이트, manifest 파일은 수정 안 함 - widen: 버전 범위 확대

베스트 프랙티스: open-pull-requests-limit를 5개 이하로 설정하세요. PR이 너무 많으면 관리가 어려워집니다.

dependabot.yml 설정 파일 작성¶

고급 설정 예시:

version: 2
updates:
  - package-ecosystem: "npm"
    directory: "/"
    schedule:
      interval: "weekly"
    # 특정 의존성만 업데이트
    allow:
      - dependency-name: "react*"
      - dependency-type: "production"
    # 특정 의존성 무시
    ignore:
      - dependency-name: "lodash"
        versions: ["4.x"]
    # 그룹화: 여러 업데이트를 하나의 PR로
    groups:
      react-ecosystem:
        patterns:
          - "react"
          - "react-dom"
          - "react-router"
    # 커밋 메시지 커스터마이즈
    commit-message:
      prefix: "deps"
      include: "scope"
    # PR 제목 템플릿
    pull-request-branch-name:
      separator: "-"

그룹화 기능:

여러 관련 패키지를 하나의 PR로 묶을 수 있습니다.

groups:
  test-dependencies:
    patterns:
      - "jest"
      - "@testing-library/*"
    update-types:
      - "minor"
      - "patch"

이렇게 하면 jest, @testing-library/react 등이 하나의 PR로 업데이트됩니다.

의존성 그래프(Dependency Graph) 활용¶

의존성 그래프는 프로젝트의 모든 의존성을 시각적으로 보여줍니다.

확인 방법:

Insights → Dependency graph

표시 정보: - 직접 의존성(Direct dependencies) - 간접 의존성(Transitive dependencies) - 의존성 버전 - 알려진 취약점 수

의존성 내보내기:

# SBOM (Software Bill of Materials) 생성
$ gh api /repos/owner/repo/dependency-graph/sbom -H "Accept: application/vnd.github+json"

SBOM은 소프트웨어 공급망 보안(Supply Chain Security)의 핵심 요소입니다.

팁: 의존성 그래프를 정기적으로 검토하여 불필요한 의존성을 제거하고 번들 크기를 줄이세요.

절 요약¶

  • Dependabot Alerts가 의존성의 보안 취약점을 자동 탐지합니다
  • Dependabot Security Updates가 취약점 패치 PR을 자동 생성합니다
  • Dependabot Version Updates로 의존성을 최신 상태로 유지합니다
  • .github/dependabot.yml로 업데이트 일정, 전략, 그룹화를 설정합니다
  • 의존성 그래프로 프로젝트의 모든 의존성을 시각적으로 파악합니다

민감 데이터 제거와 히스토리 정리¶

도입¶

실수로 API 키를 커밋하고 원격에 push했다면, 단순히 파일을 삭제하고 다시 커밋하는 것으로는 부족합니다. Git 히스토리에 여전히 남아있기 때문입니다. 이럴 때는 히스토리를 재작성하여 완전히 제거해야 합니다.

핵심 개념 설명¶

실수로 커밋된 민감 데이터 대응 절차¶

즉시 조치 사항:

  1. 비밀 즉시 무효화:
  2. API 키 재발급
  3. 비밀번호 변경

  4. 토큰 revoke

  5. 영향 범위 파악:

  6. 누가 접근 가능한가?
  7. 얼마 동안 노출되었나?

  8. 악용 흔적은 없는가?

  9. 히스토리 재작성 (아래 설명)

  10. 팀 공지:

  11. 팀원에게 상황 알림
  12. 저장소 다시 클론 요청

주의: 비밀을 무효화하는 것이 가장 중요합니다. 히스토리에서 제거해도 누군가 이미 복사했을 수 있습니다.

git filter-repo로 파일 완전 제거¶

git filter-repo는 Git 히스토리를 재작성하는 권장 도구입니다. 구버전의 git filter-branch보다 빠르고 안전합니다.

설치:

# pip로 설치
$ pip install git-filter-repo

# macOS
$ brew install git-filter-repo

# 버전 확인
$ git filter-repo --version

특정 파일 완전 제거:

# 1. 로컬 백업
$ git clone https://github.com/user/repo.git repo-backup

# 2. 작업용 클론
$ git clone https://github.com/user/repo.git repo-clean
$ cd repo-clean

# 3. 모든 브랜치에서 파일 제거
$ git filter-repo --path secrets.txt --invert-paths

# 4. 히스토리 재작성 완료
# secrets.txt가 모든 커밋에서 제거됨

--invert-paths는 "지정한 파일을 제외한 모든 것 유지"를 의미합니다.

여러 파일 제거:

$ git filter-repo --path .env --path config/secrets.yml --invert-paths

디렉토리 전체 제거:

$ git filter-repo --path private/ --invert-paths

파일 내용 대체:

민감한 내용만 마스킹하고 파일은 유지하려면:

# 비밀번호를 REDACTED로 대체
$ git filter-repo --replace-text <(echo "password123==>REDACTED")

BFG Repo Cleaner 활용¶

BFG Repo Cleaner는 대용량 파일이나 민감 데이터를 제거하는 또 다른 도구입니다. git filter-repo보다 간단하지만 기능은 제한적입니다.

설치:

# Java 필요
$ brew install bfg  # macOS
# 또는
$ wget https://repo1.maven.org/maven2/com/madgag/bfg/1.14.0/bfg-1.14.0.jar

사용법:

# 1. bare 클론
$ git clone --mirror https://github.com/user/repo.git

# 2. BFG 실행
$ java -jar bfg.jar --delete-files secrets.txt repo.git

# 3. 히스토리 정리
$ cd repo.git
$ git reflog expire --expire=now --all
$ git gc --prune=now --aggressive

# 4. force push
$ git push --force

비밀번호 대체:

# passwords.txt 파일에 민감한 문자열 나열
$ echo "password123" > passwords.txt
$ java -jar bfg.jar --replace-text passwords.txt repo.git

팁: 간단한 작업은 BFG가 빠르고, 복잡한 히스토리 재작성은 git filter-repo가 강력합니다.

히스토리 재작성의 영향과 팀 협업 시 주의사항¶

히스토리 재작성의 영향:

  • 모든 커밋 해시 변경
  • 브랜치 포인터 변경
  • 원격 저장소와 로컬 저장소 불일치

force push:

$ git push --force --all
$ git push --force --tags

경고: force push는 원격 히스토리를 덮어씁니다. 팀원의 로컬 작업이 손실될 수 있습니다.

팀원 동기화 가이드:

히스토리 재작성 후 팀원에게 다음 절차를 안내하세요:

  1. 현재 작업 커밋 또는 stash:
$ git add .
$ git commit -m "WIP: 히스토리 재작성 전 백업"
  1. 기존 저장소 삭제하고 다시 클론:
$ cd ..
$ rm -rf old-repo
$ git clone https://github.com/user/repo.git new-repo
  1. 또는 hard reset (위험):
$ git fetch origin
$ git reset --hard origin/main

브랜치 보호 규칙 임시 해제:

force push를 하려면 브랜치 보호 규칙을 임시로 해제해야 할 수 있습니다.

Settings → Branches → Edit rule → 임시 비활성화 → force push 후 다시 활성화

force push 후 팀원 동기화 가이드¶

자동화 스크립트:

팀원에게 제공할 동기화 스크립트:

sync-after-history-rewrite.sh:

#!/bin/bash
set -e

echo "⚠️  Git 히스토리가 재작성되었습니다."
echo "현재 작업을 백업합니다..."

# 현재 브랜치 이름 저장
CURRENT_BRANCH=$(git branch --show-current)

# 작업 백업
git stash save "Before history rewrite sync"

# 모든 원격 변경 가져오기
git fetch --all

# 현재 브랜치를 원격과 동기화
git reset --hard origin/$CURRENT_BRANCH

# Stash 복원 시도
if git stash list | grep -q "Before history rewrite sync"; then
  echo "작업을 복원합니다..."
  git stash pop || echo "⚠️  충돌이 발생했습니다. 수동으로 해결하세요."
fi

echo "✅ 동기화 완료!"

사용:

$ chmod +x sync-after-history-rewrite.sh
$ ./sync-after-history-rewrite.sh

베스트 프랙티스: 히스토리 재작성은 마지막 수단입니다. 가능하면 비밀을 무효화하고 앞으로 주의하는 것이 더 안전합니다.

절 요약¶

  • 민감 데이터가 커밋되면 즉시 비밀을 무효화하세요
  • git filter-repo로 히스토리에서 파일을 완전히 제거할 수 있습니다
  • BFG Repo Cleaner는 간단한 대용량 파일/비밀 제거에 적합합니다
  • 히스토리 재작성 후 force push하면 팀원도 동기화해야 합니다
  • 브랜치 보호 규칙을 임시 해제하고 force push 후 다시 활성화하세요

Git LFS와 대용량 파일 관리¶

도입¶

Git은 텍스트 파일 버전 관리에 최적화되어 있지만, 이미지, 동영상, 3D 모델 같은 대용량 바이너리 파일에는 비효율적입니다. 큰 파일을 커밋할 때마다 저장소 크기가 급증하고, 클론 속도가 느려집니다. Git LFS는 이 문제를 해결합니다.

핵심 개념 설명¶

Git이 대용량 파일에 약한 이유¶

Git의 스냅샷 모델:

Git은 매 커밋마다 전체 프로젝트의 스냅샷을 저장합니다. 동일 파일은 재사용하지만, 바이너리 파일이 조금만 변경되어도 전체를 다시 저장해야 합니다.

도서관의 대형 지도 보관과 같습니다. 큰 지도(대용량 파일)는 별도의 지도실(LFS 서버)에 보관하고, 서가에는 '지도실 A-3번에 있음'이라는 메모(포인터)만 꽂아둡니다.

문제 상황:

# 100MB 이미지 파일 커밋
$ git add large-image.png
$ git commit -m "Add image"

# 이미지 살짝 수정
$ git add large-image.png
$ git commit -m "Update image"

# 저장소 크기가 200MB로 증가!

Git은 델타 압축을 하지만, 바이너리 파일은 압축률이 낮습니다.

Git LFS 없이 대용량 파일 관리 시 문제: - 저장소 크기 급증 - 클론 시간 증가 (GB 단위 다운로드) - GitHub 저장소 크기 제한 (파일당 100MB, 저장소 권장 1GB) - 느린 푸시/풀

Git LFS 설치와 설정¶

Git LFS(Large File Storage)는 대용량 파일을 별도 서버에 저장하고, Git 저장소에는 포인터만 유지하는 확장 기능입니다.

설치:

# Windows
$ git lfs install

# macOS
$ brew install git-lfs
$ git lfs install

# Linux
$ sudo apt install git-lfs  # Ubuntu/Debian
$ sudo yum install git-lfs  # CentOS/RHEL
$ git lfs install

확인:

$ git lfs version
git-lfs/3.4.0 (GitHub; linux amd64; go 1.20.1)

저장소에서 LFS 활성화:

$ cd my-repo
$ git lfs install
Updated git hooks.
Git LFS initialized.

이 명령은 .git/hooks/에 LFS 훅을 설치합니다.

.gitattributes로 LFS 추적 패턴 지정¶

.gitattributes 설정:

어떤 파일을 LFS로 관리할지 지정합니다.

# 특정 확장자 추적
$ git lfs track "*.psd"
$ git lfs track "*.ai"
$ git lfs track "*.mp4"
$ git lfs track "*.zip"

# 특정 디렉토리의 모든 파일
$ git lfs track "assets/**"

# 추적 패턴 확인
$ git lfs track
Listing tracked patterns
    *.psd (.gitattributes)
    *.ai (.gitattributes)
    *.mp4 (.gitattributes)
    *.zip (.gitattributes)

이 명령은 .gitattributes 파일을 자동 생성/수정합니다:

.gitattributes:

*.psd filter=lfs diff=lfs merge=lfs -text
*.ai filter=lfs diff=lfs merge=lfs -text
*.mp4 filter=lfs diff=lfs merge=lfs -text
*.zip filter=lfs diff=lfs merge=lfs -text

LFS 파일 커밋:

$ git add design.psd
$ git add .gitattributes
$ git commit -m "feat: LFS로 디자인 파일 관리"
$ git push

LFS 파일은 자동으로 LFS 서버에 업로드되고, Git 저장소에는 포인터만 저장됩니다.

포인터 파일 내용:

version https://git-lfs.github.com/spec/v1
oid sha256:4d7a214614ab2935c943f9e0ff69d22eadbb8f32b1258daaa5e2ca24d17e2393
size 123456789

LFS 사용량과 비용¶

GitHub LFS 무료 한도:

  • 스토리지: 1GB
  • 대역폭: 월 1GB (다운로드)

초과 시 과금:

  • 스토리지: $5/월당 50GB
  • 대역폭: $5/월당 50GB

사용량 확인:

Settings → Billing → Git LFS Data

최적화 팁:

  1. 정말 필요한 파일만 LFS로:
  2. 작은 이미지(<1MB)는 Git으로 관리 가능

  3. 자주 변경되지 않는 파일만 LFS

  4. 로컬에서 LFS 파일 스킵:

# LFS 파일 없이 클론 (포인터만)
$ GIT_LFS_SKIP_SMUDGE=1 git clone https://github.com/user/repo.git

# 나중에 필요한 LFS 파일만 다운로드
$ git lfs pull --include="assets/logo.png"
  1. 오래된 LFS 객체 정리:
$ git lfs prune

주의: 프로젝트가 커지면 LFS 비용이 빠르게 증가할 수 있습니다. 대안으로 외부 CDN이나 클라우드 스토리지를 고려하세요.

기존 저장소에 LFS 도입하기¶

이미 커밋된 대용량 파일을 LFS로 마이그레이션:

# 1. LFS 설치 및 활성화
$ git lfs install

# 2. 추적 패턴 설정
$ git lfs track "*.psd"

# 3. 기존 파일을 LFS로 마이그레이션
$ git lfs migrate import --include="*.psd" --everything

# 4. force push (히스토리 재작성됨)
$ git push --force --all
$ git lfs push --all origin

--everything은 모든 브랜치와 태그를 처리합니다.

마이그레이션 영향: - 커밋 해시 변경 - 저장소 크기 감소 - 팀원 동기화 필요

팀원 동기화:

$ cd ..
$ rm -rf old-repo
$ git clone https://github.com/user/repo.git new-repo

베스트 프랙티스: LFS 도입은 프로젝트 초기에 하세요. 나중에 마이그레이션하면 히스토리 재작성이 필요합니다.

절 요약¶

  • Git은 바이너리 대용량 파일에 비효율적입니다
  • Git LFS는 대용량 파일을 별도 서버에 저장하고 포인터만 Git에 유지합니다
  • .gitattributes에 LFS 추적 패턴을 지정합니다
  • GitHub LFS는 1GB 스토리지와 월 1GB 대역폭이 무료입니다
  • 기존 저장소에 LFS 도입 시 git lfs migrate로 마이그레이션합니다

FAQ¶

Q1: GPG 키 비밀번호를 매번 입력하기 번거로워요. 자동화할 수 없나요?

A1: gpg-agent를 설정하여 일정 시간 비밀번호를 캐시할 수 있습니다. ~/.gnupg/gpg-agent.conf에 default-cache-ttl 3600 (1시간 캐시)을 추가하고 gpg-connect-agent reloadagent /bye로 적용하세요. 또는 SSH 키 서명(Git 2.34+)을 사용하면 SSH 에이전트가 자동으로 처리합니다.

Q2: .env 파일을 실수로 커밋했는데, 이미 원격에 push했습니다. 어떻게 해야 하나요?

A2: 다음 순서로 조치하세요. (1) .env의 모든 Secret 즉시 무효화/재발급, (2) git filter-repo로 히스토리에서 .env 완전 제거, (3) force push, (4) 팀원에게 저장소 다시 클론 요청. 가장 중요한 것은 1번입니다. 히스토리에서 제거해도 누군가 이미 복사했을 수 있습니다.

Q3: Dependabot PR이 너무 많이 올라와서 부담스러워요. 어떻게 조절하나요?

A3: .github/dependabot.yml에서 open-pull-requests-limit을 5 이하로 설정하고, schedule.interval을 weekly 또는 monthly로 변경하세요. 또한 ignore 설정으로 메이저 버전 업데이트를 제외하거나, groups 기능으로 관련 패키지를 하나의 PR로 묶을 수 있습니다.

Q4: Git LFS 비용이 걱정됩니다. 대안이 있나요?

A4: 여러 대안이 있습니다. (1) AWS S3나 Cloudinary 같은 외부 스토리지에 파일 저장하고 URL만 Git에 커밋, (2) 디자인 파일은 Figma나 Sketch Cloud 사용, (3) 빌드 산출물은 Git에 포함하지 않고 CI/CD에서 생성, (4) 필수가 아닌 대용량 파일은 별도 저장소로 분리. LFS는 정말 필요한 경우에만 사용하세요.

Q5: Secret Scanning이 false positive를 너무 많이 내요. 끌 수 있나요?

A5: Secret Scanning을 끄는 것보다 특정 Alert를 "False positive"로 표시하는 것이 좋습니다. Security → Secret scanning alerts에서 Alert를 "Dismissed as false positive"로 처리하면 같은 패턴에 대해 다시 알림이 오지 않습니다. 테스트용 더미 키는 주석으로 표시하세요. (# dummy-key-for-testing).

이번 장에서는 Git과 GitHub의 보안 기능을 배웠습니다. GPG/SSH로 커밋을 서명하여 작성자를 증명하고, GitHub Secrets와 Secret Scanning으로 민감 데이터를 안전하게 관리하며, Dependabot으로 의존성 취약점을 자동 탐지했습니다. 또한 git filter-repo로 히스토리에서 민감 데이터를 완전히 제거하고, Git LFS로 대용량 파일을 효율적으로 관리하는 방법을 익혔습니다. 이제 보안이 강화된 저장소를 운영할 수 있습니다.
