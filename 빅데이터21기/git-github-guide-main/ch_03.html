03장: 브랜치와 병합의 기초

입문

브랜치와 병합의 기초¶

학습 목표 - 브랜치가 커밋을 가리키는 경량 포인터임을 이해할 수 있다 - git switch와 git branch로 브랜치를 생성, 전환, 삭제할 수 있다 - Fast-Forward 병합과 3-Way 병합의 차이를 설명할 수 있다 - 병합 충돌을 침착하게 해결하고 병합을 완료할 수 있다 - 기본적인 브랜치 전략의 종류를 알고 적절한 전략을 선택할 수 있다

브랜치의 개념과 필요성¶

도입¶

"동시에 여러 기능을 개발하고 싶은데 서로 영향을 주지 않게 할 수 있을까?" 브랜치(Branch)는 이 문제의 해답입니다. 브랜치를 사용하면 안정적인 코드를 유지하면서 새로운 기능을 실험할 수 있습니다.

핵심 개념 설명¶

브랜치란 무엇인가: 평행 세계 비유¶

브랜치(Branch)는 커밋 히스토리에서 분기하여 독립적인 작업 흐름을 만드는 메커니즘입니다.

소설의 평행 세계와 같습니다. 주인공이 선택의 갈림길에서 각기 다른 세계가 펼쳐지듯, 브랜치를 만들면 독립적인 작업 흐름이 생깁니다. 나중에 원하는 세계를 선택하거나 합칠 수 있습니다.

브랜치는 커밋을 가리키는 포인터¶

많은 사람이 브랜치를 만들면 프로젝트 전체가 복사된다고 생각하지만, Git 브랜치는 단순히 특정 커밋을 가리키는 40바이트짜리 포인터에 불과합니다. 따라서 브랜치 생성은 거의 즉시 완료되며 디스크 공간도 거의 차지하지 않습니다.

# 현재 브랜치 확인
$ git branch
* main

# 브랜치 목록과 각각이 가리키는 커밋 확인
$ git branch -v
* main    7f3a9e8 feat: 로그인 기능 추가
  feature 5d2c1a3 feat: 회원가입 폼 작성 중

HEAD: 현재 위치를 나타내는 특수 포인터¶

HEAD는 현재 체크아웃된 브랜치(또는 커밋)를 가리키는 특수한 참조 포인터입니다. "지금 내가 어디에 있는가"를 나타냅니다.

'현재 위치' 표시가 있는 지도와 같습니다. 지도 위 어디에 있든 현재 위치 아이콘은 항상 나의 위치를 표시합니다.

# HEAD가 가리키는 커밋 확인
$ git log --oneline -1
7f3a9e8 (HEAD -> main) feat: 로그인 기능 추가

이 출력에서 HEAD -> main은 HEAD가 main 브랜치를 가리키고, main 브랜치는 7f3a9e8 커밋을 가리킨다는 의미입니다.

왜 브랜치가 필요한가: 실무 시나리오¶

실무에서 브랜치가 필요한 상황들:

  1. 기능 개발: 새 기능을 개발하는 동안 main은 안정적인 상태 유지
  2. 버그 수정: 긴급 버그를 수정하는 동안 개발 중인 기능과 분리
  3. 실험: 불확실한 아이디어를 안전하게 테스트
  4. 협업: 여러 개발자가 서로 방해하지 않고 동시 작업

예시: 웹사이트를 운영 중인데 새로운 결제 시스템을 개발해야 합니다. 하지만 기존 결제 시스템은 계속 작동해야 합니다. 브랜치를 사용하면 main에서는 기존 시스템이 작동하고, feature/payment 브랜치에서는 새 시스템을 개발할 수 있습니다.

베스트 프랙티스: main 브랜치는 항상 배포 가능한 안정적인 상태로 유지하고, 모든 작업은 별도 브랜치에서 진행하세요.

절 요약¶

  • 브랜치는 독립적인 작업 흐름을 만드는 메커니즘입니다
  • Git 브랜치는 특정 커밋을 가리키는 경량 포인터입니다
  • HEAD는 현재 체크아웃된 브랜치를 가리킵니다
  • 브랜치는 기능 개발, 버그 수정, 실험, 협업에 필수적입니다

브랜치 생성과 전환¶

도입¶

브랜치의 개념을 이해했으니 이제 실제로 만들어보고 사용해봅시다. Git 2.23부터는 git switch라는 새로운 명령어가 도입되어 브랜치 전환이 더 직관적이 되었습니다.

핵심 개념 설명¶

git branch: 브랜치 생성 및 목록 확인¶

git branch는 브랜치를 생성하고 관리하는 명령어입니다.

# 현재 브랜치 목록 확인
$ git branch
* main
  feature/login

# 새 브랜치 생성 (전환하지 않음)
$ git branch feature/signup

# 브랜치와 마지막 커밋 확인
$ git branch -v
* main           7f3a9e8 feat: 로그인 기능 추가
  feature/login  5d2c1a3 feat: 로그인 폼 작성
  feature/signup 7f3a9e8 feat: 로그인 기능 추가

# 모든 브랜치 (원격 포함) 확인
$ git branch -a

* 표시는 현재 체크아웃된 브랜치를 나타냅니다.

git switch: 브랜치 전환 (Git 2.23+)¶

git switch는 브랜치를 전환하는 명령어로, Git 2.23에서 git checkout의 브랜치 전환 기능을 분리하여 만들어졌습니다.

TV 채널을 변경하는 것과 같습니다. 리모컨으로 채널을 바꾸면 다른 프로그램이 나오듯, git switch로 브랜치를 바꾸면 해당 브랜치의 파일 상태가 작업 디렉토리에 나타납니다.

# 기존 브랜치로 전환
$ git switch feature/login

# 현재 브랜치 확인
$ git branch
  main
* feature/login
  feature/signup

git switch -c: 생성과 전환을 동시에¶

가장 자주 사용하는 패턴은 브랜치를 만들고 바로 전환하는 것입니다.

# 새 브랜치 생성과 동시에 전환
$ git switch -c feature/dashboard

# 위 명령은 다음 두 명령을 합친 것과 같음
$ git branch feature/dashboard
$ git switch feature/dashboard

git checkout과의 차이점¶

Git 2.23 이전에는 git checkout이 브랜치 전환과 파일 복원을 모두 담당했습니다. 이는 혼란스러웠기 때문에 두 기능이 분리되었습니다.

  • git switch: 브랜치 전환 전용
  • git restore: 파일 복원 전용 (2장에서 배움)
  • git checkout: 여전히 사용 가능 (호환성 유지)
# 구식 방법 (여전히 작동함)
$ git checkout feature/login
$ git checkout -b feature/new

# 현대적 방법 (권장)
$ git switch feature/login
$ git switch -c feature/new

팁: Git 2.23 이상을 사용한다면 git switch와 git restore를 사용하세요. 더 명확하고 안전합니다.

브랜치 삭제: git branch -d / -D¶

더 이상 필요 없는 브랜치는 삭제할 수 있습니다.

# 병합된 브랜치 삭제 (안전)
$ git branch -d feature/login

# 병합되지 않은 브랜치 강제 삭제 (주의)
$ git branch -D feature/abandoned

-d 옵션은 병합되지 않은 브랜치를 삭제하려 하면 경고를 표시하여 실수를 방지합니다. -D는 경고 없이 강제 삭제합니다.

주의: 브랜치를 삭제해도 커밋은 일정 기간 동안 reflog에 남아있어 복구가 가능합니다 (7장에서 배움).

절 요약¶

  • git branch는 브랜치를 생성하고 목록을 확인합니다
  • git switch <branch>는 브랜치를 전환합니다
  • git switch -c <branch>는 브랜치 생성과 전환을 동시에 합니다
  • Git 2.23+에서는 git switch가 git checkout보다 권장됩니다
  • git branch -d는 안전하게, -D는 강제로 브랜치를 삭제합니다

브랜치 병합하기: merge¶

도입¶

별도 브랜치에서 작업을 완료했다면 이제 main 브랜치에 합쳐야 합니다. 이 과정을 병합(Merge)이라고 하며, Git은 두 가지 병합 방식을 자동으로 선택합니다.

핵심 개념 설명¶

Fast-Forward 병합: 직선 히스토리¶

Fast-Forward 병합은 분기 이후 main 브랜치에 새로운 커밋이 없을 때 발생합니다. Git은 단순히 main 브랜치 포인터를 앞으로 이동시킵니다.

# main에서 시작
$ git switch main

# feature 브랜치 병합
$ git merge feature/login
Updating 7f3a9e8..a1b2c3d
Fast-forward
 login.html | 50 ++++++++++++++++++++++++++++++++++++++++++++++++++
 login.js   | 30 ++++++++++++++++++++++++++++++
 2 files changed, 80 insertions(+)

"Fast-forward"라는 메시지는 단순히 포인터가 이동했음을 의미합니다. 새로운 병합 커밋은 생성되지 않습니다.

3-Way 병합: 두 갈래 합치기¶

3-Way 병합은 main과 feature 브랜치가 각각 다른 커밋을 가지고 있을 때 발생합니다. Git은 두 브랜치와 공통 조상을 비교하여 새로운 병합 커밋을 생성합니다.

$ git merge feature/signup
Merge made by the 'recursive' strategy.
 signup.html | 60 +++++++++++++++++++++++++++++++++++++++++++++++++++++
 signup.js   | 40 ++++++++++++++++++++++++++++++++++
 2 files changed, 100 insertions(+)

병합 커밋(Merge Commit)의 의미¶

3-Way 병합에서 생성되는 병합 커밋은 두 개의 부모 커밋을 가집니다. 이는 "이 시점에서 두 브랜치가 합쳐졌다"는 역사적 기록입니다.

$ git log --oneline --graph
*   d5e6f7a (HEAD -> main) Merge branch 'feature/signup'
|\
| * b4c5d6e feat: 회원가입 폼 완성
| * a3b4c5d feat: 회원가입 유효성 검사 추가
* | c4d5e6f docs: README 업데이트
|/
* 7f3a9e8 feat: 로그인 기능 추가

--no-ff 옵션으로 명시적 병합 커밋 생성¶

Fast-Forward가 가능한 상황에서도 병합 커밋을 만들고 싶다면 --no-ff 옵션을 사용합니다.

$ git merge --no-ff feature/login

이렇게 하면 브랜치의 존재가 히스토리에 명확히 기록됩니다. 많은 팀이 이 방식을 선호합니다.

베스트 프랙티스: 팀에서 브랜치 히스토리를 명확히 유지하고 싶다면 --no-ff를 기본값으로 설정하세요. bash $ git config --global merge.ff false

절 요약¶

  • Fast-Forward 병합은 포인터만 이동하며 새 커밋을 만들지 않습니다
  • 3-Way 병합은 두 브랜치를 합친 새 병합 커밋을 생성합니다
  • 병합 커밋은 두 개의 부모 커밋을 가집니다
  • --no-ff 옵션으로 Fast-Forward를 방지하고 명시적 병합 커밋을 만들 수 있습니다

병합 충돌 해결하기¶

도입¶

두 브랜치가 같은 파일의 같은 부분을 다르게 수정했다면 Git은 어느 쪽이 맞는지 판단할 수 없습니다. 이때 병합 충돌(Merge Conflict)이 발생하며, 개발자가 직접 해결해야 합니다. 충돌은 두렵지 않습니다. 체계적으로 접근하면 누구나 해결할 수 있습니다.

핵심 개념 설명¶

충돌이 발생하는 이유¶

두 명의 편집자가 같은 문단을 서로 다르게 수정한 원고를 받은 편집장과 같습니다. 어느 쪽을 택할지, 혹은 둘을 어떻게 조합할지 직접 판단해야 합니다.

충돌 예시:

$ git merge feature/update-title
Auto-merging index.html
CONFLICT (content): Merge conflict in index.html
Automatic merge failed; fix conflicts and then commit the result.

충돌 마커 읽기: <<<<<<<, =======, >>>>>>>¶

충돌이 발생한 파일을 열면 다음과 같은 마커가 표시됩니다.

<title>
<<<<<<< HEAD
  Git & GitHub 완벽 가이드
=======
  Git과 GitHub 마스터하기
>>>>>>> feature/update-title
</title>
  • <<<<<<< HEAD: 현재 브랜치(main)의 내용 시작
  • =======: 구분선
  • >>>>>>> feature/update-title: 병합하려는 브랜치의 내용 끝

충돌 해결 단계별 가이드¶

  1. 충돌 파일 확인
$ git status
Unmerged paths:
  (use "git add <file>..." to mark resolution)
        both modified:   index.html

  1. 파일 열어서 충돌 마커 찾기 에디터로 index.html을 엽니다.

  2. 원하는 내용으로 수정 충돌 마커를 제거하고 최종 내용을 작성합니다.

<title>
  Git & GitHub 올인원 가이드
</title>
  1. 파일 저장하고 스테이징
$ git add index.html
  1. 병합 완료
$ git commit

커밋 메시지가 자동으로 생성되지만 필요시 수정할 수 있습니다.

병합 중단하기: git merge --abort¶

충돌 해결이 복잡하거나 잘못 진행 중이라면 병합을 중단할 수 있습니다.

$ git merge --abort

이 명령은 병합 전 상태로 완전히 되돌립니다.

팁: 충돌 해결에 자신이 없다면 주저하지 말고 git merge --abort로 중단한 후, 팀원에게 도움을 요청하세요.

VS Code에서 충돌 해결하기¶

VS Code는 충돌 해결을 위한 편리한 UI를 제공합니다.

  1. 충돌 파일을 열면 다음 버튼이 표시됩니다:
  2. Accept Current Change: 현재 브랜치 내용 선택
  3. Accept Incoming Change: 병합 브랜치 내용 선택
  4. Accept Both Changes: 양쪽 모두 유지

  5. Compare Changes: 비교 뷰 열기

  6. 원하는 옵션을 클릭하거나 직접 편집합니다.

  7. 저장 후 터미널에서 git add 및 git commit으로 완료합니다.

베스트 프랙티스: 큰 충돌은 GUI 도구(VS Code, GitKraken, Sourcetree 등)를 사용하면 훨씬 쉽게 해결할 수 있습니다.

절 요약¶

  • 병합 충돌은 같은 파일의 같은 부분을 다르게 수정했을 때 발생합니다
  • 충돌 마커는 <<<<<<<, =======, >>>>>>>로 표시됩니다
  • 충돌 해결 순서: 파일 확인 → 수정 → add → commit
  • git merge --abort로 병합을 중단하고 이전 상태로 되돌릴 수 있습니다
  • VS Code 같은 GUI 도구를 사용하면 충돌 해결이 더 쉽습니다

브랜치 전략 소개¶

도입¶

브랜치를 어떻게 관리할 것인가는 팀의 생산성에 큰 영향을 미칩니다. 이 절에서는 기본적인 브랜치 전략을 소개합니다. 11장에서 더 자세한 전략을 배웁니다.

핵심 개념 설명¶

기능 브랜치(Feature Branch) 패턴¶

가장 기본적인 패턴은 각 기능마다 별도 브랜치를 만드는 것입니다.

# main에서 새 기능 브랜치 생성
$ git switch -c feature/user-profile

# 작업 후 커밋
$ git commit -m "feat: 사용자 프로필 페이지 추가"

# main으로 돌아와 병합
$ git switch main
$ git merge feature/user-profile

# 브랜치 삭제
$ git branch -d feature/user-profile

브랜치 네이밍 규칙¶

일관된 네이밍 규칙은 협업의 첫걸음입니다.

일반적인 접두사: - feature/ - 새 기능 (예: feature/login, feature/payment) - bugfix/ - 버그 수정 (예: bugfix/login-error) - hotfix/ - 긴급 수정 (예: hotfix/security-patch) - docs/ - 문서 작업 (예: docs/api-guide) - refactor/ - 리팩터링 (예: refactor/user-service) - test/ - 테스트 추가 (예: test/login-validation)

좋은 브랜치 이름 예시:

feature/add-dark-mode
bugfix/fix-login-redirect
hotfix/critical-security-update
docs/update-contributing-guide

피해야 할 브랜치 이름:

my-branch          (무엇을 하는지 불명확)
test               (어떤 테스트인지 불명확)
fix                (무엇을 고치는지 불명확)

베스트 프랙티스: 브랜치 이름은 Issue 번호를 포함하면 더 명확합니다. 예: feature/123-add-user-profile, bugfix/456-fix-login-error

주요 브랜치 전략 미리보기 (Git Flow, GitHub Flow)¶

Git Flow: 복잡하지만 체계적 - main, develop, feature, release, hotfix 브랜치 사용 - 릴리스 주기가 긴 프로젝트에 적합

GitHub Flow: 단순하고 빠름 - main과 feature 브랜치만 사용 - PR 기반 협업, 빈번한 배포

11장에서 각 전략을 자세히 배웁니다.

어떤 전략을 선택할 것인가¶

  • 소규모 팀, 빠른 배포: GitHub Flow
  • 대규모 팀, 정기 릴리스: Git Flow
  • 초대형 프로젝트: Trunk-Based Development

팁: 처음에는 단순한 전략으로 시작하고, 팀이 성장하면서 점진적으로 복잡도를 높이세요.

절 요약¶

  • 기능 브랜치 패턴은 각 기능마다 별도 브랜치를 만드는 기본 전략입니다
  • 일관된 브랜치 네이밍 규칙은 협업을 원활하게 합니다
  • Git Flow는 복잡하지만 체계적이고, GitHub Flow는 단순하고 빠릅니다
  • 팀의 규모와 배포 주기에 맞는 전략을 선택하세요

FAQ¶

Q1: 브랜치를 전환하려는데 "변경 사항이 덮어써질 수 있다"는 경고가 나옵니다.

A1: 현재 브랜치에 커밋하지 않은 변경 사항이 있고, 전환하려는 브랜치와 충돌하기 때문입니다. 해결 방법은 세 가지입니다. (1) 변경 사항을 커밋, (2) git stash로 임시 저장 (7장에서 배움), (3) git restore .로 변경 취소.

Q2: Fast-Forward 병합과 3-Way 병합 중 어느 것이 더 좋나요?

A2: 상황에 따라 다릅니다. Fast-Forward는 히스토리가 선형적이고 간결하지만 브랜치 존재가 기록되지 않습니다. 3-Way 병합은 히스토리가 복잡하지만 브랜치의 흐름이 명확히 보입니다. 많은 팀이 git merge --no-ff로 항상 병합 커밋을 만드는 것을 선호합니다.

Q3: 병합 충돌이 자주 발생합니다. 어떻게 줄일 수 있나요?

A3: (1) 브랜치를 오래 유지하지 말고 자주 병합하세요. (2) main의 변경 사항을 정기적으로 feature 브랜치로 가져오세요(git merge main). (3) 팀원과 작업 영역을 조율하여 같은 파일을 동시에 수정하지 않도록 하세요. (4) 코드 리뷰를 통해 변경 범위를 미리 파악하세요.

Q4: 실수로 브랜치를 삭제했는데 복구할 수 있나요?

A4: 네, 브랜치를 삭제해도 커밋은 일정 기간 남아있습니다. git reflog로 삭제된 브랜치의 마지막 커밋 해시를 찾은 후 git switch -c <브랜치명> <커밋해시>로 브랜치를 다시 만들 수 있습니다. 자세한 내용은 7장에서 배웁니다.

Q5: main 브랜치에 있을 때 실수로 파일을 수정했습니다. 이 변경을 feature 브랜치로 옮길 수 있나요?

A5: 네, 변경 사항을 커밋하지 않았다면 간단합니다. (1) git stash로 변경 저장, (2) git switch feature/my-feature, (3) git stash pop으로 변경 복원. 이미 커밋했다면 git cherry-pick을 사용합니다 (7장에서 배움).

장 요약¶

이번 장에서는 Git의 강력한 기능인 브랜치를 배웠습니다. 브랜치는 단순히 커밋을 가리키는 경량 포인터이며, git switch와 git branch로 쉽게 생성하고 전환할 수 있습니다. 병합은 Fast-Forward와 3-Way 두 가지 방식이 있으며, 충돌이 발생해도 체계적으로 해결할 수 있습니다. 기능 브랜치 패턴과 네이밍 규칙을 익혀 팀 협업의 기초를 다졌습니다.
