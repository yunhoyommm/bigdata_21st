12장: 모노레포와 대규모 저장소 관리

실무

모노레포와 대규모 저장소 관리¶

학습 목표 - 모노레포와 멀티레포의 장단점을 비교하고 적절한 전략을 선택할 수 있다 - Git 서브모듈을 추가, 업데이트, 관리할 수 있다 - sparse-checkout과 partial clone으로 대규모 저장소를 효율적으로 다룰 수 있다 - Git 객체 모델(blob, tree, commit, tag)의 구조를 이해할 수 있다 - git gc와 성능 최적화 옵션으로 저장소 성능을 개선할 수 있다

모노레포 vs 멀티레포¶

도입¶

프로젝트가 성장하면서 "여러 프로젝트를 하나의 저장소에 관리할까, 각각 독립 저장소로 분리할까?"라는 선택의 기로에 서게 됩니다. 이 절에서는 모노레포(Monorepo)와 멀티레포(Multi-repo)의 차이를 명확히 이해하고, 각각의 장단점을 비교합니다.

핵심 개념 설명¶

모노레포란 무엇인가¶

모노레포(Monorepo)는 여러 프로젝트(서비스, 라이브러리 등)의 소스 코드를 하나의 저장소에서 관리하는 전략입니다.

대형 쇼핑몰과 같습니다. 의류, 식품, 전자제품 등 다양한 매장(프로젝트)이 하나의 건물(저장소)에 있어 공용 시설(공통 코드)을 함께 사용하고 관리합니다.

monorepo/
├── packages/
│   ├── frontend-web/
│   │   ├── src/
│   │   └── package.json
│   ├── frontend-mobile/
│   │   ├── src/
│   │   └── package.json
│   ├── backend-api/
│   │   ├── src/
│   │   └── package.json
│   └── shared-components/
│       ├── src/
│       └── package.json
├── tools/
├── .github/
└── README.md

반면 멀티레포(Multi-repo)는 각 프로젝트를 독립된 저장소로 관리하는 전통적인 방식입니다.

organization/
├── repo-frontend-web/
├── repo-frontend-mobile/
├── repo-backend-api/
└── repo-shared-components/

모노레포의 장점: 코드 공유, 원자적 변경, 일관성¶

1. 코드 공유가 쉬움

공통 컴포넌트, 유틸리티, 타입 정의 등을 여러 프로젝트에서 즉시 사용할 수 있습니다. 별도 패키지 배포 과정 없이 직접 참조합니다.

// frontend-web/src/App.js
import { Button } from '../../shared-components/src/Button';

2. 원자적 변경(Atomic Changes)

여러 프로젝트에 걸친 변경 사항을 하나의 커밋으로 묶을 수 있습니다. API 스키마를 변경하면서 프론트엔드와 백엔드를 동시에 수정할 수 있습니다.

$ git commit -m "feat: Add user avatar field to API and frontend"
# 하나의 커밋에 backend-api/, frontend-web/ 변경이 모두 포함

3. 일관성 유지

린트 규칙, 코드 스타일, 의존성 버전을 전체 프로젝트에 일괄 적용할 수 있습니다.

// 루트 package.json
{
  "workspaces": ["packages/*"],
  "devDependencies": {
    "eslint": "^8.50.0",
    "prettier": "^3.0.0"
  }
}

4. 리팩터링 용이

전체 코드베이스를 검색하고 한 번에 리팩터링할 수 있습니다. 함수 이름을 변경하면 모든 사용처가 즉시 업데이트됩니다.

5. 단순한 의존성 관리

npm/yarn workspaces, lerna 등의 도구로 의존성을 중앙 집중식으로 관리합니다.

베스트 프랙티스: 모노레포에서는 Turborepo, Nx, Bazel 같은 빌드 시스템 도구를 사용하여 변경된 프로젝트만 재빌드하는 증분 빌드(Incremental Build)를 구현하세요.

모노레포의 단점: 성능, 권한 관리, 빌드 복잡도¶

1. 저장소 크기 증가

모든 프로젝트의 히스토리가 누적되어 clone 시간이 길어집니다. 수백 GB에 달하는 저장소도 흔합니다.

2. 권한 관리 어려움

Git은 저장소 단위로 권한을 관리하므로, 특정 팀에게 특정 디렉토리만 접근 권한을 주기 어렵습니다. GitHub의 CODEOWNERS로 부분적으로 해결 가능합니다.

3. 빌드 시간 증가

전체 프로젝트를 빌드하면 시간이 오래 걸립니다. 증분 빌드 도구 없이는 비효율적입니다.

4. CI/CD 복잡도

어떤 프로젝트가 변경되었는지 감지하고, 해당 프로젝트만 테스트/배포하는 로직이 필요합니다.

5. Git 성능 저하

수만 개의 파일이 있는 저장소에서 git status, git diff 등의 명령어가 느려질 수 있습니다.

멀티레포와의 비교: 선택 기준¶

기준 모노레포 멀티레포
코드 공유 쉬움 (직접 참조) 어려움 (패키지 배포 필요)
원자적 변경 가능 (하나의 커밋) 불가능 (여러 PR 필요)
권한 관리 어려움 (저장소 단위) 쉬움 (저장소별 권한)
빌드 속도 느림 (전체 빌드) 빠름 (독립 빌드)
저장소 크기 큼 작음
도구 필요성 높음 (빌드 시스템) 낮음
팀 독립성 낮음 (공유 의존성) 높음 (독립 의존성)

모노레포가 적합한 경우: - 프론트엔드/백엔드/모바일 앱이 긴밀히 연결된 프로젝트 - 공유 컴포넌트 라이브러리가 많은 경우 - 팀 간 협업이 빈번한 경우 - Google, Meta처럼 빌드 인프라가 잘 구축된 경우

멀티레포가 적합한 경우: - 독립적으로 배포되는 마이크로서비스 - 팀이 완전히 분리되어 있는 경우 - 저장소 크기와 빌드 시간이 중요한 경우 - 외부 오픈소스 프로젝트 (권한 관리 필요)

Google, Microsoft의 모노레포 사례¶

Google: - 20억 줄 이상의 코드를 하나의 모노레포에서 관리 - Blaze(현재 Bazel)라는 자체 빌드 시스템 사용 - 매일 수만 건의 커밋 처리

Microsoft: - Windows 소스 코드를 Git 기반 모노레포로 전환 (2017년) - VFS for Git(현재 Scalar)로 Git 성능 개선 - 300GB 이상의 저장소를 효율적으로 관리

Meta(Facebook): - Mercurial 기반 모노레포 사용 - Sapling이라는 Git 호환 VCS 개발 - 수백만 개의 파일을 관리

팁: 소규모 팀이라면 모노레포의 복잡도를 과소평가하지 마세요. 처음에는 멀티레포로 시작하고, 코드 공유 필요성이 명확해지면 모노레포로 전환하는 것도 좋은 전략입니다.

절 요약¶

  • 모노레포는 여러 프로젝트를 하나의 저장소에서 관리하는 전략입니다
  • 코드 공유, 원자적 변경, 일관성 유지가 장점입니다
  • 저장소 크기, 권한 관리, 빌드 복잡도가 단점입니다
  • 팀 협업이 빈번하고 공유 코드가 많으면 모노레포가 유리합니다
  • Google, Microsoft, Meta 같은 대기업들이 모노레포를 성공적으로 운영하고 있습니다

Git 서브모듈¶

도입¶

외부 라이브러리나 공유 컴포넌트를 프로젝트에 포함시키고 싶지만, 복사-붙여넣기는 유지보수가 어렵습니다. Git 서브모듈(Submodule)은 하나의 저장소 안에 다른 저장소를 중첩시켜 관리하는 기능입니다.

핵심 개념 설명¶

서브모듈의 개념과 사용 시나리오¶

Git 서브모듈(Submodule)은 하나의 Git 저장소 안에 다른 Git 저장소를 중첩시켜 관리하는 기능입니다.

아파트의 세입자와 같습니다. 아파트(부모 저장소)에 세입자(서브모듈)가 살고 있으며, 세입자는 자신만의 생활(독립 저장소)을 하지만 아파트 규칙(특정 커밋 참조)을 따릅니다.

사용 시나리오: - 외부 오픈소스 라이브러리를 프로젝트에 포함 - 팀 간 공유 컴포넌트 라이브러리 관리 - 프로젝트 간 공통 설정 파일 공유 - 문서 저장소를 메인 프로젝트에 포함

서브모듈 추가: git submodule add¶

# 현재 프로젝트에 외부 라이브러리 추가
$ git submodule add https://github.com/user/awesome-lib.git libs/awesome-lib

# 출력:
Cloning into '/home/user/myproject/libs/awesome-lib'...
remote: Enumerating objects: 100, done.
remote: Counting objects: 100% (100/100), done.
Receiving objects: 100% (100/100), done.

이 명령은 다음 작업을 수행합니다: 1. libs/awesome-lib 디렉토리에 외부 저장소를 clone 2. .gitmodules 파일에 서브모듈 정보 기록 3. 서브모듈 디렉토리를 특수한 "gitlink" 타입으로 추적

.gitmodules 파일:

[submodule "libs/awesome-lib"]
    path = libs/awesome-lib
    url = https://github.com/user/awesome-lib.git

커밋 확인:

$ git status
On branch main
Changes to be committed:
  new file:   .gitmodules
  new file:   libs/awesome-lib

$ git commit -m "feat: Add awesome-lib as submodule"

부모 저장소는 서브모듈의 특정 커밋 해시를 기록합니다. 서브모듈 저장소가 업데이트되어도 부모 저장소는 고정된 버전을 가리킵니다.

서브모듈 포함 저장소 클론: --recurse-submodules¶

일반적으로 clone하면 서브모듈 디렉토리는 비어있습니다.

# 잘못된 방법
$ git clone https://github.com/myteam/myproject.git
$ cd myproject
$ ls libs/awesome-lib
# 비어있음!

올바른 방법:

# 방법 1: 클론 시 --recurse-submodules 옵션 사용
$ git clone --recurse-submodules https://github.com/myteam/myproject.git

# 방법 2: 이미 클론한 경우 서브모듈 초기화
$ git clone https://github.com/myteam/myproject.git
$ cd myproject
$ git submodule init
$ git submodule update

# 방법 2의 단축 명령
$ git submodule update --init --recursive

--recursive 옵션은 서브모듈 안에 또 다른 서브모듈이 있는 경우를 처리합니다.

주의: 팀원들에게 항상 --recurse-submodules 옵션을 사용하도록 안내하세요. 그렇지 않으면 빌드 실패의 원인이 됩니다.

서브모듈 업데이트와 동기화¶

서브모듈의 원격 저장소에 새 커밋이 추가되었을 때 업데이트하는 방법:

1. 서브모듈 디렉토리로 이동하여 수동 업데이트:

$ cd libs/awesome-lib
$ git fetch
$ git merge origin/main
$ cd ../..
$ git add libs/awesome-lib
$ git commit -m "chore: Update awesome-lib to v2.1.0"

2. 부모 저장소에서 한 번에 업데이트:

# 모든 서브모듈을 원격 브랜치의 최신으로 업데이트
$ git submodule update --remote

# 특정 서브모듈만 업데이트
$ git submodule update --remote libs/awesome-lib

# 변경 사항 커밋
$ git add .gitmodules libs/awesome-lib
$ git commit -m "chore: Update submodules"

3. 서브모듈의 특정 버전(태그)으로 고정:

$ cd libs/awesome-lib
$ git checkout v2.1.0  # 특정 태그로 이동
$ cd ../..
$ git add libs/awesome-lib
$ git commit -m "chore: Pin awesome-lib to v2.1.0"

팀원이 업데이트 받기:

$ git pull
$ git submodule update --init --recursive

베스트 프랙티스: 서브모듈은 항상 릴리스 태그로 고정하세요. main 브랜치를 추적하면 예기치 않은 변경이 포함될 수 있습니다.

서브모듈의 함정과 대안 (subtree)¶

서브모듈의 주요 문제점:

  1. 복잡한 워크플로우: 초기화, 업데이트, 푸시 등의 작업이 직관적이지 않습니다.
  2. Detached HEAD: 서브모듈 디렉토리는 기본적으로 특정 커밋을 가리키므로 브랜치가 아닌 Detached HEAD 상태입니다.
  3. 실수 위험: 서브모듈 안에서 작업 후 푸시를 잊어버리면 팀원이 해당 커밋을 받을 수 없습니다.
  4. 도구 지원 부족: 일부 IDE나 도구가 서브모듈을 잘 지원하지 않습니다.

대안: git subtree

git subtree는 서브모듈의 대안으로, 외부 저장소를 부모 저장소의 일부로 병합합니다.

# 외부 저장소를 subtree로 추가
$ git subtree add --prefix=libs/awesome-lib https://github.com/user/awesome-lib.git main --squash

# 업데이트
$ git subtree pull --prefix=libs/awesome-lib https://github.com/user/awesome-lib.git main --squash

# 변경 사항 업스트림에 푸시
$ git subtree push --prefix=libs/awesome-lib https://github.com/user/awesome-lib.git main

subtree 장점: - 일반 디렉토리처럼 취급되어 워크플로우가 단순 - 클론 시 별도 초기화 불필요

subtree 단점: - 히스토리가 부모 저장소에 병합되어 복잡해짐 - 업스트림 기여가 복잡함

팁: 읽기 전용으로 외부 라이브러리를 포함한다면 subtree, 양방향 동기화가 필요하다면 서브모듈을 고려하세요. 단, 가능하면 패키지 매니저(npm, pip 등)를 사용하는 것이 가장 좋습니다.

절 요약¶

  • Git 서브모듈은 저장소 안에 다른 저장소를 중첩시키는 기능입니다
  • git submodule add로 추가하고, .gitmodules 파일에 기록됩니다
  • git clone --recurse-submodules로 서브모듈 포함 클론합니다
  • git submodule update --remote로 서브모듈을 업데이트합니다
  • 서브모듈은 복잡하므로 대안으로 subtree나 패키지 매니저를 고려하세요

Sparse Checkout과 Partial Clone¶

도입¶

수 GB에 달하는 대규모 저장소를 클론하는 것은 시간과 디스크 공간의 낭비입니다. 특히 모노레포에서 프론트엔드 개발자가 백엔드 코드까지 모두 받을 필요는 없습니다. Git은 이를 위한 두 가지 기능을 제공합니다.

핵심 개념 설명¶

sparse-checkout: 필요한 디렉토리만 체크아웃¶

sparse-checkout은 저장소의 일부 디렉토리만 작업 디렉토리에 가져오는 기능입니다.

# 저장소 클론 (작업 디렉토리 없이)
$ git clone --filter=blob:none --no-checkout https://github.com/myorg/monorepo.git
$ cd monorepo

# sparse-checkout 활성화 (cone 모드 권장)
$ git sparse-checkout init --cone

# 필요한 디렉토리만 선택
$ git sparse-checkout set frontend shared-components

# 출력:
# frontend/
# shared-components/
# (backend/, mobile/ 등은 작업 디렉토리에 없음)

$ git checkout main

이제 frontend와 shared-components 디렉토리만 작업 디렉토리에 나타납니다. 다른 디렉토리는 Git 히스토리에는 있지만 파일 시스템에는 없습니다.

패턴 추가/제거:

# 디렉토리 추가
$ git sparse-checkout add backend/api

# 현재 패턴 확인
$ git sparse-checkout list

# sparse-checkout 해제 (전체 체크아웃)
$ git sparse-checkout disable

sparse-checkout 설정과 패턴¶

cone 모드 vs non-cone 모드:

cone 모드 (권장): - 디렉토리 단위로만 선택 - 성능 최적화됨 - 간단하고 예측 가능

$ git sparse-checkout set packages/web packages/mobile

non-cone 모드 (고급): - 파일 단위 세밀한 패턴 지정 가능 - .gitignore 문법 사용

$ git sparse-checkout init  # (--cone 없이)
$ git sparse-checkout set '/*' '!/backend' '!/docs'
# 루트 모든 파일 포함, backend와 docs 제외

실무 예시:

# 프론트엔드 개발자용
$ git sparse-checkout set frontend shared-components tools/scripts

# 백엔드 개발자용
$ git sparse-checkout set backend shared-lib tools/scripts

# CI/CD용
$ git sparse-checkout set .github tools/ci

베스트 프랙티스: cone 모드를 사용하고, 각 팀에 맞는 sparse-checkout 설정을 문서화하여 공유하세요.

partial clone: 필요한 객체만 다운로드¶

partial clone은 저장소의 일부 객체(커밋, 트리, 블롭)만 다운로드하는 기능입니다.

세 가지 필터 모드:

1. --filter=blob:none (treeless clone): - 가장 일반적인 모드 - 커밋과 트리는 모두 다운로드, 블롭(파일 내용)은 필요 시 다운로드 - 히스토리 탐색은 빠르지만 파일은 지연 로드

$ git clone --filter=blob:none https://github.com/myorg/large-repo.git

# 출력:
Cloning into 'large-repo'...
remote: Enumerating objects: 100000, done.
remote: Counting objects: 100% (5000/100000), done.
# 블롭은 checkout 시점에 다운로드됨

2. --filter=tree:0 (blobless clone): - 커밋만 다운로드, 트리와 블롭은 필요 시 다운로드 - 가장 적은 초기 다운로드 - 히스토리 탐색 시 네트워크 요청 발생

$ git clone --filter=tree:0 https://github.com/myorg/large-repo.git

3. --filter=blob:limit=: - 특정 크기 이상의 블롭만 지연 로드 - 작은 텍스트 파일은 즉시, 큰 바이너리는 나중에

$ git clone --filter=blob:limit=1m https://github.com/myorg/large-repo.git
# 1MB 이상 파일만 지연 로드

partial clone의 장점: - 초기 클론 속도 대폭 향상 (수 GB → 수십 MB) - 디스크 공간 절약 - 브랜치 전환, 히스토리 탐색은 여전히 가능

주의사항: - 네트워크 연결 필수 (오프라인 작업 제한) - 파일 접근 시 지연 발생 가능 - 일부 도구가 partial clone을 지원하지 않을 수 있음

--filter=blob:none (treeless clone)¶

가장 실용적인 모드인 --filter=blob:none을 자세히 살펴봅니다.

# 일반 클론 (10GB)
$ git clone https://github.com/myorg/huge-repo.git
Cloning into 'huge-repo'...
Receiving objects: 100% (1000000/1000000), 10.2 GiB | 50.00 MiB/s, done.

# Partial clone (500MB)
$ git clone --filter=blob:none https://github.com/myorg/huge-repo.git
Cloning into 'huge-repo'...
Receiving objects: 100% (50000/50000), 502 MiB | 50.00 MiB/s, done.
# 20배 빠름!

작업 흐름:

$ cd huge-repo

# 파일 편집 시 자동으로 블롭 다운로드
$ vim src/main.py  # 블롭이 필요하면 자동 fetch

# 히스토리 탐색은 즉시 가능
$ git log --oneline --graph  # 빠름 (커밋은 이미 있음)

# 특정 파일의 히스토리 보기
$ git log -- README.md  # 필요한 블롭만 fetch

수동 블롭 fetch:

# 특정 경로의 블롭 미리 가져오기
$ git sparse-checkout set frontend
$ git checkout main
# frontend/ 아래 모든 블롭이 다운로드됨

팁: CI/CD 환경에서는 --filter=blob:none과 sparse-checkout을 조합하면 빌드 시간을 극적으로 단축할 수 있습니다.

대규모 저장소에서의 작업 효율화¶

최적의 조합: partial clone + sparse-checkout

# 1. Partial clone (블롭 제외)
$ git clone --filter=blob:none --sparse https://github.com/myorg/monorepo.git
$ cd monorepo

# 2. Sparse-checkout 설정
$ git sparse-checkout init --cone
$ git sparse-checkout set packages/frontend tools

# 3. Checkout (필요한 블롭만 다운로드)
$ git checkout main

이 방법은: - 초기 클론: 수 GB → 수십 MB (50배 이상 빠름) - 디스크 사용량: 10GB → 200MB (50배 절약) - 작업 속도: git status, git add 등이 훨씬 빠름

GitHub Codespaces/Gitpod에서 활용: 클라우드 개발 환경에서는 이 기법이 필수입니다.

# .devcontainer/devcontainer.json
{
  "postCreateCommand": "git sparse-checkout set frontend shared"
}

Microsoft VFS for Git (현재 Scalar): Microsoft가 개발한 도구로, Windows 저장소 같은 초대형 모노레포를 다룰 때 사용합니다.

# Scalar 설치 (Git 2.38+ 내장)
$ scalar clone https://github.com/myorg/huge-repo.git
# 자동으로 partial clone, sparse-checkout, FSMonitor 등 설정

절 요약¶

  • sparse-checkout은 필요한 디렉토리만 작업 디렉토리에 가져옵니다
  • cone 모드로 디렉토리 단위 선택이 권장됩니다
  • partial clone은 필요한 객체만 다운로드합니다
  • --filter=blob:none이 가장 실용적인 partial clone 모드입니다
  • 두 기능을 조합하면 대규모 모노레포를 효율적으로 다룰 수 있습니다

Git 성능 최적화¶

도입¶

Git 저장소가 커지면서 git status가 느려지거나 디스크 공간이 부족해지는 문제에 직면할 수 있습니다. 이 절에서는 Git의 내부 구조를 이해하고 성능을 최적화하는 방법을 배웁니다.

핵심 개념 설명¶

Git 객체 모델 이해하기: blob, tree, commit, tag¶

Git 객체 모델(Object Model)은 Git이 데이터를 저장하는 내부 구조입니다.

파일 캐비닛 시스템과 같습니다. blob은 문서 원본, tree는 폴더, commit은 특정 시점의 폴더 구조 전체를 사진 찍은 것, tag는 그 사진에 붙인 이름표입니다.

네 가지 객체 타입:

1. blob (Binary Large Object): - 파일의 내용을 저장 - 파일 이름이나 경로는 저장하지 않음 - SHA-1 해시로 식별

$ echo "Hello Git" | git hash-object --stdin
8d0e41234f24b6da002d962a26c2495ea16a425f

2. tree: - 디렉토리 구조를 저장 - 파일 이름, 권한, 해당 blob/tree의 해시를 기록

tree
├── README.md (blob: a1b2c3d...)
├── src/ (tree: e4f5g6h...)
│   └── main.py (blob: i7j8k9l...)
└── .gitignore (blob: m0n1o2p...)

3. commit: - 특정 시점의 스냅샷 - tree 해시, 부모 커밋 해시, 작성자, 메시지 등 메타데이터 포함

$ git cat-file -p HEAD
tree 4b825dc642cb6eb9a060e54bf8d69288fbee4904
parent c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0c1d2
author John Doe <john@example.com> 1609459200 +0000
committer John Doe <john@example.com> 1609459200 +0000

feat: Add user authentication

4. tag: - 특정 커밋에 이름표를 붙임 - Annotated tag는 메타데이터 포함

$ git tag -a v1.0.0 -m "Release 1.0.0"
$ git cat-file -p v1.0.0
object 7f3a9e8d1c2b3a4f5e6d7c8b9a0e1f2d3c4b5a6e
type commit
tag v1.0.0
tagger John Doe <john@example.com> 1609459200 +0000

Release 1.0.0

객체 저장 위치:

$ ls -la .git/objects/
drwxr-xr-x   8a/
drwxr-xr-x   1c/
drwxr-xr-x   4b/
...

각 디렉토리는 해시의 첫 2자, 파일은 나머지 38자입니다.

$ git cat-file -t 8a0e41234f24b6da002d962a26c2495ea16a425f
blob
$ git cat-file -p 8a0e41234f24b6da002d962a26c2495ea16a425f
Hello Git

주의: Git은 파일의 변경분(diff)이 아닌 전체 스냅샷을 저장합니다. 하지만 동일한 내용은 같은 SHA-1 해시를 가지므로 한 번만 저장됩니다.

git gc와 팩 파일(Pack File)¶

시간이 지나면서 .git/objects/ 디렉토리에 수많은 loose object가 쌓입니다. git gc(garbage collection)는 이를 정리합니다.

$ git gc
Enumerating objects: 10000, done.
Counting objects: 100% (10000/10000), done.
Delta compression using up to 8 threads
Compressing objects: 100% (5000/5000), done.
Writing objects: 100% (10000/10000), done.
Total 10000 (delta 3000), reused 10000 (delta 3000)

gc가 하는 일: 1. 패킹(Packing): Loose objects를 pack 파일로 압축 2. 델타 압축: 비슷한 객체를 델타(차이)로 저장하여 공간 절약 3. 참조 정리: 오래된 reflog 항목 삭제 4. 도달 불가능 객체 제거: 어느 브랜치에서도 도달할 수 없는 객체 삭제

Pack 파일 확인:

$ ls -lh .git/objects/pack/
-r--r--r--  pack-a1b2c3d4e5f6.idx   (인덱스 파일)
-r--r--r--  pack-a1b2c3d4e5f6.pack  (압축된 데이터)

수동 gc 실행:

# 일반 gc
$ git gc

# 공격적 gc (더 많은 시간 소요, 더 좋은 압축)
$ git gc --aggressive --prune=now

자동 gc: Git은 loose objects가 일정 수 이상 쌓이면 자동으로 gc를 실행합니다. git config gc.auto로 조절 가능합니다.

베스트 프랙티스: 대규모 저장소에서는 주기적으로 git gc를 실행하여 디스크 공간을 절약하고 성능을 개선하세요. CI/CD에서 매일 실행하는 것도 좋은 전략입니다.

commit-graph와 multi-pack-index¶

Git 2.18+에서 도입된 고급 최적화 기능입니다.

commit-graph: 커밋 히스토리를 그래프 형태로 미리 계산하여 git log, git merge-base 등의 속도를 10배 이상 향상시킵니다.

$ git commit-graph write --reachable --changed-paths

# 출력:
Computing commit graph generation numbers: 100% (10000/10000), done.
Expanding reachable commits in commit graph: 100% (10000/10000), done.
Writing changed paths Bloom filters: 100% (10000/10000), done.

multi-pack-index: 여러 pack 파일의 인덱스를 통합하여 객체 검색 속도를 향상시킵니다.

$ git multi-pack-index write

자동 활성화:

$ git config --global feature.manyFiles true
$ git config --global core.commitGraph true
$ git config --global gc.writeCommitGraph true

파일 시스템 모니터(fsmonitor) 활용¶

대규모 저장소에서 git status가 느린 이유는 모든 파일을 스캔하기 때문입니다. FSMonitor는 파일 시스템 변경 알림을 이용하여 변경된 파일만 확인합니다.

Watchman 설치 및 설정 (Meta 개발):

# macOS
$ brew install watchman

# Linux
$ sudo apt install watchman

# Git 설정
$ git config core.fsmonitor watchman
$ git config core.untrackedcache true

성능 향상: - git status: 10초 → 0.1초 (100배 향상) - 100만 개 파일이 있는 저장소에서 특히 효과적

팁: Windows 사용자는 Git 2.38+에 내장된 Built-in FSMonitor를 사용할 수 있습니다. git config core.fsmonitor true만 설정하면 됩니다.

shallow clone과 --depth 옵션¶

전체 히스토리가 필요 없다면 shallow clone으로 최신 커밋만 가져올 수 있습니다.

# 최신 1개 커밋만 클론
$ git clone --depth 1 https://github.com/user/repo.git

# 최신 10개 커밋만 클론
$ git clone --depth 10 https://github.com/user/repo.git

장점: - 클론 속도 대폭 향상 - 디스크 공간 절약

단점: - 히스토리 탐색 제한 - git blame, git log 등이 제한적 - 일부 Git 작업 불가능

Shallow clone 해제:

# 전체 히스토리 가져오기
$ git fetch --unshallow

CI/CD에서 활용:

# GitHub Actions
- uses: actions/checkout@v3
  with:
    fetch-depth: 1  # Shallow clone

베스트 프랙티스: CI/CD나 일회성 작업에는 --depth 1을 사용하고, 개발 환경에서는 전체 히스토리를 유지하세요.

절 요약¶

  • Git 객체 모델은 blob, tree, commit, tag 네 가지 타입으로 구성됩니다
  • git gc는 loose objects를 pack 파일로 압축하여 디스크 공간을 절약합니다
  • commit-graph와 multi-pack-index는 대규모 저장소의 성능을 크게 향상시킵니다
  • FSMonitor는 git status를 100배 빠르게 만듭니다
  • shallow clone은 클론 속도를 향상시키지만 히스토리가 제한됩니다

Git 내부 구조 탐험¶

도입¶

Git의 마법을 진정으로 이해하려면 내부 구조를 직접 들여다봐야 합니다. 이 절에서는 .git 디렉토리를 탐험하고, Git이 어떻게 데이터를 저장하고 관리하는지 이해합니다.

핵심 개념 설명¶

Git은 콘텐츠 주소 지정 파일 시스템¶

Git은 단순히 버전 관리 시스템이 아니라 콘텐츠 주소 지정 파일 시스템(Content-Addressable Filesystem)입니다.

도서관의 청구 기호와 같습니다. 책(파일)은 내용의 해시(청구 기호)로 저장되며, 제목이나 저자가 바뀌어도 내용이 같으면 같은 위치에 있습니다.

콘텐츠 주소 지정의 의미:

$ echo "Hello World" | git hash-object --stdin -w
557db03de997c86a4a028e1ebd3a1ceb225be238

# 같은 내용은 항상 같은 해시
$ echo "Hello World" | git hash-object --stdin
557db03de997c86a4a028e1ebd3a1ceb225be238

이 특성 덕분에: - 데이터 중복이 자동으로 제거됩니다 - 데이터 무결성이 보장됩니다 (해시가 다르면 내용이 다름) - 분산 시스템에서 일관성 유지가 쉽습니다

git cat-file로 객체 내부 살펴보기¶

git cat-file은 Git 객체를 검사하는 저수준 명령어입니다.

# 최신 커밋 해시 확인
$ git rev-parse HEAD
a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0

# 객체 타입 확인
$ git cat-file -t a1b2c3d4
commit

# 객체 내용 출력
$ git cat-file -p a1b2c3d4
tree 4b825dc642cb6eb9a060e54bf8d69288fbee4904
parent c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0c1d2
author John Doe <john@example.com> 1609459200 +0000
committer John Doe <john@example.com> 1609459200 +0000

feat: Add user profile page

# 커밋이 가리키는 tree 탐험
$ git cat-file -p 4b825dc6
100644 blob 557db03de997c86a  README.md
040000 tree e4f5a6b7c8d9e0f1  src
100644 blob i7j8k9l0m1n2o3p4  .gitignore

# README.md 내용 확인
$ git cat-file -p 557db03d
Hello World

객체 크기 확인:

$ git cat-file -s a1b2c3d4
245

팁: git cat-file -p는 객체를 "예쁘게(pretty)" 출력합니다. Git 내부를 이해하는 데 가장 유용한 명령어입니다.

참조(refs) 시스템: 브랜치, 태그, HEAD¶

브랜치와 태그는 실제로는 커밋 해시를 가리키는 파일입니다.

$ ls .git/refs/heads/
main
feature/login

$ cat .git/refs/heads/main
a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0

$ cat .git/refs/tags/v1.0.0
7f8a9b0c1d2e3f4a5b6c7d8e9f0a1b2c3d4e5f6a

$ cat .git/HEAD
ref: refs/heads/main

참조 시스템의 계층:

.git/
├── HEAD -> refs/heads/main
├── refs/
│   ├── heads/
│   │   ├── main -> a1b2c3d4
│   │   └── feature/login -> e5f6a7b8
│   ├── tags/
│   │   └── v1.0.0 -> 7f8a9b0c
│   └── remotes/
│       └── origin/
│           └── main -> c9d0e1f2
└── objects/
    └── a1/
        └── b2c3d4e5f6... (커밋 객체)

수동으로 브랜치 만들기:

# 새 브랜치 만들기 (저수준 명령)
$ echo "a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0" > .git/refs/heads/new-branch

# 확인
$ git branch
  main
* new-branch

Packfile과 델타 압축¶

Git은 비슷한 객체를 델타(차이)로 저장하여 공간을 절약합니다.

델타 압축 예시: 파일 README.md의 두 버전: - v1: "Hello World" (11 bytes) - v2: "Hello World from Git" (21 bytes)

Git은 v2를 "v1 + 추가된 10바이트"로 저장할 수 있습니다.

$ git verify-pack -v .git/objects/pack/pack-*.idx | head -20

a1b2c3d4 commit 245 167 12
e5f6a7b8 tree   150 120 179
c9d0e1f2 blob   21 10 299 1 a1b2c3d4
# 21바이트 파일이 10바이트로 압축됨 (델타)

델타 체인: 여러 버전을 거치면 델타 체인이 형성됩니다.

v1 (base) <- v2 (delta) <- v3 (delta) <- v4 (delta)

긴 델타 체인은 성능에 영향을 줄 수 있으므로 Git은 주기적으로 재패킹합니다.

Git의 무결성 보장 메커니즘¶

Git은 SHA-1 해시를 이용하여 데이터 무결성을 보장합니다.

무결성 검증 과정: 1. 파일 내용을 SHA-1로 해싱 2. 해시를 파일 이름으로 사용하여 저장 3. 읽을 때 다시 해싱하여 해시 일치 확인

# 객체 무결성 확인
$ git fsck --full
Checking object directories: 100% (256/256), done.
Checking objects: 100% (10000/10000), done.

손상된 객체 감지:

# 의도적으로 객체 손상
$ echo "corrupted" > .git/objects/a1/b2c3d4...

$ git fsck
error: sha1 mismatch a1b2c3d4e5f6...
error: a1b2c3d4: invalid sha1 pointer in cache-tree

SHA-1 충돌 문제: 2017년 Google이 SHA-1 충돌을 시연했습니다. Git은 SHA-256으로 전환 중이며, Git 2.29+에서 실험적으로 지원합니다.

# SHA-256 저장소 생성 (실험적)
$ git init --object-format=sha256

주의: SHA-256 저장소는 아직 실험 단계이며, 대부분의 호스팅 서비스가 지원하지 않습니다. 프로덕션 사용은 권장되지 않습니다.

Git의 데이터 보장: - Cryptographic integrity: 해시로 내용 검증 - Immutability: 커밋은 불변 (새 커밋 생성만 가능) - Distributed backup: 모든 클론이 전체 히스토리의 백업

절 요약¶

  • Git은 콘텐츠 주소 지정 파일 시스템으로, 내용의 해시로 데이터를 저장합니다
  • git cat-file -p로 객체 내용을 탐험할 수 있습니다
  • 브랜치와 태그는 커밋 해시를 가리키는 참조 파일입니다
  • Pack 파일은 델타 압축으로 디스크 공간을 절약합니다
  • Git은 SHA-1 해시로 데이터 무결성을 보장하며, SHA-256으로 전환 중입니다

FAQ¶

Q1: 모노레포는 언제 사용해야 하나요?

A1: 모노레포는 다음 상황에 적합합니다. (1) 여러 프로젝트 간 코드 공유가 빈번한 경우, (2) API와 클라이언트를 동시에 변경해야 하는 경우, (3) 팀이 긴밀히 협업하는 경우, (4) 일관된 도구와 스타일을 유지하고 싶은 경우. 하지만 빌드 시스템(Bazel, Nx 등)과 CI/CD 최적화가 필수이므로 인프라 투자가 가능한 팀에게 권장됩니다.

Q2: 서브모듈 vs subtree vs 패키지 매니저 중 무엇을 선택해야 하나요?

A2: (1) 패키지 매니저(npm, pip): 가장 추천. 버전 관리가 명확하고 간단합니다. (2) subtree: 읽기 전용 외부 라이브러리 포함 시. 히스토리가 부모에 병합되어도 괜찮은 경우. (3) 서브모듈: 양방향 동기화가 필요하고, 별도 저장소 유지가 중요한 경우. 단, 서브모듈은 복잡하므로 신중히 선택하세요.

Q3: sparse-checkout과 partial clone을 함께 사용할 수 있나요?

A3: 네, 가능하며 강력히 권장됩니다. git clone --filter=blob:none --sparse <url> 후 git sparse-checkout set <path>로 설정하면 초기 클론은 partial clone으로 빠르게 하고, 작업 디렉토리는 sparse-checkout으로 필요한 부분만 유지할 수 있습니다. 대규모 모노레포에서 최적의 조합입니다.

Q4: git gc를 수동으로 실행해야 하나요?

A4: 일반적으로는 Git이 자동으로 gc를 실행하므로 수동 실행이 불필요합니다. 하지만 다음 경우 수동 실행을 고려하세요. (1) 대량의 파일을 추가/삭제한 후, (2) 디스크 공간이 부족한 경우, (3) git filter-repo로 히스토리를 정리한 후, (4) 저장소를 배포하기 전 최적화할 때. git gc --aggressive는 시간이 오래 걸리므로 정기 배포 전에만 사용하세요.

Q5: Git 객체 모델을 이해하면 실무에 어떤 도움이 되나요?

A5: (1) 문제 해결: "커밋을 삭제했는데 복구할 수 있나요?"에 답할 수 있습니다. 객체는 삭제되지 않고 참조가 제거될 뿐이므로 reflog로 복구 가능합니다. (2) 성능 최적화: 왜 pack 파일이 생기고, 델타 압축이 무엇인지 이해하면 저장소를 최적화할 수 있습니다. (3) 고급 작업: git filter-repo, 객체 수동 복구 등 고급 작업을 자신 있게 수행할 수 있습니다.

장 요약¶

이번 장에서는 대규모 프로젝트와 엔터프라이즈 환경에서 Git을 효율적으로 관리하는 방법을 배웠습니다. 모노레포와 멀티레포의 장단점을 비교하고, 서브모듈로 외부 저장소를 관리하며, sparse-checkout과 partial clone으로 대규모 저장소를 빠르게 다루는 기법을 익혔습니다. Git의 내부 구조(blob, tree, commit, tag)를 탐험하고, gc, commit-graph, FSMonitor 등의 성능 최적화 도구를 활용하는 방법도 배웠습니다. 이제 수백 GB에 달하는 저장소도 자신 있게 관리할 수 있습니다.
