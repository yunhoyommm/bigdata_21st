09장: GitHub 프로젝트 관리

중급

GitHub 프로젝트 관리¶

학습 목표 - GitHub Projects(V2)로 작업을 시각적으로 관리할 수 있다 - Milestones로 릴리스 일정을 계획하고 추적할 수 있다 - Release Notes를 작성하고 자동 Changelog를 설정할 수 있다 - GitHub Pages로 프로젝트 문서 사이트를 배포할 수 있다 - CODEOWNERS와 자동화 워크플로우로 반복 작업을 줄일 수 있다

GitHub Projects로 작업 관리하기¶

도입¶

개발 프로젝트가 커질수록 수십, 수백 개의 Issue와 PR을 관리하는 것이 어려워집니다. "지금 어떤 작업이 진행 중이지?", "다음 릴리스에 무엇을 포함해야 하지?"라는 질문에 답하기 위해서는 체계적인 프로젝트 관리 도구가 필요합니다. GitHub Projects는 Issue와 PR을 시각적으로 관리할 수 있는 강력한 도구입니다.

핵심 개념 설명¶

GitHub Projects(V2) 소개¶

GitHub Projects(V2)는 2022년에 새롭게 출시된 프로젝트 관리 도구로, 기존 Projects(Classic)을 대폭 개선한 버전입니다. Issue와 PR을 다양한 관점에서 시각화하고, 커스텀 필드를 추가하여 팀의 워크플로우에 맞게 조정할 수 있습니다.

화이트보드에 포스트잇을 붙여 작업을 관리하는 칸반 보드와 같습니다. 각 포스트잇(Issue)을 'To Do', 'In Progress', 'Done' 열로 이동시키며 진행 상황을 추적합니다.

프로젝트 생성과 뷰 설정¶

프로젝트 생성 방법:

  1. 저장소 레벨 프로젝트:
  2. 저장소 페이지에서 Projects 탭 → New project

  3. 해당 저장소의 Issue와 PR만 관리

  4. 사용자/조직 레벨 프로젝트:

  5. 프로필 또는 조직 페이지 → Projects → New project

  6. 여러 저장소의 Issue와 PR을 통합 관리 가능

  7. 템플릿 선택:

  8. Blank: 빈 프로젝트에서 시작
  9. Team backlog: 스프린트 관리용
  10. Roadmap: 타임라인 기반 계획

GitHub CLI로 프로젝트 생성:

# 조직 프로젝트 생성
$ gh project create --owner myorg --title "Q1 2025 Sprint"

# 프로젝트 목록 확인
$ gh project list --owner myorg

보드 뷰 vs 테이블 뷰 vs 로드맵 뷰¶

GitHub Projects는 같은 데이터를 다양한 관점에서 볼 수 있는 세 가지 주요 뷰를 제공합니다.

1. 보드 뷰(Board View): 칸반 스타일

칸반 보드 형태로 작업을 열(Column)로 구분하여 시각화합니다. 드래그 앤 드롭으로 작업을 이동시키며 상태를 변경할 수 있습니다.

기본 열 구성 예시: - Backlog: 앞으로 할 작업 - To Do: 다음 스프린트에 할 작업 - In Progress: 현재 진행 중 - In Review: 코드 리뷰 중 - Done: 완료

팁: 열은 Issue의 Status 필드와 연동됩니다. Issue를 열 사이로 드래그하면 Status가 자동으로 변경됩니다.

2. 테이블 뷰(Table View): 스프레드시트 스타일

모든 Issue와 PR을 표 형태로 나열하며, 정렬, 필터링, 그룹화가 자유롭습니다. 스프레드시트처럼 셀을 직접 편집할 수 있습니다.

주요 기능: - 정렬: 우선순위, 담당자, 마감일 기준 - 필터링: 특정 라벨, 담당자, Sprint만 표시 - 그룹화: Status, Assignee, Milestone 별로 그룹화

3. 로드맵 뷰(Roadmap View): 타임라인 스타일

Issue의 시작일과 종료일을 시간축에 표시하여 프로젝트 일정을 한눈에 파악할 수 있습니다. 간트 차트(Gantt Chart)와 유사합니다.

사용 시나리오: - 분기별 기능 개발 계획 - 여러 팀의 작업 일정 조율 - 릴리스 마일스톤 시각화

커스텀 필드 추가 (Status, Priority, Sprint)¶

기본 Issue 필드(Title, Assignee, Labels) 외에도 프로젝트별로 필요한 커스텀 필드를 추가할 수 있습니다.

주요 커스텀 필드 타입:

  1. Text: 짧은 텍스트 (예: Sprint 이름)
  2. Number: 숫자 (예: Story Point, 예상 시간)
  3. Date: 날짜 (예: 마감일, 시작일)
  4. Single select: 드롭다운 선택 (예: Priority, Status)
  5. Iteration: 스프린트 주기

Status 필드 추가 예시:

프로젝트 설정 → + Add field → Single select
- 이름: Status
- 옵션:
  - Backlog (회색)
  - To Do (노란색)
  - In Progress (파란색)
  - In Review (주황색)
  - Done (초록색)

Priority 필드 추가:

옵션:
- P0: Critical (빨간색)
- P1: High (주황색)
- P2: Medium (노란색)
- P3: Low (회색)

Sprint 필드 추가 (Iteration):

- 반복 주기: 2주
- 시작일: 2025-01-06
- 반복 횟수: 6개 (Q1)

베스트 프랙티스: 필드를 너무 많이 추가하면 관리가 복잡해집니다. 팀에 꼭 필요한 3-5개 필드만 추가하세요.

Issues와 PR을 프로젝트에 연결¶

Issue 추가 방법:

  1. 프로젝트 내에서 직접 추가:
  2. 프로젝트 보드에서 + Add item → Issue 제목 입력 → Enter

  3. 자동으로 해당 저장소에 Issue 생성됨

  4. 기존 Issue 연결:

  5. 프로젝트 보드에서 + Add item → # 입력 → Issue 번호 선택

  6. 여러 저장소의 Issue를 검색하여 추가 가능

  7. Issue 페이지에서 연결:

  8. Issue 우측 사이드바 → Projects → 프로젝트 선택

GitHub CLI로 Issue 추가:

# 프로젝트에 Issue 추가
$ gh project item-add 1 --owner myorg --url https://github.com/myorg/repo/issues/42

# 현재 저장소의 Issue를 프로젝트에 추가
$ gh issue list --json number,url --jq '.[].url' | xargs -I {} gh project item-add 1 --owner myorg --url {}

자동화 설정:

Projects 설정에서 자동화 규칙을 설정할 수 있습니다: - Issue 생성 시: 자동으로 Backlog 열에 추가 - PR 생성 시: In Progress 열에 추가 - PR 머지 시: Done 열로 이동 - Issue 종료 시: Done 열로 이동

주의: 자동화는 프로젝트 설정에서 활성화해야 동작합니다. Workflows → Built-in workflows에서 설정하세요.

절 요약¶

  • GitHub Projects(V2)는 Issue와 PR을 시각적으로 관리하는 도구입니다
  • 보드, 테이블, 로드맵 세 가지 뷰로 데이터를 다르게 볼 수 있습니다
  • Status, Priority, Sprint 같은 커스텀 필드를 추가하여 팀 워크플로우에 맞출 수 있습니다
  • Issue는 프로젝트에 수동 또는 자동으로 연결할 수 있습니다

Milestones와 릴리스 관리¶

도입¶

소프트웨어 개발은 작은 단계들을 거쳐 큰 목표를 달성하는 과정입니다. "v1.0 릴리스에 어떤 기능이 포함되어야 하지?", "다음 분기까지 완료해야 할 작업은 무엇이지?"라는 질문에 답하기 위해 Milestones와 Releases를 사용합니다.

핵심 개념 설명¶

Milestone으로 이정표 설정¶

Milestone(마일스톤)은 프로젝트의 중요한 이정표를 나타내는 Issue의 그룹입니다. 보통 릴리스 버전, 스프린트, 분기별 목표에 해당합니다.

앨범 제작 과정과 같습니다. 여러 곡(Issue)을 녹음하고, 앨범(Milestone)으로 묶어 발매(Release)합니다.

Milestone 생성 방법:

  1. 저장소 페이지 → Issues 탭 → Milestones → New milestone
  2. 정보 입력:
  3. Title: v1.0.0 - 첫 공식 릴리스
  4. Due date: 2025-03-31
  5. Description: 이 릴리스의 목표와 주요 기능 설명
# GitHub CLI로 Milestone 생성
$ gh api repos/:owner/:repo/milestones -f title="v1.0.0" -f due_on="2025-03-31T23:59:59Z" -f description="첫 공식 릴리스"

Milestone 네이밍 규칙: - 버전 기반: v1.0.0, v1.1.0, v2.0.0 - 날짜 기반: 2025 Q1, Sprint 23 - 기능 기반: 모바일 지원, 다크 모드 출시

Milestone과 Issue 연결¶

Issue 생성 또는 편집 시 우측 사이드바에서 Milestone을 선택합니다.

# 새 Issue를 Milestone과 함께 생성
$ gh issue create --title "사용자 로그인 기능" --milestone "v1.0.0"

# 기존 Issue에 Milestone 할당
$ gh issue edit 42 --milestone "v1.0.0"

Milestone 진행 상황 추적:

Milestone 페이지에서 진행률을 시각적으로 확인할 수 있습니다: - Open issues: 남은 작업 - Closed issues: 완료된 작업 - Progress bar: 완료율 (예: 12 / 20 = 60%)

팁: Milestone 마감일이 다가오면 GitHub가 자동으로 알림을 보냅니다. 팀이 일정을 지킬 수 있도록 도와줍니다.

Release 생성과 Release Notes 작성¶

Release(릴리스)는 소프트웨어의 특정 버전을 배포 가능한 상태로 패키징하는 것입니다. Git Tag에 설명, 바이너리, Changelog를 추가한 배포 패키지입니다.

Release 생성 단계:

  1. Tag 생성:
# Annotated Tag 생성 (권장)
$ git tag -a v1.0.0 -m "첫 공식 릴리스"

# 원격에 push
$ git push origin v1.0.0
  1. GitHub에서 Release 생성:
  2. 저장소 → Releases → Draft a new release
  3. Tag 선택: v1.0.0
  4. Release title: v1.0.0 - First Official Release
  5. Description: Release Notes 작성 (아래 참조)

GitHub CLI로 Release 생성:

$ gh release create v1.0.0 --title "v1.0.0" --notes "첫 공식 릴리스

## 주요 기능
- 사용자 로그인/회원가입
- 프로필 이미지 업로드
- 다크 모드 지원

## 버그 수정
- #42: 로그아웃 버그 수정
- #45: 모바일 레이아웃 깨짐 수정

## 알려진 이슈
- iOS Safari에서 간헐적 로딩 지연
"

Release Notes 작성 가이드:

좋은 Release Notes는 사용자에게 변경 사항을 명확히 전달합니다.

구조:

# v1.0.0 - 릴리스 제목

## 🎉 주요 기능 (Features)
- **로그인 시스템**: OAuth 2.0 기반 소셜 로그인 지원 (#12, #34)
- **다크 모드**: 시스템 설정에 따라 자동 전환 (#56)

## 🐛 버그 수정 (Bug Fixes)
- 회원가입 폼 검증 오류 수정 (#42)
- 이미지 업로드 실패 문제 해결 (#45)

## ⚡ 성능 개선 (Performance)
- 초기 로딩 속도 30% 개선
- 번들 크기 15% 감소

## 📚 문서 (Documentation)
- API 문서 업데이트
- 설치 가이드 추가

## ⚠️ 주의사항 (Breaking Changes)
- API 엔드포인트 변경: `/user` → `/api/v1/user`
- 최소 Node.js 버전: 18.x

## 📦 의존성 (Dependencies)
- React 18.2.0 → 18.3.0
- TypeScript 5.0.0 → 5.3.0

**전체 Changelog**: https://github.com/owner/repo/compare/v0.9.0...v1.0.0

베스트 프랙티스: 사용자 관점에서 작성하세요. "리팩터링" 같은 내부 변경보다 "로딩 속도 개선" 같은 사용자 혜택을 강조하세요.

자동 Changelog 생성¶

GitHub는 이전 Release 이후의 모든 PR과 커밋을 바탕으로 Changelog를 자동 생성할 수 있습니다.

자동 생성 활성화: - Release 작성 화면에서 Generate release notes 버튼 클릭 - 이전 Tag와 현재 Tag 사이의 모든 PR이 자동으로 목록화됩니다

.github/release.yml 설정:

Changelog 생성 규칙을 커스터마이즈할 수 있습니다.

changelog:
  exclude:
    labels:
      - ignore-for-release
      - dependencies
  categories:
    - title: 🎉 주요 기능
      labels:
        - feature
        - enhancement
    - title: 🐛 버그 수정
      labels:
        - bug
        - fix
    - title: 📚 문서
      labels:
        - documentation
    - title: 🔧 기타 변경
      labels:
        - "*"

이 설정을 사용하면 PR 라벨에 따라 Changelog가 자동으로 카테고리별로 분류됩니다.

Pre-release와 Latest 관리¶

Pre-release(사전 릴리스):

정식 릴리스 전에 테스터나 얼리 어답터에게 공개하는 버전입니다.

# Pre-release 생성
$ gh release create v1.0.0-beta.1 --title "v1.0.0 Beta 1" --notes "베타 테스트 버전" --prerelease

# Latest 플래그 없이 생성 (자동으로 latest가 되지 않음)
$ gh release create v1.0.0-rc.1 --title "Release Candidate 1" --notes "최종 테스트 버전" --prerelease

Release 페이지에서: - Pre-release는 "Pre-release" 배지가 표시됩니다 - Latest release는 페이지 상단에 고정됩니다

Semantic Versioning(SemVer):

버전 번호는 MAJOR.MINOR.PATCH 형식을 따릅니다: - MAJOR: 하위 호환 불가 변경 (1.0.0 → 2.0.0) - MINOR: 하위 호환 가능 기능 추가 (1.0.0 → 1.1.0) - PATCH: 버그 수정 (1.0.0 → 1.0.1)

Pre-release 표기: - Alpha: 1.0.0-alpha.1 (내부 테스트) - Beta: 1.0.0-beta.1 (공개 베타) - Release Candidate: 1.0.0-rc.1 (최종 후보)

주의: Pre-release는 안정성이 보장되지 않으므로 프로덕션 환경에서 사용하지 마세요.

절 요약¶

  • Milestone은 Issue를 릴리스, 스프린트, 목표별로 그룹화합니다
  • Release는 Tag에 설명과 바이너리를 추가한 배포 패키지입니다
  • Release Notes는 사용자 관점에서 변경 사항을 명확히 전달해야 합니다
  • GitHub의 자동 Changelog 생성과 .github/release.yml로 효율을 높일 수 있습니다
  • Pre-release로 베타 버전을 관리하고, Semantic Versioning을 따라야 합니다

Discussions와 커뮤니티 소통¶

도입¶

Issue는 구체적인 버그나 작업을 추적하는 데 적합하지만, 일반적인 질문이나 아이디어 논의에는 과도합니다. "이 기능이 필요할까요?"라는 질문을 Issue로 만들면 Issue 목록이 혼잡해집니다. 이때 사용하는 것이 GitHub Discussions입니다.

핵심 개념 설명¶

Discussions 활성화와 카테고리 설정¶

Discussions는 커뮤니티와 자유롭게 소통할 수 있는 포럼 기능입니다. Q&A, 공지사항, 아이디어 토론 등 Issue보다 가벼운 주제에 적합합니다.

활성화 방법:

  1. 저장소 → Settings → Features → Discussions 체크
  2. 자동으로 Discussions 탭이 생성됩니다

기본 카테고리: - Announcements: 공지사항 (관리자만 작성 가능) - General: 일반 토론 - Ideas: 기능 제안 - Q&A: 질문과 답변 - Show and tell: 프로젝트 소개

커스텀 카테고리 추가:

Settings → Discussions → Categories → New category

예시: - Troubleshooting: 문제 해결 도움 - Feature Requests: 기능 요청 (투표 가능) - Off-topic: 자유 주제

팁: 카테고리별로 이모지를 설정하면 시각적으로 구분하기 쉽습니다.

공지, Q&A, 아이디어 토론 활용¶

1. Announcements(공지사항):

주요 릴리스, 중요한 변경 사항을 알립니다.

예시:

# 🎉 v2.0.0 릴리스 발표

새로운 메이저 버전이 출시되었습니다!

## 주요 변경사항
- 완전히 새로운 UI
- 성능 50% 개선
- API v2 도입

## 마이그레이션 가이드
기존 사용자는 [마이그레이션 가이드](링크)를 참고하세요.

베스트 프랙티스: Announcements는 Pin(고정)하여 Discussion 페이지 상단에 항상 표시되도록 하세요.

2. Q&A(질문과 답변):

Q&A 카테고리는 Stack Overflow 스타일로 동작합니다. 질문자가 답변을 "Accepted Answer"로 표시할 수 있습니다.

워크플로우: 1. 사용자가 질문 작성 2. 커뮤니티가 답변 3. 질문자가 해결된 답변에 ✅ 표시 4. 나중에 같은 문제를 겪은 사람이 검색하여 참고

**Q**: API 인증 토큰을 어떻게 갱신하나요?

**A** (Accepted): 토큰은 만료 1시간 전에 `/api/refresh` 엔드포인트로 갱신할 수 있습니다.

3. Ideas(아이디어):

기능 제안을 받고 커뮤니티 투표로 우선순위를 정합니다.

**제안**: 다크 모드 자동 전환 기능

시스템 설정에 따라 자동으로 라이트/다크 모드를 전환하는 기능이 있으면 좋겠습니다.

👍 찬성 투표를 눌러주세요!

팁: GitHub의 Upvote 기능으로 인기 있는 아이디어를 파악하고, 로드맵에 반영하세요.

Issue vs Discussion: 언제 무엇을 쓸 것인가¶

특성 Issue Discussion
목적 구체적 작업, 버그 추적 질문, 토론, 아이디어
상태 Open / Closed 종료 개념 없음
추적 Milestone, Projects 연결 카테고리별 분류
알림 모든 변경 알림 답변 알림
적합한 주제 "로그인 버튼 안 눌림" "로그인 방식 개선 제안"

Issue로 만들어야 하는 경우: - 명확한 버그 보고 - 구체적인 기능 개발 작업 - 릴리스 일정에 포함될 작업

Discussion으로 만들어야 하는 경우: - 사용법 질문 ("어떻게 설정하나요?") - 기능 제안 초기 단계 ("이런 기능 어때요?") - 일반적인 토론 ("이 디자인 vs 저 디자인") - 커뮤니티 공지

베스트 프랙티스: Discussion에서 충분히 논의된 기능 제안은 Issue로 전환하여 구체적으로 작업할 수 있습니다.

Polls 기능으로 의견 수렴¶

Polls(투표)는 Discussion 내에서 간단한 선택지 투표를 만들 수 있는 기능입니다.

투표 만들기:

Discussion 작성 시 본문에 다음 형식을 사용합니다:

다음 릴리스에서 우선 개발할 기능을 투표해주세요!

- [ ] 다크 모드
- [ ] 모바일 앱
- [ ] 다국어 지원
- [ ] 성능 개선

GitHub가 자동으로 체크박스를 투표 UI로 변환합니다.

사용 시나리오: - 기능 우선순위 결정 - 디자인 옵션 선택 - 릴리스 일정 조율

주의: Polls는 Discussion 본문에만 사용할 수 있으며, 댓글에는 작동하지 않습니다.

절 요약¶

  • Discussions는 Issue보다 가벼운 소통 채널입니다
  • Announcements, Q&A, Ideas 등 카테고리별로 주제를 분류합니다
  • Issue는 구체적 작업 추적, Discussion은 질문과 아이디어 논의에 적합합니다
  • Q&A 카테고리에서 Accepted Answer로 해결된 질문을 표시합니다
  • Polls로 커뮤니티 의견을 수렴할 수 있습니다

GitHub Pages와 Wiki¶

도입¶

오픈소스 프로젝트를 운영하다 보면 사용자에게 문서를 제공해야 할 필요가 생깁니다. README 파일 하나로는 부족하고, 설치 가이드, API 문서, 튜토리얼 등 체계적인 문서 사이트가 필요합니다. GitHub Pages와 Wiki는 이를 위한 두 가지 도구입니다.

핵심 개념 설명¶

GitHub Pages로 프로젝트 문서 사이트 만들기¶

GitHub Pages는 GitHub 저장소에서 정적 웹사이트를 무료로 호스팅하는 서비스입니다. Markdown 파일을 HTML로 자동 변환하여 웹사이트로 제공합니다.

활성화 방법:

  1. 소스 선택 (3가지 방식):

a) GitHub Actions 방식 (권장): Settings → Pages → Source: GitHub Actions

b) 브랜치 방식: Settings → Pages → Source: Deploy from a branch Branch: main / docs (또는 gh-pages)

c) 전용 브랜치: bash $ git checkout --orphan gh-pages $ git rm -rf . $ echo "Hello GitHub Pages" > index.html $ git add index.html $ git commit -m "docs: GitHub Pages 초기 설정" $ git push origin gh-pages

  1. 사이트 확인:
  2. https://<username>.github.io/<repository>/
  3. 배포 완료까지 1-2분 소요

프로젝트 문서 구조 예시:

docs/
├── index.md              # 홈페이지
├── getting-started.md    # 시작 가이드
├── api/
│   ├── authentication.md
│   └── endpoints.md
├── guides/
│   ├── installation.md
│   └── configuration.md
└── _config.yml           # Jekyll 설정 (선택)

GitHub Actions로 배포:

.github/workflows/deploy-docs.yml:

name: Deploy Docs

on:
  push:
    branches:
      - main
    paths:
      - 'docs/**'

permissions:
  contents: read
  pages: write
  id-token: write

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/configure-pages@v4
      - uses: actions/jekyll-build-pages@v1
        with:
          source: ./docs
          destination: ./_site
      - uses: actions/upload-pages-artifact@v3

  deploy:
    needs: build
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - uses: actions/deploy-pages@v4
        id: deployment

팁: 문서 변경 시에만 배포하도록 paths: - 'docs/**' 필터를 사용하면 불필요한 빌드를 줄일 수 있습니다.

Jekyll 테마 적용¶

Jekyll은 GitHub Pages의 기본 정적 사이트 생성기입니다. Markdown을 HTML로 변환하고 테마를 적용합니다.

지원 테마:

GitHub Pages는 다음 공식 테마를 기본 지원합니다: - Cayman: 깔끔한 단일 페이지 - Minimal: 미니멀리즘 - Slate: 다크 테마 - Time Machine: 기술 문서용 - Architect: 대형 프로젝트용

테마 설정:

docs/_config.yml:

title: My Project Documentation
description: 종합 사용 가이드
theme: jekyll-theme-cayman

# 네비게이션
navigation:
  - title: Home
    url: /
  - title: Getting Started
    url: /getting-started
  - title: API
    url: /api

커스텀 테마 사용:

더 많은 테마는 Jekyll Themes에서 찾을 수 있습니다.

# Gemfile
source "https://rubygems.org"
gem "jekyll-theme-just-the-docs"

# _config.yml
remote_theme: just-the-docs/just-the-docs

베스트 프랙티스: 프로젝트 문서는 일관된 구조를 유지하세요. 설치 → 설정 → 사용법 → API 레퍼런스 순서가 일반적입니다.

커스텀 도메인 설정¶

기본 <username>.github.io/<repo> 대신 자신의 도메인을 사용할 수 있습니다.

절차:

  1. 도메인 등록 서비스에서 DNS 설정:

A 레코드 추가: 185.199.108.153 185.199.109.153 185.199.110.153 185.199.111.153

또는 CNAME 레코드: docs.myproject.com → <username>.github.io

  1. GitHub Pages 설정:
  2. Settings → Pages → Custom domain
  3. docs.myproject.com 입력

  4. Enforce HTTPS 체크 (Let's Encrypt 인증서 자동 발급)

  5. CNAME 파일 생성:

docs/CNAME: docs.myproject.com

HTTPS 적용:

GitHub Pages는 Let's Encrypt를 통해 HTTPS를 자동 지원합니다. 커스텀 도메인 설정 후 몇 분이면 인증서가 발급됩니다.

주의: CNAME 파일은 저장소 루트 또는 docs 디렉토리에 있어야 하며, 배포 시 삭제되지 않도록 주의하세요.

Wiki 페이지 작성과 관리¶

Wiki는 저장소에 내장된 간단한 문서 시스템입니다. GitHub Pages보다 설정이 간단하지만 기능은 제한적입니다.

활성화:

Settings → Features → Wikis 체크

Wiki 페이지 구조:

Wiki는 자동으로 다음 페이지를 생성합니다: - Home: 위키 홈페이지 - _Sidebar: 좌측 사이드바 (선택) - _Footer: 페이지 하단 (선택)

페이지 작성:

  1. Wiki 탭 → New page
  2. 페이지 제목: Installation Guide
  3. Markdown으로 내용 작성
  4. Save page

페이지 간 링크:

자세한 내용은 [[API Reference]]를 참고하세요.

GitHub Wiki는 [[페이지 제목]] 형식으로 내부 링크를 생성합니다.

Wiki 로컬 편집:

Wiki도 Git 저장소이므로 로컬에서 편집할 수 있습니다.

# Wiki 클론
$ git clone https://github.com/username/repo.wiki.git

# 편집 후 커밋
$ git add .
$ git commit -m "docs: Installation 가이드 업데이트"
$ git push

Wiki vs GitHub Pages 비교:

특성 Wiki GitHub Pages
설정 복잡도 매우 간단 중간 (Jekyll 설정 필요)
커스텀 디자인 제한적 완전 커스터마이징
검색 엔진 최적화 약함 강함 (커스텀 도메인 가능)
버전 관리 Git 히스토리 Git 히스토리
적합한 용도 간단한 내부 문서 공식 프로젝트 문서

베스트 프랙티스: 소규모 프로젝트는 Wiki, 대규모 프로젝트는 GitHub Pages를 권장합니다.

절 요약¶

  • GitHub Pages는 정적 웹사이트를 무료로 호스팅하는 서비스입니다
  • Jekyll 테마로 문서 사이트를 예쁘게 꾸밀 수 있습니다
  • 커스텀 도메인과 HTTPS를 설정하여 전문적인 문서 사이트를 만들 수 있습니다
  • Wiki는 간단한 문서 작성에 적합하며 설정이 매우 쉽습니다
  • Wiki는 내부 문서, GitHub Pages는 공식 문서 사이트에 적합합니다

자동화 워크플로우: 프로젝트 관리 자동화¶

도입¶

프로젝트 관리에는 반복적인 작업이 많습니다. "새 Issue에 자동으로 라벨을 붙이고 싶다", "PR이 머지되면 연결된 Issue를 자동으로 닫고 싶다"라는 요구가 생깁니다. GitHub Actions와 자동화 기능으로 이런 반복 작업을 줄일 수 있습니다.

핵심 개념 설명¶

Issue 자동 할당 (CODEOWNERS)¶

CODEOWNERS는 코드의 특정 영역에 대한 소유권을 정의하는 파일입니다. PR이 생성되면 해당 영역의 소유자가 자동으로 리뷰어로 지정됩니다.

파일 위치:

.github/CODEOWNERS

기본 문법:

# 기본 소유자 (모든 파일)
* @default-owner

# 특정 디렉토리
/frontend/        @frontend-team
/backend/         @backend-team
/docs/            @docs-team

# 특정 파일 타입
*.js              @javascript-experts
*.py              @python-developers
*.md              @technical-writers

# 특정 파일
/package.json     @infra-team
/.github/         @devops-team

# 중첩 규칙 (더 구체적인 규칙이 우선)
/frontend/components/  @ui-team
/frontend/api/         @api-team

# 여러 소유자
/security/        @security-team @cto

# 이메일로 지정 (Organization만 가능)
/legal/           legal@company.com

동작 방식:

  1. 개발자가 /frontend/Login.js 파일을 수정하여 PR 생성
  2. GitHub가 CODEOWNERS 확인
  3. @frontend-team이 자동으로 리뷰어로 추가됨
  4. @frontend-team 멤버가 알림 수신

베스트 프랙티스: CODEOWNERS를 브랜치 보호 규칙과 함께 사용하세요. "CODEOWNERS 승인 필수"로 설정하면 해당 영역 전문가의 리뷰를 강제할 수 있습니다.

브랜치 보호 규칙 설정:

Settings → Branches → Add rule
- Branch name pattern: main
- ✅ Require pull request reviews before merging
  - ✅ Require review from Code Owners

PR 머지 시 Issue 자동 닫기¶

GitHub는 PR과 Issue를 자동으로 연결하는 키워드를 지원합니다.

키워드: - Closes #123 - Fixes #123 - Resolves #123

PR 설명 또는 커밋 메시지에 이 키워드를 사용하면, PR이 머지될 때 연결된 Issue가 자동으로 닫힙니다.

PR 설명 예시:

## 변경 사항
로그인 버그를 수정했습니다.

## 관련 Issue
Fixes #42
Closes #45, #47

커밋 메시지 예시:

$ git commit -m "fix: 로그인 세션 만료 버그 수정

사용자가 로그인 후 1시간이 지나면 세션이 자동 갱신되도록 수정했습니다.

Fixes #42"

여러 Issue 동시 닫기:

Closes #10, closes #12, fixes #15

팁: 키워드는 대소문자를 구분하지 않습니다. fixes, Fixes, FIXES 모두 동작합니다.

라벨 기반 자동 분류¶

새로운 Issue가 생성되면 제목이나 본문의 키워드를 바탕으로 자동으로 라벨을 추가할 수 있습니다.

.github/workflows/auto-label.yml:

name: Auto Label Issues

on:
  issues:
    types: [opened]

jobs:
  label:
    runs-on: ubuntu-latest
    permissions:
      issues: write
    steps:
      - uses: actions/github-script@v7
        with:
          script: |
            const issue = context.payload.issue;
            const title = issue.title.toLowerCase();
            const body = issue.body ? issue.body.toLowerCase() : '';
            const labels = [];

            // 키워드 기반 라벨 추가
            if (title.includes('bug') || body.includes('버그')) {
              labels.push('bug');
            }
            if (title.includes('feature') || title.includes('기능')) {
              labels.push('enhancement');
            }
            if (body.includes('도움') || body.includes('help')) {
              labels.push('help wanted');
            }
            if (title.includes('security') || title.includes('보안')) {
              labels.push('security');
            }

            // 기본 라벨
            labels.push('triage');

            // 라벨 추가
            if (labels.length > 0) {
              await github.rest.issues.addLabels({
                owner: context.repo.owner,
                repo: context.repo.repo,
                issue_number: issue.number,
                labels: labels
              });
            }

동작 흐름: 1. 사용자가 Issue 생성: "버그: 로그인 안 됨" 2. 워크플로우 트리거 3. 제목에서 "버그" 키워드 감지 4. bug, triage 라벨 자동 추가

Stale bot으로 오래된 이슈 관리¶

오래 동안 활동이 없는 Issue나 PR을 자동으로 라벨링하고 닫는 봇입니다.

.github/workflows/stale.yml:

name: Close Stale Issues

on:
  schedule:
    - cron: '0 0 * * *'  # 매일 자정(UTC) 실행

permissions:
  issues: write
  pull-requests: write

jobs:
  stale:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/stale@v9
        with:
          repo-token: ${{ secrets.GITHUB_TOKEN }}

          # Issue 설정
          stale-issue-message: |
            이 Issue는 60일 동안 활동이 없어 자동으로 `stale` 라벨이 추가되었습니다.
            7일 이내에 활동이 없으면 자동으로 닫힙니다.
            계속 진행하려면 댓글을 남겨주세요.
          close-issue-message: |
            이 Issue는 활동이 없어 자동으로 닫혔습니다.
            필요하다면 다시 열어주세요.
          days-before-stale: 60
          days-before-close: 7
          stale-issue-label: 'stale'
          exempt-issue-labels: 'pinned,security,roadmap'

          # PR 설정
          stale-pr-message: |
            이 PR은 30일 동안 활동이 없습니다.
          close-pr-message: |
            이 PR은 활동이 없어 자동으로 닫혔습니다.
          days-before-pr-stale: 30
          days-before-pr-close: 7
          stale-pr-label: 'stale'
          exempt-pr-labels: 'work-in-progress'

설정 옵션: - days-before-stale: 며칠 후 stale 라벨 추가 - days-before-close: stale 후 며칠 후 닫기 - exempt-issue-labels: 제외할 라벨 (영구 보존)

주의: Stale bot은 신중히 설정하세요. 중요한 Issue를 실수로 닫지 않도록 exempt-issue-labels를 잘 설정해야 합니다.

프로젝트 보드 자동 업데이트¶

Issue의 상태가 변경되면 GitHub Projects 보드를 자동으로 업데이트할 수 있습니다.

.github/workflows/project-automation.yml:

name: Project Automation

on:
  issues:
    types: [opened, closed, reopened]
  pull_request:
    types: [opened, closed, ready_for_review]

jobs:
  update-project:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/add-to-project@v0.5.0
        with:
          project-url: https://github.com/orgs/myorg/projects/1
          github-token: ${{ secrets.PROJECT_TOKEN }}

      - name: Update Status
        uses: actions/github-script@v7
        with:
          github-token: ${{ secrets.PROJECT_TOKEN }}
          script: |
            // Issue가 닫히면 Done 열로 이동
            if (context.payload.action === 'closed') {
              // 프로젝트 보드 업데이트 로직
            }

자동화 예시: - Issue 생성 → Backlog 열에 추가 - PR 생성 → In Progress 열로 이동 - PR 머지 → Done 열로 이동 - Issue 재개 → To Do 열로 이동

베스트 프랙티스: 자동화는 팀 워크플로우에 맞게 커스터마이즈하세요. 모든 프로젝트에 맞는 만능 자동화는 없습니다.

절 요약¶

  • CODEOWNERS로 코드 영역별 소유자를 지정하고 자동 리뷰어 할당
  • PR에 Closes #123 키워드를 사용하면 머지 시 Issue 자동 종료
  • GitHub Actions로 라벨 자동 분류, Stale Issue 관리 등을 자동화
  • Stale bot으로 오래된 Issue를 정리하여 백로그 관리
  • GitHub Projects 보드를 Issue 상태 변화에 따라 자동 업데이트

FAQ¶

Q1: GitHub Projects와 Jira의 차이는 무엇인가요? 어떤 것을 사용해야 하나요?

A1: GitHub Projects는 GitHub 생태계에 깊이 통합되어 Issue와 PR을 직접 관리하며, 간단한 프로젝트 관리에 적합합니다. Jira는 복잡한 워크플로우, 고급 보고서, 엔터프라이즈 기능이 필요한 대규모 조직에 적합합니다. 중소규모 팀이나 오픈소스 프로젝트는 GitHub Projects로 충분하며, 대기업 또는 복잡한 프로세스가 필요한 경우 Jira를 고려하세요.

Q2: Release를 잘못 만들었는데 삭제할 수 있나요?

A2: 네, Release는 삭제할 수 있습니다. 저장소 → Releases → 해당 Release → Delete this release. 단, Git Tag는 별도로 삭제해야 합니다(git push --delete origin v1.0.0). 이미 배포된 Release를 삭제하면 사용자에게 혼란을 줄 수 있으므로, 대신 새로운 패치 버전을 출시하는 것이 좋습니다.

Q3: Discussions를 활성화했는데 아무도 사용하지 않아요. 어떻게 활성화할 수 있나요?

A3: 다음 방법을 시도하세요. (1) README에 Discussions 링크와 사용법 추가, (2) 관리자가 먼저 질문을 올리고 답변하여 예시 제공, (3) Issue 템플릿에 "질문은 Discussions로" 안내 추가, (4) 커뮤니티에 흥미로운 주제로 토론 시작 (예: "다음 기능 투표"), (5) Q&A에 답변하고 Accepted Answer 표시하여 유용성 강조.

Q4: GitHub Pages 사이트가 404 오류가 나요. 어떻게 해결하나요?

A4: 다음을 확인하세요. (1) Settings → Pages에서 올바른 Source 설정 (브랜치 또는 GitHub Actions), (2) 배포 완료 여부 확인 (Actions 탭에서 워크플로우 상태 확인), (3) index.html 또는 index.md 파일이 루트에 존재하는지 확인, (4) CNAME 파일이 잘못되지 않았는지 확인, (5) 배포 후 1-2분 대기 (GitHub CDN 캐시 업데이트 시간 필요).

Q5: CODEOWNERS 파일을 만들었는데 자동 리뷰어 할당이 안 됩니다.

A5: 다음을 점검하세요. (1) 파일 위치가 정확한지 확인 (.github/CODEOWNERS, 저장소 루트, 또는 docs/ 중 하나), (2) 파일이 main 브랜치에 커밋되었는지 확인 (PR의 base 브랜치에 있어야 함), (3) 문법 오류 확인 (공백, 경로 등), (4) 지정된 사용자가 저장소 접근 권한이 있는지 확인 (읽기 권한 이상 필요), (5) Organization의 경우 팀 이름이 올바른지 확인 (@org/team-name).

장 요약¶

이번 장에서는 GitHub의 프로젝트 관리 도구들을 배웠습니다. GitHub Projects로 작업을 시각적으로 관리하고, Milestones와 Releases로 릴리스 계획을 수립하며, Discussions로 커뮤니티와 소통하는 방법을 익혔습니다. GitHub Pages로 전문적인 문서 사이트를 만들고, CODEOWNERS와 자동화 워크플로우로 반복 작업을 줄이는 방법도 배웠습니다. 이제 프로젝트를 체계적으로 관리하고 팀 생산성을 높일 수 있습니다.
