01장: Git과 버전 관리 시작하기

입문

Git과 버전 관리 시작하기¶

학습 목표 - 버전 관리의 필요성과 Git의 장점을 설명할 수 있다 - 자신의 운영체제에 Git을 설치하고 초기 설정을 완료할 수 있다 - git init으로 새 저장소를 생성하고 .git 디렉토리의 역할을 이해할 수 있다 - Git의 세 가지 영역(Working Directory, Staging Area, Repository)의 관계를 설명할 수 있다

버전 관리란 무엇인가 (ch_01_s01)¶

도입¶

파일을 수정하다가 실수로 중요한 내용을 삭제한 경험이 있으신가요? '보고서_최종.docx', '보고서_최종_수정.docx', '보고서_진짜최종.docx'처럼 파일명으로 버전을 관리해 본 적이 있으신가요? 이 절에서는 이러한 불편함을 해결하는 버전 관리 시스템의 개념과 필요성을 알아봅니다.

핵심 개념 설명¶

버전 관리(Version Control)란¶

버전 관리(Version Control)는 파일의 변경 이력을 체계적으로 기록하고 관리하여, 이전 상태로 복원하거나 변경 내용을 추적할 수 있게 하는 시스템입니다.

문서를 작성할 때 '보고서_최종.docx', '보고서_최종_수정.docx', '보고서_진짜최종.docx'처럼 파일을 복사하는 대신, 하나의 파일에 모든 변경 히스토리를 자동으로 기록하는 타임머신과 같습니다.

버전 관리가 필요한 이유

  1. 실수 복구: 잘못된 수정을 이전 상태로 쉽게 되돌릴 수 있습니다.
  2. 변경 추적: 누가, 언제, 무엇을, 왜 변경했는지 모든 히스토리를 기록합니다.
  3. 협업 지원: 여러 사람이 동시에 작업해도 변경 사항을 안전하게 합칠 수 있습니다.
  4. 실험 안전성: 새로운 기능을 시도할 때 원본을 손상시키지 않고 실험할 수 있습니다.

팁: 버전 관리는 개발자만 사용하는 도구가 아닙니다. 문서, 디자인, 데이터 등 모든 종류의 파일 관리에 활용할 수 있습니다.

버전 관리 시스템의 역사와 종류¶

버전 관리 시스템은 크게 두 가지로 분류됩니다.

1. 중앙 집중식 VCS (예: SVN, CVS)

중앙 서버에만 전체 히스토리가 저장되고, 클라이언트는 최신 스냅샷만 가져옵니다. 네트워크 연결이 필수이며, 서버 장애 시 작업이 불가능합니다.

2. 분산형 VCS (예: Git, Mercurial)

모든 클라이언트가 전체 히스토리를 포함한 완전한 저장소를 가집니다. 오프라인 작업이 가능하며, 서버 장애에도 복구가 용이합니다.

중앙 집중식 vs 분산형 비교¶

구분 중앙 집중식(SVN) 분산형(Git)
저장소 위치 중앙 서버 1개 모든 클라이언트에 완전한 복제본
네트워크 의존성 필수 (커밋 시 연결 필요) 선택적 (오프라인 커밋 가능)
속도 느림 (서버 통신 필요) 빠름 (로컬 작업)
백업 중앙 서버만 백업 모든 클라이언트가 백업
브랜치 비용 높음 (디렉토리 복사) 낮음 (포인터만 생성)

베스트 프랙티스: 현대 소프트웨어 개발에서는 분산형 버전 관리 시스템(특히 Git)이 사실상의 표준입니다. 브랜치 기반 협업과 빠른 속도가 주요 장점입니다.

절 요약¶

  • 버전 관리는 파일의 변경 이력을 체계적으로 기록하는 시스템입니다.
  • 실수 복구, 변경 추적, 협업 지원, 실험 안전성을 제공합니다.
  • 분산형 VCS(Git)는 중앙 집중식(SVN)보다 속도, 백업, 브랜치 면에서 우수합니다.

Git 설치와 환경 설정 (ch_01_s02)¶

도입¶

Git을 사용하기 전에 먼저 컴퓨터에 설치하고 기본 설정을 완료해야 합니다. 이 절에서는 Windows, macOS, Linux 각 운영체제별 설치 방법과 필수 초기 설정을 단계별로 안내합니다.

핵심 개념 설명¶

Windows에서 Git 설치 (Git for Windows)¶

1단계: 설치 파일 다운로드

https://git-scm.com/download/win에서 최신 버전을 다운로드합니다.

2단계: 설치 실행

설치 마법사에서 다음 옵션을 권장합니다.

  • 에디터 선택: VS Code 또는 자신이 선호하는 에디터
  • PATH 환경 변수: "Git from the command line and also from 3rd-party software" 선택
  • 줄바꿈 변환: "Checkout Windows-style, commit Unix-style line endings" (권장)
  • 터미널 에뮬레이터: "Use Windows' default console window" 또는 "Use MinTTY"

3단계: 설치 확인

$ git --version
# 출력: git version 2.43.0

주의: Windows에서는 CRLF(캐리지 리턴 + 라인 피드) 줄바꿈을 사용하지만, Git은 내부적으로 LF만 사용합니다. 설치 시 자동 변환 설정을 권장합니다.

macOS에서 Git 설치 (Homebrew / Xcode CLI)¶

방법 1: Homebrew 사용 (권장)

$ brew install git

방법 2: Xcode Command Line Tools

$ xcode-select --install

터미널에서 위 명령을 실행하면 설치 다이얼로그가 나타납니다.

설치 확인

$ git --version
# 출력: git version 2.43.0

팁: Homebrew를 사용하면 최신 Git 버전을 쉽게 유지할 수 있습니다. Xcode CLI는 macOS에 기본 포함되지만 버전이 구식일 수 있습니다.

Linux에서 Git 설치 (apt/yum/dnf)¶

Debian/Ubuntu

$ sudo apt update
$ sudo apt install git

Fedora/RHEL/CentOS

$ sudo dnf install git
# 또는
$ sudo yum install git

설치 확인

$ git --version
# 출력: git version 2.43.0

필수 초기 설정: 사용자 이름과 이메일¶

Git은 모든 커밋에 작성자 정보를 기록합니다. 처음 설치 후 반드시 사용자 이름과 이메일을 설정해야 합니다.

$ git config --global user.name "홍길동"
$ git config --global user.email "hong@example.com"

설정 확인

$ git config --global user.name
# 출력: 홍길동

$ git config --global user.email
# 출력: hong@example.com

주의: --global 옵션은 현재 사용자의 모든 프로젝트에 적용됩니다. 특정 프로젝트에만 다른 설정을 적용하려면 프로젝트 디렉토리에서 --global 없이 실행합니다.

에디터, 줄바꿈(CRLF/LF) 설정¶

기본 에디터 설정

# VS Code 사용 시
$ git config --global core.editor "code --wait"

# Vim 사용 시
$ git config --global core.editor "vim"

# nano 사용 시
$ git config --global core.editor "nano"

줄바꿈 설정

OS별 권장 설정입니다.

# Windows
$ git config --global core.autocrlf true

# macOS/Linux
$ git config --global core.autocrlf input

모든 설정 확인

$ git config --list
# 출력:
# user.name=홍길동
# user.email=hong@example.com
# core.editor=code --wait
# core.autocrlf=true

베스트 프랙티스: 팀 협업 시 줄바꿈 설정이 통일되지 않으면 불필요한 diff가 발생합니다. 프로젝트에 .gitattributes 파일로 줄바꿈 규칙을 명시하는 것을 권장합니다.

절 요약¶

  • Windows는 Git for Windows, macOS는 Homebrew, Linux는 패키지 매니저로 설치합니다.
  • 설치 후 git config --global로 사용자 이름과 이메일을 반드시 설정해야 합니다.
  • OS별 줄바꿈 차이(CRLF/LF)를 core.autocrlf로 설정하여 협업 문제를 방지합니다.

첫 번째 Git 저장소 만들기 (ch_01_s03)¶

도입¶

Git 설치가 완료되었다면 이제 실제로 Git 저장소를 만들어 볼 차례입니다. 이 절에서는 git init 명령어로 프로젝트를 Git 저장소로 초기화하고, .git 디렉토리의 구조를 살펴봅니다.

핵심 개념 설명¶

git init으로 저장소 초기화¶

저장소(Repository)는 프로젝트의 모든 파일과 변경 히스토리를 저장하는 공간으로, .git 디렉토리에 메타데이터와 객체 데이터베이스를 포함합니다.

도서관의 서고와 같습니다. 서고에는 모든 책(파일)이 보관되어 있고, 카드 카탈로그(.git)에는 각 책의 위치, 저자, 입고 날짜 등의 정보가 기록되어 있습니다.

새 저장소 만들기

원하는 프로젝트 디렉토리에서 git init 명령을 실행합니다.

$ mkdir my-first-repo
$ cd my-first-repo
$ git init
# 출력: Initialized empty Git repository in /path/to/my-first-repo/.git/

현재 상태 확인

$ ls -la
# 출력:
# .
# ..
# .git/

숨김 디렉토리 .git이 생성되었습니다. 이 디렉토리가 Git 저장소의 핵심입니다.

팁: git init은 기존 프로젝트를 Git 저장소로 만들 수도 있습니다. 이미 파일이 있는 디렉토리에서 실행해도 문제없습니다.

.git 디렉토리 탐험하기¶

.git 디렉토리의 주요 구성 요소를 살펴봅니다.

$ ls -la .git/
# 출력:
# HEAD
# config
# description
# hooks/
# objects/
# refs/

주요 항목 설명

항목 설명
HEAD 현재 체크아웃된 브랜치(또는 커밋)를 가리키는 포인터
config 이 저장소의 설정 파일 (프로젝트별 설정)
objects/ 모든 콘텐츠(커밋, 파일, 트리)를 저장하는 데이터베이스
refs/ 브랜치와 태그의 포인터를 저장
hooks/ Git 이벤트에 반응하는 스크립트를 저장

HEAD 파일 확인

$ cat .git/HEAD
# 출력: ref: refs/heads/main

현재 main 브랜치를 가리키고 있습니다. (아직 커밋이 없어 refs/heads/main 파일은 존재하지 않습니다.)

주의: .git 디렉토리를 직접 수정하지 마세요. Git 명령어를 통해서만 조작해야 저장소가 손상되지 않습니다.

작업 디렉토리와 저장소의 관계¶

작업 디렉토리(Working Directory)는 실제 파일이 존재하는 공간입니다. 개발자가 직접 파일을 편집하는 곳입니다.

.git 디렉토리는 Git이 버전 관리에 필요한 모든 정보를 저장하는 숨겨진 공간입니다.

my-first-repo/          ← 작업 디렉토리
├── .git/               ← Git 저장소 (숨김)
│   ├── HEAD
│   ├── objects/
│   └── refs/
├── README.md           ← 실제 프로젝트 파일
└── main.py

베스트 프랙티스: .git 디렉토리만 삭제하면 일반 디렉토리가 되고 Git 히스토리는 모두 사라집니다. 백업 시 .git 디렉토리를 포함해야 합니다.

절 요약¶

  • git init 명령으로 디렉토리를 Git 저장소로 초기화합니다.
  • .git 디렉토리에 모든 버전 관리 정보가 저장됩니다.
  • 작업 디렉토리는 실제 파일이 있는 곳, .git은 Git의 데이터베이스입니다.

Git의 세 가지 영역 이해하기 (ch_01_s04)¶

도입¶

Git을 효과적으로 사용하려면 파일이 어떤 상태에 있는지, 어떤 경로로 저장되는지 이해해야 합니다. 이 절에서는 Git의 핵심 개념인 세 가지 영역(작업 디렉토리, 스테이징 영역, 저장소)의 관계와 파일 흐름을 알아봅니다.

핵심 개념 설명¶

작업 디렉토리(Working Directory): 실제 파일이 존재하는 곳¶

작업 디렉토리(Working Directory)는 프로젝트의 특정 버전을 체크아웃하여 실제 파일로 펼쳐 놓은 공간입니다.

개발자가 직접 파일을 생성, 수정, 삭제하는 곳입니다.

$ echo "# My First Project" > README.md
$ ls
# 출력: README.md

현재 README.md 파일은 작업 디렉토리에만 존재하고, Git은 아직 이 파일을 추적하지 않습니다.

스테이징 영역(Staging Area / Index): 커밋 준비 공간¶

스테이징 영역(Staging Area)은 커밋에 포함시킬 변경 사항을 미리 선별하여 준비하는 중간 영역으로, 인덱스(Index)라고도 불립니다.

장을 보고 돌아와서 냉장고(저장소)에 넣기 전에 식탁(스테이징 영역)에 올려놓고 무엇을 냉장고에 넣을지 정리하는 과정과 같습니다.

파일을 스테이징 영역에 추가

$ git add README.md

이제 README.md는 스테이징 영역에 추가되었습니다. 다음 커밋에 포함될 준비가 완료되었습니다.

팁: 스테이징 영역이 불필요한 단계라고 생각할 수 있지만, 여러 파일을 수정한 후 관련된 변경만 골라서 커밋할 수 있게 해주는 핵심 기능입니다.

저장소(.git): 커밋 히스토리 저장소¶

저장소(Repository)는 스테이징 영역의 스냅샷을 영구적으로 저장하는 공간입니다. .git 디렉토리에 모든 커밋 히스토리가 저장됩니다.

스테이징된 파일을 커밋

$ git commit -m "Add README file"
# 출력:
# [main (root-commit) a1b2c3d] Add README file
#  1 file changed, 1 insertion(+)
#  create mode 100644 README.md

이제 README.md의 현재 상태가 저장소에 영구적으로 기록되었습니다.

세 영역 간 파일 흐름 실습¶

세 영역 간 파일 이동을 실습으로 확인합니다.

1단계: 파일 수정 (작업 디렉토리)

$ echo "Hello Git!" >> README.md

2단계: 상태 확인

$ git status
# 출력:
# On branch main
# Changes not staged for commit:
#   modified:   README.md

수정된 파일이 작업 디렉토리에만 있고, 아직 스테이징되지 않았습니다.

3단계: 스테이징 (작업 디렉토리 → 스테이징 영역)

$ git add README.md
$ git status
# 출력:
# On branch main
# Changes to be committed:
#   modified:   README.md

4단계: 커밋 (스테이징 영역 → 저장소)

$ git commit -m "Add greeting message"
# 출력:

전체 흐름 요약

[작업 디렉토리]  →  git add  →  [스테이징 영역]  →  git commit  →  [저장소]
   파일 편집                         커밋 준비                      영구 저장

베스트 프랙티스: 논리적으로 관련된 변경만 묶어서 하나의 커밋으로 만드세요. 스테이징 영역을 활용하면 여러 파일을 수정해도 관련된 변경만 선택하여 커밋할 수 있습니다.

절 요약¶

  • 작업 디렉토리는 파일을 편집하는 실제 공간입니다.
  • 스테이징 영역은 커밋에 포함할 변경을 선별하는 준비 공간입니다.
  • 저장소는 커밋으로 영구 저장된 히스토리를 보관합니다.
  • git add로 스테이징, git commit으로 저장소에 기록합니다.

FAQ¶

Q1: Git과 GitHub는 같은 것인가요?

아닙니다. Git은 로컬 컴퓨터에서 실행되는 버전 관리 도구이고, GitHub는 Git 저장소를 호스팅하는 웹 서비스입니다. Git은 1명이 사용해도 유용하지만, GitHub는 협업을 위한 플랫폼입니다. GitLab, Bitbucket 등 다른 호스팅 서비스도 있습니다.

Q2: .git 디렉토리가 보이지 않아요.

.git은 숨김 디렉토리입니다. Windows 탐색기에서는 "보기 → 숨김 항목"을 체크하고, macOS Finder에서는 Cmd + Shift + .을 누르면 보입니다. 터미널에서는 ls -la 명령으로 확인할 수 있습니다.

Q3: git init을 실수로 잘못된 디렉토리에서 실행했어요. 어떻게 취소하나요?

.git 디렉토리를 삭제하면 됩니다. rm -rf .git (Linux/macOS) 또는 탐색기에서 .git 폴더를 삭제하세요. Git 저장소 초기화가 완전히 취소됩니다. 단, 이미 커밋한 히스토리도 모두 사라지므로 주의하세요.

Q4: 스테이징 영역이 왜 필요한가요? 바로 커밋하면 안 되나요?

스테이징 영역이 있으면 여러 파일을 수정한 후 관련된 변경만 선택하여 논리적으로 의미 있는 커밋을 만들 수 있습니다. 예를 들어 버그 수정과 새 기능을 동시에 작업했다면, 각각 별도의 커밋으로 분리할 수 있습니다. 이는 나중에 히스토리를 검토할 때 매우 유용합니다.

Q5: 회사 이메일과 개인 이메일을 프로젝트별로 다르게 설정할 수 있나요?

가능합니다. --global 없이 프로젝트 디렉토리에서 git config user.email "work@company.com"을 실행하면 해당 프로젝트에만 적용됩니다. 설정 우선순위는 로컬(프로젝트) > 전역(--global) > 시스템(--system) 순입니다.

장 요약¶

이번 장에서는 Git과 버전 관리의 기초를 배웠습니다. 버전 관리 시스템이 필요한 이유와 중앙 집중식과 분산형의 차이를 이해했습니다. 운영체제별로 Git을 설치하고 필수 설정(사용자 이름, 이메일, 줄바꿈)을 완료했습니다. git init으로 첫 저장소를 만들고, .git 디렉토리의 역할을 살펴보았습니다. 마지막으로 Git의 세 가지 핵심 영역(작업 디렉토리, 스테이징 영역, 저장소)과 파일 흐름을 실습했습니다.
