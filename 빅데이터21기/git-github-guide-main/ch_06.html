06장: Pull Request와 코드 리뷰

기초

Pull Request와 코드 리뷰¶

학습 목표 - Pull Request의 목적과 코드 리뷰의 중요성을 설명할 수 있다 - 웹 UI와 GitHub CLI로 PR을 생성하고 관리할 수 있다 - 효과적인 코드 리뷰를 수행하고 Suggestion 기능을 활용할 수 있다 - 세 가지 머지 전략의 차이를 이해하고 상황에 맞게 선택할 수 있다 - PR 충돌을 웹 UI와 로컬 환경에서 해결할 수 있다

Pull Request의 개념과 워크플로우 (ch_06_s01)¶

도입¶

코드를 작성한 후 바로 main 브랜치에 병합하면 어떤 일이 발생할까요? 버그가 숨어 들어가거나, 코딩 스타일이 일관되지 않거나, 더 나은 방법을 놓칠 수 있습니다. 이 절에서는 이러한 문제를 해결하는 GitHub의 핵심 협업 메커니즘인 Pull Request를 배웁니다.

핵심 개념 설명¶

Pull Request란 무엇인가¶

Pull Request(PR)는 브랜치의 변경 사항을 다른 브랜치에 병합하기 전에 팀원에게 리뷰를 요청하는 GitHub의 협업 메커니즘입니다.

논문 심사 과정과 같습니다. 연구자(개발자)가 논문(코드)을 제출하면, 심사위원(팀원)이 검토하고 피드백을 주고, 수정 후 최종 승인되면 학술지(main 브랜치)에 게재됩니다.

PR은 단순한 병합 요청을 넘어서, 코드 리뷰, 자동화된 테스트, 토론, 문서화가 이루어지는 협업의 중심입니다.

PR 기반 협업의 장점¶

1. 코드 품질 향상

동료 개발자가 코드를 검토하면 버그, 성능 문제, 보안 취약점을 조기에 발견할 수 있습니다. 4명의 눈이 2명의 눈보다 버그를 더 잘 찾아냅니다.

2. 지식 공유와 학습

코드 리뷰를 통해 팀원이 서로의 코드를 이해하고 새로운 기술을 배웁니다. 시니어 개발자의 피드백은 주니어의 성장을 가속화합니다.

3. 변경 사항 문서화

PR 설명과 리뷰 코멘트는 "왜 이렇게 구현했는가"를 기록합니다. 6개월 후 코드를 다시 볼 때 맥락을 빠르게 파악할 수 있습니다.

4. CI/CD 자동화 통합

PR 생성 시 자동으로 테스트, 린트, 빌드를 실행하여 품질을 보장합니다. 병합 전에 문제를 발견하여 main 브랜치를 항상 안정적으로 유지합니다.

5. 비동기 협업 지원

서로 다른 시간대에서 일하는 팀원도 PR을 통해 효율적으로 협업할 수 있습니다. 리뷰어는 자신의 시간에 맞춰 검토하고 피드백을 남길 수 있습니다.

팁: PR은 코드가 완성된 후에만 만드는 것이 아닙니다. Draft PR로 작업 중인 코드를 미리 공유하여 초기 피드백을 받을 수 있습니다.

PR 워크플로우: 브랜치 생성부터 머지까지¶

PR 기반 협업의 전체 흐름을 단계별로 살펴봅니다.

1단계: 브랜치 생성

$ git switch -c feature/user-profile
# feature/user-profile 브랜치 생성 및 전환

2단계: 코드 작성 및 커밋

$ git add src/components/UserProfile.js
$ git commit -m "feat: 사용자 프로필 컴포넌트 추가"

3단계: 원격에 push

$ git push -u origin feature/user-profile

4단계: PR 생성

GitHub 웹 UI 또는 CLI로 PR을 생성합니다. (다음 절에서 상세 설명)

5단계: 리뷰 요청

Reviewer를 지정하고 리뷰를 요청합니다.

6단계: 피드백과 수정

리뷰어의 코멘트를 확인하고 필요한 수정을 진행합니다.

$ git add .
$ git commit -m "refactor: 리뷰 피드백 반영"
$ git push

PR에 새 커밋이 자동으로 추가되며, 리뷰어에게 알림이 갑니다.

7단계: 승인(Approve)

리뷰어가 코드를 승인합니다.

8단계: 머지

PR을 main 브랜치에 병합합니다.

9단계: 브랜치 삭제

병합된 브랜치를 삭제하여 저장소를 깔끔하게 유지합니다.

$ git switch main
$ git pull
$ git branch -d feature/user-profile

베스트 프랙티스: PR 제목에 타입을 명시하세요. feat:, fix:, docs:, refactor: 등 Conventional Commits 규약을 따르면 변경 사항을 한눈에 파악할 수 있습니다.

좋은 PR 작성법: 제목, 설명, 크기¶

작게 나누기

한 PR에 하나의 목적만 담으세요. 300줄 이내를 권장합니다. 큰 PR은 리뷰하기 어렵고, 버그를 찾기 어려우며, 충돌 가능성이 높습니다.

명확한 제목

좋은 예:
- feat: 사용자 프로필 페이지 추가
- fix: 로그인 시 토큰 만료 오류 수정
- refactor: API 클라이언트를 axios로 전환

나쁜 예:
- 업데이트
- 수정
- 작업 완료

상세한 설명

## 변경 사항
사용자 프로필 페이지를 추가했습니다. 프로필 사진, 이름, 이메일, 가입일을 표시합니다.

## 변경 이유
Issue #123에서 요청된 기능입니다. 사용자가 자신의 정보를 확인할 수 있어야 합니다.

## 테스트 방법
1. 로그인 후 우측 상단 프로필 아이콘 클릭
2. 프로필 페이지에서 정보 확인
3. 편집 버튼으로 이름 변경 테스트

## 스크린샷
![프로필 페이지](screenshots/profile.png)

## 체크리스트
- [x] 로컬에서 테스트 완료
- [x] 단위 테스트 추가
- [x] 문서 업데이트
- [x] 관련 이슈: Closes #123

관련 이슈 연결

Closes #123, Fixes #456, Resolves #789 키워드를 사용하면 PR 머지 시 자동으로 이슈가 닫힙니다.

주의: 여러 기능을 하나의 PR에 넣으면 리뷰어가 부담을 느끼고, 리뷰 시간이 길어지며, 일부만 수정하고 싶어도 전체를 다시 검토해야 합니다.

절 요약¶

  • PR은 코드 병합 전 리뷰를 요청하는 GitHub의 핵심 협업 도구입니다.
  • 코드 품질 향상, 지식 공유, 문서화, CI 자동화의 장점이 있습니다.
  • 브랜치 생성 → 커밋 → push → PR 생성 → 리뷰 → 수정 → 승인 → 머지의 워크플로우를 따릅니다.
  • 좋은 PR은 작고, 명확한 제목과 상세한 설명을 가지며, 관련 이슈를 연결합니다.

PR 생성과 관리 (ch_06_s02)¶

도입¶

PR의 개념을 이해했다면 이제 실제로 PR을 생성하고 관리하는 방법을 배울 차례입니다. 이 절에서는 웹 UI와 GitHub CLI 두 가지 방법으로 PR을 생성하고, Draft PR과 PR Template을 활용하는 실무 기술을 익힙니다.

핵심 개념 설명¶

웹 UI에서 PR 생성하기¶

GitHub 웹사이트에서 PR을 만드는 가장 직관적인 방법입니다.

1단계: 저장소 이동

GitHub에서 프로젝트 저장소로 이동합니다.

2단계: Pull requests 탭 클릭

상단 메뉴에서 "Pull requests" 탭을 선택합니다.

3단계: New pull request 클릭

초록색 "New pull request" 버튼을 클릭합니다.

4단계: 브랜치 선택

  • base: 병합 대상 브랜치 (일반적으로 main 또는 develop)
  • compare: 내 작업 브랜치 (예: feature/user-profile)

GitHub은 두 브랜치 간 차이를 자동으로 표시합니다.

5단계: Create pull request 클릭

변경 사항을 확인한 후 "Create pull request" 버튼을 클릭합니다.

6단계: 제목과 설명 작성

  • 제목: 50자 이내의 명확한 요약
  • 설명: 변경 이유, 테스트 방법, 관련 이슈 등 상세 정보

7단계: 메타데이터 설정

우측 사이드바에서 다음을 지정할 수 있습니다.

  • Reviewers: 리뷰를 요청할 팀원 (최소 1명 권장)
  • Assignees: PR 담당자 (보통 작성자 본인)
  • Labels: 타입 분류 (bug, enhancement, documentation 등)
  • Projects: 프로젝트 보드에 연결
  • Milestone: 릴리스 일정과 연결

8단계: Create pull request

모든 정보를 입력한 후 "Create pull request" 버튼을 최종 클릭합니다.

팁: PR 생성 후에도 제목, 설명, 메타데이터를 언제든지 수정할 수 있습니다. 리뷰 도중 추가 정보가 필요하면 편집하세요.

GitHub CLI(gh pr create)로 PR 생성하기¶

터미널을 선호하는 개발자를 위한 빠른 방법입니다.

기본 사용법

$ git push -u origin feature/user-profile
$ gh pr create

인터랙티브 모드에서 질문에 답변하며 PR을 생성합니다.

? Where should we push the 'feature/user-profile' branch? myusername/myrepo
? Title feat: 사용자 프로필 페이지 추가
? Body <Received>
? What's next? Submit

명령줄에서 모든 옵션 지정

$ gh pr create \
  --title "feat: 사용자 프로필 페이지 추가" \
  --body "Issue #123을 해결합니다. 프로필 페이지를 추가했습니다." \
  --base main \
  --head feature/user-profile \
  --reviewer alice,bob \
  --label enhancement \
  --assignee @me

파일에서 설명 읽기

$ gh pr create --title "feat: 프로필 추가" --body-file pr-description.md

PR 생성 후 브라우저로 열기

$ gh pr create --web

베스트 프랙티스: 팀의 워크플로우에 맞는 방법을 선택하세요. 웹 UI는 시각적이고 직관적이며, CLI는 빠르고 스크립트 자동화에 유리합니다.

Draft PR: 작업 중인 PR 표시¶

Draft PR은 아직 리뷰 준비가 완료되지 않은 PR을 표시하는 기능입니다.

사용 시나리오

  • 초기 피드백을 받고 싶을 때
  • CI 테스트를 먼저 실행하고 싶을 때
  • 작업 진행 상황을 팀에게 공유하고 싶을 때
  • 나중에 본격적인 리뷰를 요청하려고 할 때

웹 UI에서 Draft PR 만들기

"Create pull request" 버튼 옆 드롭다운에서 "Create draft pull request"를 선택합니다.

GitHub CLI로 Draft PR 만들기

$ gh pr create --draft

Draft에서 Ready for review로 전환

PR 페이지 하단에서 "Ready for review" 버튼을 클릭하면 리뷰어에게 알림이 전송됩니다.

# CLI로 전환
$ gh pr ready <PR번호>

주의: Draft PR은 병합할 수 없습니다. Ready for review로 전환해야 Merge 버튼이 활성화됩니다.

PR Template 설정¶

PR Template을 설정하면 PR 생성 시 자동으로 템플릿 내용이 설명란에 채워집니다.

템플릿 파일 위치

.github/pull_request_template.md 파일을 저장소 루트에 생성합니다.

기본 템플릿 예시

## 변경 사항 요약
<!-- 무엇을 바꿨는지 간단히 설명 -->

## 변경 이유
<!-- 왜 이 변경이 필요한지 설명 -->

## 테스트 방법
<!-- 리뷰어가 어떻게 테스트할 수 있는지 안내 -->
1.
2.
3.

## 스크린샷 (선택)
<!-- UI 변경이 있다면 스크린샷 첨부 -->

## 체크리스트
- [ ] 로컬에서 테스트 완료
- [ ] 단위 테스트 추가/업데이트
- [ ] 문서 업데이트 (필요 시)
- [ ] 관련 이슈: Closes #

## 추가 참고사항
<!-- 리뷰어가 알아야 할 추가 정보 -->

여러 템플릿 사용 (고급)

.github/PULL_REQUEST_TEMPLATE/ 디렉토리에 여러 템플릿을 저장하고 선택할 수 있습니다.

.github/
└── PULL_REQUEST_TEMPLATE/
    ├── bug_fix.md
    ├── feature.md
    └── documentation.md

PR 생성 시 쿼리 파라미터로 템플릿 지정:

https://github.com/user/repo/compare/main...feature?template=feature.md

베스트 프랙티스: PR Template은 팀의 리뷰 프로세스를 표준화하고, PR 작성자가 중요한 정보를 누락하지 않도록 도와줍니다.

Reviewer, Assignee, Label 지정¶

Reviewer

리뷰를 요청할 팀원을 지정합니다. 일반적으로 1~2명을 지정하며, 코드 소유자나 해당 영역의 전문가를 선택합니다.

$ gh pr create --reviewer alice,bob

Assignee

PR의 담당자를 지정합니다. 보통 PR 작성자 본인을 지정하여 책임 소재를 명확히 합니다.

$ gh pr create --assignee @me

Label

PR의 타입이나 우선순위를 분류합니다.

  • bug: 버그 수정
  • enhancement: 기능 추가
  • documentation: 문서 작업
  • breaking change: 호환성이 깨지는 변경
  • needs review: 리뷰 필요
  • work in progress: 작업 중
$ gh pr create --label bug,urgent

팁: CODEOWNERS 파일을 설정하면 특정 파일 수정 시 자동으로 해당 코드 소유자가 리뷰어로 지정됩니다.

절 요약¶

  • 웹 UI에서 "New pull request"로 직관적으로 PR을 생성할 수 있습니다.
  • GitHub CLI(gh pr create)로 터미널에서 빠르게 PR을 만들 수 있습니다.
  • Draft PR로 작업 중인 PR을 표시하고 초기 피드백을 받을 수 있습니다.
  • PR Template으로 PR 작성을 표준화하고 중요 정보 누락을 방지합니다.
  • Reviewer, Assignee, Label을 지정하여 PR을 체계적으로 관리합니다.

코드 리뷰 수행하기 (ch_06_s03)¶

도입¶

PR을 만들었다면 이제 리뷰어의 역할을 이해해야 합니다. 효과적인 코드 리뷰는 버그를 줄이고 코드 품질을 높이며 팀 전체의 역량을 끌어올립니다. 이 절에서는 GitHub의 코드 리뷰 기능을 활용하여 건설적인 피드백을 제공하는 방법을 배웁니다.

핵심 개념 설명¶

Files Changed 탭에서 변경 사항 검토¶

Files Changed 탭 열기

PR 페이지에서 "Files changed" 탭을 클릭하면 모든 변경 사항을 파일별로 확인할 수 있습니다.

코드 색상

  • 초록색 배경: 추가된 줄 (+)
  • 빨간색 배경: 삭제된 줄 (-)
  • 흰색 배경: 맥락을 위한 변경되지 않은 줄

뷰 모드 전환

  • Unified: 변경 전후를 하나의 창에 표시 (기본값)
  • Split: 변경 전/후를 좌우로 나눠 표시
Unified 뷰:
  10  function login(username, password) {
- 11    return api.post('/login', { username, password });
+ 11    return api.post('/auth/login', { username, password });
  12  }

Split 뷰:
[이전]                          [변경 후]
function login(user, pass) {    function login(username, password) {
  return api.post('/login',       return api.post('/auth/login',
    { user, pass });                { username, password });
}                               }

변경 파일 필터링

파일이 많을 때 체크박스로 검토 완료한 파일을 표시할 수 있습니다. "Viewed" 체크를 하면 해당 파일이 접히고 진행 상황을 추적할 수 있습니다.

팁: 큰 PR은 논리적인 순서로 파일을 검토하세요. 예를 들어 모델 → 컨트롤러 → 뷰 순서로 리뷰하면 맥락을 이해하기 쉽습니다.

라인별 코멘트 달기¶

단일 라인 코멘트

  1. 코멘트를 달고 싶은 줄 번호 옆 + 아이콘 클릭
  2. 코멘트 작성
  3. 선택:
  4. Add single comment: 즉시 게시 (다른 사람이 바로 볼 수 있음)
  5. Start a review: 리뷰에 추가 (나중에 일괄 게시)

여러 라인에 코멘트

  1. 시작 줄의 줄 번호 클릭 후 드래그하여 끝 줄 선택
  2. + 아이콘 클릭하여 코멘트 작성
  3. 선택한 여러 줄에 대한 코멘트 작성

코멘트 작성 요령

좋은 코멘트:
- 이 부분에서 `username`이 null일 경우를 처리하는 게 좋을 것 같습니다. 예를 들어:
  ```javascript
  if (!username) {
    throw new Error('Username is required');
  }
  ```

나쁜 코멘트:
- 이거 틀렸어요.
- 왜 이렇게 했나요?

응답하기

코멘트에 대한 토론이 가능합니다. 리뷰어와 작성자가 주고받으며 최선의 해결책을 찾습니다.

베스트 프랙티스: 긍정적이고 건설적인 피드백을 제공하세요. "이렇게 하면 어떨까요?"라는 제안이 "왜 이렇게 했나요?"라는 질문보다 효과적입니다.

Suggestion 기능으로 코드 수정 제안¶

Suggestion이란

리뷰어가 코드를 직접 수정하여 제안하는 기능입니다. PR 작성자는 클릭 한 번으로 제안을 적용할 수 있습니다.

Suggestion 작성

코멘트 작성 중 툴바에서 ± 아이콘(Insert a suggestion)을 클릭하거나 Markdown 직접 작성:

이 부분을 상수로 분리하는 게 좋겠습니다:

​```suggestion
const MAX_LOGIN_ATTEMPTS = 3;
if (attempts > MAX_LOGIN_ATTEMPTS) {
  throw new Error('Too many login attempts');
}
​```

GitHub은 자동으로 현재 코드와 제안 코드를 비교하여 표시합니다.

Suggestion 적용

PR 작성자는 "Commit suggestion" 버튼을 클릭하면 제안이 새 커밋으로 자동 추가됩니다.

여러 Suggestion 한 번에 적용

"Add suggestion to batch" 버튼으로 여러 제안을 모은 후 "Commit suggestions" 클릭하여 하나의 커밋으로 적용할 수 있습니다.

주의: Suggestion은 단일 파일의 연속된 줄에만 적용됩니다. 여러 파일에 걸친 변경은 수동으로 커밋해야 합니다.

리뷰 완료: Approve, Request Changes, Comment¶

리뷰를 마치면 전체 피드백을 제출합니다.

Finish your review 클릭

"Files changed" 탭 상단의 "Finish your review" 또는 "Review changes" 버튼을 클릭합니다.

세 가지 리뷰 타입

1. Comment (코멘트만)

의견을 남기지만 승인이나 거부는 하지 않습니다. 질문이나 제안만 있을 때 사용합니다.

2. Approve (승인)

코드가 병합 가능한 상태라고 승인합니다. 브랜치 보호 규칙에서 필수 승인 수를 설정한 경우, 해당 수만큼 승인이 모여야 병합할 수 있습니다.

3. Request changes (변경 요청)

반드시 수정이 필요한 문제가 있을 때 사용합니다. 변경 요청 상태에서는 병합이 차단됩니다. (브랜치 보호 규칙 설정 시)

리뷰 코멘트 작성

전반적으로 좋은 구현입니다! 몇 가지 개선 제안을 남겼습니다.

주요 의견:
- 입력 검증 로직 추가 필요
- 에러 처리 개선 권장
- 단위 테스트가 누락된 것 같습니다

수정 후 재요청 부탁드립니다.

제출

"Submit review" 버튼을 클릭하여 모든 코멘트를 한 번에 게시합니다.

베스트 프랙티스: 사소한 문제(코딩 스타일, 오타)는 Comment로, 버그나 보안 문제는 Request changes로 처리하세요.

효과적인 코드 리뷰를 위한 체크리스트¶

기능성 - [ ] 코드가 의도한 대로 동작하는가? - [ ] 엣지 케이스를 처리하는가? - [ ] 에러 처리가 적절한가?

가독성 - [ ] 변수/함수 이름이 명확한가? - [ ] 복잡한 로직에 주석이 있는가? - [ ] 코딩 스타일 가이드를 따르는가?

성능 - [ ] 불필요한 반복이나 연산이 없는가? - [ ] 데이터베이스 쿼리가 효율적인가? - [ ] 메모리 누수 가능성은 없는가?

보안 - [ ] 사용자 입력을 검증하는가? - [ ] SQL 인젝션, XSS 등 취약점이 없는가? - [ ] 민감한 정보가 로그에 노출되지 않는가?

테스트 - [ ] 새 기능에 대한 테스트가 있는가? - [ ] 기존 테스트가 모두 통과하는가? - [ ] 테스트 커버리지가 충분한가?

문서 - [ ] README나 API 문서가 업데이트되었는가? - [ ] 복잡한 알고리즘에 설명이 있는가?

팁: 리뷰에 30분 이상 걸린다면 PR이 너무 큰 것입니다. 작성자에게 더 작은 PR로 나눌 것을 제안하세요.

절 요약¶

  • Files Changed 탭에서 변경 사항을 파일별로 검토합니다.
  • 라인별 코멘트로 구체적인 피드백을 제공합니다.
  • Suggestion 기능으로 코드 수정을 직접 제안할 수 있습니다.
  • Approve, Request changes, Comment 세 가지 리뷰 타입으로 최종 의견을 제출합니다.
  • 효과적인 리뷰는 기능성, 가독성, 성능, 보안, 테스트, 문서를 모두 검토합니다.

머지 전략 선택하기 (ch_06_s04)¶

도입¶

PR이 승인되었다면 이제 main 브랜치에 병합할 차례입니다. 그런데 GitHub은 세 가지 머지 전략을 제공합니다. 어떤 전략을 선택하느냐에 따라 커밋 히스토리의 모양이 완전히 달라집니다. 이 절에서는 각 전략의 특징을 이해하고 프로젝트에 맞는 선택을 할 수 있도록 안내합니다.

핵심 개념 설명¶

Create a Merge Commit: 모든 커밋 보존¶

동작 방식

feature 브랜치의 모든 커밋을 유지하고 새로운 병합 커밋을 생성합니다.

main:     A --- B --- C ----------- M
                       \           /
feature:                D --- E ---

M은 병합 커밋이며, 두 부모(C와 E)를 가집니다.

장점

  • 완전한 히스토리 보존: 모든 커밋과 작업 과정이 기록됩니다
  • 브랜치 흐름 명확: 어디서 분기했고 어디서 합쳐졌는지 명확합니다
  • 되돌리기 쉬움: git revert M으로 전체 기능을 한 번에 되돌릴 수 있습니다

단점

  • 히스토리 복잡도: 브랜치가 많으면 그래프가 복잡해집니다
  • 병합 커밋 증가: 작은 PR도 병합 커밋을 만들어 히스토리가 길어집니다

사용 시나리오

  • 작은 팀에서 상세한 히스토리를 원할 때
  • 긴 수명의 기능 브랜치를 개발할 때
  • 병합 시점을 명확히 기록하고 싶을 때

베스트 프랙티스: feature 브랜치의 커밋이 논리적으로 의미 있게 나뉘어 있다면 Merge Commit이 좋습니다.

Squash and Merge: 커밋을 하나로 압축¶

동작 방식

feature 브랜치의 모든 커밋을 하나의 커밋으로 합친 후 main에 추가합니다.

main:     A --- B --- C --- S

feature:                D --- E ---  (병합 후 삭제)

S는 D와 E의 변경 사항을 모두 포함한 단일 커밋입니다.

장점

  • 깔끔한 선형 히스토리: main 브랜치가 간결하고 읽기 쉽습니다
  • PR 단위 추적: 각 PR이 하나의 커밋으로 표현되어 변경 단위가 명확합니다
  • 작은 커밋 정리: 실험적 커밋, 오타 수정 등이 히스토리에서 사라집니다

단점

  • 세부 히스토리 소실: feature 브랜치의 커밋 메시지와 작업 과정이 사라집니다
  • 원본 작성자 정보: Co-authored-by 태그를 수동으로 추가해야 합니다
  • 되돌리기 어려움: 큰 기능을 부분적으로만 되돌리기 어렵습니다

Squash 커밋 메시지 작성

feat: 사용자 프로필 페이지 추가 (#123)

사용자가 자신의 프로필 정보를 확인하고 편집할 수 있는 페이지를 추가했습니다.

변경 사항:
- 프로필 조회 API 엔드포인트
- 프로필 편집 폼 컴포넌트
- 프로필 이미지 업로드 기능

테스트:
- 단위 테스트 추가
- E2E 테스트 추가

사용 시나리오

  • 오픈소스 프로젝트: 깔끔한 히스토리를 원할 때
  • 큰 팀: 많은 PR이 병합되는 프로젝트
  • PR 단위 관리: 각 PR을 하나의 논리적 단위로 관리할 때

팁: Squash and Merge 사용 시 PR 설명을 상세히 작성하세요. 이것이 커밋 메시지의 본문이 됩니다.

Rebase and Merge: 선형 히스토리 유지¶

동작 방식

feature 브랜치의 커밋들을 main의 최신 커밋 위에 다시 적용합니다.

병합 전:
main:     A --- B --- C
                       \
feature:                D --- E

병합 후:
main:     A --- B --- C --- D' --- E'

D'와 E'는 D와 E의 내용은 같지만 새로운 커밋 해시를 가집니다.

장점

  • 완전 선형 히스토리: 병합 커밋이 없어 히스토리가 일직선입니다
  • 모든 커밋 보존: feature의 각 커밋이 main에 그대로 포함됩니다
  • bisect 친화적: 선형 히스토리에서 이진 탐색이 효율적입니다

단점

  • 커밋 해시 변경: 원본 커밋과 다른 해시를 가져 혼란 가능
  • Rebase 이해 필요: 팀원이 rebase 개념을 이해해야 합니다
  • 병합 시점 불명확: 언제 병합되었는지 히스토리만으로는 알기 어렵습니다

사용 시나리오

  • 선형 히스토리를 선호하는 팀
  • 각 커밋이 의미 있고 독립적일 때
  • Git 고급 사용자로 구성된 팀

주의: Rebase and Merge는 커밋 해시를 변경하므로, feature 브랜치를 다른 개발자와 공유 중이라면 주의해야 합니다.

세 가지 전략 비교와 선택 기준¶

구분 Merge Commit Squash and Merge Rebase and Merge
히스토리 브랜치 구조 유지 선형, PR 단위 완전 선형, 커밋 단위
커밋 수 모두 보존 + 병합 커밋 1개로 압축 모두 보존
되돌리기 병합 커밋 revert Squash 커밋 revert 각 커밋 revert
복잡도 높음 (브랜치 많을 때) 낮음 낮음
학습 곡선 낮음 낮음 높음 (rebase 이해 필요)

프로젝트 유형별 권장

프로젝트 유형 권장 전략 이유
오픈소스 Squash and Merge 깔끔한 히스토리, 기여자 커밋 스타일 통일
큰 조직 Squash and Merge 많은 PR 관리, PR 단위 추적
작은 팀 (3~5명) Merge Commit 상세 히스토리, 협업 과정 기록
선형 히스토리 선호 Rebase and Merge Git 고급 사용자, bisect 활용

Auto-merge 설정¶

Auto-merge란

필수 조건(리뷰 승인, CI 통과)이 충족되면 자동으로 병합하는 기능입니다.

사용법

1. PR 페이지에서 "Enable auto-merge" 클릭
2. 머지 전략 선택 (Merge commit / Squash / Rebase)
3. 조건 충족 시 자동 병합 실행

CLI로 설정

$ gh pr merge <PR번호> --auto --squash

베스트 프랙티스: 팀 전체가 하나의 머지 전략을 일관되게 사용하세요. 프로젝트 Settings → General → Pull Requests에서 허용할 전략을 제한할 수 있습니다.

절 요약¶

  • Merge Commit은 모든 커밋을 보존하고 병합 커밋을 생성합니다.
  • Squash and Merge는 PR의 모든 커밋을 하나로 압축합니다.
  • Rebase and Merge는 커밋을 main 위에 재적용하여 선형 히스토리를 만듭니다.
  • 프로젝트 특성과 팀 규모에 따라 적절한 전략을 선택하세요.
  • Auto-merge로 조건 충족 시 자동 병합을 설정할 수 있습니다.

PR 충돌 해결과 업데이트 (ch_06_s05)¶

도입¶

PR을 만든 후 다른 팀원이 먼저 같은 파일을 수정한 PR을 병합하면 어떻게 될까요? 충돌이 발생합니다. 충돌은 두려워할 일이 아니라 협업 과정에서 자연스럽게 발생하는 상황입니다. 이 절에서는 PR에서 충돌을 해결하고 리뷰 피드백을 반영하는 방법을 배웁니다.

핵심 개념 설명¶

PR에서 충돌이 발생했을 때¶

충돌 알림

PR 페이지에 다음 메시지가 표시됩니다.

This branch has conflicts that must be resolved
Use the web editor or the command line to resolve conflicts before the branch can be merged.

충돌 발생 원인

  1. 같은 파일의 같은 줄을 다르게 수정
  2. 한쪽에서 파일을 삭제하고 다른 쪽에서 수정
  3. 파일 이름을 다르게 변경

충돌 해결 방법 선택

  • 간단한 충돌: 웹 UI에서 해결
  • 복잡한 충돌: 로컬 환경에서 해결

웹 UI에서 충돌 해결하기¶

1단계: Resolve conflicts 클릭

PR 페이지의 "Resolve conflicts" 버튼을 클릭합니다.

2단계: 충돌 마커 확인

GitHub 에디터에서 충돌이 발생한 파일이 열립니다.

function greet(name) {
<<<<<<< feature/greeting
  return `Hello, ${name}!`;
=======
  return `Hi, ${name}!`;
>>>>>>> main
}
  • <<<<<<< feature/greeting: 내 브랜치의 변경
  • =======: 구분선
  • >>>>>>> main: 대상 브랜치(main)의 변경

3단계: 충돌 해결

충돌 마커를 제거하고 최종 코드를 작성합니다.

function greet(name) {
  return `Hello, ${name}!`;
}

또는 두 변경을 조합:

function greet(name) {
  return `Hello and hi, ${name}!`;
}

4단계: Mark as resolved

우측 상단의 "Mark as resolved" 버튼을 클릭합니다.

5단계: Commit merge

모든 충돌을 해결한 후 "Commit merge" 버튼을 클릭하여 충돌 해결 커밋을 추가합니다.

팁: 웹 UI는 간단한 텍스트 충돌에만 적합합니다. 여러 파일에 걸친 복잡한 충돌은 로컬에서 해결하는 것이 좋습니다.

로컬에서 충돌 해결 후 push하기¶

1단계: 브랜치 전환

$ git switch feature/my-feature

2단계: main의 최신 변경 가져오기

$ git fetch origin

3단계: main 브랜치 병합

$ git merge origin/main
# 출력:
# Auto-merging src/greet.js
# CONFLICT (content): Merge conflict in src/greet.js
# Automatic merge failed; fix conflicts and then commit the result.

4단계: 충돌 파일 확인

$ git status
# 출력:
# Unmerged paths:
#   both modified:   src/greet.js

5단계: 에디터에서 충돌 해결

VS Code, IntelliJ 등 에디터의 병합 도구를 사용하면 시각적으로 충돌을 해결할 수 있습니다.

VS Code 예시:

Accept Current Change | Accept Incoming Change | Accept Both Changes | Compare Changes

<<<<<<< HEAD (Current Change)
  return `Hello, ${name}!`;
=======
  return `Hi, ${name}!`;
>>>>>>> origin/main (Incoming Change)

6단계: 해결 완료 후 스테이징

$ git add src/greet.js

7단계: 병합 커밋 생성

$ git commit
# 기본 병합 커밋 메시지 사용 또는 수정

8단계: push

$ git push

PR에 충돌 해결 커밋이 자동으로 추가되며, 충돌 경고가 사라집니다.

베스트 프랙티스: 충돌 해결 후 테스트를 다시 실행하세요. 두 변경을 합친 결과가 예상대로 동작하는지 확인해야 합니다.

리뷰 반영 후 PR 업데이트하기¶

피드백 확인

PR 페이지의 "Files changed" 탭에서 리뷰 코멘트를 확인합니다.

로컬에서 수정

$ git switch feature/my-feature
# 파일 수정
$ git add .
$ git commit -m "refactor: 리뷰 피드백 반영 - 입력 검증 추가"
$ git push

PR 자동 업데이트

push하면 PR에 새 커밋이 자동으로 추가되고, 리뷰어에게 알림이 갑니다.

Suggestion 일괄 적용

여러 Suggestion을 "Add to batch"로 모은 후 "Commit suggestions" 클릭하여 한 번에 적용할 수 있습니다.

Re-review 요청

수정이 완료되면 리뷰어에게 다시 검토를 요청합니다.

$ gh pr ready <PR번호>  # Draft에서 Ready로 전환

또는 웹 UI에서 리뷰어에게 "Re-request review" 버튼을 클릭합니다.

팁: 리뷰 피드백에 대한 응답을 코멘트로 남기세요. "수정 완료했습니다" 또는 "다른 방법으로 구현했습니다"처럼 소통하면 리뷰어가 변경 사항을 빠르게 파악할 수 있습니다.

Force push가 필요한 상황과 주의사항¶

Force push가 필요한 경우

  • 인터랙티브 rebase로 커밋을 정리한 후
  • 커밋 메시지를 수정한 후 (git commit --amend)
  • 실수로 push한 민감한 정보를 제거한 후

안전한 force push: --force-with-lease

$ git push --force-with-lease

--force-with-lease는 원격 브랜치가 예상과 다르면 push를 거부하여 다른 사람의 변경을 덮어쓰는 것을 방지합니다.

일반 force push (위험)

$ git push --force

이 명령은 원격 브랜치를 강제로 덮어씁니다. 다른 사람의 변경을 날릴 수 있으므로 주의해야 합니다.

Force push 안전 규칙

  1. PR 작업 브랜치에서만 사용
  2. main, develop 등 공유 브랜치에는 절대 사용 금지
  3. 다른 팀원과 같은 브랜치를 공유 중이라면 사전 협의
  4. 가능하면 --force-with-lease 사용

예시 시나리오

# 커밋 메시지 오타 수정
$ git commit --amend -m "feat: 사용자 프로필 페이지 추가 (오타 수정)"
$ git push --force-with-lease

경고: main 브랜치에 force push하면 다른 팀원의 작업이 날아가고 저장소가 혼란에 빠집니다. 브랜치 보호 규칙으로 force push를 차단하는 것을 권장합니다.

절 요약¶

  • PR 충돌은 웹 UI 또는 로컬 환경에서 해결할 수 있습니다.
  • 웹 UI는 간단한 충돌에 적합하고, 복잡한 충돌은 로컬에서 해결합니다.
  • 충돌 마커(<<<<<<<, =======, >>>>>>>)를 제거하고 최종 코드를 작성합니다.
  • 리뷰 피드백 반영 후 push하면 PR이 자동 업데이트됩니다.
  • Force push는 PR 작업 브랜치에서만 제한적으로 사용하며, --force-with-lease를 권장합니다.

FAQ¶

Q1: PR을 너무 크게 만들어버렸는데, 나눌 수 있나요?

A1: PR을 닫고 작은 PR들로 나누는 것이 좋습니다. 또는 리뷰어에게 커밋별로 리뷰를 요청하세요. 향후에는 작업 시작 전에 큰 기능을 작은 단위로 나누는 습관을 들이세요. Stacked PRs 기법을 사용하면 의존성이 있는 작은 PR들을 순차적으로 만들 수 있습니다.

Q2: 리뷰어가 승인했는데 머지 버튼이 비활성화되어 있습니다.

A2: 브랜치 보호 규칙 때문일 수 있습니다. Settings → Branches에서 확인하세요. CI 검사 통과, 최소 리뷰 승인 수, 최신 상태 유지, 관리자 승인 등이 요구될 수 있습니다. "View details"를 클릭하면 어떤 조건이 충족되지 않았는지 확인할 수 있습니다.

Q3: Squash and Merge 후 feature 브랜치를 삭제했는데, 로컬에 흔적이 남아있습니다.

A3: 로컬 브랜치도 삭제하고 원격 추적 정보도 정리하세요.

$ git switch main
$ git pull
$ git branch -d feature/my-feature
$ git fetch --prune  # 삭제된 원격 브랜치 정리

Q4: PR에서 특정 파일을 리뷰에서 제외하고 싶습니다.

A4: GitHub은 파일별 제외 기능을 제공하지 않지만, .gitattributes에서 언어 감지를 조정하거나, 대용량 생성 파일은 별도 커밋으로 분리하여 리뷰 부담을 줄일 수 있습니다. 또는 리뷰어에게 "generated/ 디렉토리는 자동 생성이므로 건너뛰셔도 됩니다"라고 코멘트로 안내하세요.

Q5: 리뷰어가 응답하지 않아 PR이 정체되어 있습니다.

A5: GitHub에서 리뷰어에게 "Re-request review"를 클릭하거나, Slack/이메일로 직접 요청하세요. 팀의 리뷰 SLA(Service Level Agreement)를 정하는 것도 좋습니다. 예: "PR은 24시간 이내 초기 리뷰, 48시간 이내 최종 승인". CODEOWNERS 파일로 자동 리뷰어 지정을 설정하면 책임이 명확해집니다.

장 요약¶

이번 장에서는 GitHub의 핵심 기능인 Pull Request를 마스터했습니다. PR의 개념과 협업에서의 장점을 이해하고, 웹 UI와 GitHub CLI로 PR을 생성하는 두 가지 방법을 배웠습니다. Draft PR로 초기 피드백을 받고, PR Template으로 작성을 표준화하는 실무 기술을 익혔습니다. 효과적인 코드 리뷰를 수행하고 Suggestion 기능으로 수정을 제안하는 방법을 실습했습니다. 세 가지 머지 전략(Merge Commit, Squash and Merge, Rebase and Merge)의 장단점을 비교하고 프로젝트에 맞게 선택하는 기준을 배웠습니다. 마지막으로 PR 충돌을 웹 UI와 로컬 환경에서 해결하고, 리뷰 피드백을 반영하여 PR을 업데이트하는 전체 프로세스를 완성했습니다.
